**The traditional perimeter-based security model** is no longer enough.

**Digital collaboration** has changed. People work from anywhere, on any device. The traditional network perimeter no longer applies.

**Security** must evolve. Identity is now the new perimeter — covering SaaS, BYOD, IoT, and external access.

> An identity is what defines a user, device, or app — including credentials like username, password, and access rights.

![3-identity-new-security-perimeter](/content/1.the-lab/2.core-principles/13.identity/3-identity-new-security-perimeter.png)

## **The Four Pillars**

A strong identity system gives us the ability:

1. **Administration** – Creating, managing, and governing identities for users, devices, and services.
2. **Authentication** – Verifying identities through credentials to confirm who they are.
3. **Authorization** – Determining what an authenticated identity can access.
4. **Auditing** – Tracking identity activity for visibility, alerts, reporting and governance of identities.

Together, these pillars form the foundation of secure identity and access management.

## Control Plan

A **control plane** decides **who can access what**.

Today, **identity** (who you are) is the best way to do that.

If we trust your identity, we can control access safely and check what they can and can’t do— in the cloud, on devices, or in apps like Office 365.

![control-plane](/content/1.the-lab/2.core-principles/13.identity/control-plane.png)

## Types Of Identity

In Microsoft Entra ID, you can assign identities to three things:

- **People** – Employees, partners, or customers (called *user identities*).
- **Software** – Apps, VMs, and services (*workload identities*).
- **Devices** – Phones, laptops, and IoT devices (*device identities*).

These help control access across your organization.

![identity-types-inline](/content/1.the-lab/2.core-principles/13.identity/identity-types-inline.png)

## Human Identities

**User identities** represent people like employees or external users (e.g. customers or partners).

They’re defined by:

- **How they sign in**: using an internal account (from your org) or an external one (from another org or provider).
- **User type**: either a **member** (internal) or **guest** (external, with fewer permissions).

### Cloud

**Why:**
Users only exist in the cloud and do not have any on-premises accounts, ideal for organizations fully embracing cloud infrastructure or for contractors and external users managed solely in Microsoft Entra ID.

**How:**

* Users are created and managed directly in Microsoft Entra ID.
* No synchronization with on-premises Active Directory.
* Authentication and access managed entirely in the cloud.

**Result:**
Users sign in with cloud-only credentials and access cloud resources without dependency on on-premises infrastructure.

### Hybrid

**Why:** Many organizations still run part of their environment on-premises (e.g., Active Directory). But users also need access to cloud services (like Microsoft 365, Teams, SharePoint) — without having two separate accounts.

**How:**

- **Provisioning:** A user is created in Entra ID based on an on-prem user.
- **Synchronization:** Keeps user info up to date via:

  - **Entra Connect Sync** (classic, heavyweight)
  - **Entra Cloud Sync** (lightweight, modern)

**Result:** The user logs in with the same credentials (e.g., UPN/password) and can access everything — both cloud and on-prem — through Single Sign-On (SSO).

## Machine Identities

### Workload

**Workload identities** are for software (like apps, services, or VMs) to access resources securely — like a user, but for code.

They’re important because:

- Software uses credentials that must be stored safely.
- It’s harder to track or revoke software identities than user ones.

**In Microsoft Entra, workload identities include:**

- **Applications**
- **Service principals** (the identity for an app)
- **Managed identities** (Microsoft manages credentials for you)
	- **System-assigned**: Tied to a resource; deleted when the resource is deleted.
	- **User-assigned**: Created separately, used by multiple resources; doesn't delete with the resource.

	<!-- ![managed-identities-inline](/content/1.the-lab/2.core-principles/13.identity/managed-identities-inline.png) -->

> By registering an app in Entra ID, it gets a **service principal** in each tenant where it's used, allowing it to authenticate and access resources securely. **Managed identities** help avoid manual credential management, reducing risk.

### Device

**Device identities** represent hardware like phones, laptops, servers, and printers, enabling secure access control and management.

**Types in Microsoft Entra ID:**

- **Entra registered**: Personal devices for BYOD scenarios, no org account required.
- **Entra joined**: Org-owned devices using organizational accounts to sign in.
- **Entra hybrid joined**: Devices connected to both on-premises Active Directory and Entra ID.

> This allows users Single Sign-On (SSO) to both cloud and on-premises resources, with IT admins using tools like **Microsoft Intune** for security and access control.

#### 🚀 **The Evolution to Microsoft 365 Groups**

**Back in the day**, setting up collaboration for a project like *ProjectDuck* was a pain:

* IT had to manually create security groups (e.g., `ProjectDuckAdmins`, `ProjectDuckRead`, etc.)
* Then create a shared mailbox (`ProjectDuck@company.com`)
* A SharePoint site with matching permissions
* A file share for documents
  👎 It was time-consuming and hard to manage.

#### ✅ **Enter Microsoft 365 Groups**

Microsoft 365 Groups streamlined all of this. When you create a group:

* It **automatically** provisions:

  * A shared mailbox + calendar
  * SharePoint site
  * OneNote
  * Planner
  * Shared document library

It also defines **3 permission roles** out of the box: *Owner, Member, and Guest*.

#### 💬 **Adding Microsoft Teams**

You can **optionally activate Teams** for a group. This:

* Adds a "General" channel
* Creates a folder called "General" in the SharePoint document library
* Embeds this folder in Teams, so people can access the same files from Teams, SharePoint, or OneDrive

**Teams-enabled M365 Groups are one-to-one** — one Team per group.
But you can add **channels** like "Support", "Development", etc., which create matching folders in SharePoint.

#### 🔐 **Private Channels & Permissions**

* If you make a **private channel** (e.g., "Leadership Only"), Microsoft creates a **separate SharePoint subsite** to keep permissions secure.
* Keep in mind: too many unique permissions can get messy — better to create a new M365 group if access needs differ significantly.

#### 🧠 **Everything Is Tied Together**

When someone joins an M365 group, they get access to:

* The mailbox
* Calendar
* SharePoint site
* Teams
* Planner
* OneNote
  All in one go. And if deleted, **everything goes** — the entire workspace is treated as a single unit.

You **can't** create a Teams workspace without a Microsoft 365 Group — Teams chats and files are stored in the underlying group (Exchange & SharePoint).

#### 🔄 **Dynamic Membership Makes It Even Better**

Use **dynamic group membership** in Entra ID (Azure AD) to simplify onboarding:

* E.g., if someone's department is “IT”, they’re automatically added to the IT group.
* Memberships can be based on **department, job title, location**, etc.

#### 🗂️ **Mixing It All Together**

You can link other SharePoint sites into Teams tabs.
Example:

* A “Policies” SharePoint site can be added to the HR Team — as long as users have access rights.

But remember: **Only one SharePoint site is tied to each M365 Group**, the rest are external links.

#### 🎯 Summary

Microsoft 365 Groups unify all collaboration tools — mail, calendar, files, chat, and permissions — into **one manageable identity**.
They replace the messy old way of stitching things together.

They’re not just “email groups” or “Teams containers” — they’re **modern workspaces** that scale with your org.

## Identity Systems

### Centralized

**Example:** Microsoft Entra ID.

- Credentials are stored and managed by a single authority (admin or group).
- Provides authentication and authorization for access to apps and resources.
- Can be cloud-based or on-premises.

**Benefits:** Unified management, secure adaptive access, simplified governance, seamless user experience.

![central-identity-system](/content/1.the-lab/2.core-principles/13.identity/central-identity-system.png)

### Decentralized

**Example:** Blockchain, ledgers.

- Users create, own, and control their digital identities (DIDs) independently.
- Identities are stored encrypted off-chain; only minimal metadata is anchored on decentralized systems like blockchains.
- No central authority; trust is based on cryptographic proofs and user control.

**Benefits:** User privacy and ownership, tamper-resistant identities, enhanced security without reliance on a single authority.

![decentralized-identity-flow](/content/1.the-lab/2.core-principles/13.identity/decentralized-identity-flow.png)

### Federation

**Example:** Logging into a site using a social media account (like X).

* A website in Domain A uses Identity Provider A (IdP-A) for login.
* A user from Domain B logs in with Identity Provider B (IdP-B).
* IdP-A trusts IdP-B, so the website accepts the user without extra login steps.
* Trust can be one-way; Domain A might trust Domain B, but not always the other way around unless set up.

**Benefits:** Users need only one set of credentials, easier access across domains, reduced password fatigue, and controlled trust between organizations.

![5-federated-identification](/content/1.the-lab/2.core-principles/12.identity/5-federated-identification.png)

## Active Directoy

> A directory is like a phone book for a network—it stores information about users, devices, and resources.

**Active Directory (AD)** is Microsoft’s system to manage users and devices in a company’s network. AD DS lets organizations manage many on-site systems using one identity per user. It verifies who you are and controls what you can access.

::important
**A server running Active Directory Domain Services (AD DS) is called a domain controller (DC)**
::

::note
**Active Directory (AD)** works well for traditional on-premises setups but doesn’t support mobile devices, cloud apps (SaaS), or modern sign-in methods easily.

- Because of the growth of cloud services and mobile devices, there’s a need for modern authentication.

**Microsoft Entra ID** (part of the Microsoft Entra family) is a cloud-based Identity as a Service (IDaaS) solution. It helps manage user identities and access across both cloud and on-site apps using newer, secure authentication methods.
::

## IAM

Microsoft Entra ID is Azure’s centralized identity and access management solution. It supports authentication, authorization, and governance for users, devices, and apps across cloud and on-premises environments.

**Why it matters:** One identity for secure access and collaboration across platforms.

![identity-basic-architecture](/content/1.the-lab/2.core-principles/13.identity/identity-basic-architecture.png)

## Identity Provider

**Modern authentication** uses token-based methods to securely verify identities between clients and servers. At the center is the **identity provider (IdP)**—a trusted system that creates, stores, and manages digital identities.

**Core IdP Functions:**

- Stores user identities
- Authenticates users (e.g., password, biometrics)
- Issues security tokens
- Supports SSO for seamless access
- Uses security protocols to defend against threats

> With modern auth, the IdP verifies a user's identity and issues a **token** the client presents to access resources. This improves security, reduces password fatigue, and supports mobile/roaming access without constant reauthentication.

**Examples of IdPs:** Microsoft Entra ID, Google, GitHub, LinkedIn.

## Licensing

**License = User Access**
A **license** gives a user access to specific Microsoft services or features.

* **Entra ID P1/P2**: Needed for advanced identity tools (e.g., PIM needs P2).
* **Microsoft 365 / Office 365 / Windows**: Required per user for product access.
* **MAU (Monthly Active User)**: Used for billing external users in Entra ID. First 50,000 MAUs are free/month (with P1 or P2).

**Subscription = Service Agreement**
A **subscription** is your organization's contract to use Microsoft cloud services.

* Can be billed **per-user** (SaaS like M365) or **by usage** (Azure).

> You can start with a free trial and convert to a paid subscription.

## Identity Adminstration

Identity administration means managing who can use what over time, either by hand or automatically—if you don’t do it, things can get messy fast.

::foldable
#title
Why

#content
1. The system can **easily change** to fit how your business works
2. It can **quickly** add or reduce resources when needed
3. **Saves money** by automating tasks and sharing management
4. **Flexible** in keeping data synced, updated, and controlled
::

::caution
Story - the life of an identity

- Juan worked at the company for years and got admin access. When he left, his account wasn’t removed because no one checked. A year later, Juan’s stolen password let hackers break in using his old, still-active account.
::

* **Identity Proliferation:** Identities exist in many places like directories and apps.
* **Provision and Deprovision:** Creating new user accounts (provision) and removing or disabling old ones (deprovision).
* **Identity Updates:** Keeping user info up to date, ideally automatically instead of manually.
* **Synchronization:** Making sure all systems have the latest user info, either by schedule or when changes happen.
* **Password Management:** How passwords are set and reset, often helped by a Service Desk.
* **Group Management:** Managing user groups that control access to resources; it can be complex and costly.
* **Application Entitlement Management:** Deciding which users can access which apps and what level of access they get.
* **User Interface:** How users can update their own info, usually still done through a Service Desk.
* **Change Control:** How identity changes are managed, sometimes manual, sometimes automated with workflows.

### Auditing

**Auditing** means keeping a record of what users do (like sign-ins or password changes) to spot problems, follow rules, and fix issues.

* Tracks activities like sign-ins, password changes, and MFA usage
* Helps detect attacks, ensure compliance, and troubleshoot issues
* Includes logs such as Entra activity, sign-in, provisioning, and audit logs
* Supports keeping identities and data secure

::important
Uses tools like Azure Monitor and Microsoft Sentinel for reporting and monitoring
::

### Governance

**Governance** means managing who has access, keeping that access safe, and regularly maintaining the identity system to run smoothly.

- It involves setting policies.
- Monitoring account usage
- Regularly updating processes.

::caution
Without governance, risks increase—like orphaned accounts that can be exploited (e.g., a former employee’s account left active). Good governance includes regular reviews with HR, access right audits, and enforcing security practices like password changes and MFA.
::
