## 1. **Collection**:

### Data Identification

We gather all the important stuff (like pictures, files, or clues) from a place where something bad happened, like a broken toy.

### Data Acquisition

In real life environment we also consider logs, anything that can we usefull to us.

::caution
When powered off, we lose the volatile data. So, we need to collect the volatile data first before powering off the system.

Think of ways to contain the data first. By making an image of the disk.

**Order Of Volatility (RFC3227)**

- Registers, cache
- Routing table, ARP cache, process table,
kernel statistics, memory
- Temporary file systems
- Disk
- Remote logging and monitoring data that
is relevant to the system in question
- Physical configuration, network topology
- Archival media
::

### Verify Integrity

We make sure not to touch anything so it doesn't get mixed up or broken.

::important
To ensure the integrity of our data, we must generate a cryptographic hash—a unique digital fingerprint. Later, if the evidence is shared or revisited, we can create a new hash and compare it to the original. This guarantees the data remains unchanged, preserving its authenticity—an essential safeguard in any legal investigation.
::

### Extraction

In this phase, we focus on retrieving relevant data from the collected evidence. This involves isolating artifacts, files, and logs that provide insights into system activity, user behavior, and potential anomalies.

> The goal is to transform raw data into structured, actionable information for deeper analysis.

- **Memory Extraction** – Use tools like **Volatility** to analyze memory dumps for active processes, network connections, and runtime artifacts.
- **Disk Image Mounting** – Leverage **KAPE** (Kroll Artifact Parser and Extractor) with predefined targets to collect critical artifacts and create a **triage folder** for further analysis. This ensures the disk content remains intact while enabling structured examination.

> **Note:** The triage folder simplifies processing and categorization, allowing deeper analysis without risking contamination of the source data.

## 3. **Analysis**:

Now we figure out what all the clues mean. We ask questions like "Who did this?" or "When did it happen?" to understand the story.

In this stage we've made a triage collection and now we need to analyze the data that we've collected.

### Sources of Evidence

![sources-of-evidence](/windows-forensics/sources-of-evidence.png)

#### **1. System & User Information**
- Overview of system configurations and user profiles.

---

#### **2. Registry**
- **Key Insights** – Unlock critical data stored in the Windows Registry.
- **Analysis Techniques** – Investigate registry hives for hidden configurations and traces.

---

#### **3. File Analysis**
- **File Metadata** – Extract timestamps and properties.
- **Deleted Files** – Recover evidence of removed or hidden files.

---

#### **4. NTFS**
- **Disk Structures** – Explore NTFS partitions and volumes.
- **Journal Analysis** – Leverage USN Journal for file activity history.

---

#### **5. Evidence of Execution**
##### **Background Activity Moderator**
- **ShimCache** – Track program execution artifacts.
- **Amcache** – Reveal application execution details.
- **Prefetch** – Analyze preloaded data for runtime footprints.

---

#### **6. Persistence Mechanisms**
- **Run Keys** – Examine registry keys for autorun scripts.
- **Startup Folder** – Detect files set to execute on boot.
- **Scheduled Tasks** – Analyze tasks for scheduled malicious activity.
- **Services** – Investigate service configurations for persistence.

---

#### **7. Event Log Analysis**
- **Windows Event Logs** – Decode logs to uncover security incidents and unauthorized actions.
- **Activity Tracking** – Map patterns and anomalies through log entries.

## 4. **Reporting**:

Finally, we write everything down so others can understand what happened, like telling a teacher or the police about the broken toy, so they know who did it and when.

#### Reporting Considerations

Key Considerations for Effective Incident Reporting

---

##### **1. Establish Expectations Early**
- **Clarify Scope:** Define the depth and level of detail required upfront.
- **Understand Requirements:** Avoid overwhelming the audience with technical jargon if it’s not necessary. Focus on delivering insights that match their expectations.

> *Tip: Not all clients care about the nuts and bolts—know when to simplify and when to go in-depth.*

---

##### **2. Consider Your Audience**
- **Technical vs Non-Technical:** Tailor the language and content based on whether the audience is technical or non-technical.
- **Key Concerns:**
  - **Non-Technical Teams:** Focus on risk, impact, and high-level findings.
  - **Legal Teams:** Provide defensible evidence to support claims.
  - **Subject Matter Experts (SMEs):** Dive into technical details and indicators of compromise.

> *Who needs answers—legal teams, executives, or security analysts? Adjust accordingly.*

---

##### **3. Explore Alternative Explanations**
- **Avoid Bias:** Always assess multiple possibilities before concluding.
- **Thorough Investigation:** Rule out false positives and examine all angles to ensure conclusions are valid and defensible.

> *Remember: Investigations require open-mindedness, not assumptions.*

---

##### **4. Provide Actionable Information**
- **Follow-Up Steps:** Highlight critical actions needed, such as:
  - Disabling compromised accounts.
  - Monitoring for lateral movement.
  - Expanding searches for additional indicators of compromise.
- **Incident Response Lifecycle:** Ensure findings feed into the next phases, enabling SMEs to act quickly and effectively.

> *Deliver insights that drive decisions, not just observations.*
