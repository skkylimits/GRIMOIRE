title: Az Powershell Attack Path
Entra ID enumeration focuses on understanding identities, roles, and permissions within an organization. The goal is to identify privileged accounts, misconfigurations, and potential lateral movement opportunities. Here's a structured attack path:
navigation: false
---

## **1️. Identify Users & Groups** 🧑‍💻👥
**Why?**
- Discover all **users** in the environment.
- Identify privileged **service accounts** (e.g., automation, backups).
- Spot users with **suspicious naming conventions** (e.g., old admin accounts that might be unmonitored).
- Enumerate **groups** to see how access is assigned at scale.

**Commands:**
```powershell
Get-AzADUser
Get-AzADGroup
Get-AzADGroupMember -ObjectId <GroupID>
```

## **2. Check for Privileged Roles** 🔐⚙️
**Why?**
- Identify **Global Administrators** who have full control over Entra ID.
- Find **Custom Directory Roles** that may have **high-privilege access** but fly under the radar.
- Look for **role assignments** on users, groups, and service principals.

**Commands:**
```powershell
Get-AzRoleAssignment
Get-AzADUser -Filter "UserType eq 'Member'"
```

## **3️. Enumerate Devices & Hybrid Identities** 💻
**Why?**
- Identify **joined devices** (Intune, Hybrid AD Join, AAD Join).
- Look for **old or inactive devices** that could be exploited.
- Find **workstations or servers** tied to privileged users.

**Commands:**
```powershell
Get-AzADDevice
```

## **4️. Discover App Registrations & Service Principals** 📜🏢
**Why?**
- Find **App Registrations** that might have access to sensitive data.
- Look for **Enterprise Applications (Service Principals)** with permissions to Entra ID or Azure Resources.
- Check for apps with **high privileges** (e.g., those that can **read emails, reset passwords, or access key vaults**).

**Commands:**
```powershell
Get-AzADApplication
Get-AzADServicePrincipal
```

## **5️. Identify Potential Privilege Escalation Paths** ⚠️
**Why?**
- If an **app registration** or **service principal** has high privileges, it could be abused to gain further access.
- If a user can **modify group memberships**, they might be able to escalate themselves into a privileged role.

**Check:**
- **Users who can modify group membership:**
  ```powershell
  Get-AzRoleAssignment | Where-Object {$_.RoleDefinitionName -match "Groups Administrator"}
  ```
- **Apps with high permissions:**
  ```powershell
  Get-AzADServicePrincipal | Select DisplayName, AppId, AppRoleAssignments
  ```

## **6️. Map Out Attack Paths & Lateral Movement** 🔄
**After gathering the above data:**
- Are there **weakly protected Global Admins**?
- Can an **App Registration be used for privilege escalation**?
- Do any **Service Principals** have unnecessary permissions?
- Are there **old, unused devices** that might have valid session tokens?

This **step-by-step attack path** helps move from **simple enumeration to privilege escalation and persistence** in an **Entra ID environment**. 🚀
