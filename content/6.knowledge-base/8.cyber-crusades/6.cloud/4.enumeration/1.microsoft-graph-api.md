## The Attack Path

`Mapping` the attack path is key. 🗝️

The primary goal is[`resource enumeration `](/knowledge-base/cyber-crusades/cloud/enumeration/attack-path-ms-graph)—understanding what's deployed in Azure and who controls it. We begin by investigating users, groups, roles, devices, and applications using Microsoft Graph to map potential attack paths

1. **Identify Compromised Users or Devices**
2. **Check Group Memberships and Roles**
3. **Look for Misconfigured Applications**
4. **Pivoting Opportunities**

## **The Command Line**
If you want to enumerate information from the command line in Microsoft environments, you would use the **Microsoft Graph PowerShell Module (MgModule)**.

This module is essentially a wrapper around the **Graph API**, which is a unified API endpoint that allows access to various Microsoft 365 services. The output of this module isn’t very user-friendly, but it is extremely useful for automating administrative tasks and security assessments.

### 🧛**Microsoft Graph API**
Microsoft has done exetremly well by bringen every thing under one API and thus consolidated almost all of its **Microsoft 365 (M365) services** under a **single API endpoint**—the **Graph API**.

Before this, different services had their own APIs:
- **Outlook API** for emails
- **Exchange API** for mailboxes
- **OneDrive API** for files
- **Azure AD Graph API** for identity management

> Now, everything is accessible via **Graph API**.
In **2023**, Microsoft introduced **Graph Activity Logs**, which means enumeration attempts can now be logged. Previously, there was no logging for enumeration, making it a blind spot for defenders.

::tip
These logs can be integrated with **Microsoft Sentinel** or any **SIEM (Security Information and Event Management) tool** to detect enumeration attempts. However, unless an organization has a **very mature security posture**, worrying about Graph API enumeration should not be a top priority.
::

### 🚫 **Azure AD Graph API is Being Deprecated**
The **Azure AD Graph API (AD Graph)** is being retired soon.
- **MgModule** replaces the older **Azure AD (AzAD) module**.
- **Azure AD Graph (graph.windows.net)** is being replaced by **Microsoft Graph (graph.microsoft.com)**.

This is **bad news for attackers**, because:
- **Azure AD Graph API had no logging.** 😈
- **It couldn't be blocked using Conditional Access Policies.**
- **It allowed enumeration without special privileges.**

For example, using **Azure AD Graph**, you could **enumerate Conditional Access Policies without being a Security Reader**, which **is no longer possible** in **Microsoft Graph API**.
This was a huge weakness that security researchers—like **Dulcan, the creator of RoadTools**—discovered.

Submodules are **installed automatically** when you install `Microsoft.Graph`. You do **not** import them manually.

**Submodules**
When you run:
```powershell
Install-Module Microsoft.Graph -Scope CurrentUser -Repository PSGallery -Force
```
PowerShell installs the **main module (`Microsoft.Graph`)** along with its **submodules** (like `Microsoft.Graph.Users`, `Microsoft.Graph.Groups`, etc.).

**Did They Get Imported Automatically?**
- **No**, they are just installed and available for use.
- PowerShell only imports the specific submodules **on demand** when you run a related command (e.g., `Get-MgUser` will auto-load `Microsoft.Graph.Users`).
- If a command fails, you **may** need to import the required submodule manually:
  ```powershell
  Import-Module Microsoft.Graph.Users
  ```

**Check Installed vs. Loaded Modules**

**See All Installed Modules**
These are installed but **not necessarily loaded yet**:
```powershell
Get-Module -ListAvailable Microsoft.Graph*
```

**See Loaded Modules in the Current Session**
These are **actively loaded** and available:
```powershell
Get-Module Microsoft.Graph*
```

#### **When to Manually Import Modules?**
- If PowerShell fails to recognize a command (e.g., `Get-MgUser` not found).
- If running in **restricted environments** where auto-import doesn’t work.
- If you **only want specific submodules** loaded for performance reasons:
  ```powershell
  Import-Module Microsoft.Graph.Users
  Import-Module Microsoft.Graph.Groups
  ```

💡 **Best Practice:** Let PowerShell auto-load submodules unless you face issues.

#### **1️⃣ Interactive Login**
```powershell
Connect-MgGraph
```
- Opens an **interactive login prompt** for authentication.

#### **2️⃣ Non-Interactive Login (Using Client Credentials)**
If you need a **non-interactive** login (for automation or scripts), use an **Azure AD App Registration** with a **Client ID & Secret** or **Certificate**:

```powershell
$TenantId = "<Your-Tenant-ID>"
$ClientId = "<Your-App-ID>"
$ClientSecret = "<Your-App-Secret>"

$SecureSecret = ConvertTo-SecureString -String $ClientSecret -AsPlainText -Force
$Credential = New-Object System.Management.Automation.PSCredential ($ClientId, $SecureSecret)

Connect-MgGraph -TenantId $TenantId -ClientId $ClientId -ClientSecret $SecureSecret
```

✅ **No user interaction required.**
🔹 Requires an **app registration** with **Graph API permissions** (`Application` type).

#### **3️⃣ Verify the Connection**

Shows token details & session info. 🚀

```powershell
Get-MgContext
```

Get details of the current tenant 🔥

```powershell
Get-MgOrganization | fl *
```

#### ⌚ **How Long is an Access Token Valid?**

- If you authenticate interactively (Connect-MgGraph without parameters), PowerShell caches the token only for that session. Once you close the session, the token is lost.
- If you authenticate non-interactively (using client credentials), you must manually request a new token when the old one expires.

### **1. Users**

You can use the `Get-MgUser` cmdlet to retrieve user details from Microsoft Entra ID (Azure AD). Here are some ways to do this:

`UserId` is the user's ID or email address.

#### **Listing All Users**

- **List all users:**
    ```powershell
    Get-MgUser -All
    ```

- **List a specific user by their User ID or email:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com
    ```

#### **Filtering Users**

- **Search for users whose DisplayName starts with a specific string:**
    (Note: Wildcards are not supported, and this uses an internal `$filter` query syntax)
    ```powershell
    Get-MgUser -Filter "startsWith(DisplayName, 'a')" -ConsistencyLevel eventual
    ```

- **Search for users with "admin" in their Display Name:**
    ```powershell
    Get-MgUser -All | Where-Object {$_.DisplayName -match "admin"}
    ```

- **Search for users by specific attributes (e.g., DisplayName contains "admin"):**
    ```powershell
    Get-MgUser -Search '"DisplayName:admin"' -ConsistencyLevel eventual
    ```

#### **Listing All User Attributes**

- **Display all properties of a specific user:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```

- **List all the attribute names of a specific user:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com | ForEach-Object {$_.PSObject.Properties.Name}
    ```

- **Search user attributes for a specific string (e.g., "password"):**
    ```powershell
    Get-MgUser -All | ForEach-Object {
        $Properties = $_
        $Properties.PSObject.Properties.Name | ForEach-Object {
            if ($Properties.$_ -match 'password') {
                "$($Properties.UserPrincipalName) - $_ - $($Properties.$_)"
            }
        }
    }
    ```

#### **Identifying Synced vs Cloud-Only Users**

- **All users who are synced from on-premises Active Directory:**
    ```powershell
    Get-MgUser -All | Where-Object {$_.OnPremisesSecurityIdentifier -ne $null}
    ```

- **All users who are native to Entra ID (not synced):**
    ```powershell
    Get-MgUser -All | Where-Object {$_.OnPremisesSecurityIdentifier -eq $null}
    ```
::note
- **Users Synced from On-Prem Active Directory (Hybrid Identity):** These users have information about their **OnPremisesSecurityIdentifier**, which can be valuable to attackers.
- **Cloud-Only Users:** These users do not have the **OnPremisesSecurityIdentifier** attribute set to a non-null value.
::

#### **Objects Created or Owned by Users**

- **Objects created by a specific user (use `-UserId` for a specific user):**
    ```powershell
    Get-MgUserCreatedObject -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```

- **Objects owned by a specific user:**
    ```powershell
    Get-MgUserOwnedObject -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```
#### **Why Are We Interested in Objects Owned by a User?**

> When analyzing attack paths, we often focus on objects owned by users because:

- **Application Ownership and Roles**: If a user owns an application that has a high-privilege role, such as **Global Administrator (GA)**, the user effectively gains those elevated privileges. For instance, if the **test user** owns an application with a **GA** role, they automatically gain **Global Admin access**. This makes it crucial to investigate object ownership, especially when we map attack paths.

- **Enterprise Applications and Roles**: Enterprise applications can also have roles like **Global Administrator**, which is common for applications intended for administrative purposes (e.g., backup services). This is why we care about **ownership** of these objects, as it may give us insight into potential privilege escalation paths.

- **Attack Path Mapping**: In many attack scenarios, understanding which user owns a given object (application, service, device, etc.) helps map out how attackers can elevate their privileges. If a **compromised user** or **target user** is connected to specific groups or roles, they could be elevated to critical privileges, including **Global Admin**.

Why should we enumarate groups?

It's interesting because when a user is part of specific **groups** or has certain **roles**, it can give attackers insight into how they might escalate their privileges.

For example:
- If a **compromised user** or **target user** is a member of a group with **elevated privileges** (like **Global Admin**), the attacker could potentially use that membership to gain those privileges.
- By identifying the **roles** assigned to a user, you can see if they have access to sensitive resources or admin-level actions.
- An attacker can leverage these **group memberships** and **roles** to elevate their access within the system, potentially moving laterally or gaining access to more critical assets.

In short, knowing which groups or roles a user is in helps attackers map out potential paths to escalate their access to higher privileges, such as Global Admin. This is why looking at **group memberships and role assignments** is crucial during a compromise investigation.
::

### **3. Roles**

You can use Microsoft Graph PowerShell to enumerate roles in Entra ID.
- `Get-MgDirectoryRoleTemplate` → Lists all available role templates.
- `Get-MgDirectoryRole` → Lists users assigned to roles.

#### **Listing Roles**
- **Get all available role templates:**
    ```powershell
    Get-MgDirectoryRoleTemplate
    ```
- **Get all enabled roles (roles must be enabled before usage):**
    ```powershell
    Get-MgDirectoryRole
    ```
- **List all custom directory roles (non-built-in roles):**
    ```powershell
    Get-MgDirectoryRole -All | Where-Object {$_.IsBuiltIn -eq $false}
    ```

#### **Finding Global Admins**
- **Find users assigned to a specific role (e.g., Global Administrator):**
    ```powershell
    $RoleId = (Get-MgDirectoryRole -Filter "DisplayName eq 'Global Administrator'").Id
    Get-MgDirectoryRoleMember -DirectoryRoleId $RoleId | Select-Object DisplayName, Id
    ```

::warning
**Important Notes:**
- Many organizations have **too many** Global Admins, creating a security risk.
- **Roles should follow the least privilege principle** to reduce attack surfaces.
::

> Most misconfigured lab environments **are is still better than many real-world production environments**. Most companies have **far too many** Global Admins, which is a security risk.

### **4. Devices**
- `Get-MgDevice` → Lists all devices in the environment.
- **Important attributes:**
  - **Compliance status** (is the device compliant with security policies?).
  - **Device join type** (Azure AD Joined, Hybrid Joined, or Registered).
  - **LastSignInDateTime** (if null, the device was never used).
  - **Device Owners** (owners are automatically Local Admins on the device).

#### **Enumerating Devices - Mg Module**

- **Count the amount of devices: in the tenant**
    ```powershell
    Get-MgDevice -All | measure
    ```

- **Get all Azure AD Joined and Registered Devices:**
    ```powershell
    Get-MgDevice -All | fl *
    ```

- **List all active devices (not stale ones):**
    ```powershell
    Get-MgDevice -All | Where-Object {$_.ApproximateLastSignInDateTime -ne $null}
    ```

- **List Registered Owners of all Devices:**
    ```powershell
    $Ids = (Get-MgDevice -All).Id
    foreach ($i in $Ids) {
        (Get-MgDeviceRegisteredOwner -DeviceId $i).AdditionalProperties
    }
    ```

    Alternatively, list owners' **userPrincipalName**:
    ```powershell
    $Ids = (Get-MgDevice -All).Id
    foreach ($i in $Ids) {
        (Get-MgDeviceRegisteredOwner -DeviceId $i).AdditionalProperties.userPrincipalName
    }
    ```

- **List Registered Users of all Devices:**
    ```powershell
    $Ids = (Get-MgDevice -All).Id
    foreach ($i in $Ids) {
        (Get-MgDeviceRegisteredUser -DeviceId $i).AdditionalProperties
    }
    ```

    Alternatively, list users' **userPrincipalName**:
    ```powershell
    $Ids = (Get-MgDevice -All).Id
    foreach ($i in $Ids) {
        (Get-MgDeviceRegisteredUser -DeviceId $i).AdditionalProperties.userPrincipalName
    }
    ```

- **List Devices Owned by a User:**
    ```powershell
    (Get-MgUserOwnedDevice -UserId michaelmbarron@defcorphq.onmicrosoft.com).AdditionalProperties
    ```

- **List Devices Registered by a User:**
    ```powershell
    (Get-MgUserRegisteredDevice -UserId michaelmbarron@defcorphq.onmicrosoft.com).AdditionalProperties
    ```

- **List Devices Managed using Intune (compliant devices):**
    ```powershell
    Get-MgDevice -All | Where-Object {$_.IsCompliant -eq $true} | fl *
    ```

::caution
**Why Are Devices Important for Attackers?**

If **User A owns a device**, and we **compromise User A**, we now have **Admin access on that device**.

Devices, especially **Azure AD-joined devices**, can be crucial in an attack path because:

- **Local Admin Access**: The **owner** of an Azure AD-joined device is automatically a **local admin** on that device. This means that compromising the **infraadminsrv** device, for example, grants the attacker local admin rights, which can then be used to move laterally within the network.

- **Lateral Movement**: If we compromise a user (like Sam C, the owner of **infraadminsrv**), we effectively "own" that device. This ownership can be leveraged to escalate privileges and move laterally within the organization.
::

### 5. **Apps & Service Principals**

Microsoft uses confusing naming conventions when it comes to **App Registrations** and **Enterprise Applications**:

| **Portal Name**        | **Command Line Name**  | **Graph API Name**   |
|------------------------|------------------------|----------------------|
| App Registrations      | Applications           | Application Objects  |
| Enterprise Applications| Service Principals     | Service Principals   |

**What’s the Difference?**

- **App Registrations** → These are **configurations** of an application (like redirect URIs, permissions, etc.).
- **Enterprise Applications (Service Principals)** → These are **instances** of applications where **roles & permissions** are assigned.

#### **Apps (Application Registrations)**

- **List all registered applications** (in App Registrations):
    ```powershell
    Get-MgApplication -All
    ```

- **Get details of a specific application** by AppId:
    ```powershell
    Get-MgApplicationByAppId -AppId <AppId> | fl *
    ```

- **Find an application by display name**:
    ```powershell
    Get-MgApplication -All | Where-Object {$_.DisplayName -match "app"}
    ```

- **List applications with a password**:
    ```powershell
    Get-MgApplication -All | Where-Object {$_.PasswordCredentials -ne $null}
    ```

- **Get owner(s) of an application**:
    ```powershell
    (Get-MgApplicationOwner -ApplicationId <Id>).AdditionalProperties.userPrincipalName
    ```

- **Get applications where a user has a role**:

    ```powershell
    Get-MgUserAppRoleAssignment -UserId <UserId> | fl *
    ```

- **Get applications where a group has a role**:
    ```powershell
    Get-MgGroupAppRoleAssignment -GroupId <GroupId> | fl *
    ```

#### **Abusing Application Ownership for Privilege Escalation**

::caution
Owning an application in Azure can be a **powerful privilege**—and a dangerous one if misused. When a user owns an application, they can **add a Client Secret**—essentially a password that allows authentication as that app. This bypasses any Conditional Access (CA) or MFA. Giving you access to the target tenant, not just a specific machine or VM. If that application has **high-level permissions** (such as **Global Admin**), the user effectively **gains those same privileges**.
::

##### **How Attackers Exploit This**
Imagine an attacker is looking for a way to escalate their access. They follow a simple but effective path:
1. **Find a high-privilege application**—one that already has Global Admin or other sensitive permissions.
2. **Check if they have ownership** of the app. If they do, they can **add a Client Secret.**
3. **Use the Client Secret to authenticate** as that application—gaining full control over whatever permissions the app had. Without MFA.

This is an easy way for an attacker to **jump from a regular user account to full administrative control**—all without triggering traditional security alarms.

::caution
##### **Audit Logs Reveal This Activity**
Fortunately, **adding a Client Secret is very verbosely logged**. Security teams can see:
- **Who** added the secret
- **When** it was added
- **Which application** it was added to

Regularly reviewing logs can help detect and stop abuse before it becomes a major breach.
::

##### **Why Are Apps Given High Privileges in the First Place?**
Not all high-privilege apps are malicious—many are created for **automation**. Admins often assign broad permissions to applications that:
- **Manage identities**
- **Provision resources**
- **Backup virtual machines**

But here’s the issue: **It’s easier to assign broad permissions than to fine-tune security**, making these applications prime targets for attackers.

::warning
##### **Long-Term Persistence**
For an attacker, adding a Client Secret isn’t just about gaining access—it’s about **keeping it**.
- A **Client Secret** can last **months or even years** if not properly monitored.
- Security teams that **fail to detect it** may leave the door open indefinitely.

Once inside, an attacker can continue using the app’s permissions **even if their original account is disabled**.
::

#### **Service Principals (Enterprise Applications)**

- **List all service principals** (in Enterprise Applications):
    ```powershell
    Get-MgServicePrincipal -All
    ```

- **Get details of a specific service principal** by ID:
    ```powershell
    Get-MgServicePrincipal -ServicePrincipalId <ServicePrincipalId> | fl *
    ```

- **Find a service principal by display name**:
    ```powershell
    Get-MgServicePrincipal -All | Where-Object {$_.DisplayName -match "app"}
    ```

- **List service principals with an application password**:
    ```powershell
    Get-MgServicePrincipal -All | Where-Object {$_.KeyCredentials -ne $null}
    ```

- **Get owner(s) of a service principal**:
    ```powershell
    (Get-MgServicePrincipalOwner -ServicePrincipalId <ServicePrincipalId>).AdditionalProperties.userPrincipalName
    ```

- **List objects owned by a service principal**:
    ```powershell
    Get-MgServicePrincipalOwnedObject -ServicePrincipalId <ServicePrincipalId>
    ```

- **List objects created by a service principal**:
    ```powershell
    Get-MgServicePrincipalCreatedObject -ServicePrincipalId <ServicePrincipalId>
    ```

- **Get group and role memberships of a service principal**:
    ```powershell
    Get-MgServicePrincipalMemberOf -ServicePrincipalId <ServicePrincipalId> | fl *
    ```

::warning
Similarly, service principals (which are instances of applications) can have roles assigned to them, not just the app registration itself. So, in addition to checking ownership, you should also check the groups and role memberships of the service principal.
::

## **Final Thoughts**
- **Enumeration is the first step** in mapping out attack paths.
- **Attackers don’t just want access; they want privilege escalation & persistence.**
- **Applications are often a weak point** because they have **high privileges & poor oversight**.
- **Defenders should monitor logs** for suspicious changes (e.g., new client secrets, role assignments, app ownership changes).
