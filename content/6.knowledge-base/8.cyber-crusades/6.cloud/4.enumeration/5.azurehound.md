::note
BloodHound is an analysis tool—the more data you feed it, the more insights you get. With a **Global Reader** or **Security Reader** role, you can access **Conditional Access policies** and other details. By default, it **won’t** show this data unless the right permissions are granted.
::

## **Reader Role**

::warning
In **on-prem environments**, once you have access as a domain user, you can see most resources unless heavily segmented.
::

In **Azure**, there's a clear separation between **Entra ID** and **Azure resources**. Even with high privileges in **Entra ID**, you can’t see **Azure resources** unless explicitly assigned a **Reader** role or higher. This makes attack path mapping harder compared to on-prem **BloodHound**.

While **Azure Hound's** (Azure & Entra ID) output may seem less detailed than **BloodHound** (On-Prem Active Directory), it still provides valuable insights. Let’s see how it works.

## **The 2 Version**

Azure Hound uses the **Azure AD module** and **AZ PowerShell module** to collect information. There are currently two free versions of **BloodHound**:
1. [BloodHound Legacy](https://github.com/SpecterOps/BloodHound-Legacy): The last version of this is available to install on your machine.
2. [BloodHound Community Edition (CE)](https://github.com/SpecterOps/BloodHound): This provides **read-only access** to the common BloodHound Community Edition UI, which can be accessed via the lab portal.

::warning
For **BloodHound CE**, the data collected is for a user that has at least the **Reader role** on all Azure subscriptions and resources. As mentioned in session one, it's important to ask your customer to provide you with the **Reader role** on all resources to be able to gather meaningful data.
::

## **Connecting**

Set up credentials:
```powershell
$passwd = ConvertTo-SecureString "p4ssw0rd" -AsPlainText -Force
$creds = New-Object System.Management.Automation.PSCredential ("test@domain.onmicrosoft.com", $passwd)
```

Authenticate to Azure:
```powershell
Connect-AzAccount -Credential $creds
```

Authenticate to Entra ID:
```powershell
Connect-AzureAD -Credential $creds
```

## Gather Data

Load **AzureHound**:
```powershell
. C:\AzAD\Tools\AzureHound\AzureHound.ps1
```

Start data collection:
```powershell
Invoke-AzureHound -Verbose
```

This will collect **Azure & Entra ID** data for analysis in **BloodHound**. 🚀 The gathered data can be uploaded to **BloodHound** (both **Legacy** and **Community Edition**).

## **`Uploading Data`**
- Open **BloodHound Application** (`C:\AzAD\Tools\BloodHound-win32-x64\BloodHound-win32-x64\BloodHound.exe`)
- Login with:
  - **URL:** `bolt://localhost:7687`
  - **Username:** `neo4j`
  - **Password:** `BloodHound`
- Upload the **ZIP archive** to **BloodHound UI** (drag and drop) and use **built-in/custom Cypher queries**.
- Alternatively, you can use **Read-Only access** to the **pre-populated BloodHound CE** – [BloodHound CE Dashboard](https://azurehound-altsecdashboard.msappproxy.net/)
  - The data uploaded here is collected using a **Reader role** on the subscription.

## **Data Analysis**

Once the data is collected, it can be analyzed. The **BloodHound CE** gives you access to a **read-only version** where you can explore the following:
- **Global Administrators**
- **Privileged roles**
- **Shortest paths to Azure resources**
- **Foreign principals in high-value targets**
- **Devices with unsupported operating systems**

### **Privileged Roles**
- A **Global Administrator** role for an application called **Create Users**.
- Shortest paths from **Entra users** to **high-value targets** such as Global Admins.

### **Path to Privilege Escalation**
In this case, a user named **Mark D. Walden** could be used to escalate privileges, at least in some cases.

### **High-Value Targets**

::tip
A **high-value target** is any role marked as a **privileged role** in Entra ID -> [Roles and administrators | All roles](https://portal.azure.com/#view/Microsoft_AAD_IAM/RolesManagementMenuBlade/~/AllRoles/adminUnitObjectId//resourceScope/%2F). These roles are predefined, and anyone holding one of these roles is considered a high-value target.
::

### **Enumerating Resources**

If you want to enumerate all **Global Administrators** and map **paths to Azure resources** such as **Key Vaults**, you can write custom queries or explore paths like:
- Shortest path to **Azure subscriptions** or **Key Vaults**.

For example, you may discover that **Global Admins** have paths to key vaults, which could be an interesting attack vector.

## **Using BloodHound CE**

If you want real control over the analysis, you can set up **BloodHound CE** on your host machine and upload the collected data from the lab. This way, you can manipulate the data further—such as marking a user as **compromised**.

You can also look into:
- **Who has control over global admins** (e.g., anyone with the **Privileged Authentication Administrator** role can reset the password of a Global Admin).
- **Who has access to privileged roles** and more.

Is using BloodHound logged?

#content

#title
Does BloodHound show information for Conditional Access?

#content
No, it does not by default. It uses the **Azure AD module**, which does not provide information on conditional access policies. However, if you run it as a **Global Reader**, you would be able to see these policies. Once you have access to them, you can create custom queries.
::
