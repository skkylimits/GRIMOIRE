## **The Command Line**
If you want to enumerate information from the command line in Microsoft environments, you would use the **Microsoft Graph PowerShell Module (MgModule)**.

This module is essentially a wrapper around the **Graph API**, which is a unified API endpoint that allows access to various Microsoft 365 services. The output of this module isn’t very user-friendly, but it is extremely useful for automating administrative tasks and security assessments.

### 🧛**Microsoft Graph API**
Microsoft has done exetremly well by bringen every thing under 1 API and thus consolidated almost all of its **Microsoft 365 (M365) services** under a **single API endpoint**—the **Graph API**.

Before this, different services had their own APIs:
- **Outlook API** for emails
- **Exchange API** for mailboxes
- **OneDrive API** for files
- **Azure AD Graph API** for identity management

> Now, everything is accessible via **Graph API**.
In **2023**, Microsoft introduced **Graph Activity Logs**, which means enumeration attempts can now be logged. Previously, there was no logging for enumeration, making it a blind spot for defenders.

::tip
These logs can be integrated with **Microsoft Sentinel** or any **SIEM (Security Information and Event Management) tool** to detect enumeration attempts. However, unless an organization has a **very mature security posture**, worrying about Graph API enumeration should not be a top priority.
::

### 🚫 **Azure AD Graph API is Being Deprecated**
The **Azure AD Graph API (AD Graph)** is being retired soon.
- **MgModule** replaces the older **Azure AD (AzAD) module**.
- **Azure AD Graph (graph.windows.net)** is being replaced by **Microsoft Graph (graph.microsoft.com)**.

This is **bad news for attackers**, because:
- **Azure AD Graph API had no logging.** 😈
- **It couldn't be blocked using Conditional Access Policies.**
- **It allowed enumeration without special privileges.**

For example, using **Azure AD Graph**, you could **enumerate Conditional Access Policies without being a Security Reader**, which **is no longer possible** in **Microsoft Graph API**.
This was a huge weakness that security researchers—like **Dulcan, the creator of RoadTools**—discovered.

### ⚡**Connection Flow (Using Microsoft Graph Only)**

#### **1️⃣ Interactive Login**
```powershell
Connect-MgGraph
```
- Opens an **interactive login prompt** for authentication.

#### **2️⃣ Non-Interactive Login (Using Client Credentials)**
If you need a **non-interactive** login (for automation or scripts), use an **Azure AD App Registration** with a **Client ID & Secret** or **Certificate**:

```powershell
$TenantId = "<Your-Tenant-ID>"
$ClientId = "<Your-App-ID>"
$ClientSecret = "<Your-App-Secret>"

$SecureSecret = ConvertTo-SecureString -String $ClientSecret -AsPlainText -Force
$Credential = New-Object System.Management.Automation.PSCredential ($ClientId, $SecureSecret)

Connect-MgGraph -TenantId $TenantId -ClientId $ClientId -ClientSecret $SecureSecret
```

✅ **No user interaction required.**
🔹 Requires an **app registration** with **Graph API permissions** (`Application` type).

#### **3️⃣ Verify the Connection**
```powershell
Get-MgContext
```
Shows token details & session info. 🚀

```powershell
Get-MgOrganization | fl *
```
Get details of the current tenant 🔥

#### ⌚ **How Long is an Access Token Valid?**

- If you authenticate interactively (Connect-MgGraph without parameters), PowerShell caches the token only for that session. Once you close the session, the token is lost.
- If you authenticate non-interactively (using client credentials), you must manually request a new token when the old one expires.

## **Enumerating Entra ID Info**

Once connected as a `regular user`, you can enumerate various resources in Entra ID, including:

- **Users**
- **Groups**
- **Roles**
- **Applications** (App Registrations & Enterprise Apps)
- **Devices**
- **Conditional Access Policies**
- **Service Principal Objects**

As a `regular user` in Entra ID, you also have several permissions, including the ability to:

- View details about users, groups, applications, devices, roles, and subscriptions (excluding private data).
- Invite guest users to your organization.
- Create security groups and view group memberships (non-hidden).
- Add guests to groups they own.
- Create new applications.
- Add up to 50 devices to Azure.

### **1. Users**

You can use the `Get-MgUser` cmdlet to retrieve user details from Microsoft Entra ID (Azure AD). Here are some ways to do this:

#### **Listing All Users**

- **List all users:**
    ```powershell
    Get-MgUser -All
    ```

- **List a specific user by their User ID or email:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com
    ```

#### **Filtering Users**

- **Search for users whose DisplayName starts with a specific string:**
    (Note: Wildcards are not supported, and this uses an internal `$filter` query syntax)
    ```powershell
    Get-MgUser -Filter "startsWith(DisplayName, 'a')" -ConsistencyLevel eventual
    ```

- **Search for users with "admin" in their Display Name:**
    ```powershell
    Get-MgUser -All | Where-Object {$_.DisplayName -match "admin"}
    ```

- **Search for users by specific attributes (e.g., DisplayName contains "admin"):**
    ```powershell
    Get-MgUser -Search '"DisplayName:admin"' -ConsistencyLevel eventual
    ```

#### **Listing All User Attributes**

- **Display all properties of a specific user:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```

- **List all the attribute names of a specific user:**
    ```powershell
    Get-MgUser -UserId test@defcorphq.onmicrosoft.com | ForEach-Object {$_.PSObject.Properties.Name}
    ```

- **Search user attributes for a specific string (e.g., "password"):**
    ```powershell
    Get-MgUser -All | ForEach-Object {
        $Properties = $_
        $Properties.PSObject.Properties.Name | ForEach-Object {
            if ($Properties.$_ -match 'password') {
                "$($Properties.UserPrincipalName) - $_ - $($Properties.$_)"
            }
        }
    }
    ```

#### **Identifying Synced vs Cloud-Only Users**

- **All users who are synced from on-premises Active Directory:**
    ```powershell
    Get-MgUser -All | Where-Object {$_.OnPremisesSecurityIdentifier -ne $null}
    ```

- **All users who are native to Entra ID (not synced):**
    ```powershell
    Get-MgUser -All | Where-Object {$_.OnPremisesSecurityIdentifier -eq $null}
    ```

#### **Objects Created or Owned by Users**

- **Objects created by a specific user (use `-UserId` for a specific user):**
    ```powershell
    Get-MgUserCreatedObject -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```

- **Objects owned by a specific user:**
    ```powershell
    Get-MgUserOwnedObject -UserId test@defcorphq.onmicrosoft.com | Format-List *
    ```

::note
**Important Notes:**
- **Users Synced from On-Prem Active Directory (Hybrid Identity):** These users have information about their **OnPremisesSecurityIdentifier**, which can be valuable to attackers.
- **Cloud-Only Users:** These users do not have the **OnPremisesSecurityIdentifier** attribute set to a non-null value.
::

### **3. Groups**
- `Get-MgGroup` → Lists all groups.
- **Interesting groups to examine:**
  - **Dynamic Groups** (automatically updated based on conditions).
  - **On-prem synced groups** (important for attackers).
  - **Groups where specific users are members** (for privilege escalation).

### **Enumerating Groups:**

1. **List All Groups:**
   ```powershell
   Get-MgGroup -All
   ```

2. **Enumerate a Specific Group:**
   ```powershell
   Get-MgGroup -GroupId <GroupId>
   ```

3. **Search for Groups:**
   - Based on DisplayName (first characters only, no wildcard support):
     ```powershell
     Get-MgGroup -ConsistencyLevel eventual -Search '"DisplayName:A"'
     ```
   - Groups with “admin” in their name:
     ```powershell
     Get-MgGroup -ConsistencyLevel eventual -Search '"DisplayName:Admin"'
     ```

4. **Find Groups with Dynamic Membership:**
   ```powershell
   Get-MgGroup | Where-Object {$_.GroupTypes -eq 'DynamicMembership'}
   ```

5. **On-Prem Synced Groups:**
   ```powershell
   Get-MgGroup -All | Where-Object {$_.OnPremisesSecurityIdentifier -ne $null}
   ```

### **4. Roles**
- `Get-MgRoleDefinition` → Lists all available role templates.
- `Get-MgRoleAssignment` → Lists users assigned to roles.

**Finding Global Admins:**
```powershell
$roleId = (Get-MgRoleDefinition | Where-Object { $_.DisplayName -eq "Global Administrator" }).Id
Get-MgRoleAssignment -RoleDefinitionId $roleId
```
**Output Example:**
- **Monica** → Tenant Admin (**Do not target** 🚫)
- **Other Admin** → Also out of scope (**Do not target** 🚫)

Even though this lab environment is intentionally misconfigured, **it is still better than many real-world production environments**.
Most companies have **far too many** Global Admins, which is a security risk.

#### **5. Enumerating Devices**
- `Get-MgDevice` → Lists all devices in the environment.
- **Important attributes:**
  - **Compliance status** (is the device compliant with security policies?).
  - **Device join type** (Azure AD Joined, Hybrid Joined, or Registered).
  - **LastSignInDateTime** (if null, the device was never used).
  - **Device Owners** (owners are automatically Local Admins on the device).

If **User A owns a device**, and we **compromise User A**, we now have **Admin access on that device**.

## **App Registrations vs. Enterprise Applications**
Microsoft **confuses everyone** with its naming conventions:

| **Portal Name**  | **Command Line Name** | **Graph API Name**  |
||-|-|
| App Registrations | Applications | Application Objects |
| Enterprise Applications | Service Principals | Service Principals |

### **What’s the Difference?**
- **App Registrations** → **Configurations** of applications (redirect URIs, permissions, etc.).
- **Enterprise Applications (Service Principals)** → **Instances** of those applications (this is where **roles & permissions are assigned**).

### **Abusing Application Ownership for Privilege Escalation**
- If a **user owns an application**, they can **add a Client Secret**.
- **Client Secrets are like passwords** for applications.
- If the **app has Global Admin permissions**, then the **user now has Global Admin access**.

#### **How Attackers Use This:**
1. **Find an Application with High Privileges.**
2. **If the user owns it, add a Client Secret.**
3. **Use the Client Secret to access Graph API as that application.**

**Logs:**
Fortunately, this action is **well-logged**.
The logs will **show exactly who added the secret, when, and to which application**.

**Why Would Someone Assign High Privileges to an Application?**
- To **automate tasks** (e.g., backing up VMs, provisioning resources, managing identities).
- It’s **easier to grant broad permissions than fine-tune security**.

**Persistence:**
- Adding a **Client Secret** is a **common persistence technique**.
- If security teams **miss it**, the attacker can **retain access** for months.

## **Final Thoughts**
- **Enumeration is the first step** in mapping out attack paths.
- **Attackers don’t just want access; they want privilege escalation & persistence.**
- **Applications are often a weak point** because they have **high privileges & poor oversight**.
- **Defenders should monitor logs** for suspicious changes (e.g., new client secrets, role assignments, app ownership changes).
