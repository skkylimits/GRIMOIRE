## **Azure CLI Token Storage**

> **Older Versions (Before 2.30.0 - January 2022)**

- Older versions of Azure CLI **stored access tokens in clear text** inside `accessTokens.json`.
- **📁 File Location:**

  You can dynamically retrieve the path of `accessTokens.json` in PowerShell using:
  ```
  C:\Users\[username]\.Azure\accessTokens.json
  ```

  ```powershell
  Join-Path $env:USERPROFILE ".Azure\accessTokens.json"
  ```

::tip
If you have **read access** to this file, you can extract tokens and reuse them.
::

- The same directory contains `azureProfile.json`, which holds **subscription details**.

- **Modifying Tokens:**
  - Attackers can replace or inject tokens into `accessTokens.json` to gain access via Azure CLI or Az PowerShell.
- **Clearing Stored Tokens:**
  - Run `az logout` to remove stored access tokens.

  ```powershell
  az logout
  ```

## **Azure PowerShell**

> **Older Versions of Az PowerShell**

- Stored access tokens in **clear text** inside `TokenCache.dat`.
- **📁 File Location:**
  ```
  C:\Users\[username]\.Azure\TokenCache.dat
  ```
  ```powershell
  Join-Path $env:USERPROFILE ".Azure\TokenCache.dat"
  ```
- If authentication used a **service principal**, `AzureRmContext.json` stored **ServicePrincipalSecrets in clear text**.

  ```
  %USERPROFILE%\AppData\Local\AzureRmContext #Windows
  ```
  ```
  ~/.azure/AzureRmContext #macOS and Linux
  ```

- **Saved Sessions:**
  - Users can persist authentication using `Save-AzContext`, are saved locally in a hidden folder in the user's home directory `AzureRmContext.json`.
  - Running `Disconnect-AzAccount` clears saved tokens.
  ```powershell
  Disconnect-AzAccount
  ```

## **Memory Dumps**

### PowerShell Processes

PowerShell is a powerful scripting environment that can store credentials and tokens in its memory. By dumping the process memory, you might be able to recover sensitive data such as access tokens or passwords.

**How to Dump:**
1. **Using Task Manager**:
    - Press `Ctrl + Shift + Esc` to open Task Manager.
    - Locate your PowerShell instance under the "Processes" tab.
    - Right-click on it and select "Create Dump File...".
2. **Using procdump.exe**:
    - You might need to run a command prompt as administrator or use the `runas` option within PowerShell to access this utility.
    - Use the following command to create a dump file: `procdump -n <PID>`. Replace `<PID>` with the process ID of the PowerShell window you wish to dump.

**Analyzing the Dump:**
- After dumping, you can use tools like `strings.exe` (part of Sysinternals Suite) or more advanced forensic tools such as Volatility to parse and analyze the memory dump.
    - `strings <filename>` will display all strings in the dump file that might be useful for identifying tokens or credentials.

### Office 365 & Graph API Tokens

Office applications like Outlook, Word, Excel can store tokens used for accessing services such as Office 365 or Graph API. These tokens are often stored in the memory of running Office apps and can be exposed by a memory dump.

**How to Dump:**
- Use the methods described above (Task Manager or procdump.exe) to create a memory dump of an active Office application window.

**Analyzing the Dump:**
- After creating the dump, use `strings` or forensic analysis tools to search for tokens. Tokens are often stored as encrypted strings in memory; you will need to look for them and then decrypt if possible to retrieve their content.
Sure! Here's a more detailed breakdown under the "Common Extraction Methods" header, splitting up the steps into the extraction and analysis processes:

### **Extraction Methods**

#### **Using procdump.exe**

**Extracting Memory Dump:**

- **Step 1:** Download and install the Sysinternals Suite from Microsoft, which includes **procdump.exe**. You can get it from the [official Sysinternals site](https://docs.microsoft.com/en-us/sysinternals/downloads/procdump).

- **Step 2:** Open a command prompt with **elevated privileges** (Run as Administrator).

- **Step 3:** Navigate to the folder where `procdump.exe` is located or add the path to the System Environment Variables for easier access.

- **Step 4:** Run the following command to capture a dump from a target process:

  ```bash
  procdump -ma <process_id> <output_file.dmp>
  ```

  - Replace `<process_id>` with the PID (Process ID) of the process you want to dump (e.g., PowerShell, Outlook, etc.).
  - Replace `<output_file.dmp>` with the desired name for the dump file (e.g., `powershell_memory.dmp`).

  To find the process ID (PID) of a running process:
  - Open **Task Manager** and go to the **Details** tab.
  - Locate your target process (e.g., PowerShell) and note the **PID** column.

#### **Using Task Manager**

**Extracting Memory Dump:**

- **Step 1:** Press **Ctrl + Shift + Esc** to open **Task Manager**.

- **Step 2:** Go to the **Processes** tab and locate the process you wish to dump (e.g., PowerShell, Outlook).

- **Step 3:** Right-click on the target process, then select **Create Dump File**.

- **Step 4:** Task Manager will generate the dump file, and you'll receive a notification showing the file's location (typically in your **Temp** folder).

  You can find it under:
  ```
  C:\Users\<your_user>\AppData\Local\Temp\<dump_file_name>.dmp
  ```

### **Analyzing the Dump**

#### **Using Strings**

Once you've extracted the memory dump file, the next step is to analyze it to search for readable strings, such as tokens or credentials.

**Analyzing the Memory Dump:**

- **Step 1:** Open a **Command Prompt** (CMD) or **PowerShell** window.

- **Step 2:** Use the `strings` command to extract readable strings from the dump file. For example:

  ```bash
  strings <dump_file_path>
  ```

  Replace `<dump_file_path>` with the full path to the dump file you created (e.g., `C:\Users\<your_user>\AppData\Local\Temp\powershell_memory.dmp`).

- **Step 3:** This will output a list of readable strings found in the dump file. Look for patterns that could indicate access tokens, authentication data, or other sensitive information.

#### **Using Forensic Analysis Tools**

For deeper analysis, forensic tools can provide more structured and advanced insights into the dump file.

**Analyzing with Volatility (Windows Memory Forensics):**

- **Step 1:** Download and install [Volatility](https://www.volatilityfoundation.org/).

- **Step 2:** Open a command prompt or PowerShell window with **administrator privileges**.

- **Step 3:** Run Volatility against the dump file with a suitable plugin, like `pslist` to list running processes or `tokens` to extract potential authentication tokens.

  Example command:

  ```bash
  volatility -f <dump_file_path> --profile=<profile_name> pslist
  ```

  - Replace `<dump_file_path>` with the path to the memory dump file.
  - Replace `<profile_name>` with the appropriate memory profile (e.g., `Win10x64_18362` for Windows 10 x64 version 18362).

- **Step 4:** Review the output from the forensic tool for any relevant tokens, API keys, or other sensitive information that could be present in the memory dump.

## Console History

Running `Get-PSReadLineOption` will display various settings, including **HistorySavePath**, which reveals the exact location of the history file.

```powershell
Get-PSReadLineOption
```

📂 **Default Location:**
```plaintext
C:\Users\%username%\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

- Modern Windows OS **stores a persistent history** of executed PowerShell commands.
- This history is managed by the **PSReadLine module** and is saved to a file on disk.
- To locate and read the history file in PowerShell, use:
  ```powershell
  Get-Content (Get-PSReadLineOption).HistorySavePath
  ```

::caution
The **history file is stored in clear text**, meaning it can be accessed by an administrator or any user with sufficient privileges.

This file is **reboot persistent**, meaning it retains command history even after system restarts.

It may contain **sensitive credentials** if commands involving secrets, passwords, or tokens were executed.
::

- **Disable PowerShell history logging**:
  ```powershell
  Set-PSReadlineOption -HistorySaveStyle SaveNothing
  ```

## Browser

Can I use a token in the browser?

You cannot use a token in the browser to access the Azure portal. However, you can do it the other way around — you could potentially steal a token from the browser and use it elsewhere.

## **Workload Identities**

Workload identities are special accounts used by apps or services to access resources, kind of like how a robot needs a key to open doors without needing a person to do it.

::caution
Workload identities are particularly vulnerable because it's challenging to implement MFA on service accounts for enforcement, and even in those cases, workload identities and service applications are often exempted from MFA requirements.
::

::important
Starting Tuesday, 15th, MFA enforcement will apply, but workload identities like service principals and applications  even in this case, will still be exempted . The recommendation is to implement a conditional access policy for these identities.
::

#title
Why Avoid User Identities?

#content
- **User identities are closely monitored** and have a higher chance of detection.
- **Access tokens expire faster** than those issued to workload identities.
- **Instead, focus on**:
  - **Managed identities** (used by applications, VMs, etc.).
  - **Service principals** (used for automation and app authentication).
::

## **Defenses**
- **Update Azure CLI & Az PowerShell** to latest versions.
- **Use `az logout` and `Disconnect-AzAccount` to clear stored tokens.**
- **Disable PowerShell history logging**:
  ```powershell
  Set-PSReadlineOption -HistorySaveStyle SaveNothing
  ```
- **Monitor for unauthorized API requests and unusual token usage.**
- **Enforce conditional access policies on workload identities (requires Premium license).**

## **Final Thought**
- **Stealing access tokens from Azure CLI and PowerShell is possible in older versions.**
- **User identities are risky** due to short token lifetimes and high detection chances.
- **Workload identities (service principals, managed identities) are a better target.**
- **PowerShell history (PSReadLine) can reveal sensitive credentials.**
- **Memory dumps of running processes (Office 365, PowerShell) can contain access tokens.**
- **Tokens can be used to access ARM and Microsoft Graph APIs.**
