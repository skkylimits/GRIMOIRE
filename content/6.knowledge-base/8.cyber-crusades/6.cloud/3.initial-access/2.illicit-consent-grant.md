## Permissions Requested
Regardless of the platform, applications must request user permissions. For example, when signing in, an app may ask:
*"Would you like to allow this application to read your profile?"*

> You've likely seen this when logging into the lab portal with your Google account—it asked for consent to access your profile. Or any other similar platform.

![permissions-requested](/content/6.knowledge-base/8.cyber-crusades/6.cloud/3.initial-access/2.illicit-consent-grant-attack/permissions-requested.png)

::warning
As a proof of concept, when you granted access, in theory, the app could read all your emails, access your files, or even delete them. That’s the power of consent.
::

Of course, we wouldn’t actually do that! But this highlights why **consent is critical in OAuth-based applications**, and Microsoft has recognized this.

## **Types of Permissions**

Microsoft realised that  that can be abused to the max. That's why if an application requests high-privileged permissions, a normal Entra ID user can only grant consent for low-impact permissions.

> There are 3 types of permissions. Low, medium and high. Microsoft designed it this way to limit potential risks.

### 1. **Low Impact Permissions (User-Granted)**
   - `Regular users can only grant consent to low-impact permissions.`
   - Only the permissions that don't need admin consent can be classified as
low impact
   - Example: [Basic sign-in](https://learn.microsoft.com/en-us/entra/identity/enterprise-apps/configure-permission-classifications?pivots=portal) (allows an app to read your email from the authentication token).
   - Another example: **User.ReadBasic.All**—lets an app read basic details (name, email, open extensions and profile photo) of all users in the tenant! It's a very interesting low impact permission.

::important
I'm simplifying this, but the way it works for the lab portal or most basic sign-ins is like this:

When you consent to the lab portal (enterprisesecurity.io) for basic sign-in, the application gets the JWT (token) that you submit when you consent. It then reads the email from the token because, in basic sign-in, it has permission to access your email. The app checks the email against the database to see if the user has registered for the course.

That's the basic idea behind most sign-ins.
::

### 2. **High/Medium Impact Permissions (Admin-Granted)**
   - `Permissions that affect multiple users or resources require admin consent.`
   - **Global Administrator, Application Administrator, Cloud App Administrator,** or **"custom roles with permission-granting privileges"** can approve these.
   - Example: Allowing an app to read **all OneDrive or SharePoint files** requires admin consent.

### **3. Application Vs. Delegated Permissions**

When configuring API permissions for an application, there are two primary types:

- **Delegated Permissions:**
  - `Runs **on behalf of a signed-in user**.`
  - Requires user consent (or admin consent for high-impact permissions).
  - Limited by the signed-in user's permissions.

- **Application Permissions:**
  - `Always requires **admin consent**.`
  - Used when an application runs as a daemon or background service.
  - Does not require a user to be signed in.

## **Default Consent Settings**

- **Default Setting** → **Allow user consent for all apps** (Not recommended but enabled by default).
- **Recommended Setting** → **Allow user consent only for verified publishers** (Reduces risk but increases IT workload).

To check or modify [these](https://entra.microsoft.com/#view/Microsoft_AAD_IAM/ConsentPoliciesMenuBlade/~/UserSettings) policies settings:
1. Go to **Enterprise Applications** in Entra ID.
2. Navigate to **Security** → **Consent & Permissions**.

Many organizations **stick with the default setting**, knowingly or unknowingly, due to convenience.

::warning
The "Allow user consent for all apps" setting is **not recommended** but is enabled by default, and it is **abusable**.

If an organization allows user consent for all apps, employees can grant apps access to basic information in their profiles (e.g., display name, email). If an organization doesn't restrict this This could potentially lead to unauthorized access if not properly controlled.
::

### **App Registration**

1. **Create the App Registration**
   - Name the app registration.
   - Register a new application in **Azure AD** as **multi-tenant**.
   - Specify the redirect URI (e.g., Web URL).

2. **Create a Secret**
   - Generate a client secret to authenticate the app.

3. **Assign API Permissions**
   - Choose between **Delegated** and **Application** permissions.
   - Example:
     - Delegated: User.ReadBasic.All
     - Application: Requires **admin consent** before use.

4. **Configure Token Settings**
   - In the **Authentication** option of the **studentx** app, check **"Access tokens (used for implicit flows)"**.
   - Click on **Save**.

   *(Note: If you want to use Refresh tokens, no additional configuration is required.)*

::warning
- Logs will clearly show **"Consent to Application"** activity. making it difficult to stay undetected.
- Targeting multi-tenant environments increases detection risk.
- More success may come from internal attacks within a compromised tenant, as logs may be ignored.
::

### Persistence

## **Defenses**

::caution
- The most visible part of such an attack is the **redirect URI**.
- If an attacker gains user consent to a malicious app, they can access sensitive data.
::

- **Prevent Users from Consenting to Unverified Applications**
  - Restrict users from approving apps from **unverified publishers**.
  - This is the most effective and **low-cost** defense.

- **Enable Detection & Alerts**
  - Create an **alert for `User.ReadBasic.All`** permission grants.
  - Monitor admin consent activity, as **admin-granted permissions pose a higher risk**.

- **Advanced Security Measures**
  - Organizations with **large security budgets** can implement automatic remediation.
  - Regularly audit **OAuth consent logs** to detect unusual app approvals.
