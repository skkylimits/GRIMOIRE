In the context of **OPSEC** (Operational Security), when someone advises to "stay away from user identities," they are referring to **user accounts** or **credentials** tied to individuals (e.g., usernames and passwords, or other forms of authentication like multi-factor authentication) that grant access to systems and services.

### What are **user identities**?
- **User identities** are the digital representations of individuals in a system or network. They are typically associated with:
  - **Usernames**: The unique identifier used to authenticate and access systems.
  - **Passwords**: The credentials used to validate the identity of a user.
  - **Multi-Factor Authentication (MFA)**: Additional layers of security, such as SMS codes or authenticator apps, tied to the user.
  - **User-Specific Permissions**: Access control policies that grant specific rights or privileges to the user (e.g., the ability to read, write, or execute certain resources).

### Why is staying away from **user identities** important in OPSEC?
1. **Increased Detection Risk**:
   - **User identities** are often heavily monitored by security systems. Actions associated with these identities are logged and can be flagged by security tools for abnormal activity (e.g., multiple failed login attempts, unfamiliar IP addresses, or suspicious file access).
   - Many organizations have strong monitoring and defense mechanisms that look for unauthorized access, lateral movement, or privilege escalation tied to user accounts.

2. **High Levels of Protection**:
   - User accounts are frequently subject to **MFA** or **conditional access policies**, which provide strong protections against unauthorized access. These protections can trigger alerts or block unauthorized attempts, making it more difficult for an attacker to bypass these security measures.

3. **Audit Trails**:
   - Actions performed by **user identities** (e.g., login, password reset, group membership change) are logged and typically monitored for unusual or malicious activity. Attacks involving **user identities** are more likely to be detected through these logs.

4. **Easier to Trace**:
   - Attackers who compromise a **user identity** (e.g., through phishing or social engineering) may leave behind a clear trail of actions linked to that specific user account, making it easier for security teams to trace malicious activity back to the attacker.

### Alternatives: **Service Accounts, Managed Identities, and Application Identities**
- **Service Accounts**: Accounts created for specific applications or services (not individual users) with limited permissions. These accounts usually do not have MFA, making them easier to manipulate.
- **Managed Identities**: Special identities in cloud environments (like **Azure Managed Identities**) that provide secure, automated access to resources without using traditional user credentials.
- **Application Identities**: Identities created for applications and services, rather than individual users. These are often less protected than user accounts but can still be valuable for attackers if compromised.

### Summary:
The recommendation to "stay away from user identities" emphasizes the high risk of detection and the challenges in evading security mechanisms when targeting individual accounts. Instead, attackers may prefer to focus on **service accounts, managed identities**, or **application identities**, which are less scrutinized and often have fewer protections in place, providing a stealthier and more effective means of attack.

### Access Tokens and Workload Identities

#### Access Token Usage:
1. **Getting Access Tokens:**
   - You can obtain an access token using several methods, but it's highly advised to **stay away from user identities**. Accessing user identities increases the chances of detection, and token expiry happens more quickly for user tokens compared to managed identities or service principals.

2. **Why Avoid User Identities:**

::caution
- User identities are closely monitored and have a higher chance of being detected. **Stay away from using user identities** for security and operational reasons.s
::

3. **Refresh Tokens:**
   - **Refresh tokens** allow you to obtain new access tokens once they expire. However, accessing user identities or extracting tokens from user machines is not recommended due to high detection risks.

#### Methods of Obtaining Access Tokens:

1. **From Older Versions of Tools:**
   - Older versions of Azure CLI (`az`) and PowerShell stored access tokens in local user directories (e.g., `.azure` folder). In these directories, you may find files like `accessTokens.json` that store tokens in clear text. If you have read access to these files, you could potentially extract tokens. However, if **easy logout** is used by the user, tokens are cleared.

2. **PowerShell Token Cache:**
   - Older versions of `AZ PowerShell` stored credentials in `tokenCache.dat` files. If you have access to this directory, you may retrieve access tokens in clear text.

3. **Memory Dumps:**
   - If a user has an Office 365 app open (e.g., Word, PowerPoint), **memory dumps** of the application can be used to extract **Graph API tokens**. This can be done if you have access to the user’s memory dump.

4. **PS Readline Console History:**
   - Windows PowerShell maintains a history of all executed commands in a file, which can be read in clear text. This is enabled by default in **PSReadLine** module. It can be used to retrieve previously executed commands and, in some cases, sensitive credentials.

#### Workload Identities:
1. **Workload Identities Explained:**
   - **Workload identities** refer to service principals and managed identities used by applications or services. They are different from user identities and are designed to facilitate secure access to resources without relying on user credentials.

2. **Why Focus on Workload Identities:**
   - **Workload identities** are more secure than user identities because they don’t rely on user credentials, and conditional access policies are more difficult to enforce on them. However, Microsoft offers limited functionality for workload identities without premium licenses.

#### Microsoft Licensing:
1. **Premium Workload Identity:**
   - To protect workload identities, you need to purchase **Premium Workload Identity** licenses. These licenses are required for every application or service principal you wish to protect with a conditional access policy.

2. **Cost Implications:**
   - Workload identity licenses are sold per app or service principal, with a cost of around **$15 per month** per application. This can be a significant cost, especially for small organizations.

#### API Access with Access Tokens:
1. **Azure Resource Manager (ARM) API:**
   - You can use access tokens to directly interact with the **Azure Resource Manager (ARM)** API. This API is used to manage resources like Virtual Machines, Key Vaults, and Storage Accounts.

2. **Microsoft Graph API:**
   - Similarly, access tokens can also be used to interact with the **Microsoft Graph API** to manage user data, organizational information, etc.

3. **End of AD Graph:**
   - **AD Graph** (graph.windows.net) is deprecated and being replaced by **Microsoft Graph** (graph.microsoft.com).

4. **Access Token Validation:**
   - When using an access token, the API validates the token’s audience and signature. Access tokens are used to authenticate requests to these APIs and ensure the requester has the necessary permissions.

#### PowerShell & Azure CLI:

1. **Using Tokens in PowerShell:**
   - When using PowerShell or Azure CLI, you can pass the access token to the appropriate APIs to authenticate requests.

2. **Token Expiration:**
   - Keep in mind that access tokens have a limited lifespan. When they expire, you can use a **refresh token** to get a new access token, provided that the refresh token is still valid.

---

### Summary:
- **Access tokens** are used to authenticate API requests after the initial authentication process, and they are valid only for a short period.
- **Workload identities** (service principals and managed identities) are a safer option than user identities for accessing resources and interacting with APIs.
- **Premium licenses** are required for protecting workload identities with conditional access policies.
- Access to tokens can be obtained from tools like Azure CLI, PowerShell, and via **memory dumps** from Office 365 apps.
