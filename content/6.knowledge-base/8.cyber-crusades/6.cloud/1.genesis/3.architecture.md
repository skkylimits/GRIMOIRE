## **Azure Architecture**

> Understanding the scope is crucial before diving into Azure services.

Azure resources are divided into four hierarchical levels:

- **Management Groups**
- **Subscriptions**
- **Resource Groups**
- **Resources**

![scope-levels](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/scope-levels.png)

> Entra ID Directory sits above the **Root Management Group** in terms of hierarchy..

## **Management Groups**

- **Management Groups** help organize and manage multiple Azure subscriptions.
- Any rules or policies applied to a **Management Group** automatically apply to all its subscriptions (**inheritance**).
- All subscriptions in a **Management Group** must be part of the same **Azure tenant**.
- **Management Groups** can be nested within each other, creating a hierarchy.
- The **Root Management Group** sits at the top of this hierarchy and controls all management groups in a directory.
  - **Note:** A **Global Administrator** can always gain access to the **Root Management Group** if needed.

## **Subscriptions**

- An **Azure Subscription** is a way to manage and organize Azure services under an account.
- It acts as a **billing** and **access control** boundary within an **Entra ID Directory**.
- A single **Entra ID Directory** can have multiple subscriptions, but each subscription is linked to only one directory.
- If you assign a **role** at the subscription level, it applies to all resources inside that subscription.

![subscriptions](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/subscriptions.jpg)

### **Changing Directories**

- You can **move a subscription** to a different directory, but only if both are linked to the same **billing profile**.
- This change mostly affects **billing** and can impact access when switching between **tenants**.

## **Resource Groups**

- **Resource Groups** are containers that organize Azure resources.
  - Every resource must be in a **Resource Group** and can belong to only one.
  - Deleting a **Resource Group** also deletes everything inside it.
- **Resource Groups** have their own **Identity and Access Management (IAM)** settings, allowing role-based access control (RBAC) for all resources within the group.

- **Azure Managed Identities** let resources like App Services, Function Apps, and Virtual Machines securely access other Azure services.
- They use **Entra ID tokens** to authenticate with services like Key Vault and Storage Accounts—no need to manage passwords or secrets.
- A **Managed Identity** is a special type of **service principal** that works with Azure resources.
  - **System-assigned**: Linked to a single resource and deleted when the resource is removed.
  - **User-assigned**: Created separately and can be shared across multiple resources.

### Authentication

- **Managed Identity** eliminates the need for manual credential management, reducing overhead and improving security.
- The security depends on how **roles** are assigned. For example, assigning a **Reader** role is more secure than **Owner** or **Contributor** roles.
- You can identify a **Managed Identity** by navigating to:
  - **Virtual Machine** -> **App** -> **Security** -> **Identity** (with authenticated access to the resource).
  ![IAM](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/IAM.png)

## **Azure Resource Manager (ARM)**

> Every time you manage a resource in Azure—whether through the **portal**, **CLI**, or an API—**ARM** processes the request.

- **Azure Resource Manager (ARM)** is the central service that manages and deploys Azure resources. It handles:
  - Creating, updating, and deleting resources (**lifecycle management**).
  - **Access control** to manage who can do what with resources.
- You can use **ARM Templates** to deploy resources in a structured, repeatable way, with built-in dependency management.

![consistent-management-layer](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/consistent-management-layer.png)

## **Entra ID**

- **Entra ID Directory** sits at the top of the hierarchy, above the **Root Management Group**.
- A **Global Admin** has the highest level of access in **Entra ID** and can always manage the **Root Management Group**.
- High-level roles in **Entra ID** allow control over all resources within the directory.

### **Access Control and Role Assignments**

- By default, regular users **do not** have access to all resources in **Entra ID**.
  - To access specific resources, **roles** must be assigned explicitly.
- Unlike **on-prem AD**, where users can see everything on a domain-joined machine, **Azure** requires specific **role assignments** for visibility.
  - If you have the **Reader** role, you can view all resources in a subscription but **cannot** make changes.

## **Entra ID Roles**

- **Entra ID roles** apply to identity-related resources like users, groups, domains, and licenses.
- There are many **Administrator roles** in Entra ID (see slide notes), and custom roles can also be created.
- The **Global Administrator** role is the most powerful and well-known admin role.
- A **Global Administrator** can elevate their permissions to become a **User Access Administrator** in Azure, gaining control over the **Root Management Group**.

![entra-id-roles](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/entra-id-roles.png)

### **The Risk of Global Administrator Compromise**

In an **on-premises Active Directory (AD)** environment, if domain controllers (DCs) are compromised, physical access can allow admins to restore the system. However, in **Entra ID (Azure AD)**, there is no backup copy of your schema—if a **Global Administrator** is compromised, recovery is extremely difficult.

::tip
**Best Practices for Securing Global Admin Accounts**

- **Avoid permanent Global Admin assignments**—use **just-in-time (JIT) access** instead.
  - A **Global Admin** role should only be assigned for short periods (e.g., **1-hour elevation**, then removal).
- **Store Global Admin credentials securely**, such as using:
  - **FIDO security keys** stored in a **secure locker**.
  - **Multi-Factor Authentication (MFA)** for added security.
  - **Certificate-based authentication** with smart card dongles.
- **Use a break-glass account** for emergency situations.
  - This account should be highly secured, only used in critical scenarios (e.g., when other methods of access are unavailable), and require multiple layers of protection, such as MFA and strong password policies.
::

#### **Real-World Examples of Global Admin Compromise**

Several major cyberattacks have shown how dangerous a **Global Admin compromise** can be:

- **APT2029 (Midnight Blizzard)** – A sophisticated nation-state attack.
- **SolarWinds Attack** – Hackers gained persistence in networks through privileged access.
- **Recent Exchange Server Attack** – Exploited high-level admin access for long-term control.

> These incidents highlight why **permanent Global Admin roles should only exist for emergency (break-glass) scenarios**.

## Entra ID Editions

![entra-editions](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.architecture/entra-editions.png)

To properly secure your Azure tenant, it's important to understand the differences between the available editions, and often, **paying for a higher-tier edition** is necessary.

- The **free edition** offers basic features, but it lacks the advanced security controls that are necessary for real protection.
- For **production environments**, at least **Azure AD Premium P1** is recommended.
  - With **P1**, you'll have access to **Conditional Access (CA)**, which is crucial when enforcing **Multi-Factor Authentication (MFA)** requirements.
- If you're willing to take risks, you could use **Security Defaults** and go with the free version, but this may not be sufficient for securing a production environment.

> In short, to ensure strong security, investing in the appropriate edition, like **P1**, is highly recommended.
