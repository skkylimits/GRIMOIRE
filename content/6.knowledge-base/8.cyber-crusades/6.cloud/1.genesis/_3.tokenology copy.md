## Microsoft Identity Platform
Plays a significant role in authentication and authorization within Azure environments.

### Key Concepts of MIP:
- **Azure Resource Manager (ARM)**: The endpoint where Azure resources communicate or when you want to interact with resources in Azure.
- Unlike on-prem Active Directory (which is Kerberos-based), the **Microsoft Identity Platform** uses **OAuth** and **OIDC** for authentication and authorization across services like **Entra IDM**, **Microsoft 365**, and more.

### Authentication Protocols:
- **OAuth**: The backbone of the Microsoft Identity Platform and the de facto authentication and authorization mechanism.
- **OIDC (OpenID Connect)**: A protocol built on top of OAuth for identity verification.
- **Other Protocols Supported**:
  - **SAML** (Security Assertion Markup Language)
  - **Legacy Authentication Protocols** for synchronization, such as header-based LDAP and Kerberos-based constraint delegation.

For now, we'll focus mainly on **OAuth** and **OIDC**, as the other protocols are more relevant for domain synchronization in the cloud, which we won't cover here.

## Understanding OAuth

If you're new to OAuth, it can be overwhelming, but don't worry—I'll break it down for you.

### What is OAuth?
OAuth is a protocol that allows third-party applications to grant users access to resources without needing to share their credentials. Instead, OAuth uses **tokens** to facilitate access.

### How OAuth Works (Basic Flow)
1. **User Authentication**:
    - A user submits their **username** and **password** to a client (e.g., a web browser) when trying to access a service like **portal.azure.com**.
    - The client (browser) sends these credentials to the **Microsoft Identity Platform** (authorization server).

2. **Token Issuance**:
    - After authentication, the **Microsoft Identity Platform** validates the credentials and issues a **bearer token** to the client.

3. **Accessing Resources**:
    - The client (browser) then sends the bearer token to a **resource server** (e.g., Graph API, ARM API) to access resources.
    - The resource server validates the token and provides access if the token is valid.

### The Importance of Bearer Tokens
- The **bearer token** is the key takeaway here. It grants access to the resources specified in the token.
- Tokens are used for authentication and authorization across Microsoft services.

## OAuth in Action: Basic Sign-In Process

Let's look at how the sign-in process works in a real-world example, like accessing **portal.azure.com**:

1. **Access Request**:
    - The user navigates to **portal.azure.com** (a web application).
    - The web app redirects the user to the Microsoft Identity Platform for authentication.

2. **User Inputs Credentials**:
    - The user enters their credentials directly in the Microsoft Identity Platform (not in the web application). This ensures credentials aren't shared with the web app.
    - If necessary, the user consents to permissions required by the application.

3. **Token Validation**:
    - After validating the credentials, the Microsoft Identity Platform issues a token (JWT).

4. **Token Submission**:
    - The browser sends the token to the **web application** (e.g., Azure Portal).
    - The web server validates the token and presents the secure page to the user.

### Key Takeaway:
- After authentication, the token is essential for accessing the portal or resources. The **token** is what allows the web application to communicate with other Azure services on behalf of the authenticated user.

## Tokens: Access, ID, and Refresh Tokens

### Types of OAuth Tokens
1. **Access Token**:
    - Used to access resources on behalf of a user.
    - Access tokens are issued for a specific **user**, **client**, and **resource**.
    - Once issued, access tokens are valid until expiry (typically 1 hour).
    - **Cannot** be revoked before expiry.

2. **ID Token**:
    - This token provides information about the **user** and is generally not of interest for resource access. It's used for authentication, not authorization.

3. **Refresh Token**:
    - Allows clients to obtain a new access token after the original access token expires.
    - It can be revoked, unlike the access token.
    - **No expiry** for active tokens; **90 days** for inactive refresh tokens.

### The Importance of Access Tokens
- **Access Tokens** are vital for accessing resources (e.g., Entra ID, Graph API, ARM endpoint).
- They cannot be revoked until they expire (typically 1 hour by default, but can last up to 24 hours for specific tokens).

#### Access Token Use Cases:
- If you have a **user’s access token**, you can use it to access resources without needing the user's credentials.
- Access tokens are very powerful and can be used by attackers if stolen.

#### Exceptions to the Access Token Rules:
1. **Expiry Duration**:
    - By default, access tokens expire in **1 hour** but may last up to **24 hours** for tokens issued by the Azure instance metadata service (IMDS).

2. **Continuous Access Evaluation (CAE)**:
    - CAE is a mechanism that allows access tokens to be **revoked before expiry** if certain conditions are met (like security breaches). This helps mitigate the risk of token abuse.

## Why Are Access Tokens Important for Security?

- **Access Tokens** are often **misused** because once they are issued, they can’t be revoked until they expire (unless using CAE).
- Attackers target **access tokens** to gain unauthorized access to resources. This is why token replay attacks are a significant concern.

### Activity Logs and Access Tokens
- **Access tokens** themselves do not create sign-in logs. This is because OAuth's flow focuses on validation, not logging in.
- However, when an **access token** is used, an activity log may be created if actions are taken (like accessing a resource), but **no sign-in log** will be present.

#### Key Points:
- Using an **access token** does not generate a **sign-in log**.
- If you access a resource with an access token, the **resource server** just validates the token without re-authenticating the user.
- **Authentication** happens when the token is issued, but using the token doesn't trigger a new sign-in event.

### When Do Activity Logs Appear?
- If a **refresh token** is used to obtain a new access token, it generates a **sign-in log** because the **refresh token** is associated with the actual authentication event.

### Activity Log Exceptions
- **Instance Metadata Service (IMDS)**:
    - Access tokens issued by the IMDS (for example, in Azure App Services) are handled differently. These tokens may not follow the standard logging behavior.

## Summary of OAuth Token Types
- **Access Tokens**: Used for accessing resources (expires after 1 hour, cannot be revoked until expiry unless using CAE).
- **ID Tokens**: Provide user information (not relevant for accessing resources).
- **Refresh Tokens**: Used to acquire new access tokens, can be revoked, typically valid for 90 days.

### Final Takeaway:
- The **token** (especially the **access token**) is key to understanding how authentication and authorization work in modern systems, including Azure.
- The **bearer token** is crucial for gaining access to resources in the Microsoft Identity Platform.

::caution
Remember, **access tokens are powerful**, and once obtained, they can be used until expiry. Attackers often target these tokens for unauthorized access.

### Overview

1. **Conditional Access Block (53003 Error)**:
    - When trying to authenticate using the test user credentials from your host machine, you were blocked by a **conditional access policy**.
    - **Conditional Access** (CA) works like an "if-else" flow, where access is either allowed or blocked based on specific conditions such as location or user status. For example, if a user is trying to access resources from an untrusted location, access is denied.
    - **Lab scenario**: When you connected from the student VM, it worked because that VM was within a trusted location, but accessing from your host machine was blocked.

2. **What is Conditional Access Policy?**
    - Conditional Access (CA) is used to define access rules for users trying to access resources (e.g., cloud apps) from untrusted locations. In your example, the policy blocked access when attempting to access the cloud apps from a non-trusted public IP.
    - The policy can enforce additional security measures such as **Multi-Factor Authentication (MFA)** or **blocking access** entirely.

3. **Why Access Tokens Bypass CA?**
    - An **access token** is used after authentication to access resources. The key point here is that tokens can **bypass conditional access policies**.
    - When you directly use the access token (rather than authenticating again), CA doesn't evaluate the request because the token has already been issued and validated, meaning the policy doesn’t reapply.

4. **The Power of Access Tokens**:
    - The access token is a **base64 encoded** string, which can be decoded to reveal valuable information, like the **audience**, **expiration time**, **scopes**, and **identity of the user**.
    - Access tokens allow you to bypass certain security measures because once you have the token, you don’t need to go through the authentication process again, even if you’re operating from an untrusted location.

5. **Token Information (Decoded Example)**:
    - When decoded, the token reveals:
      - **Audience**: The Azure Resource Manager (ARM) or API you're interacting with.
      - **Expiration**: When the token expires (e.g., 90 minutes).
      - **Authentication Method**: The method (e.g., password) used to authenticate the user.
      - **Scope**: The resources and actions the token allows.
    - This decoded information also includes the **user identity**, and **IP address** the token was issued from.

6. **Security Considerations**:
    - Once an attacker has an access token, they can bypass the conditional access policies (such as location-based restrictions) and interact with the resources associated with that token.
    - This can be a major vulnerability, as even if conditional access policies block authentication attempts, the attacker can still **replay the token**.

---

### CAE (Continuous Access Evaluation)
- **CAE** is a new feature Microsoft is introducing to mitigate token replay attacks.
- **CAE** works by constantly checking for changes in user context, including IP address, location, device health, and session risk. It can revoke access tokens or block further actions if the conditions change, providing an additional layer of protection.

    - **Example**: If a user logs in from a trusted location, but an attacker tries to use an old token from an untrusted location, CAE can revoke that token and block access.

- **Limitation**: If **CAE** isn't active or doesn't catch the replayed token in time, the attacker can still use it to perform malicious actions.
- **Why Doesn’t It Always Work?**: CAE is a relatively new feature, and not every action is subject to this continuous evaluation. Depending on how the token was issued or the nature of the API being accessed, CAE might not prevent the attacker from using the token.

### Key Takeaways
- **Access Tokens**: While useful, they can be a major security risk if not handled properly. They allow users to bypass conditional access policies, and attackers can exploit this to perform unauthorized actions.
- **Conditional Access Policies**: These are meant to enforce security measures based on specific conditions like location or device health. However, they cannot block access once an attacker has obtained a valid access token.
- **CAE**: Continuous Access Evaluation is a newer tool designed to mitigate token abuse, but it isn’t perfect. It provides continuous monitoring of session activities to block abnormal access, but the use of stolen tokens is still a challenge.
- **Defense**: As a defensive measure, it's crucial to monitor user activity closely, especially when tokens are used. However, because token use often doesn't generate logs until actions are performed, detection might be difficult without active monitoring tools.

### Microsoft Identity Platform (MIP) - **Authentication** vs **Authorization**

#### Authentication (AuthN) and Authorization (AuthZ) Roles:
- **Authentication (AuthN)**: Verifies the identity of a user or system (i.e., confirming who the user is).
- **Authorization (AuthZ)**: Determines what actions or resources the authenticated entity is allowed to access.

### In MIP:

1. **OAuth 2.0**:
   - OAuth 2.0 is primarily used for **authorization (AuthZ)**.
   - It allows third-party applications to request permission to access resources on behalf of the user, without needing to share the user’s credentials.
   - OAuth 2.0 **does not directly handle authentication**. It just grants access based on tokens.

2. **OIDC (OpenID Connect)**:
   - OIDC is built on top of **OAuth 2.0** and adds the functionality for **authentication (AuthN)**.
   - It verifies the user's identity by adding a standardized identity layer on top of OAuth 2.0, providing information like user details (name, email, etc.) in the **ID token**.

### So to summarize:
- **OAuth 2.0**: Primarily used for **authorization** (AuthZ), not authentication.
- **OIDC**: Extends OAuth 2.0 and handles **authentication** (AuthN), providing a user identity verification mechanism.

### Other Protocols:
- **SAML 2.0** is often used for **authentication**, especially in enterprise contexts, where it’s used in **Single Sign-On (SSO)** scenarios.
- **Legacy Methods** like **LDAP** or **Kerberos** are used for **synchronization** between on-premise systems and cloud services but are not directly involved in authentication or authorization in the same way OAuth or OIDC are.

### What's True?
- OAuth 2.0 is for **authorization** (AuthZ).
- OIDC is for **authentication** (AuthN) on top of OAuth.
- **SAML** is another option for **authentication**.
