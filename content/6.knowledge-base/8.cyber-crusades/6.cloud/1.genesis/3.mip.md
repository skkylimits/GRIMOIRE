## Microsoft Identity Platform (MIP) Overview
The Microsoft Identity Platform (MIP) is a core part of Azure's authentication and authorization system. It uses **OAuth** and **OIDC** to secure access to services like **Entra ID**, **Microsoft 365**, and others.

> By the end of this file, you should have a clear understanding of the **Microsoft Identity Platform**, **OAuth** flow, the **types of tokens** used, and the potential security risks associated with them.

### Authentication Protocols
- **`OAuth 2.0`**:
   - OAuth 2.0 is primarily used for **authorization (AuthZ)**.
   - It allows third-party applications to request permission to access resources on behalf of the user, without needing to share the user’s credentials.
   - OAuth 2.0 **does not directly handle authentication**. It just grants access based on tokens.

- **`OIDC` (OpenID Connect)**:
   - OIDC is built on top of **OAuth 2.0** and adds the functionality for **authentication (AuthN)**.
   - It verifies the user's identity by adding a standardized identity layer on top of OAuth 2.0, providing information like user details (name, email, etc.) in the **ID token**.

- **`Other Protocols`**:
  - **SAML 2.0**: Primarily used for **authentication** (AuthN), especially in Single Sign-On (SSO) scenarios.
  - Legacy methods like **LDAP**, **Kerberos**, and **Header-based** protocols are used for synchronization between cloud services and on-prem systems.

## Roles in OAuth 2.0

Four parties are generally involved in an OAuth 2.0 and OpenID Connect authentication and authorization exchange. These exchanges are often called authentication flows or auth flows.

![protocols-roles](/content/6.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.mip/protocols-roles.svg)

- **Authorization Server**: The identity provider (e.g., Microsoft identity platform) that manages user authentication and issues tokens for access control.
- **Client**: The app requesting access to protected resources (e.g., web app, mobile app).
- **Resource Owner**: The end-user who owns the protected data and can grant or deny access to it.
- **Resource Server**: The server (usually an API) that hosts or provides access to the resource owner’s data.
## How OAuth Works: The Basics

OAuth is a protocol that allows third-party applications to request **access** to a user’s resources **without needing the user’s credentials**. Instead, it uses **tokens**.

### The OAuth Flow
1. **User Login**: A user enters their credentials (username/password) into a client (e.g., browser) to access a service.
2. **Token Issuance**: After verifying the credentials, the **Microsoft Identity Platform** sends back a **bearer token** to the client.
3. **Access Resources**: The client uses this bearer token to request access to resources (e.g., Graph API, Azure Resource Manager).

### **Bearer Tokens**
A **bearer token** is a security token that grants the user access to a resource. It's like a "key" that opens the door to various services.

## OAuth in Action: Accessing Resources

### Step-by-Step Sign-In
1. **Request Access**: A user navigates to an application like **portal.azure.com**.
2. **Redirect to Authentication**: The application redirects the user to the **Microsoft Identity Platform** for authentication.
3. **Credential Validation**: The user enters their credentials, and if correct, a **token** is issued.
4. **Token Validation**: The client sends the token to the app’s backend to access the user’s resources, and the server validates the token.

## Types of Tokens in OAuth

### 1. **Access Token**
- Used to access resources on behalf of a user.
- **Expires** after a set time (usually 1 hour).
- Cannot be revoked until it expires, unless **Continuous Access Evaluation (CAE)** is used.

### 2. **ID Token**
- Contains basic **user information** (e.g., name, email).
- Used primarily for **authentication**, not resource access.

### 3. **Refresh Token**
- Used to request new access tokens after the old ones expire.
- Can be revoked but usually has a long lifespan (e.g., 90 days of inactivity).

## Token Security: Why It Matters

### Why Access Tokens Are Important
Access tokens are powerful because they let you access resources without needing to log in again. However, they can be **misused** if stolen by attackers.

- **Access tokens** are tied to a **user**, a **client**, and a **resource**. If an attacker steals a valid token, they can access resources like the legitimate user without needing to log in.

### Security Risks
- **Token Replay**: Once an attacker gets an access token, they can use it until it expires—**bypassing conditional access policies** (like location or MFA checks).
- **Log Activity**: Accessing a resource with an access token doesn’t create a **new sign-in log**. So, the system won’t know if an attacker is using the token.

### CAE
- **CAE** Continuous Access Evaluation allows access tokens to be revoked before they expire if there’s a change in the user's context (e.g., location or device health).
- This helps to **prevent token abuse** by ensuring that a token is only valid as long as the user's session is secure.

## CA and Access Tokens

### Conditional Access (CA) Policies
- These policies can block access based on factors like **location** or **user status** (e.g., requiring **MFA**).
- Once a user has an **access token**, it can bypass **Conditional Access policies**. This means that even if access is blocked due to the user's location or device, the attacker can still use the token to access resources.

## How Attackers Exploit Tokens
Once attackers have a valid **access token**, they can **replay** it and gain access to resources. This is a **security vulnerability** because the token allows them to bypass the conditional access policies set for authentication.

## Summary of OAuth Token Types
- **Access Tokens**: Used for resource access; expires after 1 hour.
- **ID Tokens**: Contain user information, not used for resource access.
- **Refresh Tokens**: Can be used to get new access tokens, valid for up to 90 days.

### Key Takeaways
1. **Access Tokens** allow applications to access resources without re-authentication.
2. **Conditional Access** policies help secure access, but once an attacker has a valid token, they can bypass these checks.
3. **Continuous Access Evaluation (CAE)** adds an additional layer of security by monitoring the user’s session in real-time, but it’s not foolproof.
4. Attackers often target **access tokens** because they are easy to reuse until expired.
