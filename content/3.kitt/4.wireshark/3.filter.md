## Compare display vs capture filters

::caution
**Comparison**
- Use Capture Filters to grab only the traffic you need **before** you start recording.
<br />
- Use Display Filters to look through everything **after** you already recorded.
::

Wireshark uses two kinds of filters:

1. **Capture Filters**

::note
**Capture Filters** are applied *before* capturing data to limit what traffic is collected, using a simpler syntax (like `tcp` or `host 10.0.0.5`).
::

* **Used before packet capture begins** (applied at the moment you're starting the capture).
* **Goal**: To limit what packets are actually captured and saved to memory/disk.
* **Syntax**: Based on **libpcap/tcpdump** syntax (more limited).
* **Examples**:

  * `tcp` – only capture TCP packets
  * `host 192.168.1.3` – only capture traffic to/from this IP

> 🧠 Think of Capture Filters like setting a camera to only record red cars on a street. If a blue car goes by, it's never recorded.

2. **Display Filters**

::note
**Display Filters** are used *after* data is captured, to analyze specific packets with more detailed conditions (like `ip.addr == 10.0.0.5 && tcp.port == 443`).
::

* **Used after the packets are already captured** (you filter what you see).
* **Goal**: To dig into specific traffic from all captured packets.
* **Syntax**: Wireshark’s own syntax (more powerful and expressive).
* **Examples**:

  * `ip.addr == 192.168.1.3`
  * `http.request.method == "GET"`

> 🧠 Think of Display Filters like using a magnifying glass to examine only the red cars in a full recording of all traffic.

🔁 Comparison Table

| Feature            | Capture Filter                | Display Filter                      |
| ------------------ | ----------------------------- | ----------------------------------- |
| When Used          | Before capture starts         | After capture has been completed    |
| Syntax             | tcpdump/libpcap (basic)       | Wireshark (advanced)                |
| Flexibility        | Limited                       | Very flexible                       |
| Performance Impact | Reduces data and memory usage | No reduction – data already loaded  |
| Use Case Example   | Only want TCP traffic         | Only want to view HTTP GET requests |

## Create capture filter (protocol/IP/MAC/port)

::caution
**Capture Filters **limit what Wireshark records**, which is helpful when:**

* You only want to see certain traffic
* You’re trying to save memory and disk space
* You already know what you’re looking for
::

Before clicking **Start**, you type your **Capture Filter** in the **"Capture Filter" box** (bottom left of the interface).

🔧 **Examples**

| What you want to filter | Capture Filter syntax          |
| ----------------------- | ------------------------------ |
| Only TCP traffic        | `tcp`                          |
| Only UDP traffic        | `udp`                          |
| Traffic from one IP     | `host 192.168.1.3`            |
| Traffic from one MAC    | `ether host 00:1A:2B:3C:4D:5E` |
| Traffic on one port     | `port 80`                      |
| Range of ports          | `portrange 1000-2000`          |

::tip
**Quick Tip**

- Capture Filters use **BPF (Berkeley Packet Filter)** syntax, which is **different** from Display Filters. Keep that in mind—they're **not interchangeable**.
::

## Create display filter (manual/right-click/drag)

| Method            | What to do                                                | Example                                             |
| ----------------- | --------------------------------------------------------- | --------------------------------------------------- |
| **Manual Entry**  | Type filter yourself in the filter bar                    | `arp`                                               |
| **Right Click**   | Right-click a field in Packet Details → *Apply as Filter* | Right-click `arp.dst.proto_ipv4`, then apply filter |
| **Drag and Drop** | Drag a field from Packet Details to the filter bar        | Drag `arp.opcode` to the filter bar                 |

## Use membership filters

::caution
**How does it work in Wireshark?**

- Wireshark uses the `in` keyword to check if a field’s value is inside a list or a subnet.

To filter HTTP or HTTPS traffic, you could write:

```
tcp.port in {80, 443}
```
::

Membership filters let you test if a value is **within a set** or **belongs to a group** of possible values.

Think of it like asking:
**“Is this IP address one of these?”**
or
**“Is this port one of these ports?”**

| Use Case                     | Filter Example                                       | What it does                                   |
| ---------------------------- | ---------------------------------------------------- | ---------------------------------------------- |
| Check if IP is in a subnet   | `ip.addr in {192.168.1.0/24}`                        | Shows packets where IP is in 192.168.1.x       |
| Check if TCP port in list    | `tcp.port in {80, 443, 8080}`                        | Matches packets with TCP port 80, 443, or 8080 |
| Check if MAC address in list | `eth.addr in {00:11:22:33:44:55, aa:bb:cc:dd:ee:ff}` | Matches packets from either MAC address        |

## Use logical operators

Logical operators let you **combine multiple filter conditions** to create complex filters.

| Operator | Symbol | Meaning                   | Example          |
| -------- | ------ | ------------------------- | ---------------- |
| **AND**  | `&&`   | Both conditions must be true | `ip.src == 192.168.1.1 && tcp.port == 80` |
| **OR**   | `||`   | Either condition is true  | `icmp || arp`    |
| **NOT**  | `!`    | Negates a condition       | `!arp`           |

::caution
**How to apply:**

* **Manual entry** in the filter bar is the main way to use logical operators.
* Combine filters to be specific, like filtering traffic from an IP AND a specific port.
* Use parentheses `( )` to group conditions and clarify precedence.
::

**Example Filters:**
* Capture HTTP or HTTPS traffic:
  `tcp.port == 80 || tcp.port == 443`
* Traffic from a certain IP AND TCP port 22:
  `ip.addr == 10.0.0.5 && tcp.port == 22`
* Show all traffic except ARP:
  `!arp`

## Create filter button

::tip
The **Create Filter button** helps you **build a new display filter based on the selected field/value**, which you can then modify or save.
::

1. **Type** your desired filter into the **Display Filter bar** (e.g., `tcp.port == 80`).
2. Click the small **plus icon (➕)** to the right of the filter bar.
3. In the popup, give your filter a **name** (e.g., "HTTP Traffic").
4. Click **OK** — now your filter shows up as a button under the filter bar.
5. You can now **click that button anytime** to instantly apply the saved filter.
6. Right-click on any filter button & Choose "Manage Display Filter Buttons" to list **filters**.

::caution
**Why use filter buttons?**

* Quick access to filters you use often
* Saves time during live captures or analysis
* Organizes your workflow for common protocols or IPs
::

## Recognize incomplete/excess filter results

* You’re **seeing too much traffic** → **Excess results**
* Or you’re **missing expected traffic** → **Incomplete results**

| Type           | Mistake                                             | Example                             | Fix                                             |
|----------------|-----------------------------------------------------|-------------------------------------|--------------------------------------------------|
| **Excess**     | Filter includes unrelated traffic                   | `ip` (shows all IP traffic)         | Use `ip.addr == 192.168.1.1`                    |
| **Incomplete** | Filter excludes needed packets due to wrong logic   | `tcp.port == 80 && tcp.port == 443` | Use `tcp.port == 80 || tcp.port == 443`         |
| **Incomplete** | Using `==` instead of `contains` for string matches | `http.host == "google"`             | Use `http.host contains "google"`               |
| **Excess**     | Forgetting to negate something                      | `http`                              | Use `http && !http.request.method == "OPTIONS"` |

::tip
**Tip to Spot Issues**

* **Expected traffic missing?** → Your filter may be too **strict**
* **Seeing random/unwanted traffic?** → Your filter may be too **loose**
* Use the **Statistics > Protocol Hierarchy** menu to get a baseline of what traffic *should* be there
::

## Explain 'not' operator behavior

**`not` Operator (`!`) Behavior**

* The **`not` operator** negates the condition that follows it.
* It **matches all packets that do NOT satisfy** the condition.
* It’s used to **exclude** certain traffic or values from your filter results.

| Filter                    | Meaning                                      | Example Use Case                       |
| ------------------------- | -------------------------------------------- | -------------------------------------- |
| `!arp`                    | Match all packets that are **not** ARP       | Show all traffic except ARP            |
| `!ip.addr == 192.168.1.1` | Match all packets **not** from or to this IP | Exclude traffic from/to IP 192.168.1.1 |
| `!(tcp.port == 80)`       | Match packets where TCP port is **not** 80   | Exclude HTTP traffic                   |

::note
**Important notes:**

* When negating complex expressions, **use parentheses** to group conditions for clarity and correctness.
  E.g. `!(tcp.port == 80 || tcp.port == 443)` means packets where the port is neither 80 nor 443.
* You can combine `!` with other logical operators (`&&`, `||`) for more precise filters.
::

## Apply filters from stats

Wireshark’s **Statistics menus** (like Protocol Hierarchy, Conversations, Endpoints) give you summaries of captured traffic.
From these stats views, you can **apply filters directly** to focus on specific protocols, IP addresses, ports, or conversations.

1. Open a stats window, for example:

   * **Statistics → Protocol Hierarchy**
   * **Statistics → Conversations**
   * **Statistics → Endpoints**
2. Find the item you want to filter by (e.g., a specific IP address or protocol).
3. **Right-click** on that item.
4. Select **“Apply as Filter”** → choose to filter **“Selected”** (only that item) or **“Not Selected”** (everything except that item).
5. Wireshark automatically creates and applies the correct display filter for you.

**Example**

* In **Conversations**, right-click an IP address → Apply as Filter → Selected
  → This creates a filter like `ip.addr == 192.168.1.10` to see only packets to/from that IP.

::caution
**Why use this?**

* Quickly focus on specific traffic without manually typing filters.
* Ideal for exploring big captures by narrowing down based on statistics summaries.
::

## Filter with generated fields

::caution
**What are they**

* **Generated fields** are fields that Wireshark **creates or interprets** from raw packet data.
* These fields often represent **decoded or derived information**, like DNS query names, HTTP methods, or TCP flags.
* You can use generated fields in **display filters** to focus on specific protocol details.
::

1. **Identify** the generated field in the **Packet Details** pane (e.g., `http.request.method`).
2. **Use it directly** in the display filter bar, like:
   `http.request.method == "GET"`
3. Combine with other fields using logical operators if needed:
   `http.request.method == "POST" && ip.addr == 192.168.1.10`
4. You can also use **right-click → Apply as Filter** on generated fields to create filters easily.

| Filter                                     | Description                           |
| ------------------------------------------ | ------------------------------------- |
| `dns.qry.name == "example.com"`            | DNS query for domain "example.com"    |
| `http.response.code == 404`                | HTTP response with error 404          |
| `tcp.flags.syn == 1 && tcp.flags.ack == 0` | TCP SYN packets (start of connection) |
