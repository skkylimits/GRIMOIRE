## Compare traffic capture methods

* **Direct capture on endpoints** is great for detailed local traffic but limited to one device.
* **Network taps** offer full passive visibility with no packet loss but need hardware.
* **Infrastructure devices** enable centralized capture but may affect device performance.
* **Port mirroring** is flexible and easy but can miss packets if overloaded.
* **Multi-point capture** provides broad visibility but increases complexity.

| Method                    | Description                                                               | Benefits                                                                                                               | Drawbacks / Notes                                                                                              |
| ------------------------- | ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Direct on Endpoint**    | Capture traffic directly on the device (e.g., with Wireshark on PC)       | - Captures all local traffic including internal process communication<br>- Easy to deploy on single device             | - Limited to that endpoint only<br>- May miss network-wide traffic<br>- Requires access to device              |
| **Network Tap**           | Physical device inserted inline on network link to passively copy traffic | - Captures all traffic on the link (full visibility)<br>- Does not affect network operation<br>- High accuracy         | - Requires hardware installation<br>- Cost and physical access needed                                          |
| **Infrastructure Device** | Capture on routers, switches, or firewalls with built-in packet capture   | - Centralized capture from network hardware<br>- Can filter and capture specific traffic                               | - May have limited buffer size<br>- Load on infrastructure devices may affect performance                      |
| **Port Mirror (SPAN)**    | Switch duplicates traffic from specified ports/VLANs to monitoring port   | - Flexible capture points<br>- No physical tap needed<br>- Good for monitoring specific segments                       | - May drop packets under high load<br>- Not all traffic types always mirrored<br>- Possible performance impact |
| **Multi-Point Capture**   | Capture traffic simultaneously from multiple points or devices            | - Comprehensive network view<br>- Correlate traffic from various sources<br>- Good for complex network troubleshooting | - More complex setup<br>- Requires synchronization<br>- Higher resource requirements                           |

## Select capture interface

To **select the appropriate interface to capture traffic in Wireshark**, based on the Wireshark documentation, you would:

1. **Open Wireshark.**

2. **View the Capture Interfaces List:**

   * On the main screen, you’ll see a list of available network interfaces under **"Capture"**.
   * Alternatively, go to **Capture → Options** or press `Ctrl + K`. Make sure you're not actively capturing.

3. **Identify the Correct Interface:**

   * Each interface usually shows its name, description, and packet activity.
   * Look for the interface that matches the network connection you want to monitor.
   * For example, if you're on Wi-Fi, look for an interface named like "Wi-Fi" or your wireless adapter.
   * You can also check packet counters to see which interface is active.

4. **Select the Interface:**

   * Click the checkbox next to the interface name.
   * You can select multiple interfaces if you want to capture from more than one.

5. **Start Capturing:**

   * Click **Start** to begin capturing packets on the selected interface(s).

::tip
**Pro tip.**

* If you’re unsure which interface to pick, watch the **traffic indicator graphs** next to the interface name. The graph will move when there is network activity.
* Use **`ipconfig`** (Windows) or **`ifconfig`/`ip addr`** (Linux/macOS) to confirm your device’s IP address and match it to the correct interface in Wireshark.
* Enable **Promiscuous Mode** in capture options if you want to capture all packets on the network segment, not just packets addressed to your machine.
::

## Start/stop/restart capture

**Start Capture**

* Open Wireshark.
* Select the desired interface from the interface list.
* Click the **blue shark fin icon** (Start capturing packets) in the toolbar, or double-click the interface name.
* Wireshark begins capturing packets immediately.

**Stop Capture**

* Click the **red square icon** (Stop capturing packets) in the toolbar.
* This stops the current capture session.

**Restart Capture**

* Click the **Restart** icon (circular arrow) in the toolbar.
* This stops the current capture and immediately starts a new capture on the same interface.

::tip
You can also use shortcuts:

* **Start:** Ctrl + E
* **Stop:** Ctrl + E (toggles start/stop)
* **Restart**: Useful when you want to clear the capture view and start fresh without closing Wireshark.
::

## Limit capture size/time

::warning
Stops the capture automatically after a set time or packet count to prevent overrun or massive files.
::

* Go to **Capture** > **Options** (or press **Ctrl + K**).
* Under the **Options** tab:

**Limit Capture by Number of Packets**

* In **Capture Options**, find the **“Stop capture automatically after”** setting.
* Enter the maximum number of packets to capture.
* Wireshark will automatically stop when that number is reached.

**Limit Capture by Duration**

* In **Capture Options**, find the **“Stop capture automatically after”** setting.
* Set a time limit in seconds (e.g., 300 for 5 minutes).
* Capture will stop automatically after the set time.

::tip
**Quick Steps Summary:**

1. **Open Capture Options (Ctrl + K)**
2. Set limits under **Stop capture after** (packets or seconds).
3. Use **Output** tab for file size/time limits
::

::important
**Using "Stop capture after" + Ring Buffer together:**
- Split your capture into multiple files (with the Ring Buffer), and

- Still tell Wireshark to stop after a specific duration or number of packets.
::

## Use ring buffer

A ring buffer lets Wireshark save captured packets into multiple smaller files instead of one huge file. Once the set number of files is reached, it **overwrites the oldest files**.

::important
This helps manage disk space and keeps recent data available.
::

1. **Open Capture Options:**
   Go to **Capture** > **Options** (or press **Ctrl + K**).

2. **Enable Ring Bugger:**
   Check the box **“Use Ring Bugger”**.

3. **Set Number of Files:**
   Enter how many files you want Wireshark to keep (e.g., 5).

4. **Set File Size or Time:**
   Choose when to switch to the next file by setting:

   * **“Next file every”** size limit (e.g., 50 MB), **or**
   * **“Next file every”** time interval (e.g., 10 minutes).

5. **Start Capture:**
   Wireshark will save captured data in a cycle, overwriting old files once the number limit is reached.

::caution
**Why Use a Ring Buffer?**

* Prevents huge capture files that are hard to handle.
* Keeps the most recent network data available.
* Useful for long-term or continuous capture scenarios.
::

## Save capture file

**Save a Capture in Wireshark**

1. **After stopping the capture**, go to the top menu.
2. Click **File** > **Save As** (or press `Ctrl + S`).
3. Choose:

   * A **location** to save the file.
   * A **filename**.
   * The **file format** (default: `.pcapng`, compatible with Wireshark).
4. Click **Save**.

::tip
**Pro Tip:**
Give meaningful filenames (e.g., `2025-06-04_dns_capture.pcapng`) to keep captures organized.
::

## Export Specified packets to file

Want to save only the **filtered** or **selected** packets? Here’s how:

1. Apply a **display filter** to show only the packets you want to export.
   *(e.g., `http`, `ip.addr == 192.168.1.1`)*
   → This will limit what’s visible in the packet list.

2. Go to **File** > **Export Specified Packets**.

3. In the export window:

   * ✅ **Packet Range**: Choose **"Displayed"** to export only filtered packets.
   * (Optional) Select other ranges if needed (e.g., marked packets, specific numbers).
   * Choose your **filename** and **format** (usually `.pcapng`).

4. Click **Save**.

::tip
**Pro Tip:**
- This is perfect for sharing minimal captures when troubleshooting or reporting issues—no need to send the entire file.
::

**Capture traffic using command line tools** with Wireshark’s companion tool, **Tshark**:

1. **Open your command prompt or terminal.**

2. Use **tshark** (Wireshark’s CLI tool) to start capturing on an interface.
   Example command:

   ```bash
   tshark -i 1 -w capture.pcapng
   ```

   * `-i 1` specifies the interface number (use `tshark -D` to list interfaces).
   * `-w capture.pcapng` saves the capture to a file.

3. (Optional) Add filters:

   ```bash
   tshark -i 1 -f "tcp port 80" -w http_capture.pcapng
   ```

   * `-f` applies a **capture filter** (like capture only HTTP traffic).

4. Press **Ctrl+C** to stop the capture.

::tip
**Tip:** Use `tshark -D` to list all available interfaces before starting your capture.
- Tshark is great for automated or remote captures without a GUI.
::

## Promiscuous Mode

**What extra data do you see with promiscuous mode?**

Normally, your network card only listens to packets sent **directly to your computer** or to everyone (broadcasts).
When you turn on **promiscuous mode**, your network card listens to **all packets on the same network segment**, even if they’re not meant for you.

**So, what extra do you get?**

* You see messages between other devices on the network, like if computer A is talking to computer B.
* You still see broadcast and group messages (which you always see).
* You also see normal messages between other devices that you wouldn’t normally get.

---

::caution
**But there’s a catch:**

* On modern networks with switches, the switch only sends messages to the device they’re for.
* So, even in promiscuous mode, you only see messages sent to your device or broadcast, unless:

  - You’re on an old-style network hub (rare now), or
  - The switch is set to copy all traffic to your port (called port mirroring or SPAN).
::

## Monitor Mode

::warning
Not all Wi-Fi adapters or drivers support monitor mode — you need compatible hardware and software.
::

**What extra data does monitor mode capture compared to normal mode?**

**Monitor mode** is specific to **wireless (Wi-Fi) adapters** and is quite different from promiscuous mode:

* Normally, your Wi-Fi adapter only captures packets **addressed to it** or broadcast/multicast packets on the wireless network it’s connected to.
* When you enable **monitor mode**, your Wi-Fi adapter captures **all wireless traffic in the air**, on the channel it’s listening to — **including packets not addressed to your device, management frames, control frames, and even traffic from other networks** using the same channel.

**What extra do you see in monitor mode?**

* **All Wi-Fi frames on the channel:**
  Data, management, and control frames from all nearby wireless devices.
* **Packets from other networks on the same channel:**
  Even if your adapter is not connected to those networks.
* **Raw 802.11 frames:**
  You get the full wireless frame, including headers normally stripped off in managed mode.
* **Ability to capture Wi-Fi management activities:**
  Like beacons, probe requests/responses, authentication, and association frames.

---
::caution
**Why is this useful?**

* It’s essential for **wireless troubleshooting, security analysis, and detailed wireless protocol analysis**.
* Without monitor mode, you only see the traffic your adapter is part of (your connected network).
::
