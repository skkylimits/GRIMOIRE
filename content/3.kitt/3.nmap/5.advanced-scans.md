## 🔐 Geavanceerde Scans (TCP/UDP/Wildcards)

**Al deze technieken hieronder zijn alternatieve scanmethodes om poorten te ontdekken; hun stealth-effect is een bijwerking, geen echte firewall-evasion.**

### SYN scan (stealth)

```bash
nmap -sS target
```

**Hoe het werkt:**
1. Send SYN packet
2. Receive SYN/ACK → port is **open**
3. Send RST (don't complete handshake)

::note
"Stealth" is relatief - moderne IDS/firewalls detecteren SYN scans makkelijk.
::

### Connect scan

```bash
nmap -sT target
```

**Nadelen:**
- Langzamer dan SYN scan
- Fully logged door target system
- Minder accurate

### UDP scan

```bash
nmap -sU target
```

::warning
UDP scans zijn **extreem traag** - geen handshake, wachten op timeouts, rate limiting.
::

**Belangrijke UDP services:**
- 53 (DNS)
- 161/162 (SNMP)
- 137/138/139 (NetBIOS)
- 67/68 (DHCP)
- 123 (NTP)

### NULL scan

```bash
nmap -sN target              # No flags set
```

**Detectie logic:**
- Closed port → RST
- Open port → No response
- Filtered → No response

::important
**TCP NULL-scans (`-sN`) stuur je zonder flags, waardoor sommige systemen per RFC verplicht zijn te reageren met een RST op gesloten poorten — handig om hosts of poortstatus te ontdekken zonder een normale connectie op te zetten.**
Ze worden vaak gebruikt voor **stealth**, omdat veel firewalls en IDS dit verkeer niet als een klassieke scan herkennen.
::

### FIN scan

```bash
nmap -sF target              # FIN flag set
```

::important
Waarom `-sF` (FIN-scan)?

**`-sF` stuurt alleen een FIN-flag, waardoor veel systemen volgens RFC *geen* antwoord mogen geven op open poorten — dit maakt het een stealth-techniek om poortstatus te bepalen zonder een normale TCP-handshake.**
Firewalls en IDS detecteren FIN-scans vaak minder goed dan SYN-scans, waardoor je stiller kunt enumereren.
::

### Xmas scan

```bash
nmap -sX target              # FIN, PSH, URG flags
```
::important
**`nmap -sX target` stuurt FIN+PSH+URG (Xmas-flags) om poortstatus te bepalen.**

* **Gesloten poorten:** sturen een **RST** terug
* **Open of gefilterde poorten:** geven **geen antwoord**

Wordt gebruikt voor **stealthy port discovery** omdat veel systemen deze pakketten niet herkennen als klassieke scans.
::

::note
NULL/FIN/Xmas scans werken **alleen op RFC 793 compliant systems**. Windows reageert anders!
::

### ACK scan (Firewall detectie)

https://nmap.org/book/scan-methods-ack-scan.html

```bash
nmap -sA target
```

**Results:**
- **Unfiltered** → No firewall blocking
- **Filtered** → Stateful firewall detected

::important
ACK scans tonen **niet** of poorten open zijn - alleen firewall detection!
::

### Custom flags

https://nmap.org/book/scan-methods-custom-scanflags.html

```bash
nmap --scanflags URG,FIN,PSH target
nmap --scanflags SYN,ACK target
```

::note
Embrace the freedom to customize and create a scan that suits your requirements

- Met `--scanflags` kun je zelf exacte TCP-flags combineren om unieke of stealthy scanpatronen te creëren. Hiermee kun je firewall-reacties testen, detectie omzeilen of gedrag analyseren dat niet door standaard Nmap-scans wordt gedekt.
::

### IP Protocol scan

https://www.iana.org/assignments/protocol-numbers/protocol-numbers.xhtml

Het stuurt een **IP-pakket met protocolnummer 143** (→ uit de IP-header!)
en kijkt of de host **iets terugstuurt**.

Dus:

* **Je scant geen poort**
* **Je scant geen TCP of UDP**
* **IP Protocol Scan (-sO) werkt op laag 3, niet laag 4**
* **Je gebruikt het protocolveld, niet het poortveld**
* **Je scant welk *IP-protocoltype* het systeem ondersteunt**

```bash
nmap -sO target
```

::note
**`-sO` scant geen poorten, maar alle mogelijke *IP-protocollen* (1–255) die een host ondersteunt.**
Hiermee ontdek je welke protocollen (zoals ICMP, IGMP, GRE, ESP, SCTP) *ingeschakeld of gefilterd* zijn op een systeem.

* **Protocol toegestaan:** host reageert → protocol *open*
* **Protocol geblokkeerd:** geen reactie of “unreachable” → protocol *filtered*

Handig voor het identificeren van ongebruikelijke of gevoelige protocollen zoals GRE (VPN), ESP/AH (IPsec), of andere IP-gebaseerde services die niet via TCP/UDP-poorten zichtbaar zijn.
::

::important
Hiermee ontdek je welke **protocollen** (zoals ICMP, IGMP, GRE, ESP, SCTP) ingeschakeld of gefilterd zijn op een systeem.
::

### Ethernet Send Mode

Doet dit automatisch wanneer nodig (bijv. op LAN’s).

[https://nmap.org/book/man-bypass-firewalls-ids.html#man-send-eth](https://nmap.org/book/man-bypass-firewalls-ids.html#man-send-eth)

```bash
nmap --send-eth target
```

::note
`--send-eth` verandert **niet je scantechniek**, maar **hoe Nmap pakketten verzendt** — namelijk rechtstreeks als Ethernet-frames op laag 2.
::


### IP Send Mode

Nmap gebruikt dit automatisch wanneer Ethernet-framing niet mogelijk is (bijv. via VPN’s, tunnels, cloud-interfaces).

[https://nmap.org/book/man-bypass-firewalls-ids.html#man-send-ip](https://nmap.org/book/man-bypass-firewalls-ids.html#man-send-ip)

```bash
nmap --send-ip target
```
::note
`--send-ip` verandert **niet de scantechniek**, maar **wel hoe de pakketten worden verzonden** — namelijk direct via de IP-laag (L3) van de kernel’s raw IP stack, zonder Ethernet-frames.

* **Raw IP sending** → Verstuurd zonder Ethernet-laag, direct via IP.
* **Werkt op non-Ethernet links** → Zoals VPN’s, tunnels en virtuele adapters.
* **Routing tests** → Handig om te zien hoe routers met pure IP-pakketten omgaan.
* **Geen L2 nodig** → Werkt zelfs als er geen Ethernet-interface beschikbaar is.
* **Betrouwbaar voor WAN/VPN scans** → Minde
::
