## List Remoted

List current remotes in the local repo:

```[terminal]
git remote
```

Should be empty if repo is not linked to remote repo yet

## Create A New Remote

```[terminal]
git remote add origin https://github.com/skkylimits/git.git
```

## Show More Remote Info

```[terminal]
git remote show origin
```

## Push Files

Use git push to upload your local code to GitHub.

```[terminal]
git push origin master
```

Note: the -u flag is used to set origin as the upstream remote in your git config so git pull can be used without any arguments in the future.

## Fetch Files

Fetch the latest changes locally. Code in editor hasn't changed to reflect that because we didn't merged yet.

If you look at the branches you will see multiple branches.

```[terminal]
git fetch
```

## Merge Files

It takes two different branches/codebases and merges it together.

```[terminal]
git merge origin/master
```

Now local code should be synced up with the remote repo.

## Merge Strategy

Fast-forward - simplest type of strategy when commits are connected together so the pointer can be moved to the most recent commits.

We'll take a look at merge again later to look at more complex merge strategy.

For now there is a much simpler one

## Git Pull

Combine the fetch & merge from the previous section with the pull command.

When you work on a software project that changes frequently you'll often be pulling down new code from the remote repo.

```[terminal]
git pull origin master
```

To do everything in a single go.

However since we used the -u flag setting up the repo earlier we don't need to specify the remote branch

The way to address that problem is to either commit your local changes first or use a technique called `stash`

### **1. Stash Your Changes**

To stash your uncommitted changes, simply run:

```[terminal]
git stash
```

This command will save your changes to a stack of stashes and revert your working directory to the last committed state. By default, `git stash` will stash both staged and unstaged changes.

### **2. List Stashes**

If you want to view a list of all stashes you have saved, use:

::tabs
::div

---

label: Code
icon: i-heroicons-code-bracket-square

---

```[terminal]
git stash list
```

::

::div

---

label: Preview
icon: i-heroicons-magnifying-glass-circle

---

```[terminal]
stash@{0}: WIP on main: 1234567 Commit message
stash@{1}: WIP on feature-branch: 890abcd Another commit message
```

::
::
Each stash is referenced by an index (e.g., `stash@{0}`).

### **3. Apply Your Stash**

To apply the most recent stash and keep it in the stash list (i.e., you can reapply it later if needed), use:

```[terminal]
git stash apply
```

To apply a specific stash from the list, specify its index:

```[terminal]
git stash apply stash@{0}
```

This will reapply the changes from the stash to your working directory.

### **4. Pop Your Stash**

If you want to apply the most recent stash and remove it from the stash list, use:

```[terminal]
git stash pop
```

This is useful if you no longer need the stash after applying it. If you want to pop a specific stash, specify its index:

```[terminal]
git stash pop stash@{0}
```

### **5. Drop a Stash**

If you want to remove a specific stash without applying it, use:

```[terminal]
git stash drop stash@{0}
```

This command deletes the specified stash from the list.

### **6. Clear All Stashes**

To remove all stashes, use:

```[terminal]
git stash clear
```

This command clears the entire stash list, so use it with caution.

### **Example Workflow**

Here’s a typical workflow for using `git stash`:

1. **Modify Files**: Make some changes to your files.
2. **Stash Changes**: Save your changes to the stash.

   ```[terminal]
   git stash
   ```

3. **Pull Latest Changes**: Fetch and merge updates from the remote repository.

   ```[terminal]
   git pull
   ```

4. **Reapply Stash**: Reapply your stashed changes.

   ```[terminal]
   git stash pop
   ```

5. **Resolve Conflicts**: If there are any conflicts between your stashed changes and the pulled updates, resolve them.

6. **Commit Your Changes**: Once you’re satisfied with the changes, stage and commit them.

   ```[terminal]
   git add .
   git commit -m "Describe your changes"
   ```

## Merge Conflict

Another thing is to be aware of is that your local code is modifying the same line of code that you're trying to pull and merge in. It could result in a merge conflict

## Git Clone

Clone a remote repository to your local machine and optionally change the name of the director.

When you clone a repo it will write and clone all the files in the remote to your local machine, but it also keeps a reference to the original repo.

That means that you can use most of the git commands on it.

```[terminal]
git clone <repo-url> <local-directory>
```

## Github Codespaces

Press `.` to activate vscode in github to commit small changes without having to clone the entire repo to your local machine.

One limitation is that you can't access the terminal.

So you can't build and serve the app.

Codespaces is a paid service and allows you to run a vm inside github so you will have access to the terminal.
