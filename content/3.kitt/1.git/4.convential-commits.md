`See how a minor change to your commit message style can make you a better programmer.`

**Conventional Commits** is a simple convention for writing commit messages, where each message follows a structured format. The main goal is to make commit history more readable and enable automatic tools to generate meaningful changelogs, determine semantic version numbers, and trigger CI/CD pipelines.

The commit messages follow this general structure:

```
<type>(<scope>): <short description>
```

## Types

- `feat`: A new feature.
- `fix`: A bug fix.
- `chore`: Routine tasks, maintenance, or refactoring that doesn't affect the codebase or production.
- `docs`: Modifies documentation without changing code.
- `style`: Code style changes (like formatting), no code functionality changes.
- `refactor`: Refactoring code without adding new features or fixing bugs.
- `test`: Adding or modifying tests, nothing affecting production code.
- `perf`: Performance improvement.

## Scopes

`<scope>` is an optional label that specifies the area of the code affected by the commit. It helps clarify which part of the codebase the change pertains to, making it easier to understand the context of the change.

## Example

- `feat(login): add login button`
- `fix(auth): resolve token expiry issue`
- `chore(deps): update dependency @nuxtjs/auth`
- `docs(readme): update setup instructions`

This structure makes commit messages consistent and understandable at a glance, while also enabling automation.

```
fix: correct typo in user login error message
^--^  ^----------------------------^
|     |
|     +-> Summary in present tense.
|
+-------> Type: chore, docs, feat, fix, refactor, style, or test.
```
