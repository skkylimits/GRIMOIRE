See how a minor change to your commit message style can make you a better programmer.

Format: `<type>(<scope>): <subject>`

`<scope>` is optional

## Example
```
fix: correct typo in user login error message
^--^  ^----------------------------^
|     |
|     +-> Summary in present tense.
|
+-------> Type: chore, docs, feat, fix, refactor, style, or test.
```

## Types

- `feat`: Introduces a new feature to the codebase.
- `fix`: Addresses a bug or issue in the code.
- `chore`: Includes routine tasks that do not affect functionality, like updating dependencies.
- `docs`: Modifies documentation without changing code.
- `style` Adjusts the code style or formatting (e.g., spacing, indentation) without affecting functionality.
- `refactor`: Changes the code structure or design without altering behavior or adding features.
- `perf`: Performance improvement.
- `test`: Adds or updates tests to ensure the code works as expected.
  
## Scopes

`<scope>` is an optional label that specifies the area of the code affected by the commit. It helps clarify which part of the codebase the change pertains to, making it easier to understand the context of the change.

**Examples**:
- `feat(auth): add JWT authentication` - `auth` is the scope, indicating changes related to authentication.
- `fix(ui): correct button alignment` - `ui` is the scope, referring to user interface adjustments.
