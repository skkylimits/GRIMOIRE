## 📌 Introduction
AI models like **DeepSeek R1** have changed the landscape of **AI accessibility** and **privacy**.
In this guide, we’ll explore:
- Why running AI models **locally** is better than using cloud services
- **How to install and run DeepSeek R1** on your system
- **Potential risks** and how to mitigate them
- **Using Docker** for enhanced security

## 🎭 Why Run AI Models Locally?
When you use **DeepSeek R1 online** or via an app, your `data is stored on their servers`. This means:

- ✅ They **own** your data.
- ✅ They **can** share, analyze, or store it.
- ✅ In the case of **DeepSeek**, their servers are in **China**, subjecting your data to **Chinese cybersecurity laws**.

### 🌐 Cloud AI vs. Local AI

| **Method**                           | **Pros**                            | **Cons**                                    |
|--------------------------------------|-------------------------------------|--------------------------------------------|
| **Cloud AI (ChatGPT, DeepSeek R1 Online)** | ✅ Easy access, no hardware needed | ❌ Data privacy concerns, reliant on external servers |
| **Local AI (DeepSeek R1 on your PC)** | ✅ Full privacy, no external tracking | ❌ Requires local resources (RAM, GPU) |

🔒 **Bottom line**: If you value **privacy and control**, running AI **locally** is the best option.

# Set log file path
$logFile = "$env:USERPROFILE\ollama_network_log.txt"

# Infinite loop to monitor connections
while ($true) {
	# Get the current timestamp
	$timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss"

	# Get all running Ollama processes
	Get-Process ollama -ErrorAction SilentlyContinue | ForEach-Object {
		$id = $_.Id
		$logEntry = "`n[$timestamp] Connections for Ollama process ID: $id`n"

		# Fetch network connections, filtering out local addresses (127.x.x.x and [::1])
		$connections = Get-NetTCPConnection | Where-Object {
			$_.OwningProcess -eq $id -and
			$_.RemoteAddress -notmatch "^(127\.|::1$|0\.0\.0\.0|::)$"
		} | Select-Object LocalAddress, LocalPort, RemoteAddress, RemotePort, State

		# If there are external connections, log them
		if ($connections) {
			$logEntry += $connections | Out-String
			$logEntry | Out-File -Append -FilePath $logFile
			Write-Host $logEntry -ForegroundColor Green
		}
	}

	# Wait for 2 seconds before checking again
	Start-Sleep -Seconds 2
}

#### **Explanation of the PowerShell Script**

This PowerShell script continuously **monitors network connections** for the `ollama` process. Let's break it down:

**Step-by-Step Breakdown**

1. **`while($true) { ... }`**
   - This creates an **infinite loop**, meaning the script runs continuously until manually stopped (Ctrl + C).

2. **`Get-Process ollama | ForEach-Object { ... }`**
   - This gets all running instances of the `ollama` process and iterates over them.
   - If `ollama` isn't running, nothing happens.

3. **`$id = $_.Id`**
   - Retrieves the **process ID (PID)** of `ollama` and stores it in `$id`.

4. **`Write-Host "\nConnections for Ollama process $id" -ForegroundColor Green`**
   - Prints a message showing the connections for the given **ollama process ID**, in **green text**.

5. **`Get-NetTCPConnection | Where-Object OwningProcess -eq $id`**
   - Retrieves **all active TCP connections** and filters them to only show those owned by the `ollama` process.

6. **`Select-Object LocalAddress, LocalPort, RemoteAddress, RemotePort, State`**
   - Displays key network details:
     - `LocalAddress`: The IP address of your machine.
     - `LocalPort`: The port your machine is using.
     - `RemoteAddress`: The **destination** IP (if the process is sending data).
     - `RemotePort`: The **destination port** (if connected to another server).
     - `State`: The connection state (e.g., **Established**, **Listening**, **Closed**).

7. **`Start-Sleep -Seconds 2`**
   - Waits **2 seconds** before checking again.

8. **`Clear-Host`**
   - Clears the PowerShell console to keep the output clean.

---

#### **How This Helps Detect Internet Usage**
- If **ollama is connecting to the internet**, you will see **non-local IP addresses** under `RemoteAddress`.
- If **ollama is truly offline**, you should **only see `0.0.0.0`, `127.0.0.1`, or local network addresses (e.g., 192.168.x.x, 10.x.x.x, etc.)**.
- If the `RemoteAddress` column shows an **external IP (e.g., 8.8.8.8 or any public IP)**, this means `ollama` **is sending data over the internet**.

---

#### **How to Test It**
1. Run the script **while Ollama is running**.
2. If `RemoteAddress` is **empty or only has local IPs**, then Ollama is **not communicating externally**.
3. If `RemoteAddress` has **public IPs**, then Ollama **is connecting to the internet** (possibly sending data).

## 🔥 AI Models in Docker

> Enhancing security. We know that Ollama reaches out to external ip when update/upgrading model on request. So what if somehow in a new model or version. It does have the malicious tendency to send all your data?

Even though local AI **seems secure**, running it directly on your OS gives it access to:

- ❌ **Network connections**
- ❌ **File system**
- ❌ **System resources**

### 🛡️ **Solution: Run AI in Docker for Isolation**
Using **Docker**, we can **sandbox** the AI model, preventing access to system files and limiting internet usage.

### **📥 Installing Docker**
1. Install [Docker](https://www.docker.com) for your OS
2. **Windows Users**: Install **WSL2 (Windows Subsystem for Linux)**
3. **Linux Users**: Install the **NVIDIA Container Toolkit** if using an NVIDIA GPU

### **🚀 Running OLLAMA in Docker**
To start **Ollama** inside a Docker container, run:

```bash
docker run -d \
--gpus all \
-v ollama:/root/.ollama \
-p 11434:11434 \
--security-opt=no-new-privileges \
--cap-drop=ALL \
--cap-add=SYS_NICE \
--memory=8g \
--memory-swap=8g \
--cpus=4 \
--read-only \
--name ollama \
ollama/ollama
```

✅ **Verify it's running:**
```bash
docker ps
```

> **Note:** If you're using **Hyper-V**, remove `--gpus all \` since GPU passthrough is not supported.

### **▶️ Executing a Model**
Once Ollama is running, execute:
```bash
docker exec -it ollama ollama run deepseek-r1
```

This ensures the model is loaded inside the container. 🚀
### **🛠️ Explanation of the Command**

| **Flag**                      | **Purpose**                                      |
|--------------------------------|--------------------------------------------------|
| `--gpus all`                  | Enables GPU acceleration                        |
| `-v ollama:/root/.ollama`     | Creates a persistent volume for OLLAMA data     |
| `-p 11434:11434`              | Maps port 11434 for external access             |
| `--security-opt no-new-privileges` | Prevents privilege escalation          |
| `--cap-drop=ALL`              | Drops all Linux capabilities for security       |
| `--cap-add=SYS_NICE`          | Allows adjusting process priority (nice values) |
| `--memory=8g`                 | Limits memory usage to 8GB                      |
| `--memory-swap=8g`            | Prevents excessive swapping beyond 8GB          |
| `--cpus=4`                    | Restricts the container to 4 CPU cores          |
| `--read-only`                 | Makes the container **read-only** for extra security |
| `--name ollama`               | Assigns the container the name **ollama**       |
| `ollama/ollama`               | Specifies the Docker image to run               |

🔒 **Now, even if the AI model had malicious behavior, it wouldn’t be able to access the internet or your system files!**

::important
NVIDIA GPU Requirements: The flag --gpus all is meant to use NVIDIA GPUs, which requires the NVIDIA Container Toolkit to be installed on your system. This allows Docker to interface with NVIDIA GPUs for GPU-accelerated applications.
::
