## UX

Users access apps easily via the **My Apps portal**, avoiding repeated logins and managing app access conveniently.

## Plan

::caution
SSO implementation depends on whether the app is cloud-hosted (SaaS) or on-premises. Cloud apps use federation protocols, while on-premises apps can use Application Proxy for SSO.
::

When you plan your SSO deployment with your applications in Microsoft Entra ID, you need to consider the following questions:

1. What are the administrative roles required for managing the application?
2. Does the Security Assertion Markup Language (SAML) application certificate need to be  renewed?
3. Who needs to be notified of changes related to the implementation of SSO?
4. What licenses are needed to ensure effective management of the application?
5. Are shared and guest user accounts used to access the application?
6. **Do I understand the options for SSO deployment?**

### Administrative Roles

Assign the least privilege roles needed for tasks. Key roles include:

* **Help Desk Admin**: View sign-in logs (no special role needed)
* **Identity/Admin**: Configure/debug (Cloud Application Administrator)
* **Application Admin & Business Owner**: Manage user permissions (no special role)
* **Infrastructure Admin**: Manage certificates (Cloud Application Administrator)

### Certificates

* SAML apps get a default 3-year certificate; customize expiration if needed.
* Set processes to renew certificates before expiry.
* Assign owners for updates, troubleshooting, and notification distribution.

### Communications

* Inform users proactively about changes, timelines, and support options.
* Tailor communication based on how users access SSO apps.

### Licensing

* Microsoft Entra SSO for pre-integrated apps is free, but additional features or objects may require licenses.
* Ensure users have proper licenses for the apps they access, matching roles and provisioning.

### Shared Accounts

* Document users and credentials.
* Create security groups for user/credential sets.
* Reset and store shared passwords securely in Entra ID, using long, complex passwords.
* Enable automatic password rollover if supported, so admins don’t need to know the password.

### SSO Options

Microsoft Entra ID supports several SSO methods depending on application type and configuration.

![dec-single-sign-on-options-ea4bc1d9-a0b7097b](/content/7.knowledge-base/4.operating-systems/5.entra-id/1.identity/3.SSO/dec-single-sign-on-options-ea4bc1d9-a0b7097b.png)

* **OpenID Connect and OAuth:** Use if the app supports these modern protocols.
	* Apps registered via App registrations may use OpenID Connect/OAuth by default and might not show SSO options.
* **SAML:** Use for existing apps not supporting OpenID Connect or OAuth.
* **Password-based:** Use for apps with HTML sign-in pages or shared accounts; supports managing passwords and custom fields.
* **Linked:** Use to add links to apps managed by another identity provider or apps without authentication; no SSO via Microsoft Entra.
* **Disabled:** Use when the app isn’t ready for SSO configuration.
	- * Disabling SSO stops SSO via My Apps portal, but users can still sign in directly to the app unless their access is revoked.
* **Integrated Windows Authentication (IWA):** Use for apps supporting IWA or claims-aware apps.
* **Header-based:** Use when the app authenticates via headers.

## Integrate

You can connect cloud SaaS apps to Microsoft Entra ID. Check the Microsoft Entra ID Marketplace to find apps that are already set up. Use the application network portal to ask for SCIM apps (for automatic user setup) or SAML/OpenID Connect apps (for single sign-on) to be added. Once you add an app, set it up and test the sign-in.
