## Avoid Detection

**Why would we use Azure Powershell if we have Microsoft Graph?**

We previously performed **Entra enumeration** using Graph, but we didn’t get to see any resources. To view resources, we can either use the **Azure Portal** or leverage the **Az PowerShell module**.

> The Azure Az PowerShell module is a rollup module. Installing it downloads the generally available Az PowerShell modules, and makes their cmdlets available for use."

::tip
Targeted enumeration avoids detection, unlike broad tools that trigger security alerts.
::

## **Prerequisites**

Before using Az PowerShell, ensure your environment meets the following prerequisites.

### **PowerShell Version**
Run the following command to determine your PowerShell version:

```powershell
$PSVersionTable.PSVersion
```

### **Execution Policy**

Before installing or using Az PowerShell, set the PowerShell execution policy to **RemoteSigned** or a less restrictive policy.

#### **Check the Current Execution Policy**
To check your current execution policy, run:

```powershell
Get-ExecutionPolicy -List
```

#### **Set the Execution Policy to RemoteSigned**
To set the execution policy to **RemoteSigned** for the current user, run:

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

## **Sign in**

::warning
Starting 2025. Microsoft plans to start enforcing MFA on Azure Powershell sign ins.
::

Az PowerShell is a Microsoft module used to manage **Azure resources** and interact with **Entra ID (formerly Azure AD)**. Before using it, you need to connect to an **Azure tenant**.

### **Interactively**

To connect interactively, use the following command:

```powershell
Connect-AzAccount
```

### **Stored Credentials**

To connect using stored credentials:

```powershell
$creds = Get-Credential
Connect-AzAccount -Credential $creds
```

### **Plaintext Password**
To connect using a plaintext password (for **demo purposes only**):

```powershell
$passwd = ConvertTo-SecureString "Inloggen01!" -AsPlainText -Force
$creds = New-Object System.Management.Automation.PSCredential ("test@qhps.nl", $passwd)
Connect-AzAccount -Credential $creds
```
### **Disconnect**
To discconnect, use the following command:

```powershell
Disconnect-AzAccount
```

## **Discovering CMDLETS**
Azure PowerShell provides numerous commands and is a HUGE module, also called **cmdlets**, to help manage Azure resources.  Az PowerShell can enumerate both Entra ID and Azure Resources

To find all commands related to **Entra ID**, use:
```powershell

```

To find all commands related to **Azure resources**, use:
```powershell
Get-Command *Az*
```

To find commands related to **virtual machines**, use:
```powershell
Get-Command *AzVM*
```

To find **Get** commands specifically for virtual machines:
```powershell
Get-Command -Noun *VM* -Verb Get
```

To find all commands related to **Applications & Service Principals**, use:
```powershell
Get-Command *AzADApplication*
Get-Command *AzADServicePrincipal*
```

To find all commands related to **App Services & Function Apps**, use:

```powershell
Get-Command *AzStorageAccount*
```

To find all commands related to **Key Vaults**, use:
```powershell
Get-Command *AzKeyVault*
```

If the module is not installed, install it with:


## **Azure Resources**

### **Checking Current Context**

Get the information about the current context (Account, Tenant,
Subscription etc.)

```powershell
Get-AzContext
```

To list all available contexts:

```powershell
Get-AzContext -ListAvailable
```

::tip
If you gain access to a VM in Azure and a session is still active(forgot to disconnect), use **Get-AzContext** to check for available access tokens. If a recent session is active, you might be able to reuse it for further access.
::

### **Listing Subscriptions and Resources**
To enumerate all **subscriptions** accessible by the user:

```powershell
Get-AzSubscription
```

To enumerate all **resources** visible to the user:

```powershell
Get-AzResource
```

This is the CLI counterpart of [All Resources](https://portal.azure.com/#browse/all)

![all-resources](/content/7.knowledge-base/8.cyber-crusades/6.cloud/4.enumeration/2.azure-powershell/all-resources.png)
![all-resources-2](/content/7.knowledge-base/8.cyber-crusades/6.cloud/4.enumeration/2.azure-powershell/all-resources-2.png)

::note
You will get a list of all the resources the current user has at least the reader role on.
::

### **Enumerating Role Assignments**
Enumerate all the exact Azure RBAC role assignments that the user has visibilty on. Now you could visualize an attack path we are now mapping path on resources:

```powershell
Get-AzRoleAssignment
```

To view role assignments for a **specific user**:

```powershell
Get-AzRoleAssignment -SignInName test@defcorphq.onmicrosoft.com
```

::important
We can take a look at this because we have the reader role. If you have reader role on a resource, you can read the `role assignments` as well.
::

## **`Entra ID Objects`**

### **Users**
To get a **list of all users**:

```powershell
Get-AzADUser
```

To get a **specific user** by UserPrincipalName:

```powershell
Get-AzADUser -UserPrincipalName test@defcorphq.onmicrosoft.com
```

To search for users whose **DisplayName contains "admin"**:

```powershell
Get-AzADUser | ?{$_.DisplayName -match "admin"}
```

### **Groups**
To get a **list of all groups**:

```powershell
Get-AzADGroup
```

To get details of a **specific group** by Object ID:

```powershell
Get-AzADGroup -ObjectId 783a312d-0de2-4490-92e4-539b0e4ee03e
```

To search for groups **containing "admin"**:

```powershell
Get-AzADGroup | ?{$_.DisplayName -match "admin"}
```

To get **members of a group**:

```powershell
Get-AzADGroupMember -ObjectId 783a312d-0de2-4490-92e4-539b0e4ee03e
```

## **`Applications and Service Principals`**

### **Registered Applications**
To list all **registered applications**:

```powershell

Get-AzADApplication -ObjectId a1333e88-1278-41bf-8145-155a069ebed0
```

To search for applications **by display name**:

```powershell
Get-AzADApplication | ?{$_.DisplayName -match "app"}
```

## **Virtual Machines**

To list all **accessible virtual machines**:

```powershell
Get-AzVM
```

To view **detailed information** about a virtual machine:

```powershell
Get-AzVM | Format-List
```

::note
⚠️ This command only lists VMs the user has permission to read. It does **not** mean there are no other VMs in the environment.
::

To check **VM networking details**:

```powershell
(Get-AzVM -Name "BkdConnect").NetworkProfile
```

::important
A VM consists of **multiple resources** (disk, network adapter, IP), so you need permissions on each to access them and read them.
::

## **App Services**

To list all **App Services and Function Apps**:

```powershell
Get-AzWebApp | Format-List
```

This command reveals:
- **App Name** (e.g., `ProcessFile`, a Linux-based function app)
- **Managed Identity** (if the app has a system-assigned identity)
- **Resource Group** (where the app is hosted)

::warning
The **Reader** role can’t access function source code since it resides in the **data plane**, not the **control plane**. You can view metadata and configs but not the code or function keys.
::

## **Storage Accounts**

To list all **storage accounts**:

```powershell
Get-AzStorageAccount | Format-List
```

### **Network Rule Insights**
Storage accounts may have **network restrictions** that only allow connections from certain IPs. Checking network rules can reveal **trusted IPs**:

```powershell
(Get-AzStorageAccount).NetworkRuleSet
```

To expand and format the **NetworkRuleSet**:

```powershell
Get-AzStorageAccount | Select-Object -ExpandProperty NetworkRuleSet
```

To specifically list **IP rules**:

```powershell
(Get-AzStorageAccount | Select-Object -ExpandProperty NetworkRuleSet).IPrules
```

### **The Firewall**

When performing enumeration pay close attention to what are the attributes that are present here, it is crucial to examine the **NetworkRuleSet** property. This property acts as a **resource firewall**, restricting access to the storage account based on defined rules.

For example, expanding **NetworkRuleSet** may reveal rules like:
- **IP Rules:** A list of specific public IP addresses allowed to connect.
- **Virtual Network Rules:** Restrictions allowing access only from designated subnets.
- **Default Action:** Specifies whether access is denied or allowed by default.

#### **The IPs**
In a **lab environment**, you may find only a few IP addresses (e.g., the public IPs of student VMs). However, in a **production environment**, this list typically contains extensive **IP ranges**. These are the trusted IPs that:
- Are whitelisted for accessing sensitive resources.
- Might also be included in **Conditional Access** policies.
- Are likely monitored by **Entra ID Protection** to detect anomalous activity.

::warning
If an attacker gains control over a machine within one of these trusted IP ranges, they could potentially bypass **Multi-Factor Authentication (MFA)** and evade detection by **Entra ID Protection**.
::

## **Key Vaults**

Azure **Key Vaults** store sensitive credentials securely. Secrets.

To list **accessible Key Vaults**:

```powershell
Get-AzKeyVault | Format-List
```

::note
⚠️ A user with **Reader** permissions can view vaults but **cannot** read stored secrets.
::

## **Final Thoughts**
1. **Az PowerShell** provides a powerful way to interact with **Entra ID** and **Azure resources**.
2. **Targeted enumeration** avoids detection, unlike broad tools that trigger security alerts.
3. **Understanding permissions** is crucial—just because you can list a resource doesn’t mean you can modify or access it.
4. **Network insights** (like trusted IPs) help understand security postures.
5. **Service Principals, Managed Identities, and Key Vaults** are important assets to check for security risks.

By following this structured enumeration approach, you can systematically gather intelligence about an **Azure environment** without setting off alarms. 🚀
