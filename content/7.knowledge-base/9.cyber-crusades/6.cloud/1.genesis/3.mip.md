## Microsoft Identity Platform (MIP) Overview
The Microsoft Identity Platform (MIP) is a core part of Azure's authentication and authorization system. It uses **OAuth** and **OIDC** to secure access to services like **Entra ID**, **Microsoft 365**, and others.

> By the end of this file, you should have a clear understanding of the **Microsoft Identity Platform**, **OAuth** flow, the **types of tokens** used, and the potential security risks associated with them.

### Authentication Protocols
- **`OAuth 2.0`**:
   - OAuth 2.0 is primarily used for **authorization (AuthZ)**.
   - It allows third-party applications to request permission to access resources on behalf of the user, without needing to share the user’s credentials.
   - OAuth 2.0 **does not directly handle authentication**. It just grants access based on tokens.

- **`OIDC` (OpenID Connect)**:
   - OIDC is built on top of **OAuth 2.0** and adds the functionality for **authentication (AuthN)**.
   - It verifies the user's identity by adding a standardized identity layer on top of OAuth 2.0, providing information like user details (name, email, etc.) in the **ID token**.

- **`Other Protocols`**:
  - **SAML 2.0**: Primarily used for **authentication** (AuthN), especially in Single Sign-On (SSO) scenarios.
  - Legacy methods like **LDAP**, **Kerberos**, and **Header-based** protocols are used for synchronization between cloud services and on-prem systems.

## OAuth 2.0

> Four parties are generally involved in an OAuth 2.0 and OpenID Connect authentication and authorization exchange. These exchanges are often called authentication flows or auth flows.

- **Authorization Server**: The identity provider (e.g., Microsoft identity platform) that manages user authentication and issues tokens for access control.
- **Client**: The app requesting access to protected resources (e.g., web app, mobile app).
- **Resource Owner**: The end-user who owns the protected data and can grant or deny access to it.
- **Resource Server**: The server (usually an API) that hosts or provides access to the resource owner’s data.

![protocols-roles](/content/7.knowledge-base/8.cyber-crusades/6.cloud/1.genesis/3.mip/protocols-roles.svg)

### The OAuth Flow
1. **User Login**: A user enters their credentials (username/password) into a client (e.g., browser) to access a service.
2. **Token Issuance**: After verifying the credentials, the **Microsoft Identity Platform** sends back a **bearer token** to the client.
3. **Access Resources**: The client uses this bearer token to request access to resources (e.g., Graph API, Azure Resource Manager).

### **Bearer Tokens**
A **bearer token** is a security token that grants the user access to a resource. It's like a "key" that opens the door to various services.

## OAuth Flow: Sign-In

1. **Request Access**: A user navigates to an application like **`portal.azure.com`**.
2. **Redirect to Authentication**: The application redirects the user to the **Microsoft Identity Platform** for authentication.
	- Look at the redirection `/oauth2/v2.0/authorize` in the url.
	- `https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/authorize?client_id={client_id}&redirect_uri={redirect_uri}&response_type={response_type}&scope={scope}`
3. **Credential Validation**: The user enters their credentials, and if correct, a **`token`** is issued.
4. **Token Validation**: The client sends the token to the app’s backend to access the user’s resources, and the server validates the token.

## OAuth Flow: API Access

> How does Azure Portal accesses resources?- Access Token Acquisition

When a user accesses **`portal.azure.com`**, Azure follows an **OAuth-based token acquisition process** to authenticate and authorize access to various services like **`Azure Key Vault`** or **`Virtual Machines`**.

1. **User Access Request**
   - The user navigates to **portal.azure.com**.
   - The web app redirects the user to the **Microsoft Identity Platform** for authentication.

2. **User Authentication**
   - The user securely enters credentials within the Microsoft Identity Platform (not the web app).
   - If needed, the user consents to required permissions.

3. **Token Issuance**
   - The **Microsoft Identity Platform** validates credentials and issues a **JWT access token**.

4. **Token Usage & API Access**
   - The **Azure Portal** (acting as the client) requests a token from `https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token`.
   - The request includes:
     - **Client ID** (app registration in Entra ID)
     - **Client Secret** (for confidential clients)
     - **Grant Type** (depends on the authentication flow)
     - **Scope** (`https://management.azure.com/.default` for ARM API)
   - If valid, the **Azure Portal** receives a **JWT access token**.

5. **Accessing Azure Resources**
   - The Azure Portal includes this token in API requests to **Azure Resource Manager (ARM API)**.
   - **ARM validates the token** and grants or denies access to requested resources (e.g., **Key Vault, Virtual Machines**).

## Types of Tokens

> Think of them like tickets. Not only MIP, but the modern web lives on these tokens.

OAuth 2.0 and OIDC use bearer tokens which are JSON Web Tokens. A bearer token, as the name suggest, grants the bearer access to a protected resource

### 1. **Bearer/Access Token**
- Used to access resources on behalf of a user.
- **Expires** after a set time (usually 1 hour). Can go up to 24 hours for specific access tokens- `Azure instance metadata service`.
- Cannot be revoked until it expires. Meaning even if the user is deleted. You will still maintain access, unless **Continuous Access Evaluation (CAE)** is used.

### 2. **ID Token**
- Contains basic **user information** (e.g., name, email).
- Used primarily for **authentication**, not resource access.

### 3. **Refresh Token**
- Used to request new access tokens after the old ones expire.
- Can be revoked but usually has a long lifespan (e.g., 90 days of inactivity).
- If you have a user's refresh token you can keep requesting access tokens using that indefinitely.

## Token Security

### Why Access Tokens Are Important
Access tokens are powerful because they let you access resources without needing to log in again. However, they can be **misused** if stolen by attackers.

- **Access tokens** are tied to a **user**, a **client**, and a **resource**. If an attacker steals a valid token, they can access resources like the legitimate user without needing to log in.

### Security Risks
- **Token Replay**: Once an attacker gets an access token, they can use it until it expires—**bypassing conditional access policies** (like location or MFA checks).
- **Log Activity**: Accessing a resource with an access token doesn’t create a **new sign-in log**. So, the system won’t know if an attacker is using the token.

### Access Token and Sign-In Logs

::caution
When using an access token, such as for a test user, no sign-in log will be created because there is no actual sign-in event triggered by the token itself.

#### Why No Sign-In Log for Access Tokens?

- **Access Token Issuance**: When an access token is issued, it does so after a sign-in process, which is logged. But once you have the token, **no additional sign-in** occurs. Instead, the token is used to authenticate subsequent requests to resource servers.
- **OAuth Flow**: OAuth works by issuing access tokens, which have a **limited lifetime** and are used to authorize access to resources.
- **Resource Server Validation**: When the access token is sent to a resource server, **the server just validates** if the token is correct and its signature is valid. **No further authentication** happens at this point, so no sign-in event is logged.

::warning
However, if the token is used to perform actions, such as **resetting a password** or **modifying a group**, those **logs will be generated** because those are actual activities within the system.
::

::important
- **Access Token Issuance**: This is the part that gets logged because **authentication** is happening.
- Once the access token is issued, **no sign-in log** is created because the token itself validates the request without requiring another sign-in process.
::

#### Conditional Access (CA) Policies
- These policies can block access based on factors like **location** or **user status** (e.g., requiring **MFA**).
	1. **IP Filtering and Tokens**: IP-based restrictions are enforced at the authentication step. If authentication succeeds, the system doesn't check the IP when replaying the token. However, conditional access policies like **location-based restrictions** will block access during the authentication process if the IP is untrusted.
- Once a user has an **access token**, it can bypass **Conditional Access policies**. This means that even if access is blocked due to the user's location or device, the attacker can still use the token to access resources.

### CAE
- **CAE** Continuous Access Evaluation allows access tokens to be revoked before they expire if there’s a change in the user's context (e.g., location or device health).
- This helps to **prevent token abuse** by ensuring that a token is only valid as long as the user's session is secure.

## **Defensive Challenges**

OAuth’s design by itself makes access token abuse hard to detect. Microsoft’s **Continuous Access Evaluation (CAE)** helps, but tokens work without constant authentication, making them hard to monitor.

### **When Should We Step In?**

Since token use isn’t logged unless it triggers an event, detection should focus on **authentication events**—the point where identity is verified.

**Key Challenges:**
- **Token replay attacks** leave no sign-in logs.
- **Microsoft Defender for Identity (MDA)** may not detect abuse of valid tokens.
- **Access token usage isn’t logged unless linked to an auditable action.**

### **Why It Hasn’t Been Fixed?**
Detection is limited, but defenders should focus on authentication monitoring and anomaly detection rather than relying on token logs.

::warning
Fixing this would break how OAuth works. If users had to authenticate every time they accessed a resource, they wouldn’t use access tokens—they’d rely on refresh tokens instead. **This is exactly how access tokens are meant to work.**
::

## Exploit Tokens

- After obtaining a valid access token, attackers can use it to access services like **Graph API**.
- For example, a token valid for **Azure Resource Manager (ARM)** cannot be used for **Graph API** unless a new token is acquired.

::caution
Once attackers have a valid **access token**, they can **replay** it and gain access to resources. This is a **security vulnerability** because the token allows them to bypass the conditional access policies set for authentication.
::
