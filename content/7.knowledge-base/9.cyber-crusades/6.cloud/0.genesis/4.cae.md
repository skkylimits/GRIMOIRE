## **Continuous Access Evaluation (CAE)**

- CAE helps in invalidating access tokens before their expiry time (default 1 hour).
- Useful in cases like:
  - **User termination** or **password change** enforced in near real-time.
  - Blocks use of access tokens outside trusted locations when location-based conditional access is present.

::warning
In CAE sessions, the access token lifetime is increased up to **28 hours**.
::

## **CAE – Claims Challenge**

- **CAE needs the client (Browser/App/CLI) to be CAE-capable.** The client must understand that the token has not expired but cannot be used.
- Supported clients include **Outlook**, **Teams**, **Office apps (other than web)**, and **OneDrive web apps**.
- If the client is **not CAE-capable**, a normal access token with a 1-hour expiry is issued.

### `xms_cc: "CP1"`
If the **xms_cc** claim includes "CP1," it shows that the client can handle token invalidation scenarios, such as location-based policies, password resets, or critical events. This ensures the client can re-authenticate or request a new token if necessary.

- **xms_cc** is a special claim used by Microsoft to identify if a client (browser, app, CLI, etc.) can handle a claims challenge triggered by CAE.
- **"CP1"** means the client can respond to a 401 error with a claims challenge and revalidate the token using a refresh token.

Example JSON access token decoder from [jwt.io](https://jwt.io/):
```json
{
	"xms_cc": [
		"CP1"
	]
}
```

This indicates the access token is issued for a CAE-capable client that can process claims challenges.

**What happened here?**

When we specified the **MS Graph access token** and connected to the **DevCorp Organisation** from the host machine, we were able to access Azure resources, but **not Entra ID**. We encountered an error: **Claims challenge required exception**. This is a result of **CAE** (Continuous Access Evaluation).

::caution
CAE invalidates access tokens in **near real-time**, meaning even if you steal a token, it could be invalidated right away under certain conditions, without waiting for its expiry time.
::

## **CAE – Critical Event Evaluation**

CAE works in two scenarios:

1. **Critical event evaluation:**
   - User account is deleted or disabled.
   - Password change or reset for a user.
   - MFA enabled/re-registered for a user.
   - Refresh token is revoked.
   - High user risk detected by **Entra ID Protection** (not supported by SharePoint Online).
   - Only **Exchange Online**, **SharePoint Online**, and **Teams** are supported. More are supported but the official docs don't mention them.

## **CAE – Conditional Access Policy Evaluation**

2. **Conditional Access policy evaluation:**
   - Only **IP-based** (both IPv4 and IPv6) **named locations** are supported.
   - Other location conditions like **MFA trusted IPs** or **country-based locations** are not supported.
   - **Exchange Online**, **SharePoint Online**, **Teams**, and **MS Graph** are supported.

## **Important – Identifying CAE Enabled Tokens**

To identify if a token has CAE enabled, there are a few key points:
1. The client (browser, app, or CLI) must be **CAE-capable**. This means the client can understand that the token hasn't expired, but it can’t be used. For example, **AZ PowerShell** is CAE-capable and will show a **claims challenge required exception** instead of just saying the token expired.

   Previously, the error message was more specific about which claims challenge triggered, but it's now more vague. However, it’s still clear that CAE was triggered.

2. The claim **xms_cc** in the token is the authoritative way to confirm if the client application can handle a claims challenge. For example, if the **xms_cc** claim in the **MSGraph token** requested using **Az PowerShell** shows the value **cp1**, it means the client is capable of handling the claims challenge.

## **Caution – Managed Identities & Non-CAE Clients**

- **Access tokens issued for managed identities** via the Azure Instance Metadata Service (**IMDS**) are **not CAE-enabled**.
  - In scenarios where access tokens are issued to managed identities (as seen in learning objectives 10, 11, and 12), they **won’t** be CAE-enabled. These tokens can still be used for Graph API access.
  - **Recommendation**: Avoid using user identities for security reasons, especially if OPSEC is a concern.

::caution
If the client is **not CAE-capable**, it defaults to a normal access token with a 1-hour expiry. This just means the client will show an error rather than the more specific CAE-related errors.

For example, when running an API call on a **non-CAE capable client**, you’ll get a **401 Unauthorized** response without further clarification, unlike CAE-capable clients, which provide specific error information related to the claims challenge.
::

## User Revocation Event Flow

![user-revocation-event-flow](/content/7.knowledge-base/9.cyber-crusades/6.cloud/1.genesis/4.cae/user-revocation-event-flow.png)

1. A **CAE-capable client** requests an access token from Microsoft Entra by presenting its credentials or a refresh token.

2. Microsoft Entra returns the access token along with other related information to the client.

3. An **Administrator** explicitly revokes all refresh tokens for the user, triggering a revocation event sent to the resource provider from Microsoft Entra.

4. The client presents the access token to the resource provider.

5. The resource provider checks the validity of the token and looks for any revocation events for the user.

6. Since the refresh tokens were revoked, the resource provider denies access and sends a **401 error with a claims challenge** back to the client.

7. The CAE-capable client understands the **claims challenge** and bypasses the cache. It sends the **refresh token** along with the challenge back to Microsoft Entra.

8. Microsoft Entra reevaluates the conditions and prompts the user to **reauthenticate** due to the revoked refresh tokens.

## User Condition Change Flow

In the following example, a Conditional Access Administrator configured a location based Conditional Access policy to only allow access from specific IP ranges:

![user-condition-change-flow](/content/7.knowledge-base/9.cyber-crusades/6.cloud/1.genesis/4.cae/user-condition-change-flow.png)

1. A **CAE-capable client** requests an access token from Microsoft Entra by presenting its credentials or a refresh token.

2. Microsoft Entra checks the **Conditional Access policies** to ensure the user and client meet the required conditions.

3. If everything is valid, Microsoft Entra returns the access token and other related information to the client.

4. The user then moves out of an allowed IP range.

5. The client tries to access the resource provider with the access token from outside the allowed IP range.

6. The resource provider checks the validity of the token and the location policy synced from Microsoft Entra.

7. Since the client is not in the allowed IP range, the resource provider denies access and sends a **401 error with a claims challenge** back to the client.

8. The CAE-capable client understands the **claims challenge** and bypasses the cache. It sends the **refresh token** along with the challenge back to Microsoft Entra.

9. Microsoft Entra reevaluates all the conditions and denies access because the client is still not meeting the conditions.

## **Is CAE Useful?**

Yes, CAE is very useful. For example, in a **red-team operation**, if you **phish** a user, get an implant on their machine, and then try to dump the user’s access token, **you won’t be able to use it**. Why? Because if the organization has a **location-based conditional access policy**, using the token from outside the trusted locations will be blocked.

### Key scenarios CAE is useful:
1. A user is deleted or their password is changed.
2. An access token is used **outside clustered locations** (if there’s a location-based conditional access policy).

In these cases, CAE ensures tokens are invalidated promptly. Also, the token lifetime may increase up to **28 hours** during CAE sessions.

## **What If You Proxy Through the User’s Machine?**

::caution
Yes. If you proxy the request through the user’s machine, it will work, even with a simple **SOCKS5 proxy**. When the user's access token is invalidated by CAE due to location-based policies, you can bypass this restriction by proxying the request through the user's machine.
::

Here’s how it works:
- CAE invalidates tokens when used outside of trusted locations (like allowed IP ranges).
- By using a **SOCKS5 proxy** to route the request through the user’s machine (which is within the trusted range), the request appears legitimate to Microsoft Entra or the resource provider.
- As a result, the request is processed without being blocked by CAE.

This shows how proxying through the user's machine can bypass CAE's location-based restrictions.
