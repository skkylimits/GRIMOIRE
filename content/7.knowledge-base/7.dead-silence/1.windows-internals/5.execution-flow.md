|  Step                    | Description                                       |
| ------------------------ | ------------------------------------------------  |
| 1. Create Handle         | User app gets a handle to communicate with driver |
| 2. Call DeviceIoControl  | User app sends command/data via API wrapper       |
| 3. Syscall               | Switch from user mode to kernel mode              |
| 4. SSDT Lookup           | Kernel maps syscall number to function            |
| 5. I/O Manager           | Prepares IRP (request packet)                     |
| 6. Driver Processes IRP  | Driver executes the command                       |
| 7. Return Results        | Response goes back to user mode app               |



## 1 Getting a Handle to the Driver

* The user mode app needs a **handle** to talk to the driver.
* This handle is created using the **`CreateFile`** function.
* Think of this handle as a way for the app to open a communication channel with the driver’s device.


## 2 Sending Data to the Driver

* The app uses the handle to call **`DeviceIoControl`** (or similar functions).
* This is an API that sends commands or data to the driver.
* `DeviceIoControl` acts like a **wrapper**: it prepares arguments and passes them down to the kernel.


## 3 Transition from User Mode to Kernel Mode

* Inside `DeviceIoControl`, a **system call (syscall)** is made.
* This syscall switches the CPU from **user mode to kernel mode** (crossing the "red line").
* It uses a special instruction (`syscall`) to request the kernel to perform the operation.


## 4 Kernel Uses System Service Descriptor Table (SSDT)

* The syscall number stored in a register (like `RAX` or `EAX`) tells the kernel which function to run.
* The kernel looks up this number in the **System Service Descriptor Table (SSDT)**.
* The SSDT maps the syscall number to the actual **kernel function** (e.g., `NtDeviceIoControlFile`).


## 5 Inside the Kernel

* The kernel function (e.g., `NtDeviceIoControlFile`) receives the arguments from user mode.
* It passes these to the **I/O Manager**.
* The I/O Manager creates an **IRP (I/O Request Packet)** — a structured request that contains:

  * The command type
  * Target device info
  * Data buffers


## 6 The Kernel Driver Receives the IRP

* The **IRP** is sent to the driver.
* The driver reads the **IRP**, figures out what needs to be done, and performs the task (e.g., read/write, control hardware).


## 7 Returning the Result

* **The driver** completes the operation.
* Results or status are sent back through the **IRP**.
* Control goes back up to the **I/O Manager**, then t**he kernel**, and finally back to the **user mode** application.

## IDA Pro to Inspect

* Load **`kernel32.dll`**, search for `DeviceIoControl`.
* See how it calls into `ntdll.dll` and prepares the syscall.
* Then load **`ntoskrnl.exe`** (Windows kernel) to find the real implementation of `NtDeviceIoControlFile`.
* You can trace how the IRP is created and sent to the driver.
