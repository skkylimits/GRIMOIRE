## 🏭 What Is A Process?

A **process** is just a *container* for running an app.
It holds important things like memory, permissions, and threads — but doesn’t run code itself. The **threads** inside it do the actual work.

![process-container](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/process-container.png)

### 🧱 What Makes a Process?

* **Container**: Separate applications from each other.
* **PID**: Unique Process ID (used to track it, not the `.exe` name).
* **Threads**: The "workers" — these are what actually run code.
* **Handles**: References to system stuff (files, registry keys, etc).
* **Token**: Security identity (who you are + what you're allowed to do).
* **Address Space**: Each process gets its own private map of virtual memory — isolated and protected.
	* **Memory**: The actual contents within that space — what the process is actively using (no sharing!).
* **Parent/Child Relationship**: A process may have a parent, but if the parent dies, the child lives on unaffected.
* **Integrity Level**: Defines the trust level and what actions the process is allowed to perform.
* **Protection Level**: Some processes are protected from termination, injection, or tampering by unauthorized code.

::tip
*You can have 5 Chrome windows open = 5 separate processes using the same file image.*
::

::caution
If a process is destroyed, its PID might be reused.
::

### 🔄 **Execution Flow (User → Kernel)**

1. Threads start in **user mode**
2. Eventually call into **kernel mode** (e.g., when accessing hardware)
3. You can trace this with tools like **Process Hacker** (call stack)

### 🧭 Behavior

* Processes don't execute code directly — **threads inside them do**.
* Each process is **isolated**, for stability + security.
* Can have **1 or many threads**, running in parallel.
* **Parent-child relationship**: One process can spawn another.

  * But if the parent dies? The child keeps going 🚶‍♂️

::foldable

## ⚙️ Process Properties

### 🛡️ Integrity Levels

Defines *how much power* the process has:

| Level         | What can it do?                                                                                                                                          | Meaning                       |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------- |
| `Low`         | - The least trusted level.<br /> - `Can only interact with other Low processes.`<br /> - **Writes to very limited places like `%USERPROFILE%\AppData\LocalLow`. Limited registry access (`HKEY_CURRENT_USER`).**        | 🔐 Very limited access (e.g., browser sandbox) |
| `Medium`      | - Default for most user apps (Explorer, Word).<br /> - `Can interact with Low & Medium processes`.<br /> - **Can write to user folders like Documents, Desktop, and some registry keys.**      | 🧑 Standard user access                        |
| `High`        | - Running apps as Administrator.<br /> - `Can interact with Low, Medium & High processes.`<br /> - **Can write to system folders (`C:\Windows`, Program Files) and full registry access.** | ⚠️ Admin-level full privileges                 |
| `System`      | - Highest level, reserved for OS and core components.<br /> - `Can interact with all levels`<br /> - **Full access to entire file system & registry.**                                        | 👑 Kernel-level access                         |
| `Unsupported` | - Integrity level is unknown or not recognized.processes.                                                                                   | ❓ Legacy or special protected process          |

![process-integrity-levels](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/process-integrity-levels.png)

::foldable
#title
Real-world example

#content
* When you open a browser, it runs at **Low integrity** to keep it contained and safe.
* Your regular apps like Notepad or Explorer run at **Medium** — enough to do everyday tasks.
* Running PowerShell or an installer “as Administrator” runs at **High**, letting it change system files.
* Core Windows services like `lsass.exe` run at **System** level, because they need full access to keep the OS secure.
::

::tip
You’ll find the Integrity Level under the Token tab in Process Explorer.
::

::caution
**Why does it work this way?**

- Windows uses these integrity levels to **protect your system** and **keep things running smoothly**. Lower levels restrict apps to “sandboxed” environments, reducing damage if something goes wrong (like malware). Higher levels are trusted more but come with risks — if a high or system-level process is compromised, it can cause serious harm.
::

### 🔒 Protected Levels

Some Windows processes are *really* important for system security and stability. To protect them from malware or accidental interference.

**How Does Windows Implement This?**

Windows assigns each protected process a **protection level** represented by a special code made up of:

* **Signer:** Who signed the process (e.g., Windows system, antivirus vendor).
* **Protection Type:** How strong the protection is (full protected or protected light).
	- **Protected (2):** The strongest protection. Very hard to stop or modify — reserved for critical OS components like the Secure Kernel.

	- **Protected Light (1):** A lighter protection mostly for security apps like antivirus or the Local Security Authority (LSASS). It prevents tampering but allows controlled system interactions.

| Protection Level                      | Hex Value | Signer          | Protection Type     | What It Does                                                             |
| ------------------------------------- | --------- | --------------- | ------------------- | ------------------------------------------------------------------------ |
| **PS\_PROTECTED\_SYSTEM**             | 0x72      | WinSystem (7)   | Protected (2)       | Full protection for core OS parts like Secure Kernel.                    |
| **PS\_PROTECTED\_LSA\_LIGHT**         | 0x41      | LSA (4)         | Protected Light (1) | Protects Local Security Authority (LSASS), preventing credential theft.  |
| **PS\_PROTECTED\_ANTIMALWARE\_LIGHT** | 0x31      | Antimalware (3) | Protected Light (1) | Protects antivirus or antimalware apps from tampering or being disabled. |

![special-process-types](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/special-process-types.png)

::caution
**Why Does This Matter?**

- These protections are crucial because they stop malware from disabling your antivirus or stealing sensitive info (like passwords or security tokens). Without this, attackers could easily compromise your system.
::

### 🧷 Mitigation Policies

Windows uses **mitigation policies** to harden processes against common exploit techniques. These settings are applied **per process** and act as guardrails to block things like memory corruption, code injection, and remote execution.

| Mitigation Policy                               | What It Does                                                                                             |
| ----------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **DEP** *(Data Execution Prevention)*           | Blocks code from running in non-executable memory (like the stack or heap). **Stops buffer overflows.**      |
| **ASLR** *(Address Space Layout Randomization)* | Randomizes where code and data are loaded in memory each time a process runs. **Making tricks like ROP (reusing existing code) much harder** |
| **CFG** *(Control Flow Guard)*                  | Ensures code execution only follows safe paths. **Prevents memory corruption** exploits.                     |
| **Microsoft Signatures Only**                   | Only allows Microsoft-signed code to run. **Blocks unauthorized or unsigned executables.**                   |
| **Image Load Restrictions**                     | Blocks loading DLLs/executables from network shares (SMB, WebDAV). **Stops remote code execution.**          |

> Open **Process Hacker** → double-click a process → go to **General** tab → click **Mitigations Policies: Details** button to view mitigation policies.

![mitigation-policies](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/mitigation-policies.png)

::tip
**Where Are These Set?**

- These policies are tied directly to a process and can be configured by developers, the OS, or tools like Windows Defender Exploit Guard.
::

::caution
**Why They Matter:**

- Mitigations add layers of defense to slow down or completely stop attackers — even if they find a vulnerability.
::

## 🌆 What Are Components?

> The Building Blocks

Think of a process like a tiny city inside your computer:
- **Virtual memory** is the city map
- **Threads** are the workers
- **Handles** are the tools and keys
- **Tokens** are the ID cards


::caution
**Why does this all exist?**

* **Memory Management:** Virtual memory gives each process its own safe space to work, preventing crashes and improving stability.
* **Concurrency & Efficiency:** Threads allow programs to do many things at once, improving speed and responsiveness.
* **Security & Isolation:** Handles and tokens keep processes from stepping on each other’s toes or accessing things they shouldn’t.
::

::foldable
#title
Example Time! 🚦

#content
Imagine you open your web browser:

* The **process** is the city.
* The **main thread** loads the homepage.
* Other **threads** fetch images, play videos, and listen for your clicks simultaneously.
* The browser uses **handles** to read files, network sockets, or registry settings.
* Its **token** ensures it doesn’t get admin powers — protecting your system.
* The browser’s **virtual memory** holds everything from code to downloaded images in neatly organized pages, managing what’s in RAM or on disk.
::

### **🗺️ Virtual Memory**

> A Giant City Map of Process Memory

Every process on Windows thinks it has access to a massive, private memory space — up to **128 TB** on 64-bit systems. This **virtual memory** starts at address
`0x00000000 00000000`
and goes up to
`0x0007FFF FFFFFFFF`.

::foldable
#title
Virtual Address Space

#content
Virtual memory needs physical storage somewhere, but it doesn’t mean all of that 128 TB is actually backed by real RAM at once.

Here’s how it works:

- The virtual address space is like a huge imaginary map (128 TB).
- Only parts of that map are actively used at a time.
- When a program accesses a part of the virtual memory, the system maps that to:
	- Physical RAM (fast, limited space)
	- Or Pagefile (disk space) — slower but much larger
	- If a page isn’t used, it might not be backed by anything physical — just reserved or free.

So the system tricks the program into thinking it has a gigantic memory space, while juggling actual physical resources smartly behind the scenes. That way, programs run smoothly without needing 128 TB of real RAM!

![virtual-address-space](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/virtual-address-space.png)
::

It’s made up of **pages** — small blocks that hold all the data a process needs, organized and protected to keep everything secure and efficient.

![virtual-memory](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/virtual-memory.png)

::note
**Process Hacker**

- Each memory entry represents a region made up of multiple pages grouped together, showing their combined state, type, size, and protection.

- Type and State are shown together because each memory page’s usage (type) depends on whether it’s currently reserved, free, or actively committed—so both details are needed to understand what that page is and how it’s being used.
::

**Page States: How Memory is Allocated**

> Each page can be in one of **three states**

* **Free** – Empty land, nothing there yet
* **Reserved** – Planned for future use
* **Committed** – Actively in use, backed by RAM or pagefile

**Page Types: Where Pages Come From**

* **Private** – Used only by this process (e.g., heap, stack)
* **Mapped** – Linked to files on disk (e.g., memory-mapped files)
* **Image** – From executables and libraries (EXEs, DLLs)

**Page Uses: What the Memory Is For**

> Pages serve different roles in the process

* 📅 **Shared system info** (USER\_SHARED\_DATA, e.g., system time)
* 📦 **Process data** (Process Environment Block - PEB, Thread Environment Block - TEB)
* 🧵 **Thread stacks** (local variables, function calls)
* 🧱 **Heaps** (dynamic memory allocations)
* 🎞️ **NLS & MUI files** (language and UI data files)
* 🗂️ **Allocated pages** (dynamically allocated memory)
* 🖼️ **Executable file image** (the main EXE loaded in memory)
* 📚 **Modules** (loaded DLLs and libraries)

**Page Protections: Who Can Do What**

> Each page has a specific access type, controlling how it can be used:

| Access Type | Meaning                                    |
| ----------- | ------------------------------------------ |
| R           | Read                                       |
| RW          | Read and Write                             |
| X           | Execute                                    |
| RX          | Read and Execute                           |
| RWX         | Read, Write, and Execute (rare and risky)  |
| No Access   | Totally off-limits                         |
| Guard       | Special trap to detect overflows or errors |

> "Pages are allocated → come from somewhere → serve a purpose → and have rules on how they can be accessed."

### **🧑🏽‍💻 Threads**

> The Workers

::caution
**What are they?**

- Threads are like workers in the city. They do the actual work — executing the code.
::

::important
**Why multiple?**

- You can have many workers working at once, doing different tasks. It’s how your program multitasks.
::

![threads](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/threads.png)

**Types:**

* **Main thread:** The city mayor, launching the main program.
* **Threads created by code:** Specialized workers hired for specific tasks (e.g., via CreateThread)
* **Worker threads:** Helpers doing chores like cleanup and resource management.

**States of workers:**

* **Waiting** (on break 🛑): Paused, waiting for an event.
* **Ready** (on standby 🎯): Ready to work but waiting for CPU time.
* **Running** (actively working 🔥): Currently executing instructions.

**Access Mode:**

* **User Mode:** The thread operates with **limited privileges**, interacting with user-space memory and executing user code. The thread
uses the **user mode stack** during this mode
* **Kernel Mode:** The thread operates with **elevated privileges**, allowing direct interaction with hardware and system resources.
When a thread switches from user mode to kernel mode (e.g., during a system call), it begins using the **kernel mode stack**

**Thread stacks:**

> **Thread Call Stack:** A list tracking the chain of function calls leading to the current instruction.

- **User Mode Stack:**

  * Lives in user space.
  * Dynamic in size.
  * Stores local variables, function params, return addresses during user execution. Like your desk space.

	> 🧠 Seen as normal app/system DLLs (`cmd.exe`, `ucrtbase.dll`, etc.) in call stacks.

- **Kernel Mode Stack:**

  * Lives in kernel space.
  * Small and fixed (12 KB on 32-bit, 24 KB on 64-bit).
  * Used during syscalls, interrupts, etc.

	> 🧠 You *enter* it via calls like `ntdll.dll!ZwXxx`, though it’s not directly shown in tools like Process Hacker.

### **🔑 Handles**

> The Keys to the City 🔑

::caution
**Why handles exist:**

- Processes don’t get to touch everything in the system directly — they need **handles**, which are like keys to access resources safely.
::

![handle](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/handle.png)

Each Handle Has:

* **Handles** are `unique value`. These keys that point to resources like files, events, or registry entries.
* **Type**: What object it refers to (e.g., file, thread, registry key).
* **Name**: Optional, but helpful for identifying well-known or named resources.
* **Granted Access**: Defines what actions are allowed:
	* Read/write/execute, create/delete, synchronize, etc.

::important
**Each process keeps its own handle table — its own keyring — so processes can’t mess with each other’s stuff.**
::

### **🆔 Tokens**

> The ID Cards

::caution
**Why tokens exist:**

- Tokens are like security badges for processes — they define who you are and what you’re allowed to do, enforcing identity, permissions, and integrity.
::


Tokens are like **ID cards** for a process — but smarter. They hold:

* **User** – Who owns the process (e.g., `skkylimits`).
* **Groups** – What teams you're part of (like `Administrators`, `Users`, etc).
* **Privileges** – Special powers granted (like `SeDebugPrivilege` to debug other processes).
* **Integrity Level** – Trust level (Low, Medium, High) that restricts what the process can touch.

::caution
`SeDebugPrivilege` – If **enabled**, lets the process **inspect and control** other processes, even those it doesn't own.
::

![token-kernel](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/token-kernel.png)

::important
Many privileges are **disabled by default** until the process **explicitly enables** them.
::

![token-user](/content/7.knowledge-base/7.dead-silence/1.windows-internals/1.user-mode/token-user.png)
