## Main Goals

::caution
The kernel is a core piece of code running in memory with the highest privileges on the system.
::

It acts as a **bridge** between _hardware_ and _user applications_, ensuring system stability, security, and efficient resource management.

- **Process Management:**
	- Controls their execution and resource allocation.
	- Ensure each process has its own separate memory spaces to prevent interference.

- **Privilege Management:**
	- Manages and enforces privileges between processes to ensure that no process exceeds its authorized permissions.

- **Security Enforcement:**
	- It establishes security barriers between processes to ensure they cannot access each other's memory or resources without explicit permission.

On x64 Windows systems, the kernel operates within a 128 TB system address space, ranging from:
  `0xFFFF8000 00000000` to `0xFFFFFFFF FFFFFFFF`.

The kernel tracks each process’s state using the `_EPROCESS` structure, which holds key information about the process.

![rings](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/rings.png)

## The \_EPROCESS Structure

The kernel tracks the state of each process using the `_EPROCESS` structure, a kernel memory data structure that describes processes and contains critical information described below ⬇️

::tip
**This structure evolves across different Windows versions as features are added or modified.**

- The [Vergilius project](https://www.vergiliusproject.com/kernels/x64) is a useful resource for comparing `_EPROCESS` structures across versions.
::

You can inspect the `_EPROCESS` of a process in [WinDBg](https://learn.microsoft.com/en-us/windows-hardware/drivers/debugger/) with commands like :

* To get the process’s `_EPROCESS` address:

  ```
  !process 0 0 explorer.exe
  ```
* To display fields within `_EPROCESS`:

  ```
  dt nt!_eprocess <EPROCESS_addr>
  ```

**Key fields**:

* `UniqueProcessId` (PID) — The process identifier.
* `ImageFileName` — The executable’s name, useful for process comparison.
* `Token` — Security token, relevant for privilege escalation.
* `Protection` — Process protection level, important for bypassing Protected Process Light (PPL).
* `ActiveProcessLinks` — Links to enumerate all active processes.
* `InheritedFromUniqueProcessId` (PPID) — The parent process ID.
* `Peb` — Process environment block.
* `ObjectTable` — Stores handles to kernel objects used by the process.
* `SectionBaseAddress` — Base address of the executable image in memory.
* `ThreadListHead` — Linked list head to iterate over threads.

## Active Process Links

In Windows, every process is internally represented by a structure called `EPROCESS`.

One of the key fields in this structure is called **`ActiveProcessLinks`**, which is part of a **doubly linked list**.

::caution
This list connects **every single running process** in the system together, allowing tools (like Task Manager, Process Explorer, etc.) to enumerate all running processes.
::
### 🔗 How Are Processes Linked Together?

Each process has its own `ActiveProcessLinks` field, which is a structure called `LIST_ENTRY`. This contains two pointers:

* `Flink` (Forward link): Points to the next process's `ActiveProcessLinks`
* `Blink` (Backward link): Points to the previous process's `ActiveProcessLinks`

The list is **circular**, meaning the last process links back to the first one.

Visualized simply:

```
[Process1] <-> [Process2] <-> [Process3] <-> ... <-> [Process1]  (and so on)
```

This makes it easy to loop through all running processes in memory.

![ActiveProcessLink](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/ActiveProcessLink.png)

### 😈 How Can a Process Be Hidden?

::caution
This technique is **used by rootkits**, malware, and advanced persistent threats to stay hidden.

* It **bypasses security monitoring** unless the tool directly parses memory or uses kernel callbacks.
::

Some advanced malware or rootkits use a technique called **ActiveProcessLinks unlinking** to hide themselves from monitoring tools.

The idea is simple:

> 🔥 **If you remove a process's `ActiveProcessLinks` from the list, it won't show up in task managers or scanners** — even though it's still running.

#### 🧪 Example: Hiding cmd.exe

::tip
To hide a process🕵️

1. Set the **previous process's `Flink`** to point to the **next process**.
2. Set the **next process's `Blink`** to point to the **previous process**.
3. (Optional) Set the hidden process's `Flink` and `Blink` to `NULL`.
::

To **hide** `cmd.exe`, we do:

1. Change `Process1.Flink` to point to `Process3`
2. Change `Process3.Blink` to point to `Process1`
3. Optionally, set `cmd.exe.Flink` and `cmd.exe.Blink` to `NULL`

Now `cmd.exe` is **unlinked** from the process list. It keeps running in memory, but **disappears from view** for system tools that rely on `ActiveProcessLinks`.

![process-hiding](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/process-hiding.png)

| Concept              | Description                                             |
| -------------------- | ------------------------------------------------------- |
| `ActiveProcessLinks` | Doubly linked list field in every Windows process       |
| Purpose              | Links all processes together for enumeration            |
| Structure            | Each has `Flink` (next) and `Blink` (prev)              |
| Hiding Method        | Unlink the target process from the list                 |
| Result               | Process still runs, but becomes invisible to many tools |

### 🛡️ How to Detect or Defend?

* Use **memory forensic tools** like Volatility or Rekall that can spot discrepancies in memory structures.
* Some **EDRs (Endpoint Detection & Response)** tools hook kernel functions to catch this.
* Regular integrity checking and baselining of system internals helps.

## Token

::caution
Gain SYSTEM access to perform high-priv operations (common in malware or red team tools).
::

The `Token` field in the **EPROCESS** structure defines a process’s **security context** of type `_EX_FAST_REF`.

* It includes:
  * **SID** (Security Identifier)
  * **User privileges**
  * **Group memberships**
* Determines **what the process can do** and **what it can access**.
* Located at **offset `0x4B8`** (may vary by Windows version).
* The token is a **pointer to the access token object** in kernel memory.

**The Value field holds the token value.**

![token](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/token.png)

### 🔼 Privilege Escalation

**Stealing SYSTEM Token**

**Prerequisities:** Be abe to use [WinDBg](http://localhost:3000/knowledge-base/dead-silence/windows-internals/kernel-mode#windbg) on target machine.

1. Find the **EPROCESS** address of the **System process** `!process 0 0 system`
2. Find the **EPROCESS** address of the **target (low-priv) process** `!process 0 0 <TARGET_PROCESS_NAME>`
3. Find the offset of the Token field within **EPROCESS**: `dt nt!_eprocess Token`
4. Read the **Token** from the **System process**:
	- `dt _EX_FAST_REF <SYSTEM_EPROCESS><TOKEN_OFFSET>`
5. Read the **Current Token** from the **Target process**:
	- `dt _EX_FAST_REF <Target_EPROCESS><TOKEN_OFFSET>`
6. Write that **Token** value into the target process's token field
	- `eq <Target_EPROCESS><TOKEN_OFFSET> <SYSTEM_TOKEN_VALUE>`
7. Verify the **Token** was updated: `dt _EX_FAST_REF <Target_EPROCESS><TOKEN_OFFSET>`
8. The target process now runs as `NT AUTHORITY\SYSTEM`

✅ **Result:** The process has full SYSTEM privileges.

## Process Protection

**Protected Process Light (PPL)**: A kernel-enforced protection level that restricts access to critical processes like `lsass.exe`.

* **Purpose**: Prevent memory dumps, termination, and code injection — especially to defend against tools like Mimikatz.
* **Location**: PPL value is stored at **offset `0x6B78`** (or `0x6878`, version-dependent) in the **EPROCESS** structure.
* **Structure**:

  * High (3 bits) → Protection level (e.g., Light, Full)
  * Audit (1 bit) → Reserved
  * Signer (4 bits) → Who signed it (e.g., Windows, AntiMalware)

Example value: `0x41` → `PPL` signed by `WinTcb` (LSA Light)

::caution
**Why it Matters**

* `lsass.exe` runs as **PPL** (`WinTcb-Lsa-Light`)
* Dumping it (e.g., via Task Manager or Mimikatz) shows:
  ❌ *“Access is denied”*
::

### 🧨 Bypassing PPL on `lsass.exe`

**Remove PPL from `lsass.exe` so its memory can be accessed**

1. 🧭 Get the **EPROCESS** address of `lsass.exe`
2. 🔍 Read the **protection value** at offset `0x6B78`
   → Expected: `0x41` (`PPL`, `WinTcb-Lsa-Light`)
3. ✂️ Overwrite this field with `0x00` (no protection)
4. 🔁 Re-read the field to confirm it's cleared

✅ `lsass.exe` is now **no longer protected**
✅ Memory can be dumped via Task Manager or Mimikatz
✅ Credentials and NTLM hashes can be extracted

## Offset

Different versions of Windows store process structure fields at different offsets within the EPROCESS structure.

::caution
Using offsets from **Windows 8** on a **Windows 11** system (or vice versa) can cause:

* Invalid memory access
* **System instability**
* **Blue Screen of Death (BSOD)**

Always ensure you're using the **correct offsets for the specific Windows version** you're working with — especially when accessing or modifying low-level kernel structures like `EPROCESS`.

::

🆔 **Unique Process ID (PID) Offset**

* **Windows 11:** `0x180`
* **Windows 8:** `0x2E0`

🔗 **ActiveProcessLinks Offset**

* **Windows 11:** `0x1F0`
* **Windows 8:** `0x2E8`

![offset-UniqueProcessId](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/offset-UniqueProcessId.png)

🗝️ **Token Offset**

* **Windows 11:** `0x4B8`
* **Windows 8:** `0x348`

![offset-Token](/content/7.knowledge-base/7.dead-silence/1.windows-internals/2.kernel-mode/offset-Token.png)

## WinDBg

1. **Install WinDbg on the Host Machine**

2. **Configure the Target Machine for Kernel Debugging over Network**

   Run these commands **as administrator** on the target machine, then **restart** it:

   ```bash
   bcdedit /debug on
   bcdedit /dbgsettings net hostip:<Host-IP> port:50000 key:1.2.3.4
   ```

3. **Launch WinDbg on the Host Machine**

   * Run `WinDbg` as Administrator.
   * Go to **File → Attach to Kernel**.
   * Enter the following settings:

     * **Port:** 50000
     * **Key:** 1.2.3.4

4. **Start Debugging**

   * Press **Break** in WinDbg to pause the target machine and begin entering commands.
