## Prerequisites

1. **Azure AD and Intune**
   - Azure AD Premium P1/P2 for dynamic groups and hybrid join.
   - Microsoft Intune for device management.
   - Admin permissions to configure Autopilot.

2. **Device Requirements**
   - Windows 10/11 Pro, Enterprise, or Education (1709+).
   - Devices registered with Autopilot via hardware hash (ZTDID).
   - Internet access during OOBE.

3. **Autopilot Device Registration**
   - Import device hashes into Endpoint Manager.
   - Verify devices appear under **Windows Autopilot Devices**.
   - Use dynamic device groups for management.

4. **Deployment Profiles**
   - Configure Autopilot profiles (Self-Deploying, User-Driven, Pre-Provisioning).
   - Assign profiles to dynamic device groups.
   - Customize OOBE settings for user experience.

5. **Licensing**
   - Windows 10/11 Enterprise E3/E5 or Microsoft 365 Business Premium/Enterprise.

6. **Network and Connectivity**
   - Allow access to required Microsoft URLs (e.g., `*.microsoftonline.com`).
   - Configure Hybrid Azure AD Join for on-premises AD environments (if needed).

7. **Optional (Recommended)**
   - Deploy apps/configurations via Intune.
   - Use White-Glove deployment for pre-configuration.
   - Ensure devices receive updates via Windows Update.

8. **Testing**
   - Test deployment on a subset of devices.
   - Verify enrollment, application deployment, and user experience.

## **Windows Autopilot Lifecycle: Understanding the Phases of Autopilot Deployment**

### 1. **OOBE / Initial Device Setup:**
   - The **Autopilot deployment profile** is applied during OOBE.
   - Device joins **Azure AD** (Microsoft Entra ID).
   - **MDM enrollment** into Intune begins.

   > **Note:** At this stage, the device is registered but not fully compliant yet. No Conditional Access enforcement occurs until the device is evaluated against compliance policies.

### 2. **MDM Enrollment into Intune:**
   - The device syncs with Intune and begins applying:
     - **Security Baselines**: Applied first to ensure key security settings (e.g., BitLocker encryption, Windows Defender, Windows Update policies).
     - **Compliance Policies**: Intune evaluates compliance (e.g., encryption, OS version, security patch levels).
     - **Configuration Policies**: Settings like Wi-Fi, VPN, and device restrictions are applied to ensure the device is ready for corporate use.

   > **Key Point:** **Security Baselines** and **Compliance Policies** are applied first to secure and validate the device, while **Configuration Policies** are applied afterward to ensure the device functions correctly in the corporate environment.

   > **Note:** Compliance evaluation and Conditional Access enforcement happen after this step. If a device is non-compliant, it will be blocked from accessing resources.

### 3. **App and Configuration Deployment:**
   - Required apps (e.g., Win32, MSI, Microsoft Store apps, LOB) are installed.
   - **Configuration Policies** push settings (e.g., Wi-Fi, VPN, restrictions) to the device.
   - These steps often occur alongside compliance checks but don't block Conditional Access unless explicitly tied to a compliance policy.

### 4. **Scripts and Remediations:**
   - PowerShell scripts and remediation policies execute to apply additional configurations or fix compliance issues.
   - This step happens concurrently with app deployments or later, depending on when scripts are assigned.

### 5. **Post-Deployment Updates and Monitoring:**
   - Compliance is continually enforced, and Conditional Access remains active.
   - Devices receive regular updates (Windows updates, feature upgrades).
   - Non-compliance triggers remediation or prevents access to resources as defined by Conditional Access policies.
## Dynamic Device Groups

### **Create a Group in Azure AD**

1. **Sign In to Microsoft Entra ID**:
   - Open a web browser and navigate to:
     ```
     https://entra.microsoft.com
     ```
   - Sign in with your **Admin@yourtenant.onmicrosoft.com** account and the default tenant password if prompted.

2. **Navigate to Groups**:
   - In the left navigation pane, expand **Identity**.
   - Under **Groups**, select **All groups**.

3. **Create a New Group**:
   - In the **Groups | All groups** blade, click **New group**.
   - Configure the following settings:
     - **Group type**: Select **Security**.
     - **Group name**: Enter **WINDOWS AUTOPILOT DEVICES**.
     - **Group description**: Enter **Devices enrolled in Autopilot for automated deployment and configuration. This group is used for testing and managing Autopilot profiles, ensuring seamless device provisioning and setup**.
     - **Membership type**: Select **Dynamic Device**.

4. **Add a Dynamic Membership Rule**:
   - Click **Add dynamic query**.
   - On the **Dynamic membership rules** blade, select **Edit** above the rule syntax box.
   - Enter the following rule in the syntax editor:
     ```
     (device.devicePhysicalIDs -any (_ -contains "[ZTDId]"))
     ```
   - Click **OK** to save the rule.

5. **Save and Create**:
   - Select **Save** to close the **Dynamic membership rules** window.
   - Click **Create** to finalize the group.

## Deployment Profiles

For standardizing **Autopilot Deployment Profile** names throughout your organization, it’s important to use clear, descriptive, and consistent naming conventions that can easily identify the profile’s purpose, target audience, and configuration settings. Here's a good naming structure to follow:

### **Recommended Naming Structure**:
```
[Type] - [Use Case] - [Region/Department]
```

#### **Elements**
1. **Type**: This defines the type of deployment (e.g., **User-Driven**, **Self-Deploying**, **Pre-Provisioned**).
2. **Use Case**: A brief description of the purpose of the profile (e.g., **Standard** for common profiles, **Test** for testing profiles, or specific use case like **Corporate**).
3. **Region/Department** (optional): The department or region it is intended for (e.g., **IT**, **Sales**, **US**, **EMEA**).

#### **Examples**
- **User-Driven - Standard - IT**
  (For standard Autopilot profiles for the IT department, version 1)

- **Self-Deploying - Corporate - US**
  (For self-deploying profiles for corporate use in the US, version 1)

- **Pre-Provisioned - Test - IT**
  (For test profiles in the IT department)

- **User-Driven - Corporate - EMEA**
  (For corporate users in the EMEA region, version 2)

- **Self-Deploying - Sales - US**
  (For self-deploying profiles used by the Sales department in the US)

### Create Deployment Profile

1. In the **Windows enrollment** blade, select **Deployment Profiles**.

2. On the **Windows Autopilot deployment profiles** blade, click **Create profile** and choose **Windows PC**.

3. In the **Basics** tab:
   - **Name**: Enter **[Type] - [Use Case] - [Region/Department]**.
   - **Description**: Enter **White Glove: IT or a partner gets everything set up first, so the user just gets a fully ready device**.
   - For **Convert all targeted devices to Autopilot**, select **No**.

4. In the **Out-of-box experience (OOBE)** tab:
   - Set **Deployment mode** to **Self-Deploying**.
   - Set **Join to Microsoft Entra ID as** to **Microsoft Entra joined**.
   - Set **Language (Region)** to **Dutch (Netherlands)**.
   - Set **Automatically configure keyboard** to **Yes**.
   - Ensure the following options are set:
     - **Microsoft Software License Terms**: Hide
     - **Privacy settings**: Hide
     - **Hide change account options**: Hide
     - **User account type**: Standard
     - **Allow pre-provisioned deployment**: Yes
     - **Apply device name template**: Yes
   - For **Enter a name**, use **NO**.

5. Click **Next**.

6. In the **Assignments** tab:
   - Under **Included groups**, select **WINDOWS AUTOPILOT DEVICES**.
   - Under **Excluded groups**, leave it as **No Excluded groups**.

7. Click **Next**.

8. On the **Review + create** tab, review the information and select **Create**.

## Device Hashes

### **Manual Import: Export Hardware Hash Using Audit Mode**

1. **Enter Audit Mode**:
   - During OOBE, press `CTRL + SHIFT + F3`.
   - The system will restart and boot into **Audit Mode**.

2. **Open Settings**:
   - Once in Audit Mode, open **Settings**.

3. **Navigate to Accounts**:
   - Go to **Accounts** > **Access work or school**.

4. **Export Logs**:
   - Select the option to **Export Logs**.
   - This will generate an **MDMDiagnostics_XXXXXXXXXXXXX.zip** file and save it to the following location:
     ```
     C:\Users\Public\Documents
     ```

5. **Locate and Extract the File**:
   - Open **File Explorer** and navigate to:
     ```
     C:\Users\Public\Documents
     ```
   - Locate the file named **MDMDiagnostics_XXXXXXXXXXXXX.zip** (the name includes a timestamp).
   - Right-click the ZIP file and select **Extract All**.

6. **Find the Hardware Hash File**:
   - In the extracted folder, locate the file named:
     ```
     Device_ComputerName.csv
     ```
   - This file contains the device's hardware hash.

7. **Save to USB Drive**:
   - Insert a USB drive into the device.
   - Copy the exported log file to the USB drive for transfer.

8. **Import to Intune**:
   - On your admin machine, upload the file to Intune:
     - Go to **Microsoft Endpoint Manager Admin Center** > **Devices** > **Windows Enrollment** > **Devices** > **Import**.
     - Select and upload the exported file.

To **exit Audit Mode**, you cannot simply restart the device—you need to use the **System Preparation Tool (Sysprep)** to properly reset the device and return it to OOBE. Here's how:

9. **Open the Sysprep Tool**:
   - If the **System Preparation Tool** does not appear automatically, launch it manually:
     - Press `Win + R` to open Run.
     - Type `sysprep` and press **Enter**.

10. **Configure Sysprep Settings**:
   - In the **Sysprep** window:
     - Set **System Cleanup Action** to **Enter System Out-of-Box Experience (OOBE)**.
     - (Optional) Check the box for **Generalize** to remove unique system identifiers (recommended for preparing the device for deployment).
     - Set **Shutdown Options** to **Restart** or **Shutdown**.

11. **Execute Sysprep**:
   - Click **OK** to apply the settings.
   - The device will restart or shut down and re-enter OOBE.

#### **Why Use Sysprep?**
- Sysprep ensures the device is prepared correctly for deployment.
- It resets the device to a clean OOBE state without leaving any traces of Audit Mode.

Simply restarting the device **without Sysprep** will keep the system in Audit Mode, which is not ideal for deployment scenarios. Let me know if you need further clarification!

#### **Benefits of Audit Mode**:
- Enables hardware hash export without completing OOBE.
- Keeps the device ready for Autopilot deployment.

### **Powershell: Export and Save the Autopilot Hash During OOBE**

You can save the hardware hash during the **OOBE phase** without having to fully set up the device. Here’s how to transfer the `DEVICE_HASH.csv` file onto a USB drive directly from the OOBE environment:

1. **Insert a USB Drive:**
   - During the OOBE phase, before running the PowerShell commands, insert a USB drive into the laptop.

2. **Run PowerShell Commands:**
   - Press `SHIFT + F10` to open a command prompt.
   - Run PowerShell using the command:
     ```cmd
     powershell
     ```
   - Execute the following commands:
     ```powershell
     Install-Script -Name Get-WindowsAutoPilotInfo
     Set-ExecutionPolicy -ExecutionPolicy RemoteSigned
     Get-WindowsAutoPilotInfo.ps1 -OutputFile C:\DEVICE_HASH.csv
     ```
   - This generates the `DEVICE_HASH.csv` file in the `C:\` drive.
   - `type C:\Computer.csv` to verify the file contents.

3. **Copy the File to the USB Drive:**
   - Identify the drive letter of the USB (e.g., `D:\`, `E:\`, etc.):
     ```cmd
     wmic logicaldisk get name
     ```
   - Copy the file to the USB drive:
     ```cmd
     copy C:\DEVICE_HASH.csv D:\
     ```
     Replace `D:\` with your USB drive's letter.

4. **Verify the File on the USB Drive:**
   - List the files on the USB drive to ensure it was copied successfully:
     ```cmd
     dir D:\
     ```

5. **Eject the USB Drive:**
   - After verifying the file, remove the USB safely by typing:
     ```cmd
     powershell (New-Object -ComObject Shell.Application).Namespace(17).ParseName("D:").InvokeVerb("Eject")
     ```
   - Physically remove the USB.

6. **Import to Intune:**
   - Insert the USB into your admin machine, then upload the `DEVICE_HASH.csv` to Intune:
     - Go to **Microsoft Endpoint Manager Admin Center** > **Devices** > **Windows Enrollment** > **Devices** > **Import**.
     - Upload the CSV file.

## User Assignment

### **Assigning a User in Intune for Autopilot Devices**

1. **Navigate to Intune:**
   - In the **Microsoft Entra admin center** (or **Microsoft Endpoint Manager Admin Center**), go to **Devices**.

2. **Find the Device:**
   - Under **Devices** > **Windows** > **Windows Autopilot** > **Devices**.
   - In the **Devices** list, find the device you want to assign a user to (you can search by **device serial number**).

3. **Select the Device:**
   - Check the box next to the **serial number** of the device you want to assign a user to.

4. **Assign the User:**
   - At the top of the **Windows Autopilot devices** blade, click **Assign user**.
   - A **user selection dialog** will appear. From the list of users, select the **user** you want to assign to the device and click **Select**.

5. **Verify the User Assignment:**
   - After assigning the user, click on the **serial number** of the device in the list.
   - In the device details pane, you should see the **assigned user** listed, and under the **Profile** section, you'll also see the deployment profile assigned to the device.

---

### **What Happens When a User Is Assigned vs. Not Assigned During Hardware Hash Import**

#### 1. **Assigning a User during Hardware Hash Import:**
When you assign a user to the device during the **hardware hash import** process:

- **Pre-enrollment User Association**: The device is **directly associated** with a **specific user** before the OOBE process.
- **User-Specific Configuration**: The device will apply the user-specific configurations and apps automatically when the device starts OOBE.
- **Seamless OOBE Experience**: The user will have a smooth experience where their settings and apps are already configured, and they can sign in with their credentials without additional setup.

#### 2. **Leaving the User Field Blank During Hardware Hash Import:**
When you leave the **user field blank** during hardware hash import:

- **No Pre-enrollment User Association**: The device is not tied to a specific user initially.
- **OOBE User Selection**: The user will be prompted to **sign in** during OOBE to associate their **Microsoft Entra ID (Azure AD)** credentials.
- **Delayed Profile Assignment**: After the user signs in, their assigned profile and user-specific settings or apps will be applied.

### Key Differences

| **Aspect**                          | **Assigned User During Import** | **No User Assigned**                          |
|-------------------------------------|---------------------------------|----------------------------------------------|
| **User Association**                | Pre-assigned during hash import | Assigned during OOBE when the user signs in |
| **User-Specific Settings**          | Automatically applied during OOBE | Applied after user signs in during OOBE      |
| **OOBE Experience**                 | Seamless, with user already assigned | User needs to sign in during OOBE            |
| **Intune Profile Handling**         | Directly associated with the user’s profile | Assigned after user signs in                |
| **Deployment Flow**                 | More streamlined for the user   | Takes longer because of delayed association |

### **Summary**

- **Assigning the user during hash import**: The device will be pre-associated with the user, so when OOBE starts, the user can sign in immediately, and their settings will be applied seamlessly.
- **Leaving the user blank**: The device will wait until the user signs in during OOBE to be associated, and only then will the user-specific configurations and apps be applied.

## Start Windows Autopilot

### Option 1: Start Autopilot Using the Windows Button

1. Press the **Windows key 6 times** to trigger the **Autopilot setup** process.

2. This will automatically begin the **Autopilot configuration** and reset the device.

   > **Note:** This option will also initiate the same reset process but directly triggers the Autopilot OOBE without manual navigation through Settings.

### Option 2: Using "Reset this PC" from Settings

1. On **SEA-WS3**, select **Start** and type **reset**, then select **Reset this PC** from the search results.

2. In the **System > Recovery** page, select **Reset PC**.

3. Choose **Remove everything** and then select **Local reinstall**. Make sure it's a full wipe of the drive and files.

4. Click **Next**, then select **Reset**.

   > **Note:** Normally, a reset is not required for a new device deployment. Autopilot information is typically provided by the manufacturer or can be obtained before the Out-of-Box Experience (OOBE). However, for this lab, a reset is necessary to simulate a new device OOBE.

   > **Note:** This process may take **30-45 minutes**, and the system will restart several times during the procedure.
