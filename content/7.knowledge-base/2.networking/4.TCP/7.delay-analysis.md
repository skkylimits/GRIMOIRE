## ⏱️ Application vs Network Delay

Dit is **de meest waardevolle skill** die TCP-analysts onderscheiden van amateurs. Als je dit beheerst, ben je **boven 90% van netwerkengineers**.

::important
**Het probleem:** Iedereen wijst naar "het netwerk" bij performance-problemen. Maar meestal is het de applicatie, niet het netwerk.
::

## 🎯 Het Fundamentele Principe

### De Waarheid

```
Application delay ≠ Network delay ≠ TCP delay

Deze zijn 3 verschillende dingen!
```

**Wat gebeurt er echt:**

```
CLIENT                     NETWORK                    SERVER
  │                                                      │
  │────── Request (SEQ=1, Len=1793) ──────────────────> │
  │                                                      │
  │                    130ms (RTT)                       │
  │                                                      │
  │ <──────────── ACK=1794 ───────────────────────────  │
  │                                                      │
  │         SERVER HEEFT REQUEST! ✓                     │
  │         (netwerk deed zijn werk)                    │
  │                                                      │
  │              ... 27ms later ...                     │
  │              (SERVER PROCESSING)                    │
  │                                                      │
  │ <────────── Response (SEQ=1) ──────────────────────  │
```

## 🔍 De 3 Delay Types

### 1. Network Delay (RTT)

**Wat is het:**
- Tijd die packets nodig hebben om over het netwerk te reizen
- Round Trip Time: Client → Server → Client

**Wireshark:**
```wireshark
tcp.analysis.ack_rtt
```

**Voorbeeld:**
```
Client stuurt:  Timestamp 0.000
Server ACK't:   Timestamp 0.130
────────────────────────────
Network RTT:    130 ms
```

**Typische waardes:**
```
< 1 ms       = LAN (same building)
1-20 ms      = Regional WAN
20-50 ms     = National WAN
50-150 ms    = Intercontinental
> 150 ms     = Satellite / problematisch
```

### 2. Application Delay

**Wat is het:**
- Tijd die server/client nodig heeft om request te verwerken
- **NADAT** data ontvangen is (na ACK)

**Herkenning:**
```
═══════════════════════════════════════════════════
         APPLICATION DELAY SCENARIO
═══════════════════════════════════════════════════
Time: 0.000   Client stuurt HTTP GET request
Time: 0.130   Server ACK't (request ontvangen!)
───────────────────────────────────────────────────
              ← SERVER IS PROCESSING →
───────────────────────────────────────────────────
Time: 0.157   Server stuurt HTTP 200 OK response
───────────────────────────────────────────────────
Application delay = 0.157 - 0.130 = 27 ms
```

**Het bewijs:**
- Server heeft data (ACK gestuurd)
- Netwerk is OK (RTT is stabiel)
- **Server wacht met antwoorden** → applicatie is traag

### 3. TCP Retransmission Delay

**Wat is het:**
- Tijd die verloren gaat door packets die opnieuw verstuurd moeten worden

**Scenario:**
```
Client stuurt:      SEQ=1000
Packet lost!        ✗
Client wacht...     (RTO timeout = 3 seconden)
Client retransmit:  SEQ=1000 (opnieuw)
Server ACK:         ACK=1001
────────────────────────────────
Delay = 3+ seconden (geen network/app!)
```

## 🛠️ Hands-On: Delay Meten in Wireshark

### Methode 1: Time Reference (Beste)

**Stappen:**

1. **Find het Server ACK** (wanneer server request ontving)
```wireshark
tcp.ack == [jouw verwachte ACK number]
```

2. **Set Time Reference**
```
Rechtsklik packet → Set/Unset Time Reference
```

3. **Find eerste Server Response**
```
ip.src == SERVER_IP && tcp.len > 0
```

4. **Lees Delta Time**
```
Kolom "Time" toont nu tijd SINDS reference
Dit is jouw application delay!
```

### Methode 2: Delta Time

**Voeg kolom toe:**
```
tcp.time_delta
```

**Let op:**
::warning
`tcp.time_delta` meet tijd sinds **vorige frame**, niet vorige frame in **deze TCP stream**!

Bij multistream captures → gebruik:
```
tcp.time_since_previous_frame_in_this_tcp_stream
```
::

**Enable this:**
```
Rechtsklik TCP packet → Protocol Preferences
→ ☑ Calculate conversation timestamps
```

## 📊 Praktische Scenario's

### Scenario A: Gezonde Verbinding

```
═══════════════════════════════════════════════════
Packet 8:   Client → Server   SEQ=1, Len=1793
            Time: 0.000

Packet 10:  Server → Client   ACK=1794
            Time: 0.130         (130ms RTT ✓)

Packet 11:  Server → Client   SEQ=1, Len=1460
            Time: 0.157         (27ms app delay ✓)
═══════════════════════════════════════════════════
```

**Interpretatie:**
- RTT: 130ms (Atlantic Ocean → acceptabel)
- App delay: 27ms (prima)
- **Geen probleem**

### Scenario B: Application Bottleneck

```
═══════════════════════════════════════════════════
Packet 45:  Client → Server   Database query
            Time: 0.000

Packet 46:  Server → Client   ACK (query ontvangen)
            Time: 0.002         (2ms RTT - LAN!)

Packet 47:  Server → Client   Response
            Time: 2.145         (2143ms app delay!)
═══════════════════════════════════════════════════
```

**Diagnose:**
- RTT: 2ms (excellent)
- App delay: 2143ms (2+ seconden!)
- **Server database is traag → niet het netwerk**

### Scenario C: Network Congestion

```
═══════════════════════════════════════════════════
Packet 100: Client → Server   Request
            Time: 0.000

Packet 101: Server → Client   ACK
            Time: 0.450         (450ms RTT!)

Packet 102: Server → Client   Response
            Time: 0.467         (17ms app delay)
═══════════════════════════════════════════════════
```

**Diagnose:**
- RTT: 450ms (veel te hoog voor LAN/WAN)
- App delay: 17ms (prima)
- **Netwerk heeft congestie/routing problemen**

### Scenario D: Retransmission Hell

```
═══════════════════════════════════════════════════
Packet 200: Client → Server   SEQ=5000
            Time: 0.000

Packet 201: Client → Server   SEQ=5000 [Retrans]
            Time: 3.012         (RTO timeout)

Packet 202: Server → Client   ACK=5001
            Time: 3.142
═══════════════════════════════════════════════════
```

**Diagnose:**
- 3 seconden delay tussen origineel en retransmit
- **Packet loss → TCP retransmission delay**

## 🔬 Advanced Analysis Workflow

### Step 1: Baseline RTT

**Bij handshake:**
```wireshark
tcp.flags.syn == 1
```

Check **iRTT** (initial RTT) in packet details:
```
TCP → SEQ/ACK Analysis → iRTT: 0.130 seconds
```

**Mental pegboard:** Onthoud deze waarde!

### Step 2: Track ACKs

**Filter op stream:**
```wireshark
tcp.stream == 0
```

**Voeg kolommen toe:**
- `tcp.time_relative` - Tijd sinds start stream
- `tcp.analysis.ack_rtt` - RTT per ACK
- `tcp.len` - Payload size

### Step 3: Identify Stalls

**Filter:**
```wireshark
tcp.time_delta > 0.02 && tcp.len > 0
```

**Dit toont:**
- Packets met > 20ms delay
- Alleen packets met payload (geen pure ACKs)

### Step 4: Classify Delay

**Voor elke stall:**

**Check 1: Is er packet loss?**
```wireshark
tcp.analysis.retransmission
tcp.analysis.duplicate_ack
```
→ JA? Dan **TCP retransmission delay**

**Check 2: Is RWND klein?**
```wireshark
tcp.window_size < 5000
```
→ JA? Dan **TCP flow control delay**

**Check 3: Stabiel RTT, maar late response?**
```
ACK komt snel → response komt laat
```
→ Dit is **Application delay**

## 📈 Time-Sequence Graph Interpretatie

**Open graph:**
```
Statistics → TCP Stream Graph → Time-Sequence (tcptrace)
```

### Pattern Recognition

**Network delay:**
```
    Data ─────────────  (constante helling)
                       ╱
                      ╱  (RTT determines slope)
                     ╱
```
Constante steilheid = stable network

**Application delay:**
```
    Data ─────┐
              │ FLAT    (server processing)
              └─────────  (data resumes)
```
Horizontale secties = app stalls

**RWND stalls:**
```
    Data ────────┐     ┌────────
                 │     │
                 └─────┘  (green RWND line touches)
```
Groene lijn raakt data = flow control stall

**Retransmissions:**
```
    Data ─────╳     (packet lost)
              ╲
               ╲ ╱  (retransmit + recovery)
                ╳
```
Terugvallende/overlappende lijnen

## 🎯 Common Delay Patterns

| Delta Time | Waarschijnlijke Oorzaak | Actie |
|------------|-------------------------|-------|
| **1-10 ms** | Normal processing | Geen probleem |
| **20-30 ms** | RWND stall (buffer full) | Check window size |
| **50-200 ms** | High RTT / WAN distance | Acceptabel voor distance |
| **300-500 ms** | Network congestion | Check routing/bandwidth |
| **3 sec** | First RTO retransmit | Check packet loss cause |
| **6 sec** | Second RTO retransmit | Serious packet loss |
| **60+ sec** | Application timeout | Database/backend issue |
| **120 sec** | Idle timeout / keepalive | Normal TCP behavior |

## 🚨 Red Flags

### 🔴 Definitely Application Problem

```
✓ ACK komt binnen < 10ms
✓ RTT is stabiel en laag
✓ Geen retransmissions
✓ RWND is groot
✗ Response komt 100ms+ later
```
→ **Server applicatie is bottleneck**

### 🔴 Definitely Network Problem

```
✓ ACK komt binnen > 500ms
✓ RTT fluctueert wild
✓ Veel retransmissions
✗ App delay is klein
```
→ **Netwerk congestie/routing problemen**

### 🔴 Definitely TCP Problem

```
✓ RWND < MSS (< 1500 bytes)
✓ Periodieke 20ms stalls
✓ Zero window events
✓ Window update packets
```
→ **Flow control bottleneck**

## 💡 Pro Tips

::tip
**Mental Pegboard Techniek**

Bij elke analyse:
1. Check iRTT in handshake (bijv. 130ms)
2. Onthoud dit getal
3. Vergelijk ALLE delays hiermee
4. Delay > RTT × 2? → Onderzoek verder!
::

::tip
**Quick Sanity Check**

Vraag jezelf af:
- "Heeft de server de data?" (Is ACK verzonden?)
- "Hoe lang duurde het tot ACK?" (= RTT)
- "Hoe lang daarna kwam response?" (= App delay)

3 vragen = volledige diagnose
::

::warning
**Veel engineers maken deze fout:**

Ze zien 500ms delay en denken "slow network".
Maar als RTT = 2ms, dan is 498ms **applicatie delay**!

**Check altijd RTT eerst.**
::
