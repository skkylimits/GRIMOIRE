## 🔄 Retransmissions & Packet Loss

TCP garandeert delivery door **retransmissions**. Maar retransmissions = performance hit. Begrijp waarom ze gebeuren = troubleshooting mastery.

## 📊 Retransmission Types

### 1. Fast Retransmit (3× Duplicate ACK)

**Trigger:**
```
Server sends:  SEQ=1, SEQ=1001, SEQ=2001, SEQ=3001
Client receives: 1, (missing 1001!), 2001, 3001

Client ACKs:
  ACK=1001  (expected after 1)
  ACK=1001  (still missing 1001, got 2001)
  ACK=1001  (still missing 1001, got 3001)
  ACK=1001  (3rd duplicate!)

Server: "3 DupACKs = loss!" → Immediate retransmit SEQ=1001
```

**Wireshark:**
```
[TCP Fast Retransmission]
tcp.analysis.fast_retransmission
```

**Timeline:**
```
    ┌─────┐       ┌─────┐       ┌─────┐
    │ SEQ │       │ SEQ │       │ SEQ │
    │ 1001│ LOST  │ 2001│       │ 3001│
    └─────┘   ✗   └─────┘       └─────┘
       │                │           │
       │            ACK=1001    ACK=1001 (DupACK 1)
       │                        ACK=1001 (DupACK 2)
       │                        ACK=1001 (DupACK 3)
       │                            ↓
       └──── FAST RETRANSMIT ──────┘
```

**Good news:** Fast recovery (~RTT delay)

### 2. Retransmission Timeout (RTO)

**Trigger:**
```
Sender: Stuurt packet
Sender: Starts timer (RTO = based on RTT)
Sender: Timer expires, no ACK received
Sender: Retransmits
```

**RTO Calculation:**
```
RTO = RTT + (4 × RTT variance)

Example:
  RTT = 130ms
  RTO ≈ 3 seconds (initial)

After timeout → RTO doubles:
  First retry:  3s
  Second retry: 6s
  Third retry:  12s
```

**Wireshark:**
```
[TCP Retransmission]
tcp.analysis.retransmission
```

**Bad news:** Slow recovery (seconds!)

### 3. Spurious Retransmission

**Wat is het:**
```
Sender: Stuurt packet
Sender: RTO expires (ACK delayed)
Sender: Retransmits
Receiver: Gets duplicate! (original arrived)
```

**Wireshark:**
```
[TCP Spurious Retransmission]
```

**Cause:** Network delay, not loss

## 🚨 Root Cause Analysis

### Is It Real Packet Loss?

**Checklist:**

**1. Check RTT Stability**
```wireshark
tcp.analysis.ack_rtt
```
- Stable RTT (±10ms) = Good network
- Jumping RTT (50ms → 500ms) = Congestion

**2. Check Pattern**
```
Random retransmits = Network loss
Burst retransmits = Congestion collapse
Periodic retransmits = Application slow
```

**3. Check Direction**
```
Only Client → Server retransmits?
  → Server dropping (firewall, buffer)

Only Server → Client retransmits?
  → Client dropping (CPU, buffer)

Both directions?
  → Network path problem
```

**4. Check Window**
```wireshark
tcp.window_size < 5000
```
- Small window + retransmits = RWND problem
- Large window + retransmits = Network problem

## 📊 Common Scenarios

### Scenario 1: Congested Network

**Symptoms:**
```
• Many retransmissions (both directions)
• Increasing RTT
• Duplicate ACKs
• CWND collapsing (small bursts)
```

**Root cause:** Network can't handle load

**Fix:** QoS, bandwidth upgrade, rate limiting

### Scenario 2: Firewall Dropping

**Symptoms:**
```
• Retransmissions ONE direction only
• Specific packet sizes affected
• Consistent pattern
```

**Root cause:** Firewall rules, stateful tracking limits

**Fix:** Firewall configuration, MTU adjustment

### Scenario 3: Application Overload

**Symptoms:**
```
• Zero windows
• Long ACK delays
• Retransmissions after timeout
• Small RWND
```

**Root cause:** Receiver can't process fast enough

**Fix:** Application tuning, hardware upgrade

## 🎯 Fast vs Timeout Comparison

| Aspect | Fast Retransmit | Timeout Retransmit |
|--------|----------------|---------------------|
| **Trigger** | 3× Duplicate ACKs | RTO timer expires |
| **Speed** | ~1 RTT | 3+ seconds |
| **Performance** | Minor impact | Major impact |
| **Cause** | Single packet loss | Multiple loss or delay |
| **Recovery** | SACK helps | Slow restart |

## 📈 Throughput Impact

### Retransmission Overhead

```
1% packet loss = ~50% throughput reduction!

Why?
• Lost packet must be retransmitted
• Subsequent packets delayed
• CWND may halve
• TCP backs off
```

### The Retransmission Spiral

```
High load → Congestion → Packet loss
    ↓
Retransmissions → More load → More loss
    ↓
CWND collapse → Throughput tanks
    ↓
Application timeouts → Retries → Even more load!
```

## 🔧 Troubleshooting Workflow

**Step 1: Quantify**
```wireshark
tcp.analysis.retransmission

Statistics → Conversations → TCP
  → Sort by "Retransmissions" column
```

**Step 2: Pattern**
```
• Random = Network quality
• Burst = Congestion event
• Periodic = Application issue
```

**Step 3: Direction**
```wireshark
ip.src == X.X.X.X and tcp.analysis.retransmission
```

**Step 4: Timeline**
```
View → Time Display Format → Seconds Since Previous Packet
Look for gaps (> 1 second = timeouts)
```

**Step 5: Root Cause**
```
Check:  RTT, Window Size, Packet Size, Flags
Correlate with: Network events, application logs
```
