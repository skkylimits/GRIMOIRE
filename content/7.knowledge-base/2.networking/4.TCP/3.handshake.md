## 🤝 The Three-Way Handshake

De **Three-Way Handshake** is de meest belangrijke 3 packets van elke TCP connectie. Master dit en je begrijpt 50% van TCP troubleshooting.

### Waarom De Handshake Belangrijk Is

**Handshake negotieert:**
- Maximum Segment Size (MSS)
- Window Scaling Factor
- SACK (Selective Acknowledgement) support
- Initial Sequence Numbers
- Network Round Trip Time (RTT)

**Performance wordt hier bepaald:**
- Verkeerde MSS = packet fragmentation
- Geen window scaling = throughput ceiling (64KB max!)
- Geen SACK = slow recovery bij packet loss

::important
**Handshake fails = connection fails.** Begrijp dit proces en je kunt 90% van connection problems diagnosticeren.
::

## 📊 The Three Packets

```
   CLIENT                                    SERVER
     │                                         │
     │  (1) SYN                               │
     │  SEQ=0, ACK=-, Flags=SYN               │
     │─────────────────────────────────────>  │
     │                                         │
     │  (2) SYN/ACK                           │
     │  SEQ=0, ACK=1, Flags=SYN+ACK           │
     │  <─────────────────────────────────────│
     │                                         │
     │  (3) ACK                               │
     │  SEQ=1, ACK=1, Flags=ACK               │
     │─────────────────────────────────────>  │
     │                                         │
     │  Connection Established!               │
     │  <────────────────────────────────────>│
```

## 🔍 Packet 1: SYN (Client → Server)

### Wat Gebeurt Er?

Client zegt: "Ik wil verbinden. Dit zijn mijn parameters."

### Wireshark Filter

```wireshark
tcp.flags.syn == 1 and tcp.flags.ack == 0
```

### Complete SYN Packet Analysis

```
═══════════════════════════════════════════════════
                   SYN PACKET
═══════════════════════════════════════════════════
Source:           192.168.1.100 (Client)
Destination:      217.160.0.69   (Server)
───────────────────────────────────────────────────
Source Port:      51234  (ephemeral - random)
Dest Port:        443    (HTTPS)
───────────────────────────────────────────────────
Sequence Number:  0              (relative)
Sequence (raw):   2847563921     (random ISN)
───────────────────────────────────────────────────
ACK Number:       Not set        (no data yet)
───────────────────────────────────────────────────
Flags:            SYN
───────────────────────────────────────────────────
Window Size:      65535
Window Scale:     7               (× 128)
Calculated:       8,388,480 bytes
───────────────────────────────────────────────────
IP TTL:           64              (likely Linux/Unix)
───────────────────────────────────────────────────
OPTIONS:
  • MSS: 1460 bytes
  • SACK Permitted: Yes
  • Timestamps: TSval=0, TSecr=0
  • Window Scale: 7 (shift count)
═══════════════════════════════════════════════════
```

### Key Fields Te Analyseren

**1. TTL (Time To Live)**
```
IP Header → Time To Live: 64
```

**Wat betekent dit:**
- Start waarde: 64, 128, or 255
- **64** = Linux/Unix/MacOS meestal
- **128** = Windows meestal
- **255** = Network devices (routers, firewalls)

**Gebruik:**
```
TTL = 64  → Started at 64, no hops (local network)
TTL = 54  → Started at 64, 10 hops away
TTL = 118 → Started at 128, 10 hops away
```

::tip
**TTL vertelt je waar je capturet!** Als client TTL=128 en server TTL=54, ben je waarschijnlijk client-side (server is 10 hops away).
::

**2. MSS (Maximum Segment Size)**
```
TCP Options → Maximum Segment Size: 1460
```

**Berekening:**
```
Ethernet MTU:     1500 bytes
  - IP Header:      20 bytes
  - TCP Header:     20 bytes
  ─────────────────────────
  = MSS:          1460 bytes
```

**Waarom belangrijk:**
- Smaller MSS = meer packets nodig
- Larger MSS = risico op fragmentation
- **Mismatched MSS** = performance nightmare

**3. Window Scaling**
```
TCP Options → Window Scale: 7 (multiply by 128)
```

**Critical fact:**
::warning
**Window Scaling wordt ALLEEN in SYN packets verzonden!** Als je dit mist in de handshake, blijf je stuck op 64KB window (throughput ceiling).
::

**Berekening:**
```
Window Size Value:  65535  (16-bit max)
Scale Factor:       7
Multiplier:         2^7 = 128
────────────────────────────────────
Calculated Window:  65535 × 128 = 8,388,480 bytes (8MB)
```

**4. SACK Permitted**
```
TCP Options → SACK Permitted
```

**Wat is SACK:**
- **Selective Acknowledgement**
- Allows acknowledging non-contiguous blocks
- Essential voor goede performance bij packet loss

**Zonder SACK:**
```
Lost packet 5

Client must wait for:
  Packet 5 timeout → retransmit
  All subsequent packets discarded!
```

**Met SACK:**
```
Lost packet 5

Client sends SACK:
  "I have 1-4, missing 5, have 6-10"

Server retransmits ONLY packet 5
```

## 🔍 Packet 2: SYN/ACK (Server → Client)

### Wat Gebeurt Er?

Server zegt: "OK, I heard you. Dit zijn MIJN parameters. Ik acknowledge jouw SYN."

### Wireshark Filter

```wireshark
tcp.flags.syn == 1 and tcp.flags.ack == 1
```

### Complete SYN/ACK Analysis

```
═══════════════════════════════════════════════════
                 SYN/ACK PACKET
═══════════════════════════════════════════════════
Source:           217.160.0.69   (Server)
Destination:      192.168.1.100 (Client)
───────────────────────────────────────────────────
Source Port:      443    (HTTPS)
Dest Port:        51234  (client's port)
───────────────────────────────────────────────────
Sequence Number:  0              (server's ISN)
Sequence (raw):   3112456789     (different from client!)
───────────────────────────────────────────────────
ACK Number:       1              (client SEQ + 1)
                                 (acknowledges SYN)
───────────────────────────────────────────────────
Flags:            SYN + ACK
───────────────────────────────────────────────────
Window Size:      65535
Window Scale:     8               (× 256, bigger than client!)
Calculated:       16,776,960 bytes
───────────────────────────────────────────────────
IP TTL:           54              (started at 64, 10 hops away)
───────────────────────────────────────────────────
Delta Time:       130 ms          (RTT!)
───────────────────────────────────────────────────
OPTIONS:
  • MSS: 1460 bytes
  • SACK Permitted: Yes
  • Timestamps: TSval=578462, TSecr=0
  • Window Scale: 8
═══════════════════════════════════════════════════
```

### Critical Analysis Points

**1. Delta Time = Network RTT**
```
Packet List → Delta: 0.130 seconds (130 ms)
```

**Dit is je eerste performance metric!**

```
< 1 ms    = LAN (same building)
1-20 ms   = Regional WAN
20-50 ms  = National WAN
50-150 ms = Intercontinental
> 150 ms  = Satellite / slow path
```

::important
**Mental pegboard:** Onthoud deze RTT! Je gaat het vergelijken met data transfer delays later.
::

**Wireshark berekent dit automatisch:**
```
TCP → SEQ/ACK Analysis → iRTT: 0.130 seconds
```

**2. Server's Sequence Number**
```
Sequence Number (raw): 3112456789
```

**Cruciale facts:**
- **Totaal anders** dan client's ISN (2847563921)
- Beide starten op "0" (relative)
- **Geen relatie** tussen de twee numbers
- Each direction tracks independently

**3. ACK Number = Client SEQ + 1**
```
Client SYN:    SEQ = 0
Server ACK:    ACK = 1     # (0 + 1)
```

**Waarom +1?**
- SYN flag acts as "ghost byte"
- No actual data, but consumes 1 sequence number
- This synchronizes the streams

**4. Ingress TTL Analysis**
```
IP TTL: 54
```

**Deduction:**
```
Started at:   64 (likely)
Decremented:  10 times
Hops away:    10 routers/L3 switches
```

**How we know it's not 128 or 255:**
- 128 → 54 = 74 hops (onrealistisch)
- 255 → 54 = 201 hops (absurd)
- 64 → 54 = 10 hops (**realistisch**)

## 🔍 Packet 3: ACK (Client → Server)

### Wat Gebeurt Er?

Client zegt: "Roger that. I acknowledge your SYN/ACK. Connection established!"

### Wireshark Filter

```wireshark
tcp.flags == 0x010    # ACK only (no other flags)
```

### Complete ACK Analysis

```
═══════════════════════════════════════════════════
                   ACK PACKET
═══════════════════════════════════════════════════
Source:           192.168.1.100 (Client)
Destination:      217.160.0.69   (Server)
───────────────────────────────────────────────────
Sequence Number:  1              (advanced from SYN)
───────────────────────────────────────────────────
ACK Number:       1              (server SEQ + 1)
───────────────────────────────────────────────────
Flags:            ACK only
───────────────────────────────────────────────────
Window Size:      502 (scaled: 64256 bytes)
───────────────────────────────────────────────────
No Options        (handshake complete)
═══════════════════════════════════════════════════
```

**Key points:**
- Client SEQ advanced to 1 (from 0)
- Client ACKs server's SEQ 0 → ACK 1
- Connection now **ESTABLISHED**
- Data can flow both ways

## 🔬 Wireshark Hands-On Analysis

### Step-by-Step Workflow

**1. Find Handshake**
```wireshark
tcp.flags.syn == 1
```

**2. Follow TCP Stream**
```
Right-click packet → Follow → TCP Stream
or
tcp.stream == 0
```

**3. Add Kolommen**

Right-click fields, "Apply as Column":
- `tcp.seq` - Sequence number
- `tcp.ack` - ACK number
- `tcp.len` - Payload length
- `tcp.analysis.ack_rtt` - RTT per ACK
- `tcp.window_size` - Window size

**4. Check Options**

Expand in packet details:
```
TCP → Options
  ├── Maximum Segment Size: 1460
  ├── SACK Permitted
  ├── Timestamps
  └── Window Scale: 7
```

**5. Verify Sequence Math**

```
SYN:      SEQ=0,     ACK=-
SYN/ACK:  SEQ=0,     ACK=1    ✓ (client 0+1)
ACK:      SEQ=1,     ACK=1    ✓ (server 0+1)
```

## 🎯 Common Handshake Problems

### Problem 1: No SYN/ACK Response

**Symptoom:**
```
Client: SYN
    (silence)
Client: SYN (retransmit)
    (silence)
Client: SYN (retransmit)
```

**Mogelijke oorzaken:**
- Firewall blokkeert
- Server down/unreachable
- Port not listening
- Routing problem

**Wireshark filter:**
```wireshark
tcp.analysis.retransmission and tcp.flags.syn==1
```

### Problem 2: MSS Mismatch

**Symptoom:**
```
Client: MSS = 1460
Server: MSS = 1460
    ... maar router verandert het naar 1432!
```

**Gevolg:**
- Packets gefragmenteerd
- Performance degradation
- Possible timeouts

**Fix:** Configure MSS clamping op router
```cisco
ip tcp adjust-mss 1432
```

### Problem 3: No Window Scaling

**Symptoom:**
```
Client: No Window Scale option
Server: Window Scale = 8
```

**Gevolg:**
- Client max window = 65535 bytes
- Throughput ceiling!
- Even on gigabit link

**Bandwidth-Delay Product:**
```
1 Gbps × 100ms RTT = 12.5 MB ideal window
Actual window: 64 KB
Result: <1% utilization!
```

## 📈 RTT Calculation

### Formula

```
RTT = (Time of SYN/ACK) - (Time of SYN)
```

**Wireshark auto-calculates:**
```
TCP → SEQ/ACK Analysis → iRTT (initial RTT)
```

### RTT Interpretation

```
< 1 ms       = Excellent (LAN)
1-50 ms      = Good (regional)
50-150 ms    = Acceptable (long distance)
150-300 ms   = Poor (satellite/congested)
> 300 ms     = Problem (investigate)
```
