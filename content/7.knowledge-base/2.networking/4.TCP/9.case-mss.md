## ⚠️ Case Study: MSS Mismatch & 21-Second Timeout

Dit is een **brilliant troubleshooting case**: gebruikers ervaren exact 21 seconden delay bij elke file access. Na opening werkt alles normaal. Maar die 21 seconden zijn **frustratingly consistent**.

::important
**Het probleem:** Een firewall/gateway die MSS wijzigt zonder beide endpoints te informeren, resulterend in packet fragmentation failure en catastrophic RTO timeouts.
::

## 🔍 Het Probleem

### Symptomen

**User Experience:**
```
1. Open network share           →  21 second delay
2. Browse files                  →  Instant (works!)
3. Open another file             →  21 second delay again
4. Close and reopen share        →  21 second delay
```

**Pattern:**
- First request per connection: 21s delay
- Subsequent requests: Normal performance
- Reopen connection: 21s delay returns
- **Exact same delay elke keer**

**Environment:**
```
• Client ←→ Firewall/VPN Gateway ←→ Server
• Gigabit links
• WAN accelerator present
• GRE tunnel or IPsec VPN
```

```
───────────────────────────────────────────────────
```

```
            SEQ=1, Len=1460 [TCP Retransmission]

```

### Step 3: The Smoking Gun

**What we see:**
1. Server sends **1460-byte** segment (SEQ=1)
2. Client receives **563-byte** segment (SEQ=563)
3. Client sends SACK: "I have 1461-2024, missing 1-1460"
4. **Server retransmits** after RTO timeout

**The gap:**
```
Server sent:     SEQ=1,   Len=1460  (bytes 1-1460)
Client received: SEQ=563, Len=563   (bytes 563-1125)

Missing:         Bytes 1-562  ← Where did these go?
```

## 🔬 Root Cause Analysis

### What's Happening

**The MSS Clamping Problem:**

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│ CLIENT   │         │ GATEWAY  │         │ SERVER   │
│ MSS=1460 │         │ (Hidden) │         │ MSS=1460 │
└──────────┘         └──────────┘         └──────────┘
     │                     │                     │
     │─── SYN (MSS=1460) ─→│                     │
     │                     │─── SYN (MSS=1460) ─→│
     │                     │                     │
     │                     │←── SYN/ACK ─────────│
     │←── SYN/ACK ─────────│   (MSS=1460)        │
     │                     │                     │
     │ (Client thinks: "OK, 1460 is fine")       │
     │ (Server thinks: "OK, 1460 is fine")       │
     │                     │                     │
     │                     │←── Data (1460) ─────│
     │                     │                     │
     │                     ╳  Gateway can't      │
     │                        fragment!          │
     │                        Packet LOST!       │
     │                     │                     │
     │←── Data (563) ──────│  Smaller packet     │
     │                        gets through       │
```

**Why Gateway Can't Fragment:**

**Possible reasons:**
1. **IPsec/VPN overhead** reduces effective MTU
   ```
   Original MTU:     1500 bytes
   - IP header:        20 bytes
   - IPsec ESP:        50+ bytes
   ──────────────────────────────
   Effective MTU:    1430 bytes
   ```

2. **GRE tunnel overhead**
   ```
   Original MTU:     1500 bytes
   - IP header:        20 bytes
   - GRE header:       24 bytes
   ──────────────────────────────
   Effective MTU:    1456 bytes
   ```

3. **DF (Don't Fragment) bit set**
   ```
   IP Flags: Don't Fragment = 1
   → Gateway CANNOT fragment
   → Must drop packet
   ```

4. **WAN Optimizer/Accelerator**
   - Device clamps MSS for tunnel compatibility
   - Forgets to inform endpoints

### The Math Behind 21 Seconds

**TCP Retransmission Timeout (RTO) backoff:**

```
═══════════════════════════════════════════════════
         EXPONENTIAL BACKOFF ALGORITHM
═══════════════════════════════════════════════════
Initial RTO:        ~3 seconds (based on RTT)
───────────────────────────────────────────────────
First retry:        RTO × 1 = 3 seconds
Second retry:       RTO × 2 = 6 seconds
Third retry:        RTO × 4 = 12 seconds
───────────────────────────────────────────────────
Total delay:        3 + 6 + 12 = 21 seconds
═══════════════════════════════════════════════════
```

**Waarom geen Fast Retransmit?**

Fast retransmit vereist **3× duplicate ACKs**:

```
Normal scenario:
  Server sends:  SEQ=1, SEQ=1001, SEQ=2001, SEQ=3001
  Client gets:   1, (missing 1001!), 2001, 3001
  Client ACKs:   1001, 1001 (dup), 1001 (dup), 1001 (dup)
  → Fast retransmit triggers!
```

**In onze case:**
```
Server sends:  SEQ=1 (1460 bytes) ← LOST
Server sends:  SEQ=563 (563 bytes) ← GETS THROUGH

Client only gets ONE small packet
→ Not enough for 3× dupACKs
→ No fast retransmit
→ Must wait for RTO timeout!
```

**Check handshake MSS:**


**Find retransmissions:**

**Find SACK packets:**
```wireshark

```

**Combine:**
```wireshark
(tcp.options.mss) || (tcp.analysis.retransmission && tcp.options.sack)
```

### Visual Indicators

**In packet list:**
- **Red/Black packets** = Retransmissions
- **SACK present** = Gaps in received data
- **Long delta times** (3s, 6s, 12s) = RTO pattern

**In packet details:**
```
TCP → Options → Maximum Segment Size
    Check BOTH SYN and SYN/ACK values

TCP → Analysis → [TCP Retransmission]
    Check "Previous segment not captured"
```

**Time sequence:**
```
Look for exponential backoff pattern:
  0s → 3s → 9s → 21s
```

## 🔧 Solutions

### Solution 1: MSS Clamping (Recommended)

**Configure gateway/router to advertise correct MSS:**

**Cisco:**
```cisco
interface GigabitEthernet0/1
  ip tcp adjust-mss 1432
```

**Linux (iptables):**
```bash
iptables -t mangle -A FORWARD \
  -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --set-mss 1432
```

**MikroTik:**
```
/ip firewall mangle
add action=change-mss chain=forward new-mss=1432 \
    protocol=tcp tcp-flags=syn tcp-mss=1433-65535
```

**Fortinet:**
```
config system interface
    edit "wan1"
        set tcp-mss 1432
    next
end
```

**What this does:**
```
Gateway intercepts SYN packets
Modifies MSS option: 1460 → 1432
Both endpoints now aware of correct MSS
No fragmentation needed!
```

### Solution 2: MTU Adjustment

**Reduce MTU on endpoints:**

**Windows:**
```powershell
netsh interface ipv4 set subinterface "Ethernet" mtu=1400
netsh interface ipv4 show subinterfaces
```

**Linux:**
```bash
ip link set dev eth0 mtu 1400
ip link show eth0
```

**Verification:**
```bash
# Ping with DF bit and specific size
ping -M do -s 1400 SERVER_IP
```

::warning
**Downside:** Affects ALL traffic, not just problematic path
::

### Solution 3: Path MTU Discovery (PMTUD)

**Enable PMTUD:**

**Linux:**
```bash
# Check current
sysctl net.ipv4.ip_no_pmtu_disc

# Enable (0 = enabled)
sysctl -w net.ipv4.ip_no_pmtu_disc=0
```

**Windows:**
```powershell
# Enable PMTUD
netsh interface ipv4 set global mtu=1500
netsh interface ipv4 set subinterface "Ethernet" mtu=1500
```

**How it works:**
```
1. Host sends packet with DF=1
2. Router can't fragment → sends ICMP "Fragmentation Needed"
3. Host reduces MSS automatically
4. Retransmits with smaller size
```

**Problem:**
```
Many firewalls block ICMP Type 3 Code 4
→ PMTUD breaks
→ Black hole routing
```

### Solution 4: Tunnel Configuration

**For VPNs/GRE tunnels:**

**Reduce tunnel MTU:**
```
# IPsec
ip mtu 1400

# GRE
tunnel mtu 1400
```

**Calculate correct value:**
```
Ethernet MTU:        1500
- IP header:           20
- TCP header:          20
- IPsec overhead:      50 (ESP mode)
─────────────────────────
Recommended TCP MSS:  1410

Or conservative:     1400 (safe for most scenarios)
```

## 🔬 Advanced Detection

### Identifying Tunnel Overhead

**Check IP header flags:**
```wireshark
ip.flags.df == 1
```

**Check for encapsulation:**
```wireshark
gre || esp || l2tp
```

**Look for ICMP:**
```wireshark
icmp.type == 3 && icmp.code == 4
```

**ICMP Type 3, Code 4 = "Fragmentation Needed and DF set"**

If you see this → PMTUD is working
If you DON'T see this → Firewall likely blocks ICMP

### Calculate Required MSS

**Formula:**
```
Tunnel MTU - IP header - TCP header = MSS

Example (IPsec):
  1500 (Ethernet)
  - 20 (outer IP)
  - 50 (ESP)
  - 20 (inner IP)
  - 20 (TCP)
  ──────────────
  = 1390 MSS
```

**Conservative approach:**
```
Round down to nearest 10
1390 → 1380

Or use standard safe value: 1400
```

## 🎯 Lessons Learned

### Key Takeaways

1. **Handshake lies**
   - MSS negotiation doesn't reflect path reality
   - Middleboxes can silently break assumptions

2. **Exact delay = RTO pattern**
   - 3s, 6s, 12s → exponential backoff
   - 21s total = classic three-retry pattern

3. **SACK reveals gaps**
   - Client knows what's missing
   - Server doesn't know WHY first packet lost

4. **DF bit prevents fallback**
   - No fragmentation allowed
   - Packet must be dropped

5. **MSS clamping is transparent**
   - Endpoints unaware
   - Gateway modifies SYN options
   - Problem solved without endpoint changes
