## 🪟 TCP Windows Explained

**TCP Windows** bepalen hoeveel data "in flight" mag zijn. Dit is **dé** bottleneck voor throughput! Zelfs op gigabit links.

::important
**The Golden Rule:**
```
Effective Window = min(RWND, CWND)
```
De **kleinste** van de twee bepaalt throughput.
::

## 📊 The Two Windows

### RWND (Receive Window)

**Wat is het:**
- Ontvanger's buffer size
- "Hoeveel data kan ik tegelijk accepteren?"
- **Zichtbaar** in TCP header

**Wireshark:**
```wireshark
tcp.window_size_value         # Raw 16-bit value
tcp.window_size_scalefactor   # Multiplier (from handshake)
tcp.window_size               # Calculated (value × scale)
```

**Voorbeeld:**
```
Window Size Value: 65535
Window Scale Factor: 7
Calculated Window: 8,388,480 bytes (8MB)
```

### CWND (Congestion Window)

**Wat is het:**
- Zender's self-imposed limit
- "Hoeveel durft ik te versturen zonder netwerk te overbelasten?"
- **NIET zichtbaar** in header (internal TCP stack logic)

**Wireshark:**
- Afleiden uit burst patterns
- Time-Sequence Graph toont groei
- Indirect zichtbaar

## 🔍 RWND Analysis

### RWND in Wireshark Vinden

**Packet details:**
```
TCP → Window: 502 (× 128 = 64256)
```

**Add as Column:**
```
tcp.window_size  # Calculated window
```

### RWND Bottleneck Herkennen

**Symptomen:**
```
Window size daalt: 65535 → 64915 → 61995 → ... → 1299
                                                    ↓
                                          Too small for MSS!
                                          Sender must PAUSE
```

**Filter:**
```wireshark
tcp.window_size < 2000          # Small windows
tcp.analysis.zero_window        # Complete stall
tcp.analysis.window_update      # Buffer cleared
```

### The Staircase Pattern

**Time-Sequence Graph:**
```
Statistics → TCP Stream Graph → Time-Sequence (tcptrace)
```

**Wat je ziet:**
```
    Data ─────┐            ┌─────
              │ STALL      │
              └────────────┘
                 (20ms)

Green line (RWND) touches data line → PAUSE
```

## 🔍 CWND Analysis

### CWND Herkennen (Indirect)

**Slow Start Pattern:**
```
Burst sizes:  1 MSS → 2 MSS → 4 MSS → 8 MSS → ...
```

**Wireshark Time-Sequence Graph:**
- Steil wordende lijn = CWND groeit
- Plateau = CWND limiet bereikt
- Terugval na loss = CWND halveert

### Slow Start Example

```
═══════════════════════════════════════════════════
                  SLOW START
═══════════════════════════════════════════════════
Round 1:  Send 1 MSS  → Wait ACK → CWND = 2
Round 2:  Send 2 MSS  → Wait ACK → CWND = 4
Round 3:  Send 4 MSS  → Wait ACK → CWND = 8
Round 4:  Send 8 MSS  → Wait ACK → CWND = 16
...
Threshold reached → Linear growth (Congestion Avoidance)
═══════════════════════════════════════════════════
```

### CWND Bottleneck Herkennen

**Symptoms:**
- Burst grootte stopt met groeien
- Retransmissions → CWND halveert
- Plateaus in throughput graph

## ⚔️ RWND vs CWND: Who Wins?

### Scenario 1: RWND Bottleneck

```
RWND: 64 KB   (receiver buffer full)
CWND: 512 KB  (sender could send more)
────────────────────────────────────
Effective: 64 KB  ← RWND limits!
```

**Result:** Low throughput, receiver is slow

### Scenario 2: CWND Bottleneck

```
RWND: 1 MB    (receiver ready)
CWND: 64 KB   (network congestion)
────────────────────────────────────
Effective: 64 KB  ← CWND limits!
```

**Result:** Network can't handle more

### Scenario 3: Optimal

```
RWND: 16 MB   (huge buffer)
CWND: 16 MB   (clean network)
────────────────────────────────────
Effective: 16 MB  ← Maximum throughput!
```

## 🚨 Window Problems

### Zero Window

**Wat gebeurt er:**
```
Receiver: RWND = 0  # "Buffer FULL! STOP!"
Sender: Pauses all transmission
Sender: Sends Zero Window Probe (keepalive)
Receiver: Eventually sends Window Update
Sender: Resumes
```

**Filter:**
```wireshark
tcp.analysis.zero_window        # Buffer full
tcp.analysis.window_full        # Window approaching zero
```

**Impact:**
- Complete throughput stall
- Can last seconds/minutes
- Application is slow processing data

### Window Scaling Missing

**Problem:**
```
Handshake: No Window Scale option
Maximum window: 65535 bytes (16-bit limit)
```

**Impact calculation:**
```
Link: 1 Gbps
RTT: 100ms
────────────────────────────────────
Bandwidth-Delay Product: 12.5 MB
Actual Window: 64 KB
────────────────────────────────────
Utilization: 0.5%  ← Disaster!
```

**Fix:** Enable window scaling op both ends

## 🎯 Practical Diagnosis

### Quick Check Workflow

**1. Check RWND:**
```wireshark
tcp.window_size < 10000
```

**2. Check for stalls:**
```wireshark
tcp.time_delta > 0.02    # > 20ms gaps
```

**3. Open Time-Sequence Graph:**
```
Statistics → TCP Stream Graph → Time-Sequence (tcptrace)
```

**4. Look for:**
- RWND touching data (stalls)
- Small burst patterns (CWND limited)
- Retransmissions (congestion)

## 🔧 Window Size Calculations

### Ideal Window Size Formula

```
Optimal Window = Bandwidth × RTT

Example:
  100 Mbps × 0.100s = 10 MB minimum window needed
```

### Current Window Check

```wireshark
tcp.window_size_value         # See raw value
tcp.window_size_scalefactor   # See multiplier
tcp.window_size               # See calculated
```

**If calculated < ideal → performance problem!**
