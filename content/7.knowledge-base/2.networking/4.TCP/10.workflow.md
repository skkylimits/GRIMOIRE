## 🔧 TCP Troubleshooting Workflow

Dit hoofdstuk combineert **alles** wat je geleerd hebt in één systematische workflow. Volg deze stappen bij **elke** TCP performance issue en je lost 95% van problemen op.

::important
**De regel:** TCP liegt nooit. De packets vertellen exact wat er gebeurt. Je moet alleen weten waar je moet kijken.
::

## 🎯 The Master Workflow

```

```

## 📋 Step-by-Step Workflow

### Step 1: Handshake Analysis

**Waarom eerst:**
- Establishes baseline parameters
- Negotiates critical options
- Provides initial RTT measurement

**Filter:**
```wireshark
tcp.flags.syn == 1
```

**Check deze fields:**

```
═══════════════════════════════════════════════════
                  SYN PACKET
═══════════════════════════════════════════════════
☑ Source Port / Dest Port      (application ID)
☑ Initial Sequence Number       (random, high value)
☑ Window Size Value             (buffer size)
☑ Window Scale Factor           (multiplier)
☑ MSS (Maximum Segment Size)    (1460 typical)
☑ SACK Permitted                (essential!)
☑ Timestamps                    (RTT measurement)
☑ IP TTL                        (hop count)
═══════════════════════════════════════════════════
```

**Red flags:**
```
✗ No Window Scaling → Max 64KB throughput ceiling
✗ No SACK          → Poor loss recovery
✗ MSS < 1400       → Potential tunnel/VPN overhead
✗ MSS mismatch     → Fragmentation risk
```

**Record deze waardes:**
```
iRTT (Initial RTT):   _______ ms
MSS Client:           _______ bytes
MSS Server:           _______ bytes
Window Scale Client:  _______
Window Scale Server:  _______
SACK Enabled:         YES / NO
```

### Step 2: RTT Measurement

**Wireshark field:**
```wireshark
tcp.analysis.ack_rtt
```

**Voeg als kolom toe:**
```
Rechtsklik → Apply as Column
```

**Analyse:**

**Check iRTT:**
```
Statistics → TCP Stream Graph → Time-Sequence
Look at first ACK delta time
```

**Classify RTT:**
```
< 1 ms       ✓ Excellent (LAN)
1-20 ms      ✓ Good (regional WAN)
20-50 ms     ✓ Acceptable (national)
50-150 ms    ⚠ High (intercontinental)
150-300 ms   ✗ Poor (satellite/congested)
> 300 ms     ✗✗ Problem (investigate!)
```

**Check stability:**
```wireshark
tcp.analysis.ack_rtt
```

Sort kolom → Look for consistency:
```
Good:    130ms, 131ms, 129ms, 132ms  (±2ms variance)
Bad:     50ms, 200ms, 75ms, 400ms    (wild swings)
```

**Wild RTT swings betekenen:**
- Network congestion
- Routing flaps
- Bufferbloat
- Packet prioritization issues

### Step 3: MSS Verification

**Check negotiated MSS:**
```wireshark
tcp.options.mss
```

**Verify:**
```
Client MSS: 1460
Server MSS: 1460
Match:      YES ✓
```

**Calculate expected:**
```
Ethernet MTU:        1500
- IP header:           20
- TCP header:          20
─────────────────────────
Expected MSS:        1460
```

**Check actual packet sizes:**
```wireshark
tcp.len
```

**Look for:**
```
Majority packets:    1460 bytes  ✓
Some smaller:        < 1460      ⚠ (investigate)
Many smaller:        < 1400      ✗ (MSS mismatch!)
```

**MSS mismatch detection:**
```wireshark
tcp.analysis.retransmission && tcp.len > 1400
```

If retransmissions only affect large packets → MSS issue!

### Step 4: RWND Monitoring

**Voeg window kolommen toe:**
```
tcp.window_size_value         # Raw value
tcp.window_size_scalefactor   # Multiplier
tcp.window_size               # Calculated
```

**Watch for declining pattern:**
```
═══════════════════════════════════════════════════
Packet    Window Size      Status
───────────────────────────────────────────────────
10        8,388,480        ✓ Healthy
50        7,935,360        ✓ OK
100       6,700,160        ⚠ Declining
150       2,598,272        ✗ Low
200       166,272          ✗✗ Critical!
───────────────────────────────────────────────────
Critical threshold: Window < MSS (1460)
═══════════════════════════════════════════════════
```

**Filters:**
```wireshark
# Small windows
tcp.window_size < 5000

# Zero windows
tcp.analysis.zero_window

# Window updates
tcp.analysis.window_update
```

**RWND bottleneck symptoms:**
```
☑ Window size decreases steadily
☑ Periodic stalls (20-30ms)
☑ Window updates
☑ RTT is low and stable
☑ No packet loss
```

→ **Application can't consume data fast enough**

### Step 5: CWND Pattern Analysis

**CWND is invisible - deduce from burst patterns:**

**Open Time-Sequence Graph:**
```
Statistics → TCP Stream Graph → Time-Sequence (tcptrace)
```

**Look for:**

**Slow Start (exponential growth):**
```
Burst 1:  1 MSS
Burst 2:  2 MSS
Burst 3:  4 MSS
Burst 4:  8 MSS
```

**Congestion Avoidance (linear growth):**
```
After threshold:  +1 MSS per RTT
```

**CWND collapse (after loss):**
```
Before:  16 MSS burst
Loss:    Retransmission
After:   8 MSS burst  (halved!)
```

**CWND bottleneck symptoms:**
```
☑ Bursts stop growing
☑ Retransmissions present
☑ Throughput plateaus
☑ RWND is large
☑ Pattern repeats
```

→ **Network can't handle full speed**

### Step 6: Stall Detection

**Enable conversation timestamps:**
```
Rechtsklik TCP packet
→ Protocol Preferences
→ ☑ Calculate conversation timestamps
```

**Add kolom:**
```wireshark
tcp.time_since_previous_frame_in_this_tcp_stream
```

**Filter for stalls:**
```wireshark
tcp.time_delta > 0.020 && tcp.len > 0
```

**Classify stalls:**

| Duration | Likely Cause | Action |
|----------|--------------|--------|
| **20-30 ms** | RWND full | Check window size |
| **~RTT** | Normal ACK wait | No problem |
| **3 seconds** | First RTO | Check packet loss |
| **9 seconds** | Second RTO | Serious loss |
| **21 seconds** | Third RTO | Critical issue |
| **60+ seconds** | Application | Check app logs |

### Step 7: Retransmission Check

**Primary filter:**
```wireshark
tcp.analysis.retransmission
```

**Classify types:**
```wireshark
# Fast retransmit (3× dupACK)
tcp.analysis.fast_retransmission

# RTO timeout
tcp.analysis.retransmission && !(tcp.analysis.fast_retransmission)

# Spurious (ACK was delayed)
tcp.analysis.spurious_retransmission
```

**Analyze pattern:**

**Random retransmissions:**
```
Packet 100:  Retransmit
Packet 500:  Retransmit
Packet 1200: Retransmit
```
→ Network quality issue (interference, bad cable)

**Burst retransmissions:**
```
Packet 100-105: All retransmit
Packet 200-208: All retransmit
```
→ Congestion event (buffer overflow)

**One direction only:**
```
Only Client → Server retransmits
```
→ Server dropping (firewall, buffer, CPU)

**Calculate retransmission rate:**
```
Count:  tcp.analysis.retransmission
Total:  All packets

Rate = Retransmits / Total

< 0.1%    ✓ Excellent
0.1-1%    ⚠ Monitor
1-5%      ✗ Problem
> 5%      ✗✗ Critical
```

### Step 8: Zero Window Detection

**Filter:**
```wireshark
tcp.analysis.zero_window
```

**Also check:**
```wireshark
tcp.analysis.window_full
tcp.analysis.window_update
```

**Timeline analysis:**
```
═══════════════════════════════════════════════════
Time      Event                  Window Size
───────────────────────────────────────────────────
0.000     Data transfer          65,535
0.500     Window shrinking       10,240
0.550     Zero window!           0
          (Sender pauses)
1.200     Window update          65,535
1.201     Data resumes
═══════════════════════════════════════════════════
Stall duration: 650ms
```

**Impact calculation:**
```
If 100 zero windows × 650ms each = 65 seconds wasted!
```

### Step 9: Out-of-Order Check

**Filter:**
```wireshark
tcp.analysis.out_of_order
```

**Ook relevant:**
```wireshark
tcp.analysis.reused_ports
tcp.analysis.lost_segment
```

**Out-of-order betekent:**
```
Expected:  SEQ=1000, SEQ=2000, SEQ=3000
Received:  SEQ=1000, SEQ=3000, SEQ=2000

Packet 2000 kwam LATER aan dan 3000
```

**Causes:**
- Multipath routing (ECMP)
- Packet reordering in switches
- Load balancing issues
- High jitter

**Is het een probleem?**
```
Occasional:   No big deal (TCP handles it)
Frequent:     Yes (triggers dupACKs, potential retransmits)
With loss:    Major (SACK helps, maar nog steeds slow)
```

### Step 10: Root Cause Decision Tree

```
START
  │
  ├─ RTT high (> 200ms)?
  │  ├─ YES → Network latency problem
  │  │         Action: Check routing, WAN links
  │  └─ NO  → Continue
  │
  ├─ Retransmissions?
  │  ├─ YES → Packet loss
  │  │  ├─ Random pattern    → Network quality
  │  │  ├─ Burst pattern     → Congestion
  │  │  └─ One direction     → Endpoint dropping
  │  └─ NO  → Continue
  │
  ├─ RWND small/zero?
  │  ├─ YES → Flow control bottleneck
  │  │         Action: App too slow OR buffer too small
  │  └─ NO  → Continue
  │
  ├─ CWND not growing?
  │  ├─ YES → Congestion window limited
  │  │         Action: Check bandwidth-delay product
  │  └─ NO  → Continue
  │
  ├─ MSS mismatch?
  │  ├─ YES → Fragmentation/MTU issue
  │  │         Action: Configure MSS clamping
  │  └─ NO  → Continue
  │
  ├─ Stalls without window/loss issues?
  │  ├─ YES → Application delay
  │  │         Action: Check server processing
  │  └─ NO  → Continue
  │
  └─ All metrics good but slow?
       → Application-limited transmission
          (Server sending slowly on purpose)
```

## 📊 Wireshark Graphs Masterclass

### Graph 1: Time-Sequence (tcptrace)

**Open:**
```
Statistics → TCP Stream Graph → Time-Sequence Graph (tcptrace)
```

**What it shows:**
```
Y-axis: Sequence numbers (data progress)
X-axis: Time
```

**Interpretation:**

**Steep line:**
```
    Data ────────
                ╱
               ╱  (fast transmission)
              ╱
```
→ Good throughput

**Flat sections:**
```
    Data ────────────
                     (no progress)
```
→ Stalls (check why!)

**Staircase pattern:**
```
    Data ──┐  ┌──┐  ┌──
           │  │  │  │
           └──┘  └──┘  (periodic stalls)
```
→ RWND bottleneck (green line touches data)

**Overlapping lines:**
```
    Data ──╳╲╱
           ╳ ╱  (retransmissions)
          ╱ ╲
```
→ Packet loss and recovery

### Graph 2: Throughput Graph

**Open:**
```
Statistics → TCP Stream Graph → Throughput Graph
```

**What it shows:**
```
Y-axis: Bytes/second
X-axis: Time
```

**Good throughput:**
```
MB/s  ████████████████  (constant high)
```

**Bad throughput:**
```
MB/s  █▁▁█▁█▁▁▁█▁▁▁█▁  (spiky, unstable)
```

**RWND stalls:**
```
MB/s  ███▁▁███▁▁███▁▁  (periodic drops)
```

### Graph 3: Stevens Graph (RTT)

**Open:**
```
Statistics → TCP Stream Graph → Stevens Graph
```

**What it shows:**
```
Sequence numbers vs RTT
```

**Use case:**
- Detect RTT spikes
- See congestion onset
- Identify jitter

### Graph 4: Window Scaling Graph

**Open:**
```
Statistics → TCP Stream Graph → Window Scaling Graph
```

**What it shows:**
```
Y-axis: Window size
X-axis: Sequence number
```

**Pattern recognition:**
- Declining window → Buffer filling
- Flat zero line → Zero window event
- Jumping up → Window update

## 🚨 Common Mistakes Engineers Make

### Mistake 1: Looking at Interface Speed, Not TCP Behavior

```
❌ "We have 10 Gbps links, so it must be fast"
✓  "What's our RWND? What's our RTT? Window × RTT = throughput ceiling"
```

### Mistake 2: Looking at Average Throughput, Not Stalls

```
❌ "Average is 50 MB/s, good enough"
✓  "But there are 100× 1-second stalls → user experience is terrible"
```

### Mistake 3: Assuming Packet Loss = Network Problem

```
❌ "Packet loss → bad network"
✓  "Check: Is receiver dropping due to full buffer?"
```

### Mistake 4: Forgetting RWND Can Be Bottleneck

```
❌ "High RTT is the problem"
✓  "RTT is 2ms (LAN), but RWND is 64KB → that's the bottleneck!"
```

### Mistake 5: Not Knowing How to Identify CWND

```
❌ "I don't see CWND in Wireshark"
✓  "CWND is indirect - look at burst patterns in Time-Sequence Graph"
```

### Mistake 6: Using Wrong Wireshark Columns

```
❌ tcp.time_delta (time since ANY previous frame)
✓  tcp.time_since_previous_frame_in_this_tcp_stream
```

### Mistake 7: Not Recognizing MSS Problems

```
❌ "Handshake says MSS=1460, we're good"
✓  "But only small packets get through → MSS clamping needed"
```

### Mistake 8: Confusing Symptoms with Cause

```
❌ "High latency is causing slow transfers"
✓  "Small RWND is causing stalls which LOOK like latency"
```

## 🎯 The Master Troubleshooter Mindset

### Question Everything

```
"TCP never lies. If throughput is bad, TCP knows why.
 I just need to ask the right questions."
```

**The Questions:**

1. **Did the handshake negotiate good parameters?**
   - Window scaling? SACK? Correct MSS?

2. **Is the network healthy?**
   - Stable RTT? No packet loss?

3. **Is TCP limiting itself?**
   - RWND small? CWND not growing?

4. **Is the application keeping up?**
   - Long delays between ACK and response?

5. **Is there a middlebox interfering?**
   - MSS clamping? Proxy? WAN optimizer?

### Trust the Packets

```
"I don't guess. I don't assume.
 I look at the packets and they tell me exactly what happened."
```

### Systematic Approach

```
"I follow the same workflow every time.
 Step 1 → Step 2 → Step 3 ...
 No shortcuts. No assumptions."
```
