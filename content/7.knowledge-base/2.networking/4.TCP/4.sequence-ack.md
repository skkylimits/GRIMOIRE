## 🔢 Sequence & Acknowledgement Numbers

Dit is het **hart van TCP reliability**. Begrijp dit en je begrijpt hoe TCP gegarandeerde, geordende delivery realiseert.

### Het Fundamentele Principe

::important
**Sequence Number** = Positie van de **eerste byte** in dit segment
**ACK Number** = Welke byte ik **verwacht als volgende**
::

## 📊 How It Works

```
CLIENT                                    SERVER
  │                                         │
  │  SEQ=1, Len=1448                       │
  │  (bytes 1-1448)                        │
  │─────────────────────────────────────>  │
  │                                         │
  │  SEQ=1449, Len=512                     │
  │  (bytes 1449-1960)                     │
  │─────────────────────────────────────>  │
  │                                         │
  │                    ACK=1961            │
  │                    (got all bytes!)    │
  │  <─────────────────────────────────────│
```

**Math:**
```
Packet 1:  SEQ=1,    Len=1448  →  Next SEQ = 1 + 1448 = 1449
Packet 2:  SEQ=1449, Len=512   →  Next SEQ = 1449 + 512 = 1961

Server ACK: ACK=1961  # "I have everything up to 1960"
```

## 🔍 Wireshark Analysis

### Kolommen Toevoegen

Right-click → Apply as Column:
- `tcp.seq` - Sequence Number (relative)
- `tcp.nxtseq` - Next Sequence Number
- `tcp.ack` - Acknowledgement Number
- `tcp.len` - Segment Length (payload)

### Relative vs Raw

**Wireshark toont beide:**

```
Sequence Number: 1 (relative)
Sequence Number (raw): 2847563921
```

**Relative** (default):
- Starts at 0 (handshake) then 1 (first data)
- Easy voor mensen om te tracken
- Wireshark calculated

**Raw:**
- Echte waarde in packet
- Random Initial Sequence Number (ISN)
- Hard te volgen manueel

::tip
**Gebruik relative!** Wireshark doet de math voor je. Alleen raw gebruiken voor packet-level debugging.
::

### Filter op Stream

```wireshark
tcp.stream == 0    # Follow entire conversation
```

## 📈 Sequence Number Tracking

### Voorbeeld Data Flow

**Scenario:** Client stuurt HTTP request

```
═══════════════════════════════════════════════════
Packet 8:  SEQ=1,    Len=1448  →  Next=1449
Packet 9:  SEQ=1449, Len=345   →  Next=1794
───────────────────────────────────────────────────
Total sent: 1793 bytes (1 + 1448 + 345 - 1)
───────────────────────────────────────────────────
Server ACK=1794  # Confirms receipt
═══════════════════════════════════════════════════
```

**Wireshark shows:**
```
Packet 8:
  Sequence Number: 1
  Next Sequence Number: 1449
  TCP Segment Length: 1448

Packet 9:
  Sequence Number: 1449
  Next Sequence Number: 1794
  TCP Segment Length: 345

Packet 10 (from server):
  Acknowledgement Number: 1794  ✓
```

## 🎯 ACK Number Rules

### Rule 1: ACK = Next Expected SEQ

```
Sent:      SEQ=1,    Len=1000
Expected:  ACK=1001  # (1 + 1000)
```

### Rule 2: Cumulative ACKs

ACK number acknowledges **all prior data**:

```
Sent packets:  SEQ=1 (L=100), SEQ=101 (L=100), SEQ=201 (L=100)

ACK=301  # Means: "I got 1-100, 101-200, 201-300"
```

**Je hoeft niet elke packet te ACKen!** Één ACK kan duizenden packets acknowledgeen.

### Rule 3: ACK Never Goes Backward

```
ACK progression:  1 → 1001 → 2001 → 5001 → ...

If ACK goes back:  5001 → 2001  # Something is wrong!
```

Backward ACK indicates:
- Out-of-order ACK arrival
- Retransmission
- Missing data (SACK situation)

## 🚨 Detecting Problems

### Gap Detection

**Missing data scenario:**

```
Received:  SEQ=1 (L=1000), SEQ=2001 (L=1000)
Missing:   SEQ=1001-2000  # 1000 bytes gap!
```

**Wireshark marks:**
```
[TCP Previous segment not captured]
[TCP ACKed unseen segment]
```

### Out-of-Order Packets

```
Expected order:  SEQ=1, SEQ=1001, SEQ=2001
Actual order:    SEQ=1, SEQ=2001, SEQ=1001
```

**Wireshark marks:**
```
[TCP Out-of-Order]
```

**Filter:**
```wireshark
tcp.analysis.out_of_order
```

### Duplicate Data

```
Sent:  SEQ=1 (L=1000)
Sent again:  SEQ=1 (L=1000)  # Retransmit
```

**Wireshark marks:**
```
[TCP Retransmission]
```

**Filter:**
```wireshark
tcp.analysis.retransmission
```

## 📊 Practical Example

### HTTP GET Request

```
═══════════════════════════════════════════════════
CLIENT SENDS HTTP REQUEST:
═══════════════════════════════════════════════════
Packet 8:  SEQ=1,    Len=1448
  "GET /index.html HTTP/1.1\r\nHost: example.com..."

Packet 9:  SEQ=1449, Len=345
  "...Connection: keep-alive\r\n\r\n"
───────────────────────────────────────────────────
Total HTTP request: 1793 bytes
═══════════════════════════════════════════════════
SERVER ACKNOWLEDGES:
═══════════════════════════════════════════════════
Packet 10: ACK=1794  # Got complete request!
           Delta: 131ms (network RTT)
───────────────────────────────────────────────────
APPLICATION PROCESSING:
───────────────────────────────────────────────────
Packet 11: ACK=1794, SEQ=1 (server first response)
           Delta: 27ms from ACK

This 27ms = **Application Response Time**
═══════════════════════════════════════════════════
SERVER SENDS RESPONSE:
═══════════════════════════════════════════════════
Packet 11: SEQ=1,    Len=1460  "HTTP/1.1 200 OK..."
Packet 12: SEQ=1461, Len=1460  "...Content-Type..."
Packet 13: SEQ=2921, Len=716   "...</html>"
───────────────────────────────────────────────────
Total response: 3636 bytes
═══════════════════════════════════════════════════
CLIENT ACKNOWLEDGES:
═══════════════════════════════════════════════════
Packet 14: ACK=3637  # Got complete response!
═══════════════════════════════════════════════════
```
