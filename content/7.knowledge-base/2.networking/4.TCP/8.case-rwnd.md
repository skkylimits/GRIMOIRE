## 💥 Case Study: Receive Window Bottleneck

Dit is een **real-world case** uit de praktijk: een backup die 8 uur duurt terwijl het 30 minuten zou moeten duren. De oorzaak? Duizenden micro-pauses van 19-20 milliseconden.

::important
**Scenario:** Twee servers in hetzelfde datacenter, gigabit links, maar throughput collapse door RWND bottleneck. Dit is **death by a thousand cuts**.
::

## 🔍 Het Probleem

### Symptomen

**Environment:**
```
• Twee servers in hetzelfde datacenter
• 1 Gbps links
• Back-up: 300-800 GB data
• Verwachte duur: 30-60 minuten
• Werkelijke duur: 8 uur!
```

**Initial Observation:**
- Throughput fluctueert wild
- Geen packet loss op netwerk
- RTT is laag en stabiel (< 1ms, LAN)
- Toch: backup kruipt vooruit

## 📊 Wireshark Analysis - Step by Step

### Step 1: Open Capture & Sort Delta

**Delta time sorteren:**
```
View → Time Display Format → Seconds Since Previous Displayed Packet
```

**Of voeg kolom toe:**
```wireshark
tcp.time_delta
```

**Wat zie je:**
```
═══════════════════════════════════════════════════
Delta Time        Protocol    Info
───────────────────────────────────────────────────
0.000145          TCP         [...]
0.000167          TCP         [...]
0.019234          TCP         ACK  ← STALL!
0.000134          TCP         [...]
0.020456          TCP         ACK  ← STALL!
0.000189          TCP         [...]
0.025789          TCP         ACK  ← STALL!
0.030123          TCP         ACK  ← STALL!
═══════════════════════════════════════════════════
```

**Pattern:**
- Normale packets: < 1ms delta
- Periodieke ACKs: **19-30ms** delta
- Dit gebeurt **duizenden keren**

### Step 2: Analyze RWND Behavior

**Voeg Window kolommen toe:**

Rechtsklik op fields → Apply as Column:
```
tcp.window_size_value         # Raw 16-bit value
tcp.window_size_scalefactor   # Multiplier
tcp.window_size               # Calculated
```

**Filter op ontvanger (degene die ACKs stuurt):**
```wireshark
ip.src == RECEIVER_IP
```

**RWND daling pattern:**
```
═══════════════════════════════════════════════════
Packet    Window Size Value    Calculated Window
───────────────────────────────────────────────────
100       65535                8,388,480  (8 MB)
102       64915                8,308,320
104       61995                7,935,360
106       59875                7,664,000
108       52345                6,700,160
...
198       20299                2,598,272  ← Critical!
200       1299                   166,272  ← Too small!
202       65535                8,388,480  ← Recovery!
═══════════════════════════════════════════════════
```

### Step 3: Identify Stall Trigger

**Critical threshold:**
```
RWND < MSS (1460 bytes) = SENDER MUST PAUSE
```

**Wat gebeurt er:**

1. **Data komt aan** → RWND daalt per packet
```
Window: 65535 → 64915 → 61995 → ...
```

2. **RWND bereikt critical level** (< MSS)
```
Window: 1299 bytes
MSS:    1460 bytes
→ Kan geen full-size segment meer ontvangen!
```

3. **Sender stopt** met verzenden
```
No more packets sent (20ms pause)
```

4. **Applicatie verwerkt data** → buffer leegmaakt
```
RWND springt terug naar 65535
```

5. **Window Update** verstuurd
```
Sender hervat verzending
```

6. **Repeat** → duizenden keren

## 🔬 Deep Dive Analysis

### Waarom Precies 19-20ms?

Dit is **niet** netwerk latency (servers zitten op dezelfde switch).

Dit is de tijd die de **applicatie** nodig had om:
```
1. Data uit TCP receive buffer lezen
2. Naar disk schrijven (langzame I/O)
3. Buffer vrijmaken
4. TCP stack kan Window Update sturen
```

**Typische waardes:**
```
16-20 ms  = Langzame disk I/O
20-40 ms  = CPU-bound applicatie
50+ ms    = Zeer trage processing / swapping
```

### RWND = 0 vs RWND Low

**In deze case:**
```
RWND never hit exactly 0
RWND lowest value: 20299 bytes
```

**Waarom is dit een probleem?**
```
RWND = 20299 bytes
MSS  = 1460 bytes

20299 / 1460 = 13.9 segments

→ Net niet 0, maar net onder MSS × 14
→ Sender kan slechts kleine bursts sturen
→ Moet pauzeren voor Window Update
```

**Filter voor zero windows:**
```wireshark
tcp.analysis.zero_window
```

**Geen zero windows? Check for:**
```wireshark
tcp.window_size < 2000
```

### Time-Sequence Graph Visualization

**Open graph:**
```
Statistics → TCP Stream Graph → Time-Sequence (tcptrace)
```

**Wat je ziet:**

```
    ┌─────────────────────────────────────────────┐
    │  GREEN LINE = RWND (Receive Window)         │
    │  BLUE LINE  = Data Sequence Numbers         │
    └─────────────────────────────────────────────┘

Data  ─────┐               ┌─────────┐
(Blue)     │ STALL (20ms)  │         │ STALL
           └───────────────┘         └──────

RWND  ─────┐         ┌─────          ┌──────
(Green)    │ dips    │ recover       │ dips
           └─────────┘               └──────

    ↑                     ↑
    Data touches RWND     Buffer clears
    = STALL               = Resume
```

**Pattern interpretatie:**
- Groene lijn (RWND) daalt → buffer vult
- Blauwe lijn (data) komt dichterbij groene lijn
- **Wanneer ze raken** → horizontale sectie (stall)
- RWND springt omhoog → data hervat
- **Dit patroon herhaalt duizenden keren**

## 🚨 Root Cause Analysis

### Waarom Gebeurt Dit?

**Problem chain:**

```
┌─────────────────────────────────────────────────┐
│ 1. Sender stuurt data at line rate (1 Gbps)    │
│    ↓                                             │
│ 2. Receiver TCP buffer vult snel               │
│    ↓                                             │
│ 3. Application leest data traag (slow disk)    │
│    ↓                                             │
│ 4. RWND daalt (buffer vol)                      │
│    ↓                                             │
│ 5. RWND < MSS → Sender MUST pause               │
│    ↓                                             │
│ 6. Application clears buffer (20ms)             │
│    ↓                                             │
│ 7. RWND recovers → Window Update                │
│    ↓                                             │
│ 8. Repeat → throughput collapse                 │
└─────────────────────────────────────────────────┘
```

**Bewijs dat het NIET het netwerk is:**
- ✅ RTT is laag en stabiel (< 1ms)
- ✅ Geen packet loss
- ✅ Geen retransmissions
- ✅ Gigabit links werken perfect
- ✅ Geen congestie op switches

**Bewijs dat het NIET TCP is:**
- ✅ Window scaling correct
- ✅ SACK enabled
- ✅ MSS correct (1460)
- ✅ TCP mechanisme werkt zoals bedoeld

**Bewijs dat het DE APPLICATIE is:**
- ✅ Applicatie consumeert data traag
- ✅ 20ms processing tijd per burst
- ✅ RWND patroon toont buffer management probleem
- ✅ Throughput correlateert met disk I/O snelheid

## 🔧 Oplossingen

### Solution 1: Applicatie Optimalisatie (Beste)

**Direct fixes:**
```
• Snellere disk (SSD vs HDD)
• RAID configuratie optimaliseren
• Filesystem tuning (block size, alignment)
• Applicatie threading verbeteren
• Buffering strategie aanpassen
```

**Backup-specific:**
```
• Parallel streams gebruiken
• Compression aan source-side
• Deduplication
• Block-level incrementals
```

### Solution 2: TCP Buffer Tuning

**Receive buffer vergroten:**

**Linux:**
```bash
# Check current
sysctl net.ipv4.tcp_rmem

# Increase (min, default, max in bytes)
sysctl -w net.ipv4.tcp_rmem="4096 87380 16777216"
```

**Windows:**
```powershell
# Check current
netsh interface tcp show global

# Enable auto-tuning
netsh interface tcp set global autotuninglevel=normal

# Check receive window
Get-NetTCPSetting | Select SettingName, AutoTuningLevelLocal
```

::warning
**Let op:** Grotere buffer = meer memory usage!
Bij duizenden connections → resource exhaustion risico.
::

### Solution 3: Window Scaling Verification

**Check handshake:**
```wireshark
tcp.flags.syn == 1
```

**Verify:**
```
TCP Options → Window Scale: X (should be ≥ 7)

Calculated window = window_size_value × (2^scale)

Example:
  65535 × 2^7  = 8,388,480  (8 MB)   ✓ Good
  65535 × 2^0  = 65,535     (64 KB)  ✗ Bad
```

### Solution 4: NIC Offloading Optimization

**Features te checken:**

**Linux:**
```bash
ethtool -k eth0 | grep offload

# Enable TCP segmentation offload
ethtool -K eth0 tso on

# Enable generic receive offload
ethtool -K eth0 gro on
```

**Windows:**
```powershell
Get-NetAdapterAdvancedProperty -Name "Ethernet"

# Enable Receive Side Scaling (RSS)
Set-NetAdapterRss -Name "Ethernet" -Enabled $true
```

### Solution 5: Alternative Protocols

**Voor grote file transfers:**
```
• UDP-based protocols (UFTP, Tsunami)
• QUIC (has own flow control)
• FTP with GridFTP optimizations
• Rsync met tuning
```

## 📈 Impact Analysis

### Throughput Calculation

**Theoretische throughput (1 Gbps link):**
```
1 Gbps = 125 MB/s
800 GB backup = 800,000 MB
────────────────────────────────
Expected time = 800,000 / 125 = 6400 seconds ≈ 107 minutes
```

**Werkelijke throughput (met 20ms stalls):**
```
Elke stall = 20ms wasted
Bij 64KB bursts tussen stalls:

Effective throughput = 64 KB / 20ms = 3.2 MB/s
────────────────────────────────
Actual time = 800,000 / 3.2 = 250,000 seconds ≈ 69 uur!
```

**Improvement potential:**
```
Eliminate stalls → 107 min (target)
vs
Current state → 8 uur (480 min)
────────────────────────────────
Potential gain: 373 minuten (6+ uur)
```

## 🎯 Lessons Learned

### Key Takeaways

1. **High bandwidth ≠ High throughput**
   - 1 Gbps link kan 3 MB/s draaien door flow control

2. **TCP werkt correct**
   - RWND mechanisme voorkomt buffer overflow
   - Probleem is applicatie, niet protocol

3. **Micro-delays add up**
   - 20ms × 40,000 iteraties = 800 seconden (13 minuten) pure stall time

4. **Application layer matters**
   - Disk I/O bottlenecks manifest als TCP window collapse

5. **Graph visualization essential**
   - Time-Sequence Graph maakt probleem immediately zichtbaar
