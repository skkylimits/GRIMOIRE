## 📊 SACK (Selective Acknowledgement) Deep Dive

### Waarom SACK Bestaat

**Het probleem zonder SACK:**

```
Server stuurt:  SEQ=1 (1460), SEQ=1461 (1460), SEQ=2921 (1460), SEQ=4381 (1460)

Client ontvangt: ✓ SEQ=1
                 ✗ SEQ=1461 (LOST!)
                 ✓ SEQ=2921
                 ✓ SEQ=4381

Client ACK:     ACK=1461  (waiting for missing segment)
                ACK=1461  (duplicate)
                ACK=1461  (duplicate)
                ACK=1461  (duplicate)
```

**Zonder SACK:**
- Server retransmits SEQ=1461 (fast retransmit after 3× dupACK)
- **Maar** server denkt ook dat 2921 en 4381 verloren zijn!
- Server retransmits EVERYTHING vanaf 1461 → inefficient!

**Met SACK:**
```
Client ACK:     ACK=1461, SACK: 2921-4381, 5841-7301

Server weet:    "Missing ONLY 1461-2920"
                "I have 2921+ already"

Server retransmits: ONLY missing segment!
```

### SACK in Wireshark

**Enable SACK check in handshake:**
```wireshark
tcp.options.sack_perm == 1
```

**SACK in gebruik:**
```wireshark
tcp.options.sack
tcp.options.sack_le  # Left edge
tcp.options.sack_re  # Right edge
```

**SACK dissection:**
```
═══════════════════════════════════════════════════
               SACK OPTION EXAMPLE
═══════════════════════════════════════════════════
ACK Number:           1461

SACK Block 1:
  Left Edge:          2921
  Right Edge:         5841

SACK Block 2:
  Left Edge:          5841
  Right Edge:         7301
═══════════════════════════════════════════════════

Meaning: "I'm waiting for 1461, but I HAVE 2921-7301"
```

**Add as columns:**
```
tcp.options.sack_le  → SACK Left
tcp.options.sack_re  → SACK Right
```

### SACK Performance Impact

**Scenario: 1% packet loss**

**Zonder SACK:**
```
1 lost packet → retransmit ALL subsequent packets
100 packets sent → 100 packets retransmitted
Throughput: ~50% reduction
```

**Met SACK:**
```
1 lost packet → retransmit ONLY that packet
100 packets sent → 1 packet retransmitted
Throughput: ~1% reduction
```

**Formula:**
```
Throughput ∝ 1 / √(loss rate)

Without SACK: Very sensitive to loss
With SACK:    Tolerates loss much better
```

## 🔄 Congestion Control Deep Dive

### TCP Congestion States

TCP doorloopt **4 states** tijdens lifetime:

```
┌─────────────┐
│ SLOW START  │  Exponential growth
│ (CWND × 2)  │  1→2→4→8→16→32 MSS
└──────┬──────┘
       │
       │ (reaches ssthresh)
       ↓
┌─────────────┐
│ CONGESTION  │  Linear growth
│ AVOIDANCE   │  +1 MSS per RTT
│ (CWND + 1)  │
└──────┬──────┘
       │
       │ (packet loss detected)
       ↓
┌─────────────┐
│ FAST        │  CWND halved
│ RECOVERY    │  ssthresh = CWND/2
└──────┬──────┘
       │
       │ (loss recovery complete)
       ↓
┌─────────────┐
│ CONGESTION  │  Resume linear growth
│ AVOIDANCE   │
└─────────────┘
```

### Slow Start Mechanics

**Initial CWND:**
```
Modern TCP:    10 MSS (RFC 6928)
Legacy TCP:    2-4 MSS
```

**Growth:**
```
Round 1:  CWND = 1   → send 1 packet
          ACK received → CWND = 2

Round 2:  CWND = 2   → send 2 packets
          2 ACKs → CWND = 4

Round 3:  CWND = 4   → send 4 packets
          4 ACKs → CWND = 8

CWND doubles every RTT until ssthresh reached
```

**Wireshark visualization:**

Time-Sequence Graph toont **steilheid die toeneemt**:

```
Seq  ┌─────────────────  (steiler = faster growth)
     │              ╱╱
     │           ╱╱
     │        ╱╱
     │     ╱╱
     │  ╱╱
     └──────────────→ Time
```

### Congestion Avoidance

**Trigger:**
```
CWND >= ssthresh (slow start threshold)
```

**Growth:**
```
Linear: +1 MSS per RTT

Not per packet! Per full window of ACKs received.
```

**Example:**
```
CWND = 16 MSS
RTT = 100ms

Send 16 packets → wait 100ms
All ACKs received → CWND = 17 MSS

Send 17 packets → wait 100ms
All ACKs received → CWND = 18 MSS
```

**Wireshark:**

Time-Sequence Graph toont **constante helling**:

```
Seq  ┌───────────────
     │           ╱  (constant slope)
     │          ╱
     │         ╱
     │        ╱
     │       ╱
     └──────────────→ Time
```

### Fast Recovery

**Trigger:**
```
3× Duplicate ACKs received
```

**Actions:**
```
1. ssthresh = CWND / 2
2. Retransmit missing segment
3. CWND = ssthresh + 3 MSS
4. Each additional dupACK: CWND += 1 MSS
5. New ACK received: CWND = ssthresh
6. Enter Congestion Avoidance
```

**Why +3 MSS?**
```
3 dupACKs mean 3 packets got through
→ Network has capacity for those 3
→ Inflate window temporarily
```

**Wireshark:**

```
═══════════════════════════════════════════════════
Packet 100:  SEQ=10001  (lost)
Packet 101:  SEQ=11461  (arrives)
Packet 102:  SEQ=12921  (arrives)
Packet 103:  SEQ=14381  (arrives)
───────────────────────────────────────────────────
Packet 104:  ACK=10001  (expecting lost packet)
Packet 105:  ACK=10001  [Dup ACK]
Packet 106:  ACK=10001  [Dup ACK]
Packet 107:  ACK=10001  [Dup ACK]  ← Third!
───────────────────────────────────────────────────
Packet 108:  SEQ=10001  [Fast Retransmission]
───────────────────────────────────────────────────
CWND halved, fast recovery begins
═══════════════════════════════════════════════════
```

**Filter:**
```wireshark
tcp.analysis.fast_retransmission
tcp.analysis.duplicate_ack
```

### RTO (Retransmission Timeout)

**When fast recovery fails:**

**Trigger:**
```
Packet loss without enough dupACKs
Timer expires without ACK
```

**Consequences (severe!):**
```
1. ssthresh = CWND / 2
2. CWND = 1 MSS          (reset to initial!)
3. RTO = RTO × 2         (exponential backoff)
4. Enter Slow Start      (start over)
```

**RTO calculation:**
```
SRTT = Smoothed RTT
RTTVAR = RTT variance

RTO = SRTT + 4 × RTTVAR

Typical: RTO starts at ~3 seconds
```

**Exponential backoff:**
```
First retry:    3 seconds
Second retry:   6 seconds
Third retry:    12 seconds
Fourth retry:   24 seconds
Fifth retry:    48 seconds
Maximum:        Usually 60-120 seconds (OS dependent)
```

**Wireshark:**
```
Look for delta times: 3s, 6s, 12s pattern
Filter: tcp.analysis.retransmission
```

## 📈 Bandwidth-Delay Product (BDP)

### The Formula

```
BDP = Bandwidth × RTT

Example:
  100 Mbps × 100ms = 10 Mb = 1.25 MB
```

**What it means:**
```
"Hoeveel data is 'in flight' tussen endpoints"
```

**Optimal window size:**
```
Window ≥ BDP

If Window < BDP → throughput limited by window, not bandwidth
```

### BDP Calculation Examples

**Example 1: LAN**
```
Bandwidth:  1 Gbps
RTT:        1 ms
────────────────────────
BDP = 1 Gbps × 0.001s = 1 Mb = 125 KB

Optimal window: ≥ 125 KB
```

**Example 2: WAN**
```
Bandwidth:  100 Mbps
RTT:        50 ms
────────────────────────
BDP = 100 Mbps × 0.050s = 5 Mb = 625 KB

Optimal window: ≥ 625 KB
```

**Example 3: Intercontinental**
```
Bandwidth:  100 Mbps
RTT:        150 ms
────────────────────────
BDP = 100 Mbps × 0.150s = 15 Mb = 1.875 MB

Optimal window: ≥ 2 MB
```

### Window Scaling Necessity

**Without window scaling:**
```
Maximum window: 65,535 bytes (16-bit limit)

Even on 1 Gbps link:
  BDP = 125 KB
  Window = 64 KB
  Utilization = 64/125 = 51%
```

**With window scaling (factor 7):**
```
Window = 65,535 × 2^7 = 8,388,480 bytes (8 MB)

Now supports:
  8 MB / RTT throughput

Example (100ms RTT):
  8 MB / 0.1s = 80 MB/s = 640 Mbps ✓
```

## 🔍 Advanced Wireshark Techniques

### TCP Follow Stream Filtering

**Follow TCP stream:**
```
Rechtsklik packet → Follow → TCP Stream
```

**But also know the underlying filter:**
```wireshark
tcp.stream eq 0
```

**Why useful:**
- Filter specific conversation
- Isolate one connection from thousands
- Combine with other filters

**Advanced:**
```wireshark
# Stream 5 with retransmissions
tcp.stream eq 5 && tcp.analysis.retransmission

# Stream with large windows
tcp.stream eq 10 && tcp.window_size > 1000000
```

### Relative Sequence Numbers

**Toggle relative sequence numbers:**
```
Edit → Preferences → Protocols → TCP
☑ Relative sequence numbers
```

**Why toggle it off?**
```
When comparing:
  - Client and server perspectives
  - Multiple captures
  - Raw packet analysis
```

**Columns to understand both:**
```
tcp.seq           # Relative (Wireshark calculated)
tcp.seq_raw       # Raw (actual in packet)
tcp.nxtseq        # Next expected (relative)
```

## 🎯 Performance Optimization Strategies

### Strategy 1: Tune Initial CWND

**Linux:**
```bash
# Check current
ip route | grep default

# Set initial CWND to 10 (modern default)
ip route change default via GATEWAY initcwnd 10
```

**Impact:**
```
Old:  Start at 2-4 MSS  → slow start takes longer
New:  Start at 10 MSS   → faster startup
```

**Use case:**
- Short-lived connections (HTTP requests)
- Low RTT environments

### Strategy 2: Enable TCP Fast Open (TFO)

**Linux:**
```bash
# Enable TFO
sysctl -w net.ipv4.tcp_fastopen=3

# 1 = Client, 2 = Server, 3 = Both
```

**What it does:**
```
Normal:      SYN → SYN/ACK → ACK → Data (3 RTTs)
With TFO:    SYN+Data → SYN/ACK+Data (1 RTT)
```

**Savings:**
```
2 RTT reduction for first request
Massive improvement for high-latency links
```

### Strategy 3: BBR Congestion Control

**What is BBR:**
```
Traditional: Loss-based (packet loss → slow down)
BBR:         Bandwidth-based (measure actual capacity)
```

**Enable on Linux:**
```bash
# Load BBR module
modprobe tcp_bbr

# Set BBR as default
sysctl -w net.ipv4.tcp_congestion_control=bbr

# Verify
sysctl net.ipv4.tcp_congestion_control
```

**Advantages:**
```
• Better throughput on lossy networks
• Faster convergence to optimal rate
• Lower latency (less bufferbloat)
```

**Wireshark detection:**
```
BBR behaves differently - look for:
• More aggressive probing
• Faster recovery from loss
• Different burst patterns
```
