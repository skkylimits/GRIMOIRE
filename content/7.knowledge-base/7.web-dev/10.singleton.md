::important
Important

- In a Three.js project, many parts (like the Camera, Renderer, World, UI, etc.) need access to the same shared data. The Singleton pattern ensures they all use one consistent Experience instance, avoiding bugs and confusion.

It solves the “losing context” issue and simplifies your code by removing the need to manually pass instances around.
::

## **Why Singleton?**

Think of **`Experience.js`** as the **Director** on a movie set. The Director manages everything, like the camera, renderer, and UI. To make sure everything runs smoothly and there’s no confusion, we use the **Singleton pattern**. This guarantees there’s **only one Director** calling the shots and making sure all parts of the app are in sync.

Imagine if we had multiple directors (multiple instances of `Experience`). The actors (the different parts of the app) might get conflicting instructions. By using a Singleton, we ensure **one consistent experience** for all parts of the app.

### **Singleton Code**:

```js
let instance = null

export default class Experience {
	constructor(canvas) {
		// Singleton
		if (instance)
			return instance // If an instance already exists, don't create a new one.
		instance = this // Set the first instance.

		// Global access to the experience instance
		window.experience = this

		// Setting up various components like the camera, renderer, etc.
		this.canvas = canvas
		this.debug = new Debug()
		this.resources = new Recources(Sources)
		this.sizes = new Sizes()
		this.time = new Time()
		this.scene = new THREE.Scene()
		this.camera = new Camera()
		this.renderer = new Renderer()
		this.world = new World()
		this.gestures = new Gestures()
		this.ui = new UI()
		this.waypoints = new Waypoints()
		this.raycaster = new Raycasting()
		this.teleport = new Teleport()
	}
}
```

### **The `new Experience()` Calls**

You might see `new Experience()` in files like `Renderer.js`, but thanks to the Singleton pattern, it doesn’t create a new instance each time.

```js
this.experience = new Experience()
```

* **First call**: Creates and stores the instance in `window.experience`.
* **Later calls**: Just return the same instance.

So, even if it *looks* like a new object, it's always the same shared `Experience`.

> 💡 Even though `new Experience()` is being used, it's not actually creating a new object every time. The Singleton pattern ensures that only the first call creates the instance — later calls just return the same one. This is cleaner than directly using `window.experience`, and avoids hardcoding access to global state across your files.

## **Losing Context?**

Imagine if the **Camera** (or any other part of the app) needed information about the scene, the time, or the canvas but didn’t know where to find it. Without the Singleton, the camera might get lost and not know which `Experience` instance to look for.

Using the Singleton pattern solves this problem. The camera doesn’t need to search for the instance anymore. It can simply **grab the global instance** that’s already set up, avoiding confusion and mistakes.

## **Global (Singleton)**:

With this approach, the `Camera` class directly accesses the global instance of `Experience` (via `window.experience`). This ensures **everyone** in the app uses the same `Experience` instance. It’s a simple way to keep everything in sync.

```js
// Global solution (Singleton Pattern)
this.experience = window.experience // Directly accesses the global instance
```

> 🤔 Why not always use `window.experience` then?

Even though both `window.experience` and `new Experience()` access the same instance (because of the Singleton), calling `new Experience()` is usually preferred:

* Keeps your code cleaner and more modular
* Avoids hard dependencies on global variables
* Centralizes control inside the `Experience` class
* Makes refactoring easier if your architecture changes later

So while both are technically "global solutions," the Singleton version via `new Experience()` is the more flexible and scalable one.

## **Parameter Solution**:

In this method, you manually pass the `Experience` instance around to every class that needs it. While this can work in smaller apps, it becomes harder to manage as the project grows. You might forget to pass the instance, which could lead to bugs.

```js
// Experience.js - Passing the Experience instance to Camera
this.camera = new Camera(this)

// Camera.js - Accepting the Experience instance as a parameter
this.experience = experience // Receives the instance manually
```
