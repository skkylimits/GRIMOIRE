---
title: Zone Transfer
description: Zone transfer is the process of copying a DNS zone from one DNS server to another.
---

Het DNS-systeem is ontworpen met een soort leider/volger-structuur. Er is dus één pimrary DNS-server (de 'leider') en één of meerdere secondary  DNS-servers (de 'volgers').

Alle servers moeten precies dezelfde DNS-gegevens teruggeven. Om dat zo te houden, worden ze op elkaar afgestemd via een proces dat `zone transfer` wordt genoemd.

::warning
Let op: dit is niet hetzelfde als een domainname-transfer — daar gaan we later in op in.
::

Bij een zone transfer vraagt een secondary DNS-server een kopie van alle DNS-gegevens op bij de primary DNS-server.

::note
Deze zone transfer gebeurt via het DNS-protocol zelf. DNS heeft hiervoor een paar speciale recordtypes.
::

## SOA

Een secondary DNS-server vraagt eerst het Start of Authority (SOA) record op van een zone die hij moet volgen. Laten we daar naar kijken.

::tabs
  ::div
  ---
  label: code
  icon: i-heroicons-code-bracket-square
  ---

  ```terminal
  dig zonetransfer.me SOA
  ```
  ::

## nslookup.io

Het is misschien wat verwarrend omdat er veel gegevens zijn zonder uitleg. Daarom gebruiken we [nslookup.io/soa-lookup](https://www.nslookup.io/soa-lookup/) om het overzichtelijker te maken.

![soa-lookup](/content/7.knowledge-base/4.networking/1.dns/1.first-principle/6.zone-transfer/soa-lookup.png)

Dit record bevat configuratiegegevens:

- `Start of Authority`: hier staat waar de primary DNS-server gehost is.

- `E-mailadres`: om de beheerder van de DNS-zone te contacteren.

- `Serienummer`: dit werkt als een versienummer, zodat secondary servers kunnen zien of ze de nieuwste gegevens hebben.

- `Refresh time`: hoe vaak een secondary server moet checken op updates via zone transfer.

- `Retry time`: hoe vaak de server het opnieuw probeert als een zone transfer mislukt.

- `Expire time`: als er binnen deze tijd geen succesvolle transfer is, wordt de zone als "offline" beschouwd en worden de gegevens verwijderd.

## AXFR

Als de secondary server een lager serienummer heeft dan de primary, vraagt hij alle DNS-records voor die zone op. Dit doet hij door een DNS-query van het type `AXFR` te sturen. De primary server antwoordt dan met alle DNS-gegevens voor die zone.

In dit geval zouden we de records opvragen bij de DNS-server nsztm1.digi.ninja. met een AXFR-query.

::tabs
  ::div
  ---
  label: code
  icon: i-heroicons-code-bracket-square
  ---

  ```terminal
  dig zonetransfer.me SOA
  ```
  ::

  ::div
  ---
  label: Preview
  icon: 5:i-heroicons-magnifying-glass-circle
  ---

  ```terminal
  ; <<>> DiG 9.18.30-0ubuntu0.24.04.2-Ubuntu <<>> zonetransfer.me nsztm1.digi.ninja. AXFR
  ;; global options: +cmd
  ;; Got answer:
  ;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 43904
  ;; flags: qr rd ra; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 1

  ;; OPT PSEUDOSECTION:
  ; EDNS: version: 0, flags:; udp: 512
  ;; QUESTION SECTION:
  ;zonetransfer.me.               IN      A

  ;; ANSWER SECTION:
  zonetransfer.me.        7200    IN      A       5.196.105.14

  ;; Query time: 36 msec
  ;; SERVER: 8.8.8.8#53(8.8.8.8) (UDP)
  ;; WHEN: Wed Apr 16 20:36:57 CEST 2025
  ;; MSG SIZE  rcvd: 60

  ; Transfer failed.
  ```
  ::
::

Inderdaad, de zone transfer stuurt alle gegevens voor die zone. Het begint met de SOA-records en eindigt ook met de SOA-records.

::important
Een belangrijk punt om op te merken is dat de transfer gebeurt via TCP. Dit is omdat de specificatie een betrouwbaar protocol vereist voor zone transfers.
::

## Improvements
In de loop der jaren zijn er een paar verbeteringen gemaakt om problemen op te lossen.

> Bijvoorbeeld, de vertraging tussen updates kan ervoor zorgen dat secondary servers verouderde DNS-gegevens blijven tonen, soms wel tot 48 uur. Om dit op te lossen, kunnen DNS-servers de secondary servers waarschuwen wanneer er iets veranderd is. Dit gebeurt via een speciale DNS notify query.

## IXFR-query + Serial Number

Een ander probleem is dat er veel data kan zijn. In dit geval kan incrementele zone transfer helpen. Dit laat de secondary servers alleen de wijzigingen opvragen sinds een bepaald serial number. In plaats van een AXFR gebruiken we dan een IXFR-query en geven we het serienummer op.

## Proprietary Database Replication Mechanisms

::note
Het is goed om te weten dat veel populaire DNS-cloudproviders zone transfers helemaal niet gebruiken. In plaats daarvan gebruiken ze een eigen mechanisme voor database-replicatie, wat meestal beter is dan zone transfer.
::

Toch is het nog steeds handig om te weten dat dit mechanisme bestaat. Het kan nuttig zijn voor het oplossen van DNS-problemen.

Daarnaast kunnen een volledige set domeinnamen en IP-adressen nuttige informatie zijn voor hackers.

::caution
Daarom moeten AXFR- en IXFR-verzoeken alleen worden geaccepteerd als ze afkomstig zijn van een vertrouwd IP-adres. Of ze moeten helemaal worden uitgeschakeld als je server een ander mechanisme heeft voor het repliceren van zonegegevens.
::
