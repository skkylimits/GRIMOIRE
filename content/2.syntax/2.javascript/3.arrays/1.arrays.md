Objects are better for storing unordered data with unique keys, while arrays are better for ordered collections where the position of elements matters.

::important
Objects vs Arrays

- While objects do allow you to add, update, or delete properties, they don't offer methods to specifically manage the order of the elements.

- Arrays, on the other hand, are designed to maintain the order of elements. They have built-in methods like .push(), .pop(), and .splice() that allow you to manage the sequence of elements, including inserting new items at specific positions.
::

## Declaration

**Create an empty array using the two syntaxes**

```js
let arr = new Array();
let arr = [];
```

**Initialize an array with elements**

```js
let fruits = ['Apple', 'Orange', 'Plum']
```

**Access array elements using their index**

```js
alert(fruits[0]) // Apple
alert(fruits[1]) // Orange
alert(fruits[2]) // Plum
```

**Replace an element in the array**

```js
fruits[2] = 'Pear' // now ["Apple", "Orange", "Pear"]
```

**Add a new element to the array**

```js
fruits[3] = 'Lemon' // now ["Apple", "Orange", "Pear", "Lemon"]
```

**Get the length of the array**

```js
alert(fruits.length) // 3
```

**Show the whole array**

```js
alert(fruits) // Apple,Orange,Plum
```

**Arrays can store different types of elements**

```js
let arr = ['Apple', { name: 'John' }, true, function () { alert('hello') }]
alert(arr[1].name) // John
arr[3]() // hello
```

::note
**Use trailing commas for better readability**

```js
let fruits = [
	'Apple',
	'Orange',
	'Plum',
]
```
::


## Accessing the Last Element with `.at()`

::warning
A recent addition

- This is a recent addition to the language. Old browsers may need [polyfills](https://javascript.info/polyfills).
::

Traditionally, to get the last element of an array in JavaScript, we calculate the index manually:

```js
let fruits = ["Apple", "Orange", "Plum"];
alert(fruits[fruits.length - 1]); // Plum
```

This approach can be a bit verbose. JavaScript now supports a cleaner method using `.at()`:

```js
let fruits = ["Apple", "Orange", "Plum"];
alert(fruits.at(-1)); // Plum
```

The `.at()` method works like bracket notation:

- `arr.at(i)` is the same as `arr[i]` for non-negative `i`.
- For negative `i`, it counts from the end of the array.

Note: This is a recent addition to the language. Older browsers may require a polyfill.

::tip
Why `.at(-1)` Works: Understanding Negative Indexing

```js
let fruits = ["Apple", "Orange", "Plum"];
```

Here’s how indexing looks:

```
Index:       0        1        2
Element:   "Apple"  "Orange"  "Plum"
```

`.at(-1)` is a shortcut for accessing the last item in an array.
Remember, `array.length` gives you the total number of **items**, not the index of the last one.
Since arrays are **zero-indexed**, the final item is always at `length - 1`.

That’s why `.at(-1)` works — it’s like saying:
> *“Give me the element 1 step from the end.”*

::

## Methods pop/push, shift/unshift

Arrays in JavaScript are versatile — they can act as both [queues](https://en.wikipedia.org/wiki/Queue_(abstract_data_type)) and [stacks](https://en.wikipedia.org/wiki/Stack_(abstract_data_type)), depending on how you add and remove elements.

### Queue (FIFO: First-In, First-Out)

A **queue** is like a line: the first item added is the first one out.

- `push()` – adds to the **end**
- `shift()` – removes from the **beginning**

![push-shift](/content/2.syntax/2.javascript/3.arrays/1.arrays/push-shift.png)

```js
let queue = ["Message1", "Message2"];
queue.push("Message3");     // Add to end
console.log(queue.shift()); // Remove from start ("Message1")
```

### Stack (LIFO: Last-In, First-Out)

A **stack** is like a pile of plates: the last item added is the first one removed.

- `push()` – adds to the **end**
- `pop()` – removes from the **end**

![push-pop](/content/2.syntax/2.javascript/3.arrays/1.arrays/push-pop.png)

```js
let stack = ["Card1", "Card2"];
stack.push("Card3");      // Add to end
console.log(stack.pop()); // Remove from end ("Card3")
```

### Deque (Double-Ended Queue)

JavaScript arrays also allow adding/removing from **both ends**, making them a [deque](https://en.wikipedia.org/wiki/Double-ended_queue) (double-ended queue):

| Method     | Action                       | End Affected     |
|------------|------------------------------|------------------|
| `push()`   | Add element(s) to the end     | End              |
| `pop()`    | Remove element from the end   | End              |
| `unshift()`| Add element(s) to the start   | Beginning        |
| `shift()`  | Remove element from the start | Beginning        |

---

### Fruits

```js
let fruits = ["Apple"];

// Add multiple elements
fruits.push("Orange", "Peach");
fruits.unshift("Pineapple", "Lemon");

console.log(fruits);
// ["Pineapple", "Lemon", "Apple", "Orange", "Peach"]

// Remove from both ends
console.log(fruits.pop());   // "Peach"
console.log(fruits.shift()); // "Pineapple"
```

> 🔍 `pop()` and `.at(-1)` both access the last item, but `pop()` **removes** it, while `.at(-1)` does **not** modify the array.

## Internals

...what makes arrays really special is their `internal` representation. The engine tries to store its elements in the `contiguous memory area`, one after another, just as depicted on the illustrations in this chapter, and there are other optimizations as well, to make arrays work really fast.

### Reference

- Arrays are **copied by reference** — changes to one reference affect all.

```js
let fruits = ["Banana"]
let arr = fruits; // copy by reference (two variables reference the same array)

alert( arr === fruits ); // true

arr.push("Pear"); // modify the array by reference

alert( fruits ); // Banana, Pear - 2 items now
```

### Avoid Breaking Arrays
For instance, technically we can do this:

```js
let fruits = []; // make an array

fruits[99999] = 5; // assign a property with the index far greater than its length

fruits.age = 25; // create a property with an arbitrary name
```

JavaScript engines optimize arrays for performance — but these optimizations break if we quit working with an array as with an “ordered collection” and start working with it as if it were a regular object.:
- You add non-numeric properties: `arr.test = 5`
- You create holes: only `arr[0]` and `arr[1000]`
- You fill it backwards: `arr[1000]`, then `arr[999]`, etc.

::tip
If you need arbitrary keys, use a plain object `{}` instead.
::

## Performance

Methods like `shift/unshift` are slow, while  `push/pop` run fast.

![array-speed](/content/2.syntax/2.javascript/3.arrays/1.arrays/array-speed.png)

### Beginning

⛔ Slow for large arrays due to more memory operations.

Working with the **end** of an array (`push`, `pop`) is **faster** than working with the **beginning** (`shift`, `unshift`).

```js
fruits.shift()
// Removes the first element
// Reindexes all remaining elements
```

::note
It’s not enough to take and remove the element with the index 0. Other elements need to be renumbered as well.
::

The `shift` operation must do 3 things:

1. Remove the element with the index `0`.
2. Move all elements to the left, renumber them from the index `1` to `0`, from `2` to `1` and so on.
3. Update the `length` property.

![array-shift](/content/2.syntax/2.javascript/3.arrays/1.arrays/array-shift.png)

::important
The more elements in the array, the more time to move them, more in-memory operations.
::

### End

✅ Fast and efficient.

What’s with `push/pop`? They do not need to move anything. To extract an element from the end, the `pop` method cleans the index and shortens `length`.

![array-pop](/content/2.syntax/2.javascript/3.arrays/1.arrays/array-pop.png)

::important
The `pop/push ` method does not need to move anything, because other elements keep their indexes. That’s why it’s blazingly fast.
::

## A Word About "Length"

The `length` property of an array is the greatest numeric index plus one, not the count of elements. It's writable and can be used to truncate or clear an array, but reducing it is irreversible.

### Large Index Changes Length

```js
let fruits = [];
fruits[123] = "Apple";

alert(fruits.length); // 124
```

### Truncating Array

> Shortening the array.

```js
let arr = [1, 2, 3, 4, 5];

arr.length = 2;
alert(arr); // [1, 2]

arr.length = 5;
alert(arr[3]); // undefined
```

::warning
Once you truncate an array by manually setting its length property, there is no way to restore the deleted elements
::

::foldable
#title
Why would you truncate arrays?

#content
- **Remove excess data**: Limit the array to the first N items, improving memory and performance.
- **Clear unnecessary or outdated data**: Reset an array to an empty state without deleting the reference.
- **Efficient memory management**: Truncate an array to free up memory by removing unused elements.
- **Simulate stack behavior**: Keep only the most recent N items in the array, discarding older ones.
- **Reset state**: After processing data, truncate the array to prepare for the next batch.
- **Fix array length errors**: Correctly set the array length by truncating it to the desired size.
::

## new Array()

::caution
Avoid `new Array()` unless necessary, as it can be confusing. Use `[]` for array creation to prevent surprises.
::

There is one more syntax to create an array:

```js
let arr = new Array("Apple", "Pear", "etc");
```

::caution
**Tricky Case**: `new Array(2)` creates an array with **length 2**, but with no elements inside.

```js
let arr = new Array(2);

console.log(arr);         // [ <2 empty items> ]
console.log(arr[0]);      // undefined
console.log(arr.length);  // 2

arr.forEach(x => console.log(x)); // Nothing logs!
```
If you're expecting a usable array of values (like [undefined, undefined]), you’ll get strange behavior when you try to loop or process it.

Use `Array.from` instead if you really need it.

```js
let arr = Array.from({ length: 2 });
```
::

## Multidimensional arrays

Arrays can have items that are also arrays. We can use it for multidimensional arrays, for example to store matrices:

```js
let matrix = [
  [1, 2, 3],
  [4, 5, 6],
  [7, 8, 9]
];

alert( matrix[0][1] ); // 2, the second value of the first inner array
```



## `toString()` Behavior

Think of an array like a box of items. When JavaScript sees something like:

```js
[] + 1
```

It goes:

> “Hmm, I can’t add an array and a number… better turn them into strings!”

What is happening here:
1. `[]` becomes a string using `toString()` → `""` (an empty string)
2. Then it sees `"" + 1` → and that’s string concatenation!
3. So you get:
   ```js
   "" + 1 = "1"
   ```

```js
let arr = [1, 2, 3];

alert(arr); // "1,2,3"
alert(String(arr) === "1,2,3"); // true
```


➕ When Arrays Are Added to Values

Arrays don't have `Symbol.toPrimitive` or a useful `valueOf`, so JavaScript falls back to `toString()`:

```js
alert([] + 1);     // "1"   → "" + 1 → "1"
alert([1] + 1);    // "11"  → "1" + 1 → "11"
alert([1, 2] + 1); // "1,21" → "1,2" + 1 → "1,21"
```

This often leads to **weird coercion** results when using `+`, especially with arrays.

## Don’t Compare Arrays with `==`

In JavaScript, comparing arrays using `==` or even `===` doesn’t compare the contents — it just checks if both variables point to the *exact same array in memory*.

### Think of It Like This:

Imagine two identical houses built in different neighborhoods — they *look* the same, but they’re not the same house. Same with arrays.

```js
[] == []         // false
[0] == [0]       // false
```

Even though they “look” the same, they are separate objects in memory → so JavaScript says ❌ *not equal*.

### So What *Does* `==` Do?

The `==` operator:
- Checks if two values are the *same* object (for non-primitives)
- Converts types if one side is a primitive

But it **doesn’t** compare arrays *by content*.

### So What *Does* `==` Do? {#no-toc}

When comparing an array with a primitive, JavaScript tries to be “helpful” and converts the array to a primitive string:

```js
0 == []        // true → [] becomes "" → 0 == "" → true
'0' == []      // false → '0' vs "" → different strings
```

So yeah, it can get weird. Arrays become `""`, and the comparison ends up being between strings or numbers.

For more details on object-to-primitive conversions and type conversions, check out these articles:

- [Object to Primitive Conversion](https://javascript.info/object-toprimitive)
- [Type Conversions](https://javascript.info/type-conversions)

### What Should You Do Instead?

If you want to check if two arrays have the same contents, you need to:

- Compare **each element manually**
- OR use a method like `.every()` or `.toString()` or `JSON.stringify()` (carefully!)

Example:

```js
[1, 2, 3].toString() === [1, 2, 3].toString() // true ✅
```

But watch out — this breaks if order or types vary even slightly.

### TL;DR:

- `==` and `===` only say arrays are equal if they’re the *exact same object*
- Comparing contents? You’ll need to write your own logic (or use a library)
