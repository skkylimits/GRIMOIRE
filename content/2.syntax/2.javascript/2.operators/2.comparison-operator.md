## Comparisons

In JavaScript, comparison operators are used to compare values and return a boolean result (`true` or `false`).

- **Greater than**: `a > b`
- **Less than**: `a < b`
- **Greater than or equal to**: `a >= b`
- **Less than or equal to**: `a <= b`
- **Equal to**: `a == b`
  ::caution
  The double equality sign `==` checks for equality, while the single equality sign `=` is used for assignment.
  ::
- **Not equal to**: `a != b`
  ::caution
  In mathematics, the notation for not equal is `≠`, but in JavaScript, it's written as `!=`.
  ::

## Boolean is the Result

All comparison operators return a **boolean** value:

- `true` – means “yes”, “correct” or “the truth”.
- `false` – means “no”, “wrong” or “not the truth”.

**Examples:**

```js
alert(2 > 1) // true (correct)
alert(2 == 1) // false (wrong)
alert(2 != 1) // true (correct)
```

A comparison result can also be **assigned to a variable**:

```js
let result = 5 > 4 // assign the result of the comparison
alert(result) // true
```

## String Comparison

To compare strings, JavaScript uses **lexicographical (dictionary-like) order**, comparing characters one by one.

**Examples:**

```js
alert('Z' > 'A') // true
alert('Glow' > 'Glee') // true
alert('Bee' > 'Be') // true
```

**How it works:**

1. Compare the first characters of both strings.
2. If they differ, that determines the result.
3. If they’re the same, move to the next character.
4. Continue until one string ends or a difference is found.
5. If strings are the same until the end but one is longer, the longer string is considered greater.

::note
Not a Real Dictionary, but [Unicode Order](https://www.unicode.org/charts/)

- JavaScript compares strings using **Unicode values**, not traditional dictionary rules.

That means:

- **Case matters**: `'A'` is not equal to `'a'`
- `'a'` is **greater** than `'A'` because it has a higher Unicode value

So while the algorithm *feels* like dictionary order, it's really based on each character's position in the Unicode table.

We'll explore more about this in the chapter on **Strings**.
::

## Comparison of different types

### Type Coercion in Comparisons

When comparing **different types**, JavaScript will **convert values to numbers**

```js
alert('2' > 1) // true → '2' becomes number 2
alert('01' == 1) // true → '01' becomes number 1
```

🧠 **Equality (`==`) and comparison (`>`, `<`) convert operands to numbers when types differ.**

### Boolean Conversion

For boolean values, `true` becomes `1` and `false` becomes `0`.

```js
alert(true == 1) // true → true becomes 1
alert(false == 0) // true → false becomes 0
```

::warning
This creates a quirky case:

```js
let a = 0
alert(Boolean(a)) // false → 0 is falsy

let b = '0'
alert(Boolean(b)) // true → non-empty string is truthy

alert(a == b) // true → "0" becomes 0 for ==, but still truthy as a string
```

- `a` and `b` are equal (`a == b`)
- But `Boolean(a)` is `false`, and `Boolean(b)` is `true`
> This happens because one using `type coercion`, and the other `boolean conversion`.
::

#### 🧪 Empty vs non-empty strings

```js
Boolean('') // false → empty string
Boolean(' ') // true  → space is a character
Boolean('0') // true  → non-empty string
Boolean('false') // true  → string ≠ boolean false
Boolean('null') // true  → still a string
```

::tip
One-liner to test different values

```js
['', '0', '1', 'false', ' ', 0, 1, null, undefined, Number.NaN, [], {}]
	.forEach(val => console.log(`${JSON.stringify(val)} →`, Boolean(val)))
```
::

## Strict Equality in JavaScript

The regular equality operator `==` can have unexpected behavior because it performs type coercion.

For example:

```js
alert( 0 == false ); // true → both are converted to 0
alert( '' == false ); // true → empty string is converted to 0
```

To differentiate between values like `0` and `false`, or an empty string and `false`, we use the **strict equality** operator `===`.

```js
alert( 0 === false ); // false → types are different, no type coercion
```

There is also a **strict non-equality** operator `!==`, which checks for inequality without type conversion.

::tip
In modern JavaScript development, `===` is preferred for most cases, but =`= can be useful in certain situations where you expect type coercion.
::

## Comparison with `null` and `undefined`

There are some quirky behaviors when comparing `null` and `undefined` with other values in JavaScript.

1. **Strict equality (`===`)**:
   - `null` and `undefined` are considered different types, so they are not equal.

   ```javascript
   alert( null === undefined ); // false
   ```

2. **Non-strict equality (`==`)**:
   - **`null` and `undefined` are equal** to each other, but they are not equal to any other value.

   ```javascript
   alert( null == undefined ); // true
   ```

3. **Comparisons (`<`, `>`, `<=`, `>=`)**:
   - Both `null` and `undefined` are converted to numbers when compared, but they behave differently:
     - `null` is treated as `0`.
     - `undefined` is treated as `NaN`, which is **not a number** and results in false comparisons.

   For example:

   ```javascript
   alert( null > 0 );  // false
   alert( null == 0 ); // false
   alert( null >= 0 ); // true

   // This behavior can be surprising because `null >= 0` is true, but `null > 0` is false.
   ```

4. **Undefined comparisons**:
   - When comparing `undefined` with other values, we get strange results because `undefined` is converted to `NaN`, and `NaN` always returns false in comparisons:

   ```javascript
   alert( null > 0 );  // false → null is treated as 0, 0 is not greater than 0
   alert( null == 0 ); // false → null is not equal to 0
   alert( null >= 0 ); // true → null is treated as 0, and 0 is equal to 0 (>= includes equality)
   ```

## Avoid Problems

You don’t need to memorize all these quirks, but here's how to avoid issues:

- Always use strict equality (`===`) when comparing `null` or `undefined`.
- Avoid comparisons (`>=`, `>`, `<`, `<=`) with variables that could be `null` or `undefined` unless you’re certain. Check for them separately first.
