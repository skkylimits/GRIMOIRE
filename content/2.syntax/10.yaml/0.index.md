# YAML — korte maar complete gids

> YAML = “YAML Ain’t Markup Language”. Een leesbare data-serialisatie­taal die vaak wordt gebruikt voor config-bestanden (CI, k8s, Ansible, Docker Compose, enz.).

## 1) Basisprincipes & syntaxis

* YAML is **whitespace-significant**: inspringen (spaces) geeft structuur. Gebruik **altijd spaces** (geen tabs).
* Standaard extensie: `.yml` of `.yaml`.
* Twee representatiestijlen: **block style** (meest gebruikt) en **flow style** (inline, JSON-achtig).

Voorbeeld (mappings & sequences):

```yaml
# mapping (dictionary)
app:
  name: myapp
  version: 1.2.3

# sequence (lijst)
servers:
  - host: 10.0.0.1
    role: db
  - host: 10.0.0.2
    role: web
```

## 2) Scalers: strings, numbers, booleans, null

* Strings: zonder quotes mogelijk, maar let op speciale tekens/spaties.
* Numbers: `123`, `-42`, `3.14` (automatisch getypeerd bij deserialisatie).
* Booleans: `true`, `false`, ook `yes/no`, `on/off` (maar deze aliasing kan verrassend zijn).
* Null: `null` of `~`.

Aanbeveling: quote strings die kunnen worden geïnterpreteerd als bool/num/date (bv. `"on"`, `"yes"`, `"2020-01-01"`).

## 3) Indentation rules

* Gebruik consistente indentatie: typisch 2 of 4 spaces per niveau.
* Geen tabs! YAML-parsers weigeren of behandelen tabs verschillend.
* Indents moeten consistent zijn binnen hetzelfde bestand.

## 4) Strings: plain, single-quoted, double-quoted, multiline

* Plain: `name: bob`
* Single quoted (`'`): geen escape sequences, handig voor tekens met `:` of `#`.
* Double quoted (`"`): ondersteunt escape sequences (`\n`, `\t`, `\uFFFF`).
* Multiline block (`|` of `>`):

  * `|` = behoudt nieuwe regels (literal block)
  * `>` = vouwt regels (folded block — nieuwe regels worden spaties)

Voorbeeld:

```yaml
literal: |
  Dit is regel 1
  Dit is regel 2

folded: >
  Dit is een lange regel
  die in YAML over meerdere regels staat
  maar bij parsing als één regel wordt gezien.
```

## 5) Anchors & Aliases (DRY)

* `&anchor` definieert; `*alias` hergebruikt.
* `<<: *anchor` kan gebruikt worden om mappings te mergen (merge-key).

Voorbeeld:

```yaml
defaults: &defs
  retries: 3
  timeout: 10

service:
  <<: *defs
  name: svc1
```

Let op: sommige parsers/versies ondersteunen merge-keys niet overal (maar de meeste moderne doen het).

## 6) Tags & types

* YAML ondersteunt expliciete tags: e.g. `!!str`, `!!int`, `!!map`, `!custom`.
* Tags geven aan hoe een waarde geïnterpreteerd moet worden door de parser.

Voorbeeld:

```yaml
value: !!str 0123    # forceer string, niet int
```

## 7) Comments

* `# comment` tot einde van lijn. Geen multi-line comment syntax.

## 8) Flow style vs Block style

* Flow style = JSON-achtige inline notatie: `{ key: value, arr: [1,2,3] }`
* Block style = indent-gebaseerd (voorkeur bij configs).

## 9) YAML vs JSON

* JSON is geldig YAML (voor subset).
* YAML is mensvriendelijker, ondersteunt comments, anchors en multi-line.
* Gebruik JSON wanneer strictness en simpele tooling nodig is; YAML voor configs die je wilt redigeren.

## 10) Common pitfalls & gotchas

* **Tabs** — nooit gebruiken.
* **Unquoted yes/no/on/off** — kunnen booleans zijn; quote strings to be safe.
* **Dates** vaak automatisch geparsed (bijv. `2019-01-01`) — quote als je een string wilt.
* **Leading zeros**: `0123` kan als octal of string geïnterpreteerd — quote als string.
* **Aliasing loops** — be careful with circular aliases.
* **Inconsistent indentation** → parse error.
* **Security**: onbeveiligd YAML laden (bijv. PyYAML `yaml.load` met default) kan leiden tot arbitrary object instantiation. Gebruik veilige loader (`safe_load`) in Python en vergelijkbare veilige opties in andere libs.

## 11) Security — wat moet je stoppen?

* **Never** `load` untrusted YAML with unsafe loaders (Python: `yaml.load` zonder `Loader=SafeLoader`).
* YAML kan object tags die code/injecties toelaten (in sommige implementaties).
* Voor services: valideer schema (bijv. JSON Schema of custom validators) en beperk toegestane tags.

## 12) Tools & libraries

* Python: `PyYAML` (`safe_load`), `ruamel.yaml` (preserve comments/order).
* Go: `gopkg.in/yaml.v3`.
* JS/Node: `js-yaml`.
* CLI: `yq` (yaml processor, jq-achtige CLI), `yamllint`, `kubectl` heeft YAML helpers.
* Editors: VSCode met YAML plugin (schema validation), YAML linters.

## 13) Best practices (voor configs / infra)

* Gebruik consistent indentation (2 of 4 spaces).
* Quote ambiguous scalars (`"on"`, `"no"`, `0123`, `2019-01-01`).
* Gebruik anchors voor herbruikbaarheid, maar niet voor complexiteit.
* Geef voorkeur aan block style voor leesbaarheid.
* Valideer met schema (JSON Schema, OpenAPI, or custom) in CI.
* Versiebeheer: klein, leesbaar commits — be careful with large auto-generated YAML (helm charts enz.).
* Avoid extremely large single YAML files; split into logical smaller files.

## 14) Quick cheatsheet — parse-proof patterns

```yaml
# safe types, quoted where ambiguous
name: "my-app"
replicas: 3
enabled: true
start_date: "2020-01-01"    # quote so not parsed as date
notes: |
  Multiline notes...
servers:
  - host: "10.0.0.1"         # quote IPs if you want strings
    roles: ["web","api"]     # flow-style list okay for short lists
```

## 15) Validatie & CI

* Lint in CI with `yamllint`.
* Validate schema with JSON Schema (many tools convert YAML->JSON and validate).
* Use `ruamel.yaml` if you need to preserve comments/ordering when round-tripping.

## 16) Short examples (k8s style and merge)

Kubernetes-like example with anchors:

```yaml
defaults: &defaults
  imagePullPolicy: IfNotPresent
  resources:
    limits:
      cpu: "1"
      memory: "512Mi"

deployment:
  metadata:
    name: web
  spec:
    replicas: 3
    template:
      spec:
        containers:
          - name: app
            image: myapp:latest
            <<: *defaults
```

## 17) When **not** to use YAML

* When configs are machine-generated and never human-edited — use JSON or binary formats.
* When security requires strict schema enforcement and no complex types — prefer JSON + schema.
