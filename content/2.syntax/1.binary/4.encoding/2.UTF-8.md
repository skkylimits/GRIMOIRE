> From ASCII to a Universal Alphabet

Zonder `UTF-8` zouden talen, emoji en symbolen nog steeds in aparte werelden leven.

## Het probleem dat Unicode oplost

`ASCII` werkte goed voor Engels, maar de wereld is groter dan 128 tekens.
Talen als Chinees, Arabisch of Hindi hebben **duizenden symbolen**.

Om dat te fixen werd in `1992` `Unicode` ontworpen:
één systeem dat elk karakter op aarde een uniek nummer geeft — een `code point`.

```text
U+0041 → 'A'
U+03A9 → 'Ω'
U+4E2D → '中'
U+1F600 → 😀
```

Elk karakter heeft dus een **Unicode-nummer**, onafhankelijk van hoe het wordt opgeslagen.

## Unicode vs Encoding

`Unicode` is geen bestandsformaat — het is een **concept**.

Het zegt:

> “Dit karakter = codepoint U+0041.”

Maar om dat karakter **in bytes op te slaan**, heb je een **encoding** nodig.

Er bestaan meerdere manieren om Unicode in bits te zetten:

* `UTF-8`
* `UTF-16`
* `UTF-32`

## Wat doet UTF-8 precies

`UTF-8` (*Unicode Transformation Format 8-bit*) zet elk Unicode-codepoint om naar **1 tot 4 bytes**.
Die bytes beginnen altijd met speciale “startbits” die aangeven **hoeveel bytes** bij dat karakter horen.

💡 De eerste `128` tekens (`0–127`) zijn identiek aan `ASCII`.
Dat betekent: elke ASCII-tekst **is automatisch geldige UTF-8**.

| Code range (Unicode) | Bytes |           Startbits patroon           | Voorbeeld          |
| :------------------: | :---: | :-----------------------------------: | :----------------- |
|   `U+0000 – U+007F`  |   1   |               `0xxxxxxx`              | `A → 41`           |
|   `U+0080 – U+07FF`  |   2   |          `110xxxxx 10xxxxxx`          | `é → C3 A9`        |
|   `U+0800 – U+FFFF`  |   3   |      `1110xxxx 10xxxxxx 10xxxxxx`     | `中 → E4 B8 AD`     |
| `U+10000 – U+10FFFF` |   4   | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` | `😀 → F0 9F 98 80` |


## Wat is een [`codepoint`](https://unicodemap.com/search.asp?)?

Een **codepoint** is simpelweg het unieke nummer dat Unicode Consortium toekent aan elk teken (zoals een letter, cijfer of symbool). Het wordt genoteerd in de vorm `U+XXXX`

Bijvoorbeeld: de letter **A** heeft code point `U+0041`.

## Zo codeert UTF-8 een `codepoint`

**Stel je voor 🎨**

We willen het lettertje **é** maken.
Elke letter heeft in de computer een **nummer**.
Voor **é** is dat **U+00E9** — dat is gewoon een getal.

### 1. We maken het nummer van é in blokjes

Dat getal (`U+00E9`) in binaire vakjes is:

```
0000 0000 1110 1001
```

(Je mag de eerste nullen wegdenken, dus we houden over:)

```
11101001
```

Dat zijn de “echte bits” van het teken.

### 2.  UTF-8 zegt — dit past niet in 1 doosje!

Omdat `é` groter is dan `127` (0x7F), past het niet in **1 byte**.
Dus UTF-8 zegt:

> “Oké, dan gebruiken we **2 doosjes**!”

En die 2 doosjes hebben vaste labels:

```
110xxxxx 10xxxxxx
```

De **x’jes** zijn lege vakjes waar we bits in moeten stoppen.
De andere stukjes (110 en 10) zijn **stickers** die zeggen:

> “Hé, dit is een 2-byte teken!”

### 3. Bits in de doosjes stoppen

We hebben 8 bits van ons teken:

```
11101001
```

Maar we hebben 11 vakjes (x’jes) in totaal:

```
xxxxx xxxxxx → 11 vakjes
```

Dus we plakken **drie nullen vooraan** om alle vakjes te vullen:

```
000 11101001
```

Dat zijn dus **11 bits**:

```
0 0 0 1 1 1 0 1 0 0 1
```

### 4. Verdeel ze in de 2 doosjes

We stoppen ze van **rechts naar links** in de x-vakjes:

```
110xxxxx 10xxxxxx
```

Nu vullen we ze in:

| Doosje | Vorm     | Gevulde bits | Wat we krijgen |
| :----- | :------- | :----------- | :------------- |
| 1      | 110xxxxx | 11000011     | C3             |
| 2      | 10xxxxxx | 10101001     | A9             |

### 5.  Klaar!

Samen vormen die twee doosjes:

```
11000011 10101001
```

Of in hex:

```
C3 A9
```

👉 Dat is hoe `é` eruitziet in UTF-8!

::note
Zo weet UTF-8 altijd hoeveel bytes een teken gebruikt: de **eerste bits van de eerste byte** zeggen alles.
::

## Waarom ASCII een subset is

ASCII gebruikt maar 7 bits (`0–127`), wat past in het eerste UTF-8 patroon:

```
0xxxxxxx
```

Daarom is:

```text
'A' = 65₁₀ = 01000001₂ = 41₁₆
```

geldige ASCII én geldige UTF-8 —
de rest van de bytes zijn nooit nodig.

## Voorbeelden in context

```text
"A"   → U+0041 → 01000001 → 41
"é"   → U+00E9 → C3 A9
"中"  → U+4E2D → E4 B8 AD
"😀"  → U+1F600 → F0 9F 98 80
```

In binaire vorm:

```text
A  = 01000001
é  = 11000011 10101001
中 = 11100100 10111000 10101101
😀 = 11110000 10011111 10011000 10000000
```

## Waarom UTF-8 won

* `Backwards compatible` met `ASCII`
* Ondersteunt alle `Unicode`-tekens
* Compact — Engelse tekst = 1 byte per karakter
* Universeel — websites, JSON, e-mails, terminals → allemaal UTF-8

## Is Unicode base16?

Een veelvoorkomende misvatting is dat `Unicode` een `base16` (hexadecimaal) systeem zou zijn.
Dat klopt niet helemaal.

`Unicode` zelf is **geen getalsysteem** — het is een **concept** dat elk karakter op aarde
een uniek nummer geeft, het zogeheten `code point`.

Wat wél waar is:
Unicode-codepoints worden **bijna altijd weergegeven in hexadecimale notatie**,
omdat dat korter en leesbaarder is dan decimale of binaire getallen.

```text
A   → 65₁₀ → U+0041₁₆
é   → 233₁₀ → U+00E9₁₆
中  → 20013₁₀ → U+4E2D₁₆
😀  → 128512₁₀ → U+1F600₁₆
```

Die `U+XXXX`-notatie is dus enkel een **schrijfwijze in hex**,
niet een aanduiding dat Unicode zelf “base16” is.

### 🧩 Vergelijking

| Term              | Betekenis                                                      | Base      |
| :---------------- | :------------------------------------------------------------- | :-------- |
| `Unicode`         | Concept dat elk karakter een uniek nummer geeft                | Geen base |
| `U+XXXX`          | Hexadecimale notatie van dat nummer                            | Base 16   |
| `UTF-8 / 16 / 32` | De binaire encodings waarmee Unicode in bytes wordt opgeslagen | Base 2    |


::important
**Kort gezegd:**
- Unicode = de kaart met alle karakters
- Hex (`U+1F600`) = hoe wij die nummers opschrijven
- UTF-8/16/32 = hoe computers ze als bits bewaren
::

```text
Character     Unicode    UTF-8 bytes   Binary
A             U+0041     41            01000001
é             U+00E9     C3 A9         11000011 10101001
中            U+4E2D     E4 B8 AD      11100100 10111000 10101101
😀            U+1F600   F0 9F 98 80    11110000 10011111 10011000 10000000
```

> `UTF-8` is als een slimme vertaler:
> klein en snel voor Engels, maar krachtig genoeg voor elke taal op aarde.

## UTF-16 en UTF-32?

`UTF-8` is de populairste Unicode-encoding,  
maar het is niet de enige manier om tekens in bytes te zetten.

Sommige systemen gebruiken nog steeds **UTF-16** of **UTF-32**,  
vooral binnen oudere Windows- en Java-omgevingen.

### 🧩 UTF-16

- Gebruikt **2 of 4 bytes per karakter**  
- Handig voor talen met veel niet-Latijnse tekens (Aziatische scripts)  
- Minder efficiënt voor Engels (want alles neemt minstens 2 bytes in)

```text
'A'   → 0041
'é'   → 00E9
'中'  → 4E2D
'😀'  → D83D DE00  (twee 16-bit “surrogate” waarden)
````

> 💡 `UTF-16` gebruikt *surrogates* — twee 16-bit waarden om codepoints boven `U+FFFF` weer te geven.

### 🧩 UTF-32

* Elk karakter krijgt **altijd 4 bytes**
* Simpel, want elke Unicode-codepoint = exact één waarde
* Maar inefficiënt: Engelse tekst wordt 4× zo groot

```text
'A'   → 00000041
'é'   → 000000E9
'中'  → 00004E2D
'😀'  → 0001F600
```

> UTF-32 is vooral nuttig in systemen waar snelheid of eenvoud belangrijker is dan opslagruimte —
> bijvoorbeeld in interne verwerking, compilers of fonts.

### ⚙️ Vergelijking

| Encoding | Bytes per teken | Voordeel                      | Nadeel                         |
| -------- | --------------- | ----------------------------- | ------------------------------ |
| UTF-8    | 1–4             | Compact, compatibel met ASCII | Variabele lengte               |
| UTF-16   | 2–4             | Goed voor Aziatische talen    | Surrogates, endian-afhankelijk |
| UTF-32   | Altijd 4        | Eenvoudig, direct             | Verspilt geheugen              |

::tip
💬 **Kort gezegd:**
> `UTF-8` is de wereldtaal,  
> `UTF-16` is een oud compromis,  
> en `UTF-32` is de brute-krachtversie voor specialisten.
::

## Encoding Aware

Als *de wereld* UTF-8 gebruikt, is dat je eerste en belangrijkste target — maar in de praktijk **moet** je encoding-aware zijn: bepaal wat het doel intern gebruikt en genereer precies díe bytes. UTF-32 wordt vaak gekozen tijdens ontwikkeling of prototyping omdat het fixed-width en simpel is — niet omdat het meer voorkomt op het netwerk.
