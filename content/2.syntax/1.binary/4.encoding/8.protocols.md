> From File to Flow

## 🧠 Wat is een Protocol?

Waar een *file format* structuur geeft aan data **op schijf**,  
geeft een *protocol* structuur aan data **in beweging**.

Een protocol definieert:
- hoe een bericht wordt opgebouwd (headers, payload, terminators)  
- in welke volgorde het wordt verzonden  
- hoe de ontvanger bevestigt, reageert of fouten afhandelt  

```text
[Data] → [Encoded] → [Formatted] → [Transmitted via Protocol]
```

::caution
Een protocol is geen software, maar een **afspraak**:
zolang beide partijen zich eraan houden, kunnen ze praten.
::

## 🌍 Waarom protocollen bestaan

Machines praten niet in emoties of intuïtie, maar in **regels en bytes**.
Zonder protocol is er geen zekerheid dat de andere kant begrijpt wat je stuurt.

Bijvoorbeeld:

* `HTTP` regelt hoe webpagina’s over het internet reizen.
* `SMTP` regelt hoe e-mails worden verzonden.
* `DNS` vertaalt domeinnamen naar IP-adressen.
* `TCP/IP` vervoert alles onder de motorkap.

## 🧱 Protocol Layers (OSI-model)

Elk protocol leeft op een **niveau in de communicatielaag**:

```
┌───────────────────────────────┐
│ Application  (HTTP, DNS, TLS) │ ← Data formats & content rules
├───────────────────────────────┤
│ Transport    (TCP, UDP)       │ ← Betrouwbare overdracht van bytes
├───────────────────────────────┤
│ Network      (IP, ICMP)       │ ← Adressering en routering
├───────────────────────────────┤
│ Data Link    (Ethernet, ARP)  │ ← Fysieke verbinding binnen één LAN
├───────────────────────────────┤
│ Physical     (Cables, Radio)  │ ← Signalen en elektrische transmissie
└───────────────────────────────┘
```

::caution
De meeste *encodings* die je eerder leerde  
— zoals Base64, URL en UTF-8 — spelen zich af in de **Application Layer**.  
Ze zorgen dat data daar *leesbaar en compatibel* blijft.
::

## 📦 Hoe data reist (voorbeeldflow)

```text
1. User types message:
   "Hello 🌍"

2. Encoding:
   UTF-8 → 48 65 6C 6C 6F 20 F0 9F 8C 8D

3. Formatting:
   JSON → {"msg":"Hello 🌍"}

4. Protocol packaging:
   HTTP → POST / GET 
   Content-Type: application/json; 
   charset=UTF-8

5. Transport:
   TCP segments the bytes → sends over IP

6. Reception:
   Browser/server reconstructs → parses JSON → decodes UTF-8 → displays "Hello 🌍"
```

::caution
Elke laag begrijpt **alleen zijn eigen niveau**.
HTTP weet niets van bits.
IP weet niets van tekst.
Maar samen vormen ze communicatie.
::

## 🧩 Protocol Anatomy

```text
┌──────────────────────────────────────────────┐
│ Header  → metadata, version, length, type    │
├──────────────────────────────────────────────┤
│ Payload → de daadwerkelijke data             │
├──────────────────────────────────────────────┤
│ Footer  → checksums, terminators, padding    │
└──────────────────────────────────────────────┘
```

Voorbeeld (HTTP):

```text
POST /api/login HTTP/1.1
Host: example.com
Content-Type: application/json; charset=UTF-8
Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

{"user":"Aladdin"}
```

## 🔐 Encoding binnen protocollen

| Protocol | Encodingtype         | Voorbeeld                                           |
| :------- | :------------------- | :-------------------------------------------------- |
| **HTTP** | Charset + Base64     | `Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==` |
| **SMTP** | MIME/Base64          | Mail attachments                                    |
| **DNS**  | ASCII-labels         | `www.example.com`                                   |
| **FTP**  | ASCII / Binary modes | File transfer types                                 |
| **TLS**  | Binary framing       | Secure encapsulation                                |

::caution
Encoding/Protocol   
`Encoding` is de *taal van data*,  
`Protocol` is de *grammatica van communicatie*.
::

## 🔄 Protocols vs Formats

| Concept      | Doel                                | Voorbeeld                 |
| :----------- | :---------------------------------- | :------------------------ |
| **Format**   | Structureert data op schijf         | `PNG`, `Markdown`, `JSON` |
| **Protocol** | Structureert data tijdens transport | `HTTP`, `SMTP`, `DNS`     |

```text
File Format → zegt "Wat is dit?"
Protocol    → zegt "Hoe verzend ik dit?"
```

## 📡 Binary & Text Protocols

| Type           | Voorbeelden          | Opmerkingen                              |
| :------------- | :------------------- | :--------------------------------------- |
| **Text-based** | HTTP, SMTP, DNS, FTP | Mens-leesbaar, gebruikt CRLF (`\r\n`)    |
| **Binary**     | TLS, SSH, SMB, QUIC  | Compact, sneller, moeilijker te debuggen |

::caution
Text-based protocollen zijn geweldig om te leren *zien wat er gebeurt*.
Binary protocollen tonen pas hun magie bij *reverse engineering*.
::

## 🧠 Samenvatting

| Niveau      | Wat gebeurt er?          | Voorbeeld           |
| :---------- | :----------------------- | :------------------ |
| Bits        | 0 en 1 over de kabel     | `01010110`          |
| Encoding    | Bytes → tekst            | UTF-8, Base64       |
| Format      | Tekst → betekenis        | JSON, HTML          |
| Protocol    | Betekenis → communicatie | HTTP, SMTP          |
| Application | Actie → ervaring         | Browser, Mailclient |

**Encoding = hoe iets wordt geschreven.**
**Protocol = hoe iets wordt verstuurd.**
::

## 🔎 Zoekstrategie RFCs

Begin bij de [RFC Editor Search](https://www.rfc-editor.org/search) om het *basisdocument* te vinden,  
zoek daarna op **keyword** in [IETF Datatracker](https://datatracker.ietf.org/)  
en controleer via *“Obsoletes”* en *“Updated by”* welke RFC de **laatste en geldige** versie is.


## ⚙️ Waarom dit ertoe doet

Als security engineer of reverser moet je:

* een **file** herkennen aan z’n *magic bytes*
* de **encoding** herkennen waarin z’n inhoud zit
* en het **protocol** herkennen waarmee hij beweegt

::caution
Want alleen door de taal én de route te begrijpen,  
kun je een aanval zien, blokkeren of nabootsen.
::
