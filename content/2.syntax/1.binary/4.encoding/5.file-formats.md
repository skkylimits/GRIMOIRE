> From Encoded Bytes to Meaningful Structure

::caution
File Format  
Een *file format* bepaalt **hoe data in een bestand is gestructureerd en geïnterpreteerd**.  
Waar `encoding` beschrijft *hoe bytes worden opgeslagen*, beschrijft een *format* **wat die bytes betekenen**.
::

## 🧠 Wat is een File Format?

Een bestand is niet zomaar een reeks bytes — het volgt een **specifieke afspraak** over:
- welke data erin staat,  
- hoe die data is gescheiden,  
- en hoe programma’s die data moeten **begrijpen**.

```text
[Bytes] → [Encoding] → [Format] → [Program Meaning]
```

Bijvoorbeeld:

| Bestand       | Encoding | Format                   | Betekenis                        |
| :------------ | :------- | :----------------------- | :------------------------------- |
| `index.html`  | UTF-8    | HTML                     | Webpagina-structuur              |
| `style.css`   | UTF-8    | CSS                      | Opmaakregels                     |
| `notes.md`    | UTF-8    | Markdown                 | Tekst + semantische markup       |
| `logo.png`    | Binary   | PNG                      | Bitmap-afbeelding met compressie |
| `program.exe` | Binary   | PE (Portable Executable) | Windows-applicatie               |

---

## 🧩 Text-based vs Binary Formats

| Type           | Kenmerk                                         | Voorbeeld                               | Leesbaar door mens? |
| :------------- | :---------------------------------------------- | :-------------------------------------- | :------------------ |
| **Text-based** | Gebruikt een *character encoding* (zoals UTF-8) | `.html`, `.json`, `.xml`, `.csv`, `.md` | ✅ Ja                |
| **Binary**     | Opgebouwd uit binaire structuren en headers     | `.png`, `.exe`, `.zip`, `.mp3`, `.pdf`  | ❌ Nee               |

Zelfs text-formats **zijn nog steeds bytes**,
maar hun *encoding* maakt ze leesbaar met een teksteditor.
::

---

### ✍️ Voorbeeld 1 — Text Format (Markdown)

Bestand: `notes.md`

```markdown
# Mijn Notities

- Leren over encoding 🔥
- Begrijpen wat een format is
```

In **UTF-8** opgeslagen ziet dat er op byte-niveau zo uit:

```text
23 20 4D 69 6A 6E 20 4E 6F 74 69 74 69 65 73 0A
0A 2D 20 4C 65 72 65 6E 20 6F 76 65 72 20 65 6E
63 6F 64 69 6E 67 20 F0 9F 94 A5
```

➡️ De **encoding** vertelt wat elke byte betekent (`0x23` = `#`, `0xF0 9F 94 A5` = 🔥)  
➡️ De **format** (Markdown) vertelt *hoe* het geïnterpreteerd wordt (titel + lijst)

---

### 🧊 Voorbeeld 2 — Binary Format (PNG)

Bestand: `logo.png`

De eerste bytes (header) van een PNG zien er zo uit:

```text
89 50 4E 47 0D 0A 1A 0A
```

In ASCII:

```
‰PNG....
```

Elke PNG begint exact zo — het is een **magic number** waarmee software herkent:

> “Ik ben een PNG-bestand.”

Daarna volgen gestructureerde *chunks* zoals `IHDR`, `IDAT`, `IEND`,
met elk hun eigen lengte, type en CRC-checksum.

ASCII-schematisch:

```text
[Header] → [IHDR] → [PLTE] → [IDAT] → [IEND]
```

Magic Bytes zijn letterlijk de *vingerafdruk* van een format.
Vergelijkbaar met hoe “MZ” (0x4D 5A) altijd het begin is van een Windows `.exe`-bestand.
::

## 🧱 File Format Anatomy

```text
┌─────────────────────────────────────────────┐
│ Header / Signature                         │ ← Magic bytes, versienummer
├─────────────────────────────────────────────┤
│ Metadata / Structure                       │ ← Velden, offsets, encoding-info
├─────────────────────────────────────────────┤
│ Payload / Data                             │ ← De eigenlijke inhoud
├─────────────────────────────────────────────┤
│ Footer / Checksum                          │ ← Einde of controlewaarde
└─────────────────────────────────────────────┘
```

## 📚 Bekende voorbeelden

| Format | Magic Bytes (hex)         | Betekenis                      |
| :----- | :------------------------ | :----------------------------- |
| `PNG`  | `89 50 4E 47 0D 0A 1A 0A` | Portable Network Graphics      |
| `EXE`  | `4D 5A`                   | MZ-header, Portable Executable |
| `ZIP`  | `50 4B 03 04`             | PKZip-container                |
| `PDF`  | `25 50 44 46`             | `%PDF` header                  |
| `JPEG` | `FF D8 FF`                | Start of Image marker          |

## ⚙️ File Encoding vs File Format

| Laag         | Vraag                           | Voorbeeld          |
| :----------- | :------------------------------ | :----------------- |
| **Encoding** | Hoe slaan we tekst op in bytes? | UTF-8, UTF-16      |
| **Format**   | Hoe interpreteren we die bytes? | Markdown, PNG, EXE |

```text
"🔥" (UTF-8) → F0 9F 94 A5  ← Encoding
# Title\n (Markdown syntax) ← Format
```

::important
Een bestand zonder encoding is onleesbaar.  
Een bestand zonder format is betekenisloos.
::

## 🧩 Waar horen File Formats thuis?

Bestandsformaten bouwen voort op **encoding**:

* Encoding bepaalt *hoe bits worden vertaald naar symbolen*
* Format bepaalt *hoe die symbolen betekenis krijgen binnen een structuur*

```text
Binary → Encoding → Format → Protocol / Application
```

Zo begrijp je als security researcher:

> Wat een payload *is* (bytes) én wat het *bedoelt te zijn* (HTML? PNG? ELF?).

## 📦 File Formats vs Media Types

Niet elk bestandstype heeft een officieel **Media Type**.  
IANA registreert enkel formaten die **via protocollen** reizen (HTTP, SMTP, FTP).  
Lokaal gebruikte formaten — zoals Blender’s `.blend` — vallen daarbuiten.

| Eigenschap | Media Type (IANA) | File Format (bijv. FileInfo) |
|:--|:--|:--|
| Doel | Uitwisseling via internetprotocollen | Opslag of gebruik in lokale apps |
| Beheerder | IANA / IETF (officiële standaard) | OS of applicatie (informele associatie) |
| Voorbeeld | `application/json`, `image/png` | `.blend`, `.exe`, `.zip` |
| Herkomst | RFC of vendor-registratie | App-specifiek of OS-mapping |

::note
`.blend` is geen geregistreerd IANA-type,  
omdat het niet bedoeld is voor netwerktransmissie.  
Daarom vind je het **wel op FileInfo**, maar **niet in de IANA-registry**.
::

## 🔮 Samenvatting

| Niveau   | Voorbeeld      | Begrip             |
| :------- | :------------- | :----------------- |
| Bits     | `01001000`     | Hardwarelaag       |
| Encoding | `0x48` → `'H'` | Tekstrepresentatie |
| Format   | `# Header`     | Context / syntaxis |
| Protocol | `HTTP 200 OK`  | Communicatielaag   |

::note
Sleutel tot file forensics en malware-analyse.   
Door **encodings** te herkennen zie je *hoe data is opgeslagen*.  
Door **formats** te herkennen zie je *wat die data wil zijn*.  
::
