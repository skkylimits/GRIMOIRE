## Wat is File Encoding?

Tot nu toe heb je gezien **wat encoding is** — een manier om bits betekenis te geven.  
Maar zodra die data in een **bestand** terechtkomt, verandert er iets belangrijks:  
de encoding wordt **fysiek opgeslagen** op schijf.

Elke tekst die je ziet — van `README.md` tot `config.json` —  
is eigenlijk niets meer dan een reeks **bytes** met een specifieke betekenis.

```text
01001000 01100101 01101100 01101100 01101111
↓
H    e    l    l    o
````

Zelfde bits, andere interpretatie = totaal andere tekst.
Daarom is **file encoding** letterlijk wat bepaalt *hoe een bestand gelezen moet worden.*


## Hoe bytes worden opgeslagen

Een bestand is in essentie een **lineaire stroom van bytes**.
Je kunt het vergelijken met een lange ketting van getallen tussen `0–255`.

```text
48 65 6C 6C 6F 20 57 6F 72 6C 64
```

Dit ziet er uit als onzin — totdat een programma zegt:

> “Lees dit als UTF-8 tekst.”

Dan pas weet de computer:

```
48 → H
65 → e
6C → l
6C → l
6F → o
```

🔹 Dezelfde bytes kunnen dus iets totaal anders betekenen in een andere encoding:

| Bytes          | Als UTF-8 | Als UTF-16 | Als Latin-1 |
| -------------- | --------- | ---------- | ----------- |

> 💡 Wanneer een bestand “rare tekens” toont, ligt dat zelden aan de data —
> maar aan **hoe de software de bytes interpreteert.**

## BOM — Byte Order Mark

Sommige bestanden beginnen met een paar onzichtbare bytes aan het begin —
een **BOM (Byte Order Mark)**.

Die dient als **handtekening** om te zeggen *“ik ben UTF-8”* of *“ik ben UTF-16-LE”*.

| Encoding  | BOM (hex) | Doel                   |
| --------- | --------- | ---------------------- |
| UTF-8     | EF BB BF  | herkenning             |
| UTF-16 LE | FF FE     | little endian volgorde |
| UTF-16 BE | FE FF     | big endian volgorde    |

Voorbeeld (UTF-8 met BOM):

```text
EF BB BF 48 65 6C 6C 6F
```

→ De eerste drie bytes (`EF BB BF`) vertellen de editor:

> “Lees de rest als UTF-8.”

Niet elke applicatie gebruikt of herkent de BOM —
vandaar dat sommige editors extra symbolen (ï»¿) tonen aan het begin van tekst.

## Endianness — volgorde van bytes

Bij multi-byte encodings (zoals UTF-16 of UTF-32) maakt de **volgorde** van bytes uit.

```
U+0041 → 'A'
```

In **Big Endian**:

```
00 41
```

In **Little Endian**:

```
41 00
```

Beide zijn geldig — maar als je ze omdraait zonder context,
lees je complete onzin. Daarom zetten sommige encodings (zoals UTF-16) een BOM vooraan om dit aan te geven.

> 🧠 *Endian* komt van “The Big-Endians and Little-Endians” uit Gulliver’s Travels —
> letterlijk: “welke kant van het ei breek je eerst open?”

## Tools: hoe je encoding herkent

Wanneer je werkt met echte bestanden (logfiles, configs, scripts),
wil je soms weten: *wat is de encoding eigenlijk?*

### 🔍 Command-line tools

**Unix/Linux**

```bash
file index.html
# → UTF-8 Unicode text, with BOM
```

**iconv**

```bash
iconv -f UTF-16 -t UTF-8 old.txt > new.txt
```

**Windows (PowerShell)**

```powershell
Get-Content file.txt -Encoding byte
```

Of gebruik editors zoals **VSCode** of **Notepad++** —
die tonen de encoding rechtsonder in de statusbalk.

Dat fenomeen heet **mojibake** — “vervormde tekst”.

> 💡 **Bytes liegen niet.**
> Alleen de interpretatie verandert.

## File encoding in de praktijk

Je komt file encodings overal tegen:

* Config-bestanden (`UTF-8` of `ANSI`)
* Logbestanden (`ASCII` zonder BOM)
* E-mailheaders (`Base64` + MIME)
* Scriptfiles (`UTF-8` zonder BOM, zodat Linux-shebangs blijven werken)
