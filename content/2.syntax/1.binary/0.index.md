# Binary, C & IP-netwerken — van bits naar praktijk

Een volledig, hands-on document zodat je écht snapt hoe **binary** werkt, waarom **C** belangrijk is en hoe **IP-adressen / subnetting** direct met bits werken. Alles inclusief uitgewerkte voorbeelden en korte code-snippets (C + Python). Lees dit door, doe de oefeningen, en je begrijpt het op het niveau dat je nodig hebt voor low-level security-werk.

---

# 1 — Wat is *binary* (bits & bytes)

* **Bit** = binary digit, waarde `0` of `1`.
* **Byte** = 8 bits. Meest gebruikte eenheid.
* Grotere: kilobyte (KB), megabyte (MB), etc. (contextueel: 1024 vs 1000).

### Waarom belangrijk?

Alles in een computer is uiteindelijk een reeks bits. In security / reverse engineering moet je vaak binaire representaties lezen (netwerkheaders, file formats, machine code).

---

# 2 — Binair → Decimaal (stap-voor-stap)

Elke positie is een macht van 2, beginnend rechts bij (2^0).

Voorbeeld: binaire `1101011`

Reken stap-voor-stap uit:

* bit posities (van rechts naar links): 0,1,2,3,4,5,6
* waarden: (1\cdot2^6 + 1\cdot2^5 + 0\cdot2^4 + 1\cdot2^3 + 0\cdot2^2 + 1\cdot2^1 + 1\cdot2^0)
* bereken: (1\cdot64 + 1\cdot32 + 0 + 1\cdot8 + 0 + 1\cdot2 + 1\cdot1 = 64+32+8+2+1 = 107)

Dus `1101011₂ = 107₁₀`.

---

# 3 — Decimaal → Binair (division-by-2 methode)

Voor 107:

* 107 /2 = 53 remainder 1 (LSB)
* 53 /2 = 26 remainder 1
* 26 /2 = 13 remainder 0
* 13 /2 = 6 remainder 1
* 6 /2 = 3 remainder 0
* 3 /2 = 1 remainder 1
* 1 /2 = 0 remainder 1 (MSB)

Lees remainder van onder naar boven: `1101011`.

---

# 4 — Hexadecimal (hex) — handig bij binaire weergave

* Hex gebruikt 4 bits per hex-digit (nibble).
* 1 hex-digit = waarden 0..15 → `0..9, A..F`.
* Voorbeeld: `0x6B` = `6` is `0110`, `B` is `1011` → gezamenlijke bits `01101011` = 107.

Gebruik hex omdat het compact binaire data toont en makkelijk naar bytes te lezen is.

---

# 5 — Twos-complement (negatieve getallen)

Veel systemen gebruiken two's-complement om negatieve integers te representeren.

8-bit voorbeeld:

* `+5` = `00000101`
* Neem bij `-5`: invert bits → `11111010` then add 1 → `11111011`.
* `11111011` (8-bit) = -5 in two's-complement.

Belangrijk: teken bit is impliciet in two's-complement; overflow regels volgen vaste patronen.

---

# 6 — Endianness (byte order)

* **Little-endian**: laagste significante byte eerst (Intel x86).

  * e.g. 32-bit waarde `0x12345678` in memory (bytes): `78 56 34 12`.
* **Big-endian**: hoogste byte eerst (network byte order).

  * `12 34 56 78`.

Network protocols gebruiken **network byte order = big-endian**. In C moet je soms `htonl()` / `ntohl()` gebruiken.

---

# 7 — Bitwise operators — praktische hulpmiddelen

* AND `&` — maskeren: `x & mask`.
* OR `|` — zetten van bits.
* XOR `^` — togglen.
* NOT `~` — invert bits.
* Shift `<<` `>>` — vermenigvuldigen/delen door 2^n of bit-plaatsing.

Voorbeeld: zet bit 3 (index vanaf 0) van `flags`:

```c
flags |= (1 << 3);     // set
flags &= ~(1 << 3);    // clear
flags ^= (1 << 3);     // toggle
```

# 8 — Binary en C: waarom C leren?

* C geeft directe controle over geheugen en bytes — essentieel om binaire structuren te maken/lezen (headers, network packets, structs in files).
* Veel low-level tools en OS-componenten zijn in C geschreven → begrijpen ervan helpt bij reverse engineering.

### Voorbeeld: lees 4 bytes uit een buffer en print hex

```c
#include <stdio.h>
#include <stdint.h>

void print_4bytes_hex(const uint8_t *buf) {
    printf("Bytes: %02x %02x %02x %02x\n",
           buf[0], buf[1], buf[2], buf[3]);
}

int main() {
    uint8_t buf[4] = {0x12, 0x34, 0x56, 0x78};
    print_4bytes_hex(buf);
    return 0;
}
```

### Bitfields / shifts in C voorbeeld: extract flags

```c
uint8_t header = 0b10110100;
// bits 7..5 = version
uint8_t version = (header >> 5) & 0x07;
```

# 9 — Memory layout basics (kort)

* **Stack** — lokale variabelen, LIFO.
* **Heap** — dynamische allocatie (`malloc`/`free`).
* **Data / BSS** — globale statische data.
* **Text** — uitgevoerd code (machine instructions).

Als je binaries reverse-engineert moet je weten waar data/code typisch leeft en hoe pointers werken.

# 10 — IP-adressen & netwerken — bits first!

IP (IPv4) is eenvoudig om in binaire termen te begrijpen: het is 32 bits, meestal weergegeven als 4 octets (dotted decimal).

Voorbeeld IP: `192.168.10.5`

Converteer elk octet:

* 192 → `11000000`
* 168 → `10101000`
* 10  → `00001010`
* 5   → `00000101`

Samen: `11000000.10101000.00001010.00000101`

---

## Subnetmasker & CIDR

* `/24` betekent dat de eerste 24 bits zijn **netwerk**. Mask: `255.255.255.0` → binair `11111111.11111111.11111111.00000000`.
* Network address = IP & mask
* Broadcast address = network_address | (~mask)

### Voorbeeld berekening

IP: `192.168.10.5` (/24)

* Mask `/24` = `255.255.255.0`
* Network = `192.168.10.0`
* Broadcast = `192.168.10.255`
* Host range = `192.168.10.1` … `192.168.10.254`

### Oefening: /26

IP `10.0.0.73/26`

* /26 → mask `255.255.255.192` → laatste 6 bits host.
* Network: behoud eerste 26 bits van IP, host bits zero:

  * 10.0.0.73 binair (last octet 73 = `01001001`)
  * mask last octet = `11000000` (192)
  * Network last octet = `01000000` = 64 → Network = `10.0.0.64`
* Broadcast = network | (~mask last octet) = `01000000 | 00111111` = `01111111` = 127 → broadcast `10.0.0.127`
* Hosts: `10.0.0.65` … `10.0.0.126` (64 = network, 127 = broadcast)

# 11 — IP in C: parse & bit-ops (voorbeelden)

Gebruik `inet_pton` / `inet_ntop` voor conversies netwerk <-> presentation.

```c
#include <stdio.h>
#include <arpa/inet.h>
#include <stdint.h>

int main() {
    const char *ip_str = "10.0.0.73";
    uint32_t ip_bin;
    inet_pton(AF_INET, ip_str, &ip_bin); // ip_bin in network byte order (big-endian)

    uint32_t ip_host = ntohl(ip_bin); // convert to host order so shifts make sense
    printf("IP host-order hex: 0x%x\n", ip_host);

    uint32_t mask = 0xFFFFFFFF << (32 - 26); // /26 mask in host order
    uint32_t network = ip_host & mask;
    uint32_t broadcast = network | ~mask;

    struct in_addr tmp;
    tmp.s_addr = htonl(network);
    printf("Network: %s\n", inet_ntoa(tmp));
    tmp.s_addr = htonl(broadcast);
    printf("Broadcast: %s\n", inet_ntoa(tmp));
    return 0;
}
```

# 12 — Python voorbeelden (sneller experimenteren)

Convert IP to binary and calculate network:

```python
import ipaddress

ip = ipaddress.IPv4Interface("10.0.0.73/26")
network = ip.network
print("IP:", ip.ip)
print("Network:", network.network_address)
print("Broadcast:", network.broadcast_address)
print("Host range:", list(network.hosts())[0], "to", list(network.hosts())[-1])
```

Binary representation of IP:

```python
def ip_to_bin(ip_str):
    return '.'.join(f"{int(octet):08b}" for octet in ip_str.split('.'))

print(ip_to_bin("192.168.10.5"))
```

## 🧩 Waarom 8 bits “genoeg” leek

Toen computers begonnen (jaren ’60–’70), werkte men vooral met **Engelse tekst**.
ASCII — het eerste grote standaardalfabet — gebruikte **7 bits** (128 symbolen):

* A–Z, a–z
* cijfers 0–9
* leestekens
* enkele controlecodes (newline, tab, enz.)

➡️ 8 bits (1 byte) was dus **ruim voldoende**: 7 bits voor het teken, 1 bit over voor controle (parity/padding).

**Conclusie destijds:**

> “We hebben alle tekens die we nodig hebben, dus 1 byte per karakter is prima.”

---

## 🌍 Waarom 8 bits te weinig werd

Toen computers wereldwijd gebruikt werden, kwamen er problemen:

* Europese accenten (é, ö, ç) pasten niet in ASCII.
* Russisch, Grieks, Arabisch, Chinees... hadden *duizenden* extra karakters.

Iedere regio probeerde het op te lossen met z’n eigen **8-bit uitbreidingen**,
zoals Latin-1, Cyrillic, Shift-JIS, etc.
Maar die werkten niet samen.
📉 Een bestand geschreven in Japans werd op een Frans systeem onleesbaar (“mojibake”).

---

## 🧠 De oplossing: Unicode & UTF

Unicode kwam in de jaren ’90 met het idee:

> “Laten we één universele code maken die *alle* talen, symbolen en emoji kan bevatten.”

In Unicode heeft elk teken een uniek **codepoint** (bijv. `U+1F600` = 😀).
Alleen… er zijn véél meer dan 256 tekens → 8 bits is niet genoeg.

Dus ontstonden **UTF-encodings** (Unicode Transformation Formats)
die meer dan één byte per karakter kunnen gebruiken:

| Encoding   | Bytes per karakter | Opmerking                                 |
| ---------- | ------------------ | ----------------------------------------- |
| **UTF-8**  | 1–4                | Dynamisch; backwards-compatible met ASCII |
| **UTF-16** | 2 of 4             | Efficiënt voor oosterse talen             |
| **UTF-32** | 4                  | Vast formaat, eenvoudig maar groot        |

---

## 🧩 Voorbeeld: UTF-8 in actie

```text
'A'       → 01000001            (1 byte)
'é'       → 11000011 10101001   (2 bytes)
'你'      → 11100100 10111000 10101001 (3 bytes)
'😀'      → 11110000 10011111 10011000 10000000 (4 bytes)
```

➡️ In UTF-8 kunnen korte tekens nog steeds 1 byte gebruiken,
maar complexe tekens gebruiken meerdere bytes — precies genoeg.

---

## ⚙️ Samenvattend

* 8 bits (1 byte) werd standaard vanwege **ASCII** en **hardware-eenvoud**.
* Maar 8 bits kan maar **256** waarden representeren.
* Unicode & UTF lossen dit op door **meerdere bytes per karakter** toe te staan.

---

**Kort gezegd:**

> 8 bits was genoeg voor de westerse wereld.
> Unicode maakte het genoeg voor *de hele wereld*. 🌍

## 🧩 Vergelijking: ASCII vs IPv4

| Concept     | Oud systeem                        | Beperkingen                                                    | Nieuwe oplossing                           |
| ----------- | ---------------------------------- | -------------------------------------------------------------- | ------------------------------------------ |
| **Tekst**   | 7-bit ASCII (→ 8-bit byte)         | Slechts 128 tekens → geen ruimte voor accenten of andere talen | **Unicode / UTF-8** (meer bytes per teken) |
| **Netwerk** | IPv4 (32 bits = 4 bytes per adres) | Slechts ~4,3 miljard unieke adressen → uitgeput                | **IPv6 (128 bits = 16 bytes per adres)**   |

## ⚙️ IPv4 in binaire vorm

Een IPv4-adres bestaat uit **4 bytes (32 bits)**.
Elk deel is één byte (0–255).

```text
192.168.0.1 =
11000000.10101000.00000000.00000001
```

→ `4 × 8 bits = 32 bits totaal`
→ `2^32 = 4.294.967.296` mogelijke adressen

Klinkt veel, maar:

* duizenden adressen per organisatie
* miljarden apparaten wereldwijd
* dus voorraad = op

---

## 🌍 IPv6 als de “Unicode van netwerken”

IPv6 breidt dit uit naar **128 bits = 16 bytes**:

```text
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

→ `2^128` adressen = 340.282.366.920.938.463.463.374.607.431.768.211.456
Dat is **meer adressen dan er zandkorrels op aarde zijn**. 🏖️

## 🧩 2️⃣ Hoe komen we dan aan “16 exabytes”?

Hier zit een **eenheden-omzetting** achter.
We drukken geheugen namelijk vaak uit in *grotere eenheden* dan bytes:
kilobyte, megabyte, gigabyte, terabyte, petabyte, **exabyte** …

Maar dat doen we niet met gewone machten van 10 (zoals 1.000),
maar met machten van **2** (zoals 1.024).

---

### 📏 Binaire conversie

| Eenheid | Aantal bytes                    | Macht van 2 |
| ------- | ------------------------------- | ----------- |
| 1 KB    | 1.024 bytes                     | 2¹⁰         |
| 1 MB    | 1.048.576 bytes                 | 2²⁰         |
| 1 GB    | 1.073.741.824 bytes             | 2³⁰         |
| 1 TB    | 1.099.511.627.776 bytes         | 2⁴⁰         |
| 1 PB    | 1.125.899.906.842.624 bytes     | 2⁵⁰         |
| 1 EB    | 1.152.921.504.606.846.976 bytes | 2⁶⁰         |

---

## 🧮 3️⃣ Deel `2^64` door `2^60`

```text
2^64 ÷ 2^60 = 2^(64−60) = 2^4 = 16
```

→ Dat betekent:
**2^64 bytes = 16 × (2^60 bytes)**
en `2^60 bytes` = **1 exabyte**

➡️ **2^64 = 16 exabytes**

---

## 🧠 4️⃣ Dus in woorden:

* `2^64` = 18.446.744.073.709.551.616 bytes
* 1 exabyte = 1.152.921.504.606.846.976 bytes
* Deel die twee getallen op:

  ```
  18.446.744.073.709.551.616 ÷ 1.152.921.504.606.846.976 ≈ 16
  ```

👉 En dat is waarom we zeggen:

> “Een 64-bit adresruimte = **16 exabytes** aan geheugen.”
