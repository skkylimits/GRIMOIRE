## shamefully-hoist

Imagine you're building a toy set, with blocks usually sorted into separate boxes (like pnpm). Some older sets expect all blocks in one big box (like npm). Using shamefully-hoist=true mixes all blocks together, helping those older sets work smoothly, even if it's a bit messy.

```[.npmrc]
shamefully-hoist=true
```

## Symbolism

Here's a brief explanation of the versioning terms used in semantic versioning (SemVer):

- **Major**: Changes when there are **backward-incompatible** changes (e.g., `2.0.0` from `1.x.x`).
- **Minor**: Changes when new **features** are added in a **backward-compatible** manner (e.g., `1.2.0` from `1.1.x`).
- **Patch**: Changes when **backward-compatible** fixes or improvements are made (e.g., `1.1.2` from `1.1.1`).

So, the version number follows this format:
**`<Major>.<Minor>.<Patch>`**

Here's a short summary of versioning symbols in npm:

- **`^` (Caret)**: Allows updates to **minor** and **patch** versions, but **not major** (e.g., `^1.2.3` allows `1.3.0` but not `2.0.0`).
- **`~` (Tilde)**: Allows only **patch** updates (e.g., `~1.2.3` allows `1.2.4`, but not `1.3.0`).
- **`*` (Wildcard)**: Allows **any version** (e.g., `*` accepts `1.0.0`, `2.0.0`, etc.).
- **`>` (Greater than)**: Allows versions **greater than** a specific version (e.g., `>1.0.0` allows `1.1.0`, `2.0.0`).
- **`<` (Less than)**: Allows versions **less than** a specific version (e.g., `<1.0.0` allows `0.9.9`).
- **`>=` (Greater than or equal to)**: Allows versions **greater than or equal to** a specific version.
- **`<=` (Less than or equal to)**: Allows versions **less than or equal to** a specific version.
- **`-` (Range)**: Defines a version **range** (e.g., `1.2.3 - 2.3.4` allows `1.2.3` to `2.3.4`).
- **`x` or `X` (Wildcard for digits)**: Allows any patch version (e.g., `1.2.x` allows `1.2.3`, `1.2.9`).

These symbols control which versions of a package are compatible with your project.
