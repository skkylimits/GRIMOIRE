## 1. **Conventional Commits**

**Conventional Commits** is a simple convention for writing commit messages, where each message follows a structured format. The main goal is to make commit history more readable and enable automatic tools to generate meaningful changelogs, determine semantic version numbers, and trigger CI/CD pipelines.

The commit messages follow this general structure:

```
<type>(<scope>): <short description>
```

Here's a breakdown:

- **Type**: Defines the purpose of the commit. Common types include:
  - `feat`: A new feature.
  - `fix`: A bug fix.
  - `chore`: Routine tasks, maintenance, or refactoring that doesn't affect the codebase or production.
  - `docs`: Documentation updates.
  - `style`: Code style changes (like formatting), no code functionality changes.
  - `refactor`: Refactoring code without adding new features or fixing bugs.
  - `test`: Adding or modifying tests, nothing affecting production code.
  - `perf`: Performance improvement.

- **Scope**: (Optional) Refers to the part of the project being modified, such as a specific module, component, or function.

- **Description**: A short, concise explanation of the change.

### Example Commit Messages:

- `feat(login): add login button`
- `fix(auth): resolve token expiry issue`
- `chore(deps): update dependency @nuxtjs/auth`
- `docs(readme): update setup instructions`

This structure makes commit messages consistent and understandable at a glance, while also enabling automation.

## 2. **How Conventional Commits Tie into Changelog and Version Releases**

By using the Conventional Commits structure, you can automate parts of your development process, especially when combined with tools like:

- **`standard-version`**: This automates versioning and changelog generation based on the Conventional Commit messages.
- **`release-it`**: Similar to `standard-version`, but also handles pushing tags and publishing releases to platforms like GitHub.

Here's how it works:

### **Changelog Generation**

A tool like `standard-version` or `semantic-release` can automatically generate a changelog based on your commit messages. For example:

- If you have a `feat` commit, it might add a new section in the changelog under "Features."
- A `fix` commit would go under the "Bug Fixes" section.

These tools typically scan your commit history since the last release and update the changelog accordingly. They generate logs like this:

```markdown
# Changelog

## [1.0.0] - 2024-09-15
## Added
- feat(auth): add OAuth login support
- feat(ui): implement dark mode

## Fixed
- fix(auth): resolve login token issue
```

### **Version Releases**

The commits can also drive **semantic versioning**, where version numbers follow the format `MAJOR.MINOR.PATCH` (e.g., `1.2.3`):

- **MAJOR**: Incremented when there are breaking changes (e.g., `feat!:` in Conventional Commits indicates a breaking change).
- **MINOR**: Incremented when you add backward-compatible features (`feat` commits).
- **PATCH**: Incremented when you make backward-compatible bug fixes (`fix` commits).

**Example:**
- `feat`: Adds a feature, bumps the **minor** version (`1.1.0`).
- `fix`: Fixes a bug, bumps the **patch** version (`1.0.1`).
- `feat!`: Introduces a breaking change, bumps the **major** version (`2.0.0`).

This automation helps you release new versions consistently and ensures that the versioning follows best practices.

## 3. **Multiple Commits and Release Management**

Yes, multiple commits usually add together to form a release. Here’s how you would manage that:

### **How to Specify a Release:**

1. **Commit Regularly**: As you're working, you commit with the appropriate message using Conventional Commits.

2. **Run Versioning Tool**: When you're ready to release a new version (e.g., after completing a set of features or bug fixes), you use a tool like `standard-version` or `semantic-release`.

3. **Version Bumping**: Based on the commit types since the last release, the tool decides which part of the version to increment (major, minor, or patch).

4. **Generate the Release**: The tool will automatically:
   - Update the version number in `package.json` (or wherever versioning is handled).
   - Generate/update the changelog with the commit history since the last release.
   - Tag the release in Git (optionally push it to GitHub or npm).

5. **Specify Releases**: When using tools like `semantic-release`, releases happen automatically based on your commits. If you prefer manual control, you can trigger it manually by running a command, e.g., `npm run release`.

### **Example Workflow:**

1. **Day 1**: You commit a bug fix:
   ```
   fix(api): correct response type on user endpoint
   ```
2. **Day 2**: You add a feature:
   ```
   feat(dashboard): add user analytics widget
   ```
3. **Day 3**: You decide it's time to release. You run:
   ```shell
   npm run release
   ```
   - This will automatically bump the version based on your commits (in this case, it might bump `1.0.0` to `1.1.0`).
   - It updates the changelog with the commits.
   - It pushes the new version and changelog to your Git repository and creates a release tag.

## 4. **Tools You Can Use for This Process**

To handle Conventional Commits, versioning, and changelog generation, you can use tools like:

- **[Commitizen](https://commitizen.github.io/cz-cli/)**: Helps you write Conventional Commits in a structured way by providing a CLI interface.
- **[Standard Version](https://github.com/conventional-changelog/standard-version)**: Automates versioning and changelog generation based on your commits.
- **[Semantic Release](https://github.com/semantic-release/semantic-release)**: Fully automates the release process, including changelog creation and pushing releases.

## Adopt Later. MVP

It’s common for teams to start implementing these practices after the initial prototyping or MVP (Minimum Viable Product) phase. Once the project stabilizes and you begin adding features or preparing for external collaboration or releases, these tools become more valuable for consistency and communication
