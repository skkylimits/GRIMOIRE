## 1 — Identificeren van Live Hosts op het Netwerk

**Doel:** Vind actieve apparaten op het netwerk — zodat je IP’s en MAC-adressen kunt in kaart brengen, belangrijke systemen zoals Domain Controllers (DCs) kunt opsporen, en ze later kunt targeten.

---

### Waar je op moet letten & waarom:

* **ARP-verkeer**: Omdat ARP IP-adressen naar MAC-adressen vertaalt binnen het lokale netwerk — hiermee zie je wie actief is.
* **DHCP-verkeer**: Laat zien welke apparaten IP’s aanvragen en welke server (waarschijnlijk een DC of DHCP-server) ze uitdeelt.
* **ICMP-verkeer (ping)**: Sommige apparaten reageren op pings, wat bevestigt dat ze online zijn.
* **NetBIOS-broadcasts**: Handig op Windows-netwerken om apparaatnamen en share-informatie te ontdekken.
* **LLMNR/mDNS-verkeer**: Alternatieve naamresolutieprotocollen op Windows-netwerken.
* **Algemeen broadcast-verkeer**: Om aankondigingen en apparaatherkenning te zien.

---

### Extra Filters of Verkeer om toe te voegen:

* **NetBIOS:** `nbns`
  Handig om Windows name service requests en apparaatnamen te zien.

* **LLMNR:** `udp.port == 5355`
  Voor lokale naamresolutie-broadcasts.

* **mDNS:** `udp.port == 5353`
  Multicast DNS, minder gebruikelijk op Windows maar nuttig in gemengde omgevingen.

---

### Samenvattingschecklist voor Stap 1:

| Wat te Capturen      | Wireshark Filter   | Waarom                                              |
| -------------------- | ------------------ | --------------------------------------------------- |
| ARP                  | `arp`              | IP-naar-MAC mapping identificeren                   |
| DHCP                 | `bootp`            | IP-leases en DHCP-serverinformatie bekijken         |
| ICMP (ping)          | `icmp`             | Live hosts identificeren die op pings reageren      |
| NetBIOS Name Service | `nbns`             | Windows hostnames ontdekken                         |
| LLMNR                | `udp.port == 5355` | Lokale naamresolutie via broadcast                  |
| mDNS                 | `udp.port == 5353` | Multicast DNS (optioneel, afhankelijk van omgeving) |

---

### Nog iets om hieraan toe te voegen?

* Misschien **TCP SYN-scans** (filter `tcp.flags.syn == 1 && tcp.flags.ack == 0`) als je op zoek bent naar actieve hosts die TCP-verbindingen proberen op te zetten.
* Let op **Broadcast of Multicast-verkeer**:
  Filter: `broadcast || multicast` om netwerkbrede berichten te spotten.

## 2 — Enumerating Hosts for Login / Domain Entry

**Goal:** Identify devices offering services or showing signs of authentication, so you can focus on potential targets for credentials or domain access.

---

### What to look for & why:

* **SMB (Server Message Block) traffic:** Core Windows file sharing and domain services — often shows login attempts, shares, domain info.
* **Kerberos traffic:** Authentication protocol in Active Directory — shows tickets, authentication attempts.
* **LDAP traffic:** Active Directory queries and directory service communication.
* **NTLM traffic:** Older Microsoft authentication method (still seen in some environments).
* **RDP traffic:** Remote Desktop connections which could be login points.
* **DNS queries:** Resolve internal hostnames, domains, and services.
* **HTTP/HTTPS traffic:** Sometimes used for domain services or web-based auth portals.

---

### Filters to use:

| What to Capture | Wireshark Filter   | Why                                 |        |                                           |       |                                              |
| --------------- | ------------------ | ----------------------------------- | ------ | ----------------------------------------- | ----- | -------------------------------------------- |
| SMB             | \`smb              |                                     | smb2\` | File sharing, login attempts, domain info |       |                                              |
| Kerberos        | `kerberos`         | Authentication tickets and requests |        |                                           |       |                                              |
| LDAP            | `ldap`             | Directory service traffic           |        |                                           |       |                                              |
| NTLM            | `ntlmssp`          | Older authentication method         |        |                                           |       |                                              |
| RDP             | `tcp.port == 3389` | Remote Desktop protocol             |        |                                           |       |                                              |
| DNS             | `dns`              | Hostname resolution                 |        |                                           |       |                                              |
| HTTP / HTTPS    | \`http             |                                     | ssl    |                                           | tls\` | Web traffic, including possible auth portals |

---

### Bonus: Other interesting filters

* **SMB Login events:** Look for `smb.cmd == 0x73` (Session Setup) packets.
* **Kerberos AS\_REQ:** `kerberos.msg_type == 10` (Authentication Service Request).
* **LDAP Bind requests:** Look for LDAP bind packets to see login attempts.
* **NTLMSSP Negotiate:** `ntlmssp.message_type == 1`.

---

### Summary checklist for step 2:

* Focus on traffic where devices authenticate or request directory info.
* Look for signs of domain controllers and authentication servers.
* Identify which hosts are active in domain services for later attack focus.

## 3 — DHCP and Network Infrastructure Identification

**Goal:** Find DHCP servers, gateways, DNS servers, and other core infrastructure devices that control network communication and client configuration.

---

### What to look for & why:

* **DHCP traffic:** Identify DHCP servers handing out IPs and lease info. Servers are usually key infrastructure devices.
* **Gateway and router IPs:** Helps map network segments and routing.
* **DNS servers:** Crucial for name resolution; may also reveal domain controllers.
* **Default gateways:** Devices routing traffic between subnets.
* **Other infrastructure protocols:** CDP (Cisco Discovery Protocol), LLDP (Link Layer Discovery Protocol) can reveal device info.

---

### Filters to use:

| What to Capture                                      | Wireshark Filter                      | Why                                  |        |                             |
| ---------------------------------------------------- | ------------------------------------- | ------------------------------------ | ------ | --------------------------- |
| DHCP                                                 | \`bootp                               |                                      | dhcp\` | DHCP requests and responses |
| Gateway IPs (identify via ARP)                       | `arp`                                 | Get MAC and IP addresses of routers  |        |                             |
| DNS Servers (look for DNS responses from server IPs) | `dns` or filter by specific server IP | Name resolution and potential DC IPs |        |                             |
| CDP (Cisco Discovery Protocol)                       | `cdp`                                 | Network device discovery             |        |                             |
| LLDP (Link Layer Discovery Protocol)                 | `lldp`                                | Network device discovery             |        |                             |

---

### Bonus: Tips

* Watch for DHCP Offer and ACK packets — they reveal server IP and assigned client IP.
* Note MAC addresses of DHCP servers and routers — helps in mapping physical devices.
* Identify if multiple DHCP servers exist — may indicate rogue servers.
* Check DNS traffic for internal domain controller IPs.

---

### Summary checklist for step 3:

* Capture DHCP packets to identify servers and assigned IPs.
* Use ARP to map routers and gateways.
* Identify DNS servers and note IPs for domain services.
* Use CDP/LLDP to discover device types and topology hints.

## 4 —**ICMP and Network Scanning**

---

### Why ICMP matters in pentesting:

* **ICMP (Internet Control Message Protocol)** is mainly used for diagnostic and control purposes — think *ping*, *traceroute*, and error messages.
* Attackers and pentesters use ICMP to discover live hosts and map network topology.
* Many networks block or limit ICMP, so seeing ICMP traffic can hint at misconfigurations or firewall rules.
* ICMP can reveal network latency, reachability, and even some fingerprinting info.

---

### What to look for & why:

* **Ping requests and replies:** Identify live hosts responding to pings.
* **ICMP error messages:** Like Destination Unreachable — can help map inaccessible networks or hosts.
* **Traceroute traffic:** Shows route taken through the network.
* **Unusual or excessive ICMP traffic:** Might indicate scanning or reconnaissance.

---

### Useful Wireshark Display Filters:

| Purpose                      | Filter Example   | Notes                                   |
| ---------------------------- | ---------------- | --------------------------------------- |
| Capture all ICMP traffic     | `icmp`           | Basic catch-all for ICMP packets        |
| ICMP Echo Request (ping)     | `icmp.type == 8` | Requests sent to check if host is alive |
| ICMP Echo Reply (pong)       | `icmp.type == 0` | Replies from live hosts                 |
| ICMP Destination Unreachable | `icmp.type == 3` | Helps find unreachable hosts/networks   |

---

### Bonus tips:

* Check the source/destination IPs of ICMP packets to map live hosts.
* Look for patterns like repeated pings to multiple IPs — might indicate scanning.
* Identify if ICMP is blocked — absence can be just as telling.
* Combine ICMP filters with other protocols to correlate traffic (e.g., TCP SYN scans combined with ICMP pings).

---

### Step 4 Summary Checklist:

* Filter for `icmp` and analyze Echo Requests/Replies.
* Note error messages (Destination Unreachable) to understand network blocks.
* Detect possible scanning behavior via volume and timing.
* Use ICMP results to update your live host list and map routes.

## 5 —**DNS Traffic Analysis**

### Why DNS is important in pentesting:

* DNS queries reveal which domains and services users and devices are accessing.
* Attackers can use DNS to discover internal network names and infrastructure.
* DNS traffic can leak info about Active Directory domains, internal servers, and security policies.
* DNS tunneling is a common covert channel for data exfiltration, so spotting suspicious DNS is key.

---

### What to look for & why:

* **Frequent DNS queries to external domains:** Could hint at malware or data leakage.
* **Internal DNS queries:** Reveal domain names, hostnames, and network architecture.
* **NXDOMAIN responses:** Could indicate scanning or probing of non-existent hosts.
* **Unusual DNS query types:** Like TXT or NULL — often used in tunneling or attacks.
* **Look for DNS over non-standard ports:** Can indicate attempts to bypass monitoring.

---

### Useful Wireshark Display Filters:

| Purpose                       | Filter Example                  | Notes                                  |
| ----------------------------- | ------------------------------- | -------------------------------------- |
| Capture all DNS traffic       | `dns`                           | Basic catch-all for DNS                |
| DNS queries only              | `dns.flags.response == 0`       | Requests sent by clients               |
| DNS responses only            | `dns.flags.response == 1`       | Replies from DNS servers               |
| Queries for a specific domain | `dns.qry.name == "example.com"` | Track lookups for certain domains      |
| Non-existent domain responses | `dns.flags.rcode == 3`          | NXDOMAIN responses                     |
| DNS over non-standard ports   | `udp.port != 53 && dns`         | Suspicious DNS traffic outside port 53 |

---

### Bonus tips:

* Watch out for DNS requests to rare or suspicious domains.
* Correlate DNS queries with IP addresses to find internal hosts.
* Detect DNS tunneling by spotting large or unusual DNS payloads.
* Use DNS traffic patterns to identify domain controllers or other key infrastructure.

---

### Step 5 Summary Checklist:

* Filter for DNS traffic with `dns`.
* Analyze queries vs responses (`dns.flags.response`).
* Look for NXDOMAIN (`dns.flags.rcode == 3`) and unusual query types.
* Spot DNS tunneling attempts by filtering non-standard ports.
* Use DNS info to map network and identify key systems.

## 6 — **SMB / Windows File Sharing Traffic**

---

### Why SMB matters in pentesting:

* SMB (Server Message Block) is the primary protocol for Windows file sharing and domain services.
* It can reveal active domain controllers, file shares, and user activity.
* SMB traffic can leak info about users, computers, and security policies.
* Vulnerabilities in SMB (like EternalBlue) have been used in major attacks.
* Identifying SMB traffic helps you map the network and find attack vectors.

---

### What to look for & why:

* **Connections to domain controllers:** Often via SMB ports (445 TCP).
* **File share enumeration:** Listing shared folders or files.
* **SMB authentication attempts:** Might reveal usernames and domain info.
* **Suspicious or failed authentication:** Could indicate brute force or recon attempts.
* **Unusual SMB versions or dialects:** Might indicate outdated or vulnerable systems.

---

### Useful Wireshark Display Filters:

| Purpose                         | Filter Example       | Notes                           |        |                        |
| ------------------------------- | -------------------- | ------------------------------- | ------ | ---------------------- |
| Capture all SMB traffic         | \`smb                |                                 | smb2\` | SMB and SMB2 protocols |
| SMB traffic on default TCP port | `tcp.port == 445`    | SMB typically uses TCP 445      |        |                        |
| SMB authentication (NTLM)       | `smb.authenticating` | Filter authentication packets   |        |                        |
| SMB session setup requests      | `smb.cmd == 0x73`    | SMB\_COM\_SESSION\_SETUP (0x73) |        |                        |
| SMB2 session setup requests     | `smb2.cmd == 1`      | SMB2\_SESSION\_SETUP            |        |                        |

---

### Bonus tips:

* Watch SMB traffic around domain controllers and file servers.
* Check for cleartext passwords or NTLM hashes if capturing at a vulnerable point.
* Analyze SMB dialect negotiation to spot old, vulnerable clients or servers.
* Look for abnormal SMB traffic spikes or error codes indicating attack attempts.

---

### Step 6 Summary Checklist:

* Filter SMB/SMB2 traffic: `smb || smb2`
* Focus on TCP port 445: `tcp.port == 445`
* Monitor authentication traffic: `smb.authenticating`
* Watch session setup commands (`smb.cmd == 0x73` or `smb2.cmd == 1`)
* Analyze for anomalies or brute force attempts.

## 8 — **LDAP Traffic Analysis**

**Why LDAP?**

* LDAP is the protocol used by Active Directory for directory services.
* It carries info about users, groups, computers, policies, etc.
* Capturing LDAP traffic can reveal domain structure and credentials (if not encrypted).

**What to look for:**

* LDAP queries and responses.
* Searches for users, groups, computers.
* Anonymous binds or cleartext authentication attempts.
* Suspicious or failed LDAP operations.

**Wireshark filters:**

| Purpose              | Filter example         |
| -------------------- | ---------------------- |
| Capture LDAP traffic | `ldap`                 |
| LDAP search requests | `ldap.message == 0x63` |
| LDAP binds           | `ldap.message == 0x60` |

---

## 9 — **Kerberos Traffic Analysis**

**Why Kerberos?**

* Kerberos handles authentication in AD.
* Tickets, AS-REQs, TGS-REQs, and responses can show active users and service access.
* Look for unusual or failed ticket requests.

**Wireshark filters:**

| Purpose                                   | Filter example            |
| ----------------------------------------- | ------------------------- |
| Capture Kerberos traffic                  | `kerberos`                |
| AS-REQ (authentication request)           | `kerberos.msg_type == 10` |
| TGS-REQ (ticket granting service request) | `kerberos.msg_type == 12` |

---

## 10 — **HTTP/HTTPS and Other Application Layer Protocols**

* Look for web services, admin portals, API calls.
* Spot credentials, tokens, or unusual activity.
* For HTTPS, look for unusual certificate errors or TLS versions.

Filters:

* HTTP: `http`
* HTTPS: `tcp.port == 443`

---

## 11 — **Other Protocols**

* SMB (Step 6 if you want to reorder), RDP, FTP, NFS, etc.
* Anything that could expose shares, files, or remote desktop sessions.
