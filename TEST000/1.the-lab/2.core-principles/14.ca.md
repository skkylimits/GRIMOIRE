## CA

Conditional Access policies are simple if-then rules—for example, if a user is in a certain group, then they must use multifactor authentication to sign in.

![conditional-access-overview](/content/1.the-lab/2.core-principles/14.ca/conditional-access-overview.png)

::caution
Conditional Access policies apply after first-factor authentication. They’re not designed to stop attacks like DoS but can use related signals to decide access.
::

## Signals

Modern security uses identity-driven signals for access control. Microsoft Entra Conditional Access gathers these signals to enforce policies, acting as Microsoft’s Zero Trust policy engine.

![conditional-access-signal-decision-enforcement](/content/1.the-lab/2.core-principles/14.ca/conditional-access-signal-decision-enforcement.png)

## Policy Components

A conditional access policy in Microsoft Entra ID consists of three components: **`signals`**, **`assignments`**, and **`access controls`**..

![conditional-access-policy-components-v3](/content/1.the-lab/2.core-principles/14.ca/conditional-access-policy-components-v3.png)

### Signals

> Think of `signals` as the `information` the system receives about the sign-in or request.

**Key data** points used to evaluate access decisions, including:

* Users or group membership
* IP location (trusted ranges, countries, named locations)
* Device platform, compliance state, and filters
* Applications and user actions
* Risk detection from Microsoft Entra ID Protection (sign-in risk, user risk, insider risk)
* Real-time monitoring via Microsoft Defender for Cloud Apps

![conditional-access-central-policy-engine-zero-trust](/content/1.the-lab/2.core-principles/14.ca/conditional-access-central-policy-engine-zero-trust.png)

### Assignments

> Assignments are how you configure a CA policy to use those `signals`

Control `who`, `what`, `where`, and `when` a policy applies.

::caution
All conditions must be met to trigger the policy.
::

* **Users:** Define who the policy affects. Can include or exclude:

  * All users
  * Specific users
  * Groups
  * Directory roles
  * External guests
  * Workload identities

* **Target resources:**

  * **Cloud apps:** Microsoft apps (Microsoft 365, Azure, Admin portals) and registered apps.
  * **User actions:** Actions like registering security info or joining devices.
  * **Global Secure Access (preview):** Secures traffic through Global Secure Access profiles.
  * **Authentication context:** Extra security requirements for accessing specific content (e.g., managed devices or terms of use).

* **Network:** Controls access based on network or location:

  * Any network or location
  * Trusted networks or IP address ranges
  * Named locations
  * Compliant networks of users and devices

* **Conditions:** Combine factors such as:

  * **Sign-in risk and user risk:** Detect suspicious sign-ins or compromised accounts via Microsoft Entra ID Protection.
  * **Insider risk:** From Microsoft Purview, considering data governance and compliance.
  * **Device platform:** Operating system (Windows, iOS, Android, etc.).
  * **Client apps:** Browsers, mobile apps, desktop clients.
  * **Filters for devices:** Target policies to specific devices, like privileged access workstations.

### Access Controls

Define **`how`** the policy is enforced.

* **Block access:** Deny the user access entirely.
* **Grant access:** Allow access either freely or with additional requirements such as:

  * Multifactor authentication (MFA)
  * Specific authentication methods
  * Device compliance policies
  * Password changes, and more
* **Session controls:** Apply limits within apps to provide a restricted experience, such as:

  * Blocking download, copy, or print of sensitive files (via Microsoft Defender for Cloud Apps)
  * Enforcing sign-in frequency
  * Restricting app features based on device state

### Adaptive Access

### Authorization Context

Authentication Context, a preview feature in Microsoft Entra ID, enhances security by enforcing conditions on app data and actions.

> For example, an organization can keep files in SharePoint sites like the lunch menu or their secret BBQ sauce recipe. Everyone has access to the lunch menu site. However, users who have access to the secret BBQ sauce recipe site are required to connect from a managed device.

::important
You can even enforce them to agree to specific terms of use.
